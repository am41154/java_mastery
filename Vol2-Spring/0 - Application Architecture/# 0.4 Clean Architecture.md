# 0.4 Clean Architecture

As enterprise applications grow, one common problem begins to emerge:

> **The business logic becomes tightly coupled to frameworks, databases, and external systems.**

Initially, this doesn't seem like a problem.

However, after several years of development:

- Business logic becomes difficult to test.
- Framework upgrades become risky.
- Database changes affect business rules.
- External integrations leak into core domain logic.

To solve these problems, **Robert C. Martin (Uncle Bob)** introduced **Clean Architecture**.

The primary philosophy is simple:

> **Business rules should not depend on frameworks, databases, or infrastructure.**

Instead:

> **Frameworks should depend on your business rules.**

This is a major shift in architectural thinking.

---

# Why Was Clean Architecture Introduced?

Consider a typical layered application.

```text
Controller

↓

Service

↓

Repository

↓

Database
```

Initially, this works well.

But over time, services start depending on:

- Spring
- Hibernate
- Kafka
- Redis
- REST clients
- AWS SDK
- Logging frameworks

Eventually, the Service Layer becomes tightly coupled to infrastructure.

Business logic is no longer independent.

---

# The Central Idea

Imagine removing Spring Boot tomorrow.

Would your business rules still work?

If the answer is:

> "No"

then your domain logic depends too heavily on the framework.

Clean Architecture aims to reverse that dependency.

---

# The Dependency Rule

This is the most important rule in Clean Architecture.

> **Dependencies always point inward.**

Outer layers depend on inner layers.

Inner layers know nothing about outer layers.

Visual representation:

```text
        +----------------------------+
        |     Frameworks & Drivers   |
        +----------------------------+

                  ↓

        +----------------------------+
        | Interface Adapters         |
        +----------------------------+

                  ↓

        +----------------------------+
        | Application Business Rules |
        +----------------------------+

                  ↓

        +----------------------------+
        | Enterprise Business Rules  |
        +----------------------------+
```

Notice:

The arrows always point toward the center.

---

# The Four Layers

Clean Architecture is commonly represented as four concentric circles.

## 1. Enterprise Business Rules

The innermost layer.

Contains:

- Core domain models
- Business policies
- Domain services
- Enterprise-wide rules

These rules should remain valid regardless of:

- Database
- UI
- Framework
- Programming language

Example:

```text
Employee

Order

Invoice

Account
```

These concepts belong to the business, not to Spring Boot.

---

## 2. Application Business Rules

Contains application-specific use cases.

Examples:

- Create Order
- Transfer Money
- Register Employee
- Approve Loan

These use cases coordinate domain objects.

They define **what the application does**.

---

## 3. Interface Adapters

Responsible for converting data between external systems and business logic.

Examples:

- REST Controllers
- DTOs
- Repository Implementations
- Kafka Consumers
- GraphQL Resolvers

This layer translates data into forms that the application understands.

---

## 4. Frameworks & Drivers

The outermost layer.

Examples:

- Spring Boot
- Hibernate
- Oracle
- PostgreSQL
- Kafka
- Redis
- REST APIs
- AWS
- Docker

These are implementation details.

Clean Architecture treats them as replaceable.

---

# Why Are Frameworks on the Outside?

Many developers think:

> "Spring Boot is my application."

Clean Architecture says:

> "Spring Boot is only a tool."

Your business should continue to exist even if:

```text
Spring Boot

↓

Micronaut
```

or

```text
Oracle

↓

PostgreSQL
```

changes.

The framework serves the application—not the other way around.

---

# Example: Money Transfer

Business rule:

```text
Transfer Money
```

Should not know:

- Oracle
- Hibernate
- REST
- Kafka

Instead:

```text
TransferMoneyUseCase

↓

Account Repository Interface

↓

Repository Implementation

↓

Oracle
```

The use case depends on an interface.

The implementation depends on the database.

---

# Dependency Inversion

Suppose:

Traditional approach:

```text
TransferService

↓

OracleRepository
```

Business logic directly depends on Oracle.

Clean Architecture:

```text
TransferUseCase

↓

AccountRepository (Interface)

↑

OracleRepository (Implementation)
```

Business logic depends only on the abstraction.

Infrastructure implements that abstraction.

---

# Package Example

```text
com.company.bank

├── domain

│     ├── model

│     ├── service

│     └── repository

├── application

│     ├── usecase

│     └── dto

├── infrastructure

│     ├── persistence

│     ├── kafka

│     ├── redis

│     └── rest

└── configuration
```

Notice:

The domain package contains **no Spring annotations**.

---

# Example

Bad:

```java
@Service
public class TransferService {

}
```

The domain class now depends on Spring.

Better:

```java
public class TransferUseCase {

}
```

Spring configuration creates the bean.

The business logic itself remains framework-independent.

---

# Repository Example

Domain:

```java
public interface EmployeeRepository {

    Employee save(Employee employee);

}
```

Infrastructure:

```java
@Repository
public class JpaEmployeeRepository
        implements EmployeeRepository {

}
```

The domain defines **what it needs**.

Infrastructure decides **how to provide it**.

---

# Benefits

## Framework Independence

Changing:

```text
Spring Boot

↓

Micronaut
```

requires minimal changes to business logic.

---

## Database Independence

Switching:

```text
Oracle

↓

PostgreSQL
```

primarily affects the infrastructure layer.

The domain remains unchanged.

---

## Easier Testing

Business logic can be tested without:

- Spring Context
- Database
- Kafka
- Redis

Simple unit tests become possible.

---

## Better Maintainability

Business rules remain:

- Small
- Focused
- Stable

Infrastructure evolves independently.

---

# Drawbacks

Clean Architecture introduces:

- More interfaces
- More packages
- Additional abstractions
- Increased initial complexity

For small CRUD applications, this overhead may not be justified.

---

# Does Spring Boot Support Clean Architecture?

Yes.

Spring Boot does not enforce any architectural style.

You can implement:

- Layered Architecture
- Clean Architecture
- Hexagonal Architecture
- Onion Architecture

using the same framework.

Spring is flexible enough to support all of them.

---

# Comparison with Layered Architecture

Layered Architecture:

```text
Controller

↓

Service

↓

Repository

↓

Database
```

Dependencies generally flow downward through concrete implementations.

Clean Architecture:

```text
Infrastructure

↓

Application

↓

Domain

↑

Interfaces
```

The domain remains at the center and infrastructure depends on it through abstractions.

---

# Real-World Decision

Clean Architecture is widely adopted in systems where:

- Business rules are complex
- Long-term maintainability is critical
- Multiple infrastructure technologies may change over time

Examples include:

- Banking platforms
- Insurance systems
- Healthcare applications
- Large SaaS products

For a simple internal CRUD application, a conventional Layered Architecture is often sufficient and easier to maintain.

The cost of abstraction should be justified by the complexity of the business domain.

---

# Common Misconceptions

## "Clean Architecture Means More Layers"

No.

The goal is not adding layers.

The goal is **controlling dependencies**.

---

## "Spring Boot Is Not Allowed"

False.

Spring Boot works perfectly with Clean Architecture.

The key is ensuring that Spring-specific concerns remain outside the core business logic.

---

## "Everything Needs an Interface"

No.

Interfaces should represent meaningful boundaries, such as repositories or external services.

Creating interfaces for every class adds unnecessary complexity.

---

# Interview Questions

### Q1. What is the primary goal of Clean Architecture?

To keep business rules independent of frameworks, databases, UI, and other infrastructure concerns by ensuring dependencies point toward the domain.

---

### Q2. What is the Dependency Rule?

Source code dependencies must always point inward. Inner layers should not depend on outer layers.

---

### Q3. Why is the domain placed at the center?

Because the domain contains the core business rules, which should remain stable even if technologies, frameworks, or databases change.

---

### Q4. Why does the domain define repository interfaces instead of implementations?

The domain defines **what it requires**. Infrastructure provides the implementation, allowing the business logic to remain independent of persistence technology.

---

### Q5. Is Clean Architecture suitable for every project?

No.

It provides significant benefits for complex, long-lived systems but may introduce unnecessary complexity for small applications with simple business logic.

---

# Senior Interview Perspective

A common architecture discussion is:

> **Interviewer:** "If your company decides to migrate from Oracle to PostgreSQL, how much business code should change?"

A strong answer is:

> "Ideally, only the infrastructure layer should change. Business rules should remain unaffected because they depend on abstractions rather than concrete database implementations."

This demonstrates an understanding of dependency inversion and separation of concerns rather than simply knowing Spring annotations.

---

# Key Takeaways

- Clean Architecture places business rules at the center of the application.
- The Dependency Rule states that dependencies always point inward.
- Frameworks, databases, and messaging systems are implementation details that should remain outside the core domain.
- Repository interfaces belong to the domain or application layer, while implementations belong to infrastructure.
- Clean Architecture improves maintainability, testability, and flexibility but introduces additional abstraction that should be justified by the application's complexity.
