# 0.7 Case Study – Designing an Enterprise Order Management System

Throughout this playbook, we have discussed various architectural styles:

- Layered Architecture
- Modular Monolith
- Clean Architecture
- Hexagonal Architecture

A common question is:

> **How do these ideas come together in a real application?**

Let's design an enterprise-grade **Order Management System**, similar to those used in e-commerce, banking, logistics, or trading platforms.

This is not intended to be the only correct solution.

Rather, it demonstrates how experienced engineers make architectural decisions based on business requirements and technical trade-offs.

---

# Business Requirements

The system should support:

- Create Orders
- Update Orders
- Cancel Orders
- View Orders
- Reserve Inventory
- Process Payments
- Notify Customers
- Generate Reports

Non-functional requirements:

- High Availability
- Scalability
- Reliability
- Auditability
- Easy Maintenance
- Horizontal Scaling
- Event Publishing

---

# Step 1 – Choosing the Overall Architecture

Possible options:

| Architecture | Suitable? |
|-------------|-----------|
| Monolith | ⚠ Small Systems |
| Modular Monolith | ✅ Yes |
| Microservices | ⚠ Future Growth |

Suppose the company has:

- 10 Developers
- One Product
- 500K Users
- Continuous Deployment
- One Database Team

A **Modular Monolith** is an excellent choice.

Reasons:

- Faster development
- Easier deployment
- Simpler debugging
- Lower operational cost
- Clear module boundaries

No unnecessary distributed-system complexity.

---

# Step 2 – Define Business Modules

Instead of organizing by technical layers first, identify business capabilities.

```text
Order

Customer

Inventory

Payment

Notification

Reporting

Authentication
```

Each module owns its own business logic.

---

# Step 3 – Internal Package Structure

Within each module:

```text
order

├── controller

├── service

├── repository

├── entity

├── dto

├── mapper

├── exception

└── validation
```

This is a **Layered Architecture** inside the module.

Developers immediately recognize the structure.

---

# Step 4 – Request Flow

Suppose:

```
POST /orders
```

Flow:

```text
Client

↓

OrderController

↓

CreateOrderService

↓

Validate Order

↓

Reserve Inventory

↓

Save Order

↓

Publish Event

↓

Return Response
```

Each responsibility is clearly defined.

---

# Step 5 – Where Business Logic Lives

Business rules belong in:

```text
OrderService
```

Examples:

- Order must contain items
- Inventory must exist
- Payment required
- Customer active
- Credit limit validation

Controllers should never implement these rules.

---

# Step 6 – Database Access

Repositories contain only persistence logic.

Example:

```java
OrderRepository

CustomerRepository

InventoryRepository
```

No business calculations.

No HTTP.

No Kafka.

---

# Step 7 – External Integrations

The application communicates with:

- Payment Gateway
- Email Service
- SMS
- Kafka
- Redis

Instead of calling them directly:

```text
OrderService

↓

Payment Port

↓

Stripe Adapter
```

Later:

```text
Payment Port

↓

Razorpay Adapter
```

No business logic changes.

This is **Hexagonal Architecture** applied selectively.

---

# Step 8 – Applying Clean Architecture

Business rules should not know:

- Spring Boot
- Oracle
- Kafka
- Redis

Instead:

```text
CreateOrderUseCase

↓

OrderRepository

↓

JpaOrderRepository
```

Business logic depends on abstractions.

Infrastructure provides implementations.

---

# Step 9 – Event Publishing

Order created:

```text
Order Service

↓

OrderCreated Event

↓

Kafka
```

Consumers:

```text
Inventory

Notification

Reporting
```

Adding a new consumer should not require changes to the Order module.

---

# Step 10 – Transaction Boundary

The transaction begins in the Service Layer.

```java
@Transactional
public Order createOrder(...)
```

Within the transaction:

- Validate
- Reserve inventory
- Save order

External notifications are generally triggered after the transaction commits, often using an event or the Outbox Pattern to avoid inconsistencies.

---

# Step 11 – Validation Strategy

Controller:

```text
@NotNull

@Email

@Positive
```

Service:

```text
Customer Active

Inventory Available

Business Rules

Credit Limit
```

Each layer validates what it owns.

---

# Step 12 – Exception Handling

Controllers should not contain:

```java
try {

...

} catch (...) {

}
```

Instead:

```text
Controller

↓

Service

↓

Global Exception Handler
```

Spring's exception handling mechanisms centralize error responses.

---

# Step 13 – Package Layout

```text
com.company.orders

├── order

├── inventory

├── payment

├── notification

├── reporting

├── security

├── common

└── configuration
```

Business modules remain independent.

---

# Step 14 – Scaling the Application

Initially:

```text
Single Application

↓

4 Instances
```

Later:

Heavy traffic on payments.

Extract:

```text
Payment Module

↓

Payment Service
```

No need to split everything.

Only the bottleneck.

---

# Step 15 – Monitoring

Production monitoring includes:

- Health Checks
- Metrics
- Logs
- Distributed Tracing
- Alerts

These concerns stay outside business logic.

- Clean Architecture protects business logic.
- Hexagonal Architecture isolates external systems.

These approaches complement each other.

---

# Why Not Start with Microservices?

Many teams make this mistake.

Instead of:

```text
20 Services
```

Start with:

```text
1 Modular Monolith
```

Reasons:

- Faster delivery
- Simpler testing
- Easier debugging
- Lower operational overhead

Extract services only when business or operational needs justify it.

---

# Real-World Perspective

A common architecture in large organizations looks like this:

```text
Modular Monolith

↓

Layered Architecture

↓

Clean Architecture Principles

↓

Hexagonal Ports

↓

Event-Driven Integration

↓

Selective Microservices
```

Notice:

No single architecture dominates.

Instead, engineers combine patterns based on practical needs.

---

# Senior Interview Scenario

**Interviewer:**

> "Design a backend for an Order Management System."

A strong response would include:

- Modular Monolith as the initial deployment model.
- Layered Architecture for clear separation of concerns.
- Business rules isolated from infrastructure using Clean Architecture principles.
- Ports & Adapters for integrations such as payments, notifications, and messaging.
- Event-driven communication for asynchronous workflows.
- A migration strategy toward microservices only if scaling or team autonomy requires it.

This demonstrates architectural judgment rather than preference for a particular trend.

---

# Key Takeaways

- Enterprise architecture is about making informed trade-offs, not following a single pattern.
- Start with the simplest architecture that satisfies current requirements.
- Use Layered Architecture for organization.
- Apply Clean Architecture to protect business rules.
- Introduce Hexagonal Ports & Adapters at infrastructure boundaries.
- Allow the system to evolve gradually as business complexity and scale increase.
