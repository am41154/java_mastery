# 0.5 Hexagonal Architecture (Ports & Adapters)

One of the biggest challenges in enterprise software is that **business logic gradually becomes dependent on external systems**.

Examples:

- Databases
- REST APIs
- Kafka
- Redis
- RabbitMQ
- Cloud SDKs
- File Systems

Over time, the application starts revolving around these technologies rather than the business itself.

To solve this problem, **Alistair Cockburn** introduced **Hexagonal Architecture**, also known as **Ports and Adapters Architecture**.

The core idea is simple:

> **The application should not know how the outside world communicates with it.**

Instead, the application defines **ports**, and the outside world plugs into those ports using **adapters**.

---

# Why Another Architecture?

Consider a typical Spring Boot application.

```text
Controller

↓

Service

↓

Repository

↓

Database
```

Looks good.

Now add:

- Kafka
- Redis
- REST Clients
- Scheduled Jobs
- Batch Processing
- WebSockets
- CLI Commands

Soon the Service Layer starts depending on everything.

```text
Service

├── Repository

├── Kafka

├── Redis

├── REST Client

├── Email

├── File Storage
```

Business logic becomes tightly coupled with infrastructure.

---

# The Main Idea

Instead of designing around technologies:

```text
Spring

↓

JPA

↓

Oracle
```

Hexagonal Architecture designs around the **business**.

Everything else becomes a plugin.

---

# Why "Hexagonal"?

The shape itself is symbolic.

The important idea is:

The application can have **multiple input and output channels**.

Example:

```text
             REST

               │

CLI ───►  Application  ◄── Kafka

               │

          Scheduled Job

               │

           WebSocket
```

Every external system communicates through a defined boundary.

---

# Core Components

Hexagonal Architecture consists of three primary concepts:

- Domain
- Ports
- Adapters

---

# Domain

The center of the application.

Contains:

- Business Rules
- Entities
- Value Objects
- Domain Services
- Use Cases

The domain knows nothing about:

- Spring Boot
- Hibernate
- Kafka
- Oracle
- REST

---

# Ports

A **Port** is simply an interface.

It defines:

> "What the application needs."

Examples:

```java
public interface PaymentPort {

    void process(Payment payment);

}
```

Notice:

No Spring annotations.

No SQL.

No HTTP.

Only business intent.

---

# Adapters

Adapters implement Ports.

Example:

```java
public class StripeAdapter
        implements PaymentPort {

}
```

Another adapter:

```java
public class RazorpayAdapter
        implements PaymentPort {

}
```

The application doesn't care which adapter is used.

---

# Primary vs Secondary Adapters

Hexagonal Architecture distinguishes between:

### Primary (Driving) Adapters

These initiate interaction with the application.

Examples:

- REST Controller
- GraphQL
- CLI
- Kafka Consumer
- Scheduler

They **drive** the application.

---

### Secondary (Driven) Adapters

These are used by the application.

Examples:

- Database
- Kafka Producer
- Email Service
- Redis
- REST Client
- Cloud Storage

The application depends on ports.

Adapters fulfill those ports.

---

The **Use Case** communicates only with Ports.

# Request Flow

Suppose a REST API creates an order.

Flow:

```text
HTTP Request

↓

REST Controller

↓

CreateOrderUseCase

↓

OrderRepositoryPort

↓

JpaOrderRepositoryAdapter

↓

Oracle
```

If Oracle is replaced by PostgreSQL:

Only the adapter changes.

The use case remains untouched.

---

# Package Structure

```text
com.company.order

├── domain

│     ├── model

│     ├── service

│     └── port

├── application

│     └── usecase

├── adapter

│     ├── in

│     │      ├── rest

│     │      ├── kafka

│     │      └── scheduler

│     └── out

│            ├── persistence

│            ├── redis

│            ├── kafka

│            └── email

└── configuration
```

This structure clearly separates business logic from infrastructure.

---

# Example

## Output Port

```java
public interface EmployeeRepositoryPort {

    Employee save(Employee employee);

}
```

---

## Adapter

```java
@Repository
public class JpaEmployeeRepositoryAdapter
        implements EmployeeRepositoryPort {

}
```

---

## Use Case

```java
public class CreateEmployeeUseCase {

    private final EmployeeRepositoryPort repository;

}
```

The use case has no knowledge of JPA or Spring Data.

---

# Input Port

Large applications often define input ports as well.

Example:

```java
public interface CreateEmployeeUseCase {

    EmployeeResponse create(
            EmployeeRequest request
    );

}
```

The REST Controller depends on this interface instead of a concrete service.

This improves testability and flexibility.

---

# Benefits

## Technology Independence

Switch:

```text
Oracle

↓

PostgreSQL
```

Only adapters change.

---

## Better Testing

The domain can be tested using simple mock implementations of ports.

No Spring Context.

No database.

---

## Easier Integration

Adding:

- Kafka
- GraphQL
- Batch Jobs

does not require changes to the core business logic.

New adapters plug into existing ports.

---

## Strong Separation

The business remains independent from:

- Spring Boot
- Hibernate
- Kafka
- Redis
- REST

---

# Drawbacks

Hexagonal Architecture introduces:

- More interfaces
- More packages
- More abstractions
- Higher learning curve

For a simple CRUD application, these costs may outweigh the benefits.

---

# Clean Architecture vs Hexagonal Architecture

Many developers confuse the two.

Clean Architecture focuses on:

> **Dependency direction**

Hexagonal Architecture focuses on:

> **Communication boundaries**

They are highly compatible.

In fact, many enterprise systems combine ideas from both.

---

# Does Spring Boot Support Hexagonal Architecture?

Absolutely.

Spring Boot is simply the framework used to wire components together.

Ports remain plain Java interfaces.

Adapters become Spring beans.

Spring handles dependency injection.

---

# Real-World Example

Suppose your payment service initially uses Stripe.

```text
Application

↓

Payment Port

↓

Stripe Adapter
```

Later, the business expands to India and adopts Razorpay.

No business logic changes.

Simply add:

```text
Payment Port

├── Stripe Adapter

└── Razorpay Adapter
```

The application continues to interact through the same port.

This is one of the strongest advantages of the Ports & Adapters approach.

---

# Real-World Decision

Many financial institutions and large product companies use a variation of Hexagonal Architecture.

Examples include:

- Banking platforms
- Payment gateways
- Trading systems
- Healthcare platforms
- SaaS products with multiple external integrations

The architecture is particularly valuable when:

- Business rules are complex.
- Infrastructure changes frequently.
- Multiple integration points exist.
- Long-term maintainability is a priority.

However, for small internal applications with limited integrations, a well-designed Layered Architecture is often the more pragmatic choice.

---

# Common Mistakes

## Creating Ports for Everything

Not every class needs a port.

Ports should represent meaningful boundaries, such as:

- Persistence
- Messaging
- External services
- Notifications

Creating unnecessary interfaces increases complexity without adding value.

---

## Putting Business Logic in Adapters

Adapters should translate between external systems and the application.

Business rules belong in the domain or use case layer.

---

## Treating Adapters as Services

An adapter is not a place for business decisions.

Its responsibility is to communicate with infrastructure and convert data.

# Interview Questions

### Q1. What is the primary goal of Hexagonal Architecture?

To isolate business logic from external systems by communicating through well-defined ports implemented by adapters.

---

### Q2. What is a Port?

A Port is an interface that defines how the application communicates with external systems or receives requests.

---

### Q3. What is an Adapter?

An Adapter is an implementation of a port that connects the application to a specific technology such as a database, REST API, Kafka, or email service.

---

### Q4. What is the difference between Primary and Secondary Adapters?

- Primary (Driving) Adapters invoke the application (e.g., REST controllers, Kafka consumers).
- Secondary (Driven) Adapters are invoked by the application (e.g., JPA repositories, REST clients, message producers).

---

### Q5. Can Hexagonal Architecture be implemented using Spring Boot?

Yes.

Spring Boot provides dependency injection and infrastructure support, while the architectural boundaries are defined by the application's package structure and dependency direction.

---

# Senior Interview Perspective

A common design discussion is:

> **Interviewer:** "What would happen if tomorrow you needed to support both Oracle and MongoDB?"

A strong answer is:

> "The business logic should continue interacting with a persistence port. Separate adapters can implement that port for Oracle and MongoDB. The core use cases remain unchanged."

This demonstrates an understanding of technology independence rather than framework-specific coding.

---

# Key Takeaways

- Hexagonal Architecture organizes applications around business capabilities rather than technologies.
- Ports define the application's required interactions, while adapters implement those interactions using specific technologies.
- Primary adapters drive the application; secondary adapters provide infrastructure services.
- The architecture enables technology independence, easier testing, and cleaner integration boundaries.
- Hexagonal Architecture works exceptionally well for systems with multiple external integrations but may introduce unnecessary complexity for smaller applications.
