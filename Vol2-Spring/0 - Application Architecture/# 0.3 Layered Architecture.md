# 0.3 Layered Architecture

If there is one architecture that every Java developer has worked with, it is the **Layered Architecture** (also known as the **N-Tier Architecture**).

Almost every Spring Boot application follows this pattern in some form.

Even organizations that use:

- Microservices
- Modular Monoliths
- Hexagonal Architecture

still use layered separation **inside each service**.

For this reason, Layered Architecture is considered the **foundation of enterprise Java development**.

---

# What is Layered Architecture?

Layered Architecture organizes an application into multiple horizontal layers.

Each layer has a well-defined responsibility and communicates only with adjacent layers.

Typical structure:

```text
                Client
                   │
                   ▼
         +-------------------+
         |   Controller      |
         +-------------------+
                   │
                   ▼
         +-------------------+
         |     Service       |
         +-------------------+
                   │
                   ▼
         +-------------------+
         |   Repository      |
         +-------------------+
                   │
                   ▼
         +-------------------+
         |    Database       |
         +-------------------+
```

Each layer knows **only what it needs to know**.

---

# Why Layers?

Imagine a controller directly writing SQL:

```java
@RestController
public class EmployeeController {

    @GetMapping("/employees")
    public List<Employee> getEmployees() {

        Connection connection = ...

        PreparedStatement statement = ...

        ResultSet rs = ...
    }
}
```

Problems:

- HTTP logic
- SQL
- Database connection
- Business rules

are all mixed together.

This violates the **Single Responsibility Principle (SRP)**.

---

# Responsibilities of Each Layer

## Presentation Layer

Handles interaction with external clients.

Examples:

- REST APIs
- GraphQL
- WebSocket
- gRPC
- MVC Views

Typical Spring components:

```java
@RestController

@Controller
```

Responsibilities:

- Accept requests
- Validate input format
- Convert DTOs
- Return responses
- Set HTTP status codes

It should **not** contain business logic.

---

## Service Layer

The heart of the application.

Responsibilities:

- Business rules
- Validation
- Transactions
- Orchestration
- Calling external systems

Typical Spring component:

```java
@Service
```

Example:

```java
public Employee createEmployee(
        EmployeeRequest request) {

    validate(request);

    repository.save(...);

    audit(...);

    publishEvent(...);

    return employee;
}
```

The Service Layer coordinates business operations.

---

## Repository Layer

Responsible only for persistence.

Typical Spring components:

```java
@Repository

JpaRepository
```

Responsibilities:

- CRUD
- Database queries
- JPQL
- SQL
- Specifications

Example:

```java
employeeRepository.findById(id);
```

Repositories should **not** implement business rules.

---

## Database Layer

Responsible for storing data.

Examples:

- Oracle
- PostgreSQL
- MySQL
- MongoDB
- Redis

The database should enforce:

- Constraints
- Indexes
- Referential integrity

Business rules still belong in the Service Layer.

---

Controller:

```java
@RestController

public class EmployeeController {

    private final EmployeeService service;

    @PostMapping
    public EmployeeResponse create(

            @RequestBody EmployeeRequest request) {

        return service.create(request);
    }
}
```

---

```java
@Service
public class EmployeeService {

    public EmployeeResponse create(
            EmployeeRequest request) {

        ...
    }
```

```java
@Repository
public interface EmployeeRepository
        extends JpaRepository<Employee, Long> {

}
```

Simple.

Readable.

Maintainable.

# Benefits of Layered Architecture

## Separation of Concerns

Each layer has a single responsibility.

---

## Easier Testing

Controller:

```text
Mock Service
```

Service:

```text
Mock Repository
```

Each layer can be tested independently.

---

## Reusability

Multiple controllers may reuse the same service.

Example:

```text
REST API

↓

EmployeeService

↓

Batch Job
```

Business logic remains centralized.

---

## Maintainability

Changing:

```text
Oracle

↓

PostgreSQL
```

usually affects only the Repository Layer.

Business logic remains unchanged.

---

## Better Team Collaboration

Frontend developers focus on:

Controllers.

Backend developers focus on:

Services.

DB specialists focus on:

Repositories and schema.

Clear boundaries reduce conflicts.

---

# Common Mistakes

## Fat Controllers

Bad:

```java
@PostMapping
public Employee create(...) {

    validate();

    calculateSalary();

    sendEmail();

    save();

    publishKafkaEvent();

    updateRedis();
}
```

Controllers become hundreds of lines long.

This makes testing and maintenance difficult.

Controllers should remain thin.

---

## Anemic Services

Bad:

```java
public Employee create(Employee e){

    repository.save(e);
}
```

If every service simply delegates to the repository, business rules are likely scattered elsewhere or missing entirely.

The Service Layer should encapsulate meaningful business behavior.

---

## Fat Repositories

Repositories should not contain:

- Validation
- Business calculations
- Email sending
- Kafka publishing

Only persistence logic belongs here.

---

# Recommended Dependency Direction

```text
Controller

↓

Service

↓

Repository
```

Never:

```text
Repository

↓

Controller
```

or

```text
Repository

↓

Service

↓

Controller
```

Dependencies should always point downward.

---

# DTOs vs Entities

A common enterprise pattern:

```text
Request DTO

↓

Controller

↓

Entity

↓

Repository

↓

Entity

↓

Service

↓

Response DTO

↓

Controller
```

Do **not** expose JPA entities directly from REST APIs.

Reasons:

- Security
- API stability
- Lazy-loading issues
- Tight coupling between persistence and API contracts

---

# Where Should Validation Go?

Input validation:

```java
@NotNull

@Size

@Email
```

belongs in the Controller layer (or request DTOs).

Business validation belongs in the Service Layer.

Example:

```text
Age >= 18

Credit Limit Available

Account Active
```

These require business knowledge.

---

# Transactions

The transaction boundary usually belongs in the Service Layer.

Example:

```java
@Transactional
public void transferMoney() {

}
```

The Service Layer coordinates all database operations within a single business transaction.

---

# Is Layered Architecture Enough?

For many applications:

Yes.

However, as applications become larger:

- Domain logic grows
- External integrations increase
- Testing becomes harder
- Infrastructure concerns spread

This leads to architectures such as:

- Clean Architecture
- Hexagonal Architecture

These build upon the same layering concepts while introducing stronger dependency rules.

---

# Real-World Decision

Most enterprise Spring Boot applications still follow Layered Architecture.

Examples include:

- Banking systems
- Insurance platforms
- ERP systems
- Internal enterprise tools
- Government applications

Even organizations using microservices typically structure **each microservice internally** as:

```text
Controller

↓

Service

↓

Repository
```

This pattern has remained effective because it is simple, well understood, and supported naturally by Spring.

# Interview Questions

### Q1. Why is Layered Architecture popular?

It separates responsibilities, improves maintainability, simplifies testing, and provides a clear structure that scales well for many enterprise applications.

---

### Q2. Where should business logic reside?

Business logic belongs primarily in the **Service Layer**.

Controllers coordinate HTTP interactions, while repositories handle persistence.

---

### Q3. Why shouldn't Controllers call Repositories directly?

Doing so mixes presentation and persistence concerns, bypasses business rules, makes testing harder, and often leads to duplicated logic.

---

### Q4. Why shouldn't JPA entities be returned directly from REST APIs?

Because entities are persistence models, not API contracts. Returning them can expose internal fields, trigger lazy-loading issues, and tightly couple clients to the database model.

---

### Q5. Where should `@Transactional` typically be placed?

At the Service Layer, where business operations spanning one or more repository calls are coordinated.

---

# Limitations of Layered Architecture

As applications grow, several challenges emerge:

- Services become very large ("God Services")
- Business logic may be scattered across services
- Infrastructure concerns leak into business logic
- Testing complex business rules becomes harder
- Tight coupling between domain logic and framework components

These limitations motivated newer architectural styles such as **Clean Architecture** and **Hexagonal Architecture**, which aim to better isolate business logic from infrastructure.

---

# Key Takeaways

- Layered Architecture is the foundation of most enterprise Spring applications.
- Each layer has a clearly defined responsibility:
  - Controller → Presentation
  - Service → Business Logic
  - Repository → Data Access
  - Database → Persistence
- Business logic should remain in the Service Layer, while controllers and repositories stay focused on their respective concerns.
- DTOs should be used at API boundaries instead of exposing JPA entities directly.
- Although Layered Architecture remains highly effective, larger and more complex systems often evolve toward Clean or Hexagonal Architecture for stronger separation of concerns.
