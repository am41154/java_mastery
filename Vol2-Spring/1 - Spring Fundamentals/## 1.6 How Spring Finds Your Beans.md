## 1.6 How Spring Finds Your Beans: Component Scanning

**Objective:** Understand how Spring discovers application components, how Component Scanning works internally, why package structure matters, and how the IoC container identifies which classes should become Spring Beans.

Until now, we have learned that:

- The IoC Container manages application objects.
- Managed objects are called **Spring Beans**.
- Spring Beans are automatically created and wired together.

This naturally leads to another important question:

> **How does Spring know which classes should become Beans?**

After all, a typical enterprise project may contain thousands of Java classes.

```text
src
 ├── controllers
 ├── services
 ├── repositories
 ├── dto
 ├── entities
 ├── configuration
 ├── exceptions
 ├── utilities
 ├── security
 ├── clients
 ├── events
 └── ...
```

Clearly, not every class should become a Spring Bean.

For example:

- DTOs should not become beans.
- JPA entities should not become beans.
- Utility classes usually should not become beans.
- Constants classes should not become beans.

Spring needs a mechanism to distinguish **application components** from ordinary Java classes.

That mechanism is called **Component Scanning**.

---

## Before Component Scanning

In the early versions of Spring, every bean had to be declared explicitly inside an XML configuration file.

```xml
<bean id="paymentService"
      class="com.company.service.PaymentService"/>

<bean id="orderService"
      class="com.company.service.OrderService"/>

<bean id="inventoryService"
      class="com.company.service.InventoryService"/>
```

Imagine maintaining this configuration for an application containing:

- 800 services
- 250 repositories
- 120 controllers

Adding a new service required:

1. Writing the Java class.
2. Updating the XML.
3. Restarting the application.

As projects grew, XML configuration became increasingly difficult to maintain.

---

## Annotation-Based Discovery

Spring introduced annotations to eliminate most manual configuration.

Instead of declaring beans in XML, developers simply annotate the class.

```java
@Service
public class PaymentService {

}
```

or

```java
@Repository
public class OrderRepository {

}
```

or

```java
@Controller
public class OrderController {

}
```

Now the developer only writes the class.

Spring discovers it automatically.

---

## What is Component Scanning?

Component Scanning is the process by which the IoC container:

1. scans specified packages,
2. identifies classes marked with Spring stereotypes,
3. creates Bean Definitions,
4. registers them with the container.

Conceptually:

```text
Project

       │

Package Scan

       │

Annotated Classes Found

       │

Create Bean Definitions

       │

Register Beans

       │

Ready for Dependency Injection
```

Notice something important.

At this stage, Spring is **discovering metadata**, not merely creating objects.

The actual bean instantiation happens later during the container startup lifecycle.

---

## Which Annotations are Detected?

By default, Component Scanning looks for classes annotated with Spring stereotypes such as:

```java
@Component
```

```java
@Service
```

```java
@Repository
```

```java
@Controller
```

```java
@RestController
```

Although these annotations serve different semantic purposes, they all ultimately identify a class as a Spring-managed component.

Later chapters will explain why each stereotype exists and the additional behavior associated with some of them.

---

## The Root of Component Scanning

Every scan starts somewhere.

In Spring Boot, scanning typically begins from the package containing the main application class.

Example:

```java
@SpringBootApplication
public class TradingApplication {

    public static void main(String[] args) {

    }

}
```

Suppose the project structure is:

```text
com.company

    TradingApplication

    controller

    service

    repository

    security

    configuration
```

Spring begins scanning from:

```text
com.company
```

and recursively scans all child packages.

Every eligible component found within this hierarchy becomes a candidate bean.

This convention is one of the reasons Spring Boot projects follow a standardized package structure.

---

## Why Package Structure Matters

Consider this project.

```text
com.company.application

        TradingApplication
```

Now suppose services are placed under:

```text
org.external.service
```

Since this package is outside the default scanning hierarchy, Spring never discovers those classes.

As a result:

- beans are not registered,
- dependency injection fails,
- startup errors occur.

This is one of the most common issues encountered by developers when reorganizing package structures.

A well-designed package hierarchy is therefore not just a matter of readability—it directly affects application startup.

---

## How Does Spring Perform the Scan?

Conceptually, the container performs something similar to the following:

```text
For each class in configured packages

    ↓

Read class metadata

    ↓

Check annotations

    ↓

Is it a Spring component?

    ↓

Yes

    ↓

Create Bean Definition

    ↓

Register with the container
```

Notice that Spring does **not** instantiate every class during scanning.

It first collects metadata describing potential beans.

This distinction becomes important when studying Bean Definitions and the bean creation lifecycle.

---

## Bean Definition vs Bean Instance

Component Scanning creates **Bean Definitions**, not necessarily bean instances.

Think of it this way.

During scanning:

```text
PaymentService

↓

Metadata Stored
```

Later:

```text
Metadata

↓

Actual Object Created
```

Separating discovery from instantiation gives Spring tremendous flexibility.

For example, the container can:

- validate dependencies,
- detect circular references,
- determine scopes,
- apply conditional configuration,
- prepare proxy information,

before creating a single object.

---

## Enterprise Perspective

A modern Spring Boot application may contain:

- hundreds of services,
- dozens of REST controllers,
- repositories,
- messaging consumers,
- schedulers,
- event listeners,
- configuration classes,
- security components.

Manually registering every bean would be error-prone and difficult to maintain.

Component Scanning allows the application to grow naturally while keeping configuration minimal.

Developers focus on writing business components, while the framework automatically discovers and registers them.

This convention-over-configuration approach is one of the primary reasons Spring Boot dramatically reduces boilerplate compared to earlier Spring applications.

---

## Key Takeaways

- Component Scanning is how Spring discovers application components.
- It replaces most manual XML bean registration.
- Spring scans configured packages for stereotype annotations.
- Scanning produces **Bean Definitions**, not immediately instantiated objects.
- The package containing the main application class usually becomes the root of the scan.
- Proper package organization is essential because Spring only scans configured package hierarchies.

---

## Interview Focus

Senior interviewers often ask:

- How does Spring discover beans automatically?
- What is Component Scanning?
- Why does moving the main application class affect bean discovery?
- What happens internally when Spring encounters a `@Service` annotation?
- What is the difference between Component Scanning and Bean Creation?

A strong answer should explain that **Component Scanning is a metadata discovery process**. It identifies candidate classes, creates Bean Definitions, and registers them with the IoC container. Actual bean creation occurs later as part of the container initialization lifecycle.
