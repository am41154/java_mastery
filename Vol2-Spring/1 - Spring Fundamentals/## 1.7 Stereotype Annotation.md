## 1.7 Stereotype Annotations: `@Component`, `@Service`, `@Repository`, and `@Controller`

**Objective:** Understand why Spring provides multiple stereotype annotations, how they differ internally, why they all ultimately derive from `@Component`, and why choosing the correct stereotype improves readability, tooling, and framework integration in enterprise applications.

After learning how **Component Scanning** discovers application components, another question naturally arises:

> **How does Spring decide which classes are application components?**

The answer lies in **stereotype annotations**.

When Spring scans your application, it doesn't automatically register every Java class as a bean.

Instead, it looks for specific annotations that indicate:

> "This class is intended to be managed by the Spring IoC Container."

These annotations are called **stereotype annotations**.

---

## What is a Stereotype Annotation?

A stereotype annotation is simply a marker that tells Spring:

> "Treat this class as a managed component."

For example:

```java
@Service
public class PaymentService {

}
```

During Component Scanning, Spring detects this annotation and registers the class as a Spring Bean.

Without a stereotype annotation (or explicit bean registration), Spring completely ignores the class.

---

## The Core Stereotype: `@Component`

At the heart of all stereotype annotations is:

```java
@Component
public class PaymentService {

}
```

`@Component` is the most generic stereotype.

It simply tells Spring:

> "Register this class as a Spring Bean."

Internally, the other commonly used stereotype annotations are themselves built upon `@Component`.

Conceptually:

```text
          @Component
          /    |    \
         /     |     \
@Service @Repository @Controller
                      |
               @RestController
```

This hierarchy allows Spring to recognize all of them during Component Scanning.

---

## Why Not Use `@Component` Everywhere?

Technically, you could.

The following classes would all be discovered successfully.

```java
@Component
public class OrderService {

}
```

```java
@Component
public class OrderRepository {

}
```

```java
@Component
public class OrderController {

}
```

The application would still work.

So why does Spring provide different annotations?

Because **they communicate intent**.

When another developer reads:

```java
@Service
```

they immediately understand:

> This class contains business logic.

Likewise,

```java
@Repository
```

implies:

> This class interacts with the database.

Good software is not only written for the compiler—it is written for other developers.

Stereotype annotations improve the readability and maintainability of large codebases.

---

## `@Component`

`@Component` is the most generic stereotype.

Use it for classes that do not naturally belong to another architectural layer.

Examples include:

- utility components
- helper classes
- custom validators
- converters
- strategy implementations
- schedulers
- infrastructure components

Example:

```java
@Component
public class JwtTokenGenerator {

}
```

There is no special framework behavior associated with `@Component`.

It simply registers the class as a Spring-managed bean.

---

## `@Service`

`@Service` represents the **business layer**.

Example:

```java
@Service
public class PaymentService {

}
```

Typical responsibilities include:

- business rules
- validations
- orchestration
- workflow management
- interaction with repositories
- interaction with external services

From a bean registration perspective,

```java
@Service
```

behaves exactly like:

```java
@Component
```

However, the annotation clearly communicates the architectural role of the class.

Many enterprise coding standards require business logic classes to use `@Service` instead of the generic `@Component`.

---

## `@Repository`

`@Repository` identifies classes responsible for persistence.

Example:

```java
@Repository
public class OrderRepository {

}
```

Unlike `@Service`, `@Repository` provides an additional framework benefit.

It participates in **exception translation**.

Suppose the underlying database throws a vendor-specific exception.

```text
SQLException
```

Spring can translate it into one of its portable data access exceptions.

For example:

```text
DataAccessException
```

This abstraction allows application code to remain independent of a specific database vendor.

We will study Spring's exception translation mechanism in detail in the Data Access chapter.

---

## `@Controller`

`@Controller` identifies presentation-layer components.

Example:

```java
@Controller
public class OrderController {

}
```

Its responsibilities typically include:

- receiving HTTP requests
- validating input
- invoking services
- preparing responses

Controllers should contain minimal business logic.

Instead, they delegate work to service-layer beans.

---

## `@RestController`

Modern REST applications usually use:

```java
@RestController
public class OrderController {

}
```

Internally,

```java
@RestController
```

is effectively equivalent to:

```java
@Controller
@ResponseBody
```

This instructs Spring MVC to write the returned object directly to the HTTP response body instead of resolving a view.

Although frequently discussed alongside stereotype annotations, `@RestController` is primarily an MVC annotation and will be explored in depth later in the Spring MVC chapters.

---

## All Roads Lead to `@Component`

Although the annotations appear different, they ultimately serve the same fundamental purpose.

```text
@Component

↓

Spring Bean
```

```text
@Service

↓

@Component

↓

Spring Bean
```

```text
@Repository

↓

@Component

↓

Spring Bean
```

```text
@Controller

↓

@Component

↓

Spring Bean
```

The container registers all of them as managed beans.

The difference lies in the semantic meaning and additional framework behavior attached to certain stereotypes.

---

## How Component Scanning Uses Stereotypes

During startup, Component Scanning performs a process similar to:

```text
Read Class

      │

Check Annotations

      │

@Component?

      │

Yes

      │

Create Bean Definition

      │

Register Bean
```

Since `@Service`, `@Repository`, and `@Controller` are meta-annotated with `@Component`, they are also discovered automatically.

This design allows Spring to support both generic and specialized component types without duplicating scanning logic.

---

## Choosing the Correct Annotation

Consider these examples.

Business logic:

```java
@Service
public class PricingService {

}
```

Persistence:

```java
@Repository
public class TradeRepository {

}
```

REST endpoint:

```java
@RestController
public class TradeController {

}
```

Infrastructure:

```java
@Component
public class RequestIdGenerator {

}
```

Although replacing every annotation with `@Component` would still produce a functioning application, doing so would hide the architectural intent of each class.

Well-structured enterprise applications use stereotypes to make the system self-documenting.

---

## Enterprise Perspective

Large enterprise applications may contain thousands of Spring Beans.

Understanding their roles becomes increasingly important as systems grow.

Imagine opening a project containing:

- 600 services
- 200 repositories
- 150 controllers
- 80 schedulers
- 120 event listeners

Without meaningful stereotypes, navigating the codebase becomes significantly harder.

Stereotype annotations provide architectural context at a glance, making onboarding, code reviews, and maintenance considerably easier.

They also enable Spring and third-party frameworks to apply specialized behavior to particular categories of beans.

---

## Key Takeaways

- Stereotype annotations identify classes that should become Spring Beans.
- `@Component` is the generic stereotype annotation.
- `@Service`, `@Repository`, and `@Controller` are specialized forms of `@Component`.
- All stereotype annotations are discovered during Component Scanning.
- `@Repository` additionally participates in Spring's exception translation mechanism.
- Choosing the appropriate stereotype improves readability and communicates architectural intent.

---

## Interview Focus

Senior interviewers frequently ask:

- What is the difference between `@Component` and `@Service`?
- Can every stereotype annotation be replaced with `@Component`?
- Why does Spring provide multiple stereotype annotations?
- What additional behavior does `@Repository` provide?
- How does Component Scanning discover `@Service` if it only scans for `@Component`?

A strong answer should explain that all stereotype annotations ultimately derive from `@Component`, making them discoverable by the IoC container. Their primary purpose is to express architectural intent, while some annotations—such as `@Repository`—also enable additional framework behavior like automatic exception translation.
