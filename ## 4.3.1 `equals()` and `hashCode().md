## 4.3.1 `equals()` and `hashCode()` – The Contract Behind `HashMap`

### Why This Matters

Sooner or later, every senior Java interview reaches this question:

> **"Why do we need both `equals()` and `hashCode()`?"**

Many developers can define each method individually.

Far fewer can explain **how they work together inside a `HashMap`.**

If you understand this relationship, you understand one of the most fundamental concepts in Java.

---

## A Common Misconception

Consider the following statement:

> **"HashMap uses `equals()` to find an object."**

This is only partially correct.

The actual lookup process is:

1. Compute the hash code.
2. Find the appropriate bucket.
3. Compare keys using `equals()` within that bucket.

Notice that `equals()` is **not** the first step.

It is only used after the correct bucket has been identified.

---

## The Lookup Process

Suppose we insert:

```java
Employee emp = new Employee(101, "John");

map.put(emp, "Developer");
```

### Step 1 – Compute Hash Code

```
Employee

↓

hashCode()

↓

4589321
```

---

### Step 2 – Find Bucket

```
4589321

↓

Bucket 9
```

```
Bucket Array

0

1

2

...

9

...

15
```

The entry is stored inside Bucket 9.

---

### Step 3 – Handle Collision (If Any)

Suppose another object also maps to Bucket 9.

```
Bucket 9

Employee(101)

Employee(205)

Employee(310)
```

Now the bucket contains multiple entries.

How does Java determine which one is correct?

It calls:

```java
equals()
```

Only the matching key is returned.

---

## Why Isn't `hashCode()` Enough?

Imagine two different employees.

```
Employee A

hashCode = 25

Employee B

hashCode = 25
```

Both objects have identical hash codes.

Does that mean they are equal?

No.

Different objects can legitimately produce the same hash code.

This is called a **hash collision**.

That is why Java performs a second check using `equals()`.

```
Same Hash

↓

equals()

↓

True or False
```

Only when `equals()` returns `true` has the correct key been found.

---

## The Contract

Java defines an important contract between these two methods.

### Rule 1

If two objects are equal,

their hash codes **must** be equal.

```
a.equals(b)

↓

true

↓

a.hashCode() == b.hashCode()
```

This rule is mandatory.

---

### Rule 2

If two objects have the same hash code,

they **do not have to be equal**.

```
hashCode()

↓

25

↓

Could still be different objects
```

This explains why collisions are perfectly valid.

---

### Rule 3

If `equals()` is overridden,

`hashCode()` must also be overridden.

Ignoring this rule breaks collections such as:

- `HashMap`
- `HashSet`
- `ConcurrentHashMap`

This is one of the most common interview questions.

---

## A Classic Bug

Consider the following class.

```java
class Employee {

    private int id;

    @Override
    public boolean equals(Object obj) {
        ...
    }

}
```

Notice something missing?

`hashCode()`.

Now insert:

```java
map.put(employee, "Developer");
```

Later,

```java
map.get(employee);
```

Unexpectedly returns:

```
null
```

Even though the object exists.

Why?

Because the lookup computes a different hash code and searches a different bucket.

The object is never found.

---

## Correct Implementation

Modern Java makes this easy.

```java
@Override
public boolean equals(Object o) {

    if (this == o)
        return true;

    if (!(o instanceof Employee))
        return false;

    Employee other = (Employee) o;

    return id == other.id;
}

@Override
public int hashCode() {

    return Objects.hash(id);
}
```

The implementation is concise, readable, and satisfies the required contract.

## Production Perspective

A team stores custom `Customer` objects as keys in a `HashMap`.

Everything works during testing.

After deployment, cache lookups begin failing intermittently.

Investigation reveals that the developer overrode `equals()` but forgot to override `hashCode()`.

The cache now stores logically identical customers in different buckets, resulting in missed lookups and duplicate entries.

The issue wasn't in the cache implementation—it was in the key class.

---

## Interview Focus

### Question

Why must `equals()` and `hashCode()` be overridden together?

### Strong Senior-Level Answer

> `HashMap` first uses `hashCode()` to locate the bucket and then uses `equals()` to identify the correct key within that bucket. If equal objects produce different hash codes, they may be stored in different buckets, making lookups fail. Overriding both methods ensures consistent behavior in hash-based collections.

---

## Common Mistakes

❌ Overriding `equals()` without overriding `hashCode()`.

❌ Assuming identical hash codes imply object equality.

❌ Using mutable fields in `hashCode()` calculations.

❌ Forgetting that changing a key after insertion can make it unreachable.

---

## Engineering Insight – Why Immutable Keys Are Preferred

Consider this example.

```java
Customer customer = new Customer(101);

map.put(customer, order);
```

Now imagine:

```java
customer.setId(102);
```

If `id` is used to compute the hash code, the object's bucket changes conceptually—but the `HashMap` has no idea the key has changed.

Subsequent lookups may fail because the object is stored in the bucket calculated from the old hash code.

This is why immutable objects, such as `String`, make excellent keys.

When using custom objects as keys, avoid modifying fields that participate in `equals()` or `hashCode()` after insertion.

---

## Engineering Takeaways

- `hashCode()` determines the bucket.
- `equals()` identifies the correct key within that bucket.
- Equal objects must have equal hash codes.
- Hash collisions are expected and handled correctly.
- Immutable keys provide predictable and reliable behavior.

---

## Quick Revision

✓ `HashMap` uses both `hashCode()` and `equals()`.

✓ `hashCode()` narrows the search to a bucket.

✓ `equals()` finds the exact key.

✓ Override both methods together.

✓ Prefer immutable objects as keys.
