## 4.6 Queue & Deque – FIFO, LIFO, and Practical Implementations

### Why This Matters

Not every application processes data in random order.

Many backend systems process work **sequentially**.

Examples include:

- Processing Kafka messages
- Scheduling background jobs
- Handling API requests
- Executing tasks in thread pools
- Breadth-First Search (BFS) algorithms

These scenarios naturally fit a **Queue**.

Sometimes, however, we need to insert or remove elements from **both ends** of the collection.

That's where a **Deque** becomes useful.

Understanding these interfaces helps you choose the right data structure for producer-consumer systems and scheduling problems.

---

# Queue

A **Queue** follows the **FIFO (First In, First Out)** principle.

The first element inserted is the first element removed.

```
Insert

A

↓

B

↓

C

↓

D

Remove

A → B → C → D
```

Think of people waiting in a billing queue.

The person who arrives first is served first.

---

## Common Queue Operations

```java
Queue<String> queue = new LinkedList<>();

queue.offer("A");
queue.offer("B");
queue.offer("C");
```

Retrieve and remove:

```java
queue.poll();
```

View the first element:

```java
queue.peek();
```

Common methods:

| Method | Purpose |
|---------|----------|
| `offer()` | Add an element |
| `poll()` | Remove and return the head |
| `peek()` | View the head without removing it |

---

## Queue Implementations

Some commonly used implementations are:

| Implementation | Typical Use Case |
|---------------|------------------|
| `LinkedList` | General-purpose queue |
| `ArrayDeque` | High-performance queue/deque |
| `PriorityQueue` | Priority-based processing |
| `ConcurrentLinkedQueue` | Lock-free concurrent queue |
| `LinkedBlockingQueue` | Producer-consumer systems |

Each implementation optimizes different characteristics such as ordering, concurrency, or blocking behavior.

---

# Deque

A **Deque (Double-Ended Queue)** allows insertion and removal from **both ends**.

```
Front

↓

A ←→ B ←→ C ←→ D

↑

Rear
```

Operations are supported at both the front and the back.

```java
Deque<String> deque = new ArrayDeque<>();

deque.addFirst("Java");

deque.addLast("Spring");

deque.removeFirst();

deque.removeLast();
```

---

## FIFO and LIFO with Deque

One of the biggest advantages of `Deque` is its flexibility.

### FIFO Queue

```
addLast()

↓

removeFirst()
```

---

### LIFO Stack

```
addFirst()

↓

removeFirst()
```

The same data structure can efficiently behave as both a queue and a stack.

This is why modern Java recommends `Deque` instead of the legacy `Stack` class.

---

## Why Prefer ArrayDeque?

Many developers use:

```java
Stack<Integer> stack = new Stack<>();
```

Although functional, `Stack` is considered a legacy class.

Instead, Java recommends:

```java
Deque<Integer> stack = new ArrayDeque<>();
```

Reasons include:

- Better performance
- No unnecessary synchronization
- Cleaner API
- More flexible operations

For single-threaded applications, `ArrayDeque` is usually the preferred implementation.

---

## What About PriorityQueue?

Unlike a normal queue, a `PriorityQueue` removes elements based on **priority**, not insertion order.

Example:

```java
PriorityQueue<Integer> queue = new PriorityQueue<>();

queue.offer(30);
queue.offer(10);
queue.offer(20);
```

Removal order:

```
10

20

30
```

Internally, `PriorityQueue` uses a **binary heap**, allowing efficient retrieval of the highest (or lowest) priority element.

Typical applications include:

- Task scheduling
- Job prioritization
- Dijkstra's shortest path algorithm
- Event simulation

---

## Production Perspective

Consider an email notification service.

Whenever a user performs an action, an email request is placed into a queue.

```
User Action

↓

Queue

↓

Worker Thread

↓

Send Email
```

The API responds immediately without waiting for the email to be sent.

Background workers continuously consume requests from the queue.

This design improves responsiveness and scalability.

Queues are a fundamental building block of asynchronous systems.

---

## Interview Focus

### Question

Why is `ArrayDeque` preferred over `Stack`?

### Strong Senior-Level Answer

> `Stack` is a legacy synchronized class that inherits from `Vector`, introducing unnecessary overhead for most applications. `ArrayDeque` provides better performance, cleaner APIs, and supports both stack (LIFO) and queue (FIFO) operations, making it the recommended choice for modern Java applications.

---

## Common Mistakes

❌ Using `Stack` in new applications.

❌ Assuming every queue processes elements strictly in insertion order (`PriorityQueue` does not).

❌ Using `LinkedList` as a stack when `ArrayDeque` is more efficient.

❌ Confusing `peek()` with `poll()`.

## Engineering Insight

Many enterprise systems rely heavily on queues.

Examples include:

- Kafka consumers
- RabbitMQ workers
- JMS listeners
- ExecutorService task queues
- ThreadPoolExecutor work queues

Although these frameworks provide additional features such as persistence and distribution, the underlying concept remains the same:

**Tasks are produced, queued, and consumed.**

Understanding Java's Queue interface provides the foundation for understanding these larger messaging systems.

---

## Engineering Takeaways

- Queue follows FIFO ordering.
- Deque supports operations at both ends.
- `ArrayDeque` is the preferred implementation for stacks and general-purpose queues.
- `PriorityQueue` processes elements based on priority.
- Queues are fundamental to asynchronous and concurrent applications.
