# 2.2 Variables, References and Object Identity

## Why This Matters

One of the biggest differences between an average Java developer and an experienced one is their mental model of objects.

Junior developers often think in terms of variables.

Senior engineers think in terms of **references**.

This distinction explains a surprising number of Java behaviors, including:

- Why changing an object inside a method affects the caller.
- Why `==` sometimes returns `false` even when two objects contain identical data.
- Why immutable objects are safer in concurrent applications.
- Why memory leaks occur.
- Why `HashMap` keys should be immutable.

Many production bugs originate from misunderstanding object identity rather than business logic.

---

## Real Interview

An interviewer writes the following code.

```java
Trade trade1 = new Trade("T1001");
Trade trade2 = trade1;

trade2.setStatus("COMPLETED");

System.out.println(trade1.getStatus());
```

The interviewer asks:

> "What is printed?"

Most candidates answer:

```
COMPLETED
```

The interviewer follows up.

> "Why?"

Many candidates respond:

> "Because both variables point to the same object."

The interviewer asks one final question.

> "Then what exactly is stored inside `trade1`?"

That question often separates candidates who understand Java's memory model from those who rely on intuition.

---

## Variables Do Not Store Objects

This is one of the most important concepts in Java.

Consider the following code.

```java
Trade trade = new Trade("T1001");
```

Many developers visualize this incorrectly.

```
trade
┌──────────────┐
│ Trade Object │
└──────────────┘
```

This is **not** how Java works.

The variable does **not** contain the object.

Instead, it stores a reference to the object.

```
Stack Frame

trade
  │
  │ Reference
  ▼

Heap

+---------------------------+
| Trade                     |
|---------------------------|
| tradeId = "T1001"         |
| status = "NEW"            |
+---------------------------+
```

The object exists independently of the variable.

The variable merely tells the JVM where the object can be found.

> **Key Insight**
>
> Variables never contain objects.
> They contain values or references.

---

## Multiple References Can Point to the Same Object

Consider the following example.

```java
Trade trade1 = new Trade("T1001");

Trade trade2 = trade1;
```

The memory now looks like this.

```
trade1 ─────┐
            │
            ▼
      +--------------------+
      | Trade Object       |
      |--------------------|
      | tradeId = T1001    |
      | status = NEW       |
      +--------------------+
            ▲
            │
trade2 ─────┘
```

Both variables reference exactly the same object.

There is only one `Trade` instance.

Changing the object through either reference affects the same underlying data.

```java
trade2.setStatus("COMPLETED");
```

Now,

```java
System.out.println(trade1.getStatus());
```

prints

```
COMPLETED
```

because both references point to the same object.

---

## Reassigning a Reference

Now consider a slightly different example.

```java
Trade trade1 = new Trade("T1001");

Trade trade2 = trade1;

trade2 = new Trade("T2001");
```

The situation has changed completely.

```
trade1
   │
   ▼
+----------------+
| Trade T1001    |
+----------------+

trade2
   │
   ▼
+----------------+
| Trade T2001    |
+----------------+
```

The original object still exists.

Only the reference stored in `trade2` changed.

This distinction becomes extremely important when discussing parameter passing later in the chapter.

## Object Identity

Every object created using `new` has its own identity.

Even if two objects contain identical data, they are still different objects.

```java
Trade trade1 = new Trade("T1001");

Trade trade2 = new Trade("T1001");
```

Visually,

```
trade1
   │
   ▼
+------------------+
| Trade T1001      |
+------------------+

trade2
   │
   ▼
+------------------+
| Trade T1001      |
+------------------+
```

The contents are identical.

The objects are different.

Java treats them as separate entities unless equality is explicitly defined.

This concept becomes the foundation for understanding `==` and `equals()` in the next section.

---

## Object Lifetime

Creating an object does not guarantee that it will exist forever.

Consider the following method.

```java
public void process() {

    Trade trade = new Trade("T1001");

}
```

After the method completes, the local variable disappears.

If no other references exist, the object becomes eligible for Garbage Collection.

Notice the wording.

The object is **eligible** for collection.

It is **not immediately removed**.

The JVM decides when memory should actually be reclaimed.

---

## Production Perspective

An inventory service begins consuming more memory every day.

Heap dumps reveal millions of `Order` objects.

After investigation, engineers discover a static cache that continually stores object references but never removes them.

The problem isn't object creation.

The problem is that references continue to exist.

As long as at least one live reference points to an object, the Garbage Collector cannot reclaim its memory.

This is one of the most common causes of memory leaks in Java applications.

> **Production Perspective**
>
> Memory leaks in Java rarely occur because objects cannot be deleted.
> They occur because applications unintentionally continue holding references to objects that are no longer needed.

## Interview Focus

**Question**

What is the difference between an object and a reference?

**Strong Senior-Level Answer**

> An object is an instance created in memory. A reference is a value that points to the location of that object. Multiple references may point to the same object, and changing the reference is different from changing the object itself.

**Common Follow-up Questions**

- Can two references point to the same object?
- When does an object become eligible for Garbage Collection?
- Does assigning one object variable to another create a copy?
- Where are references stored?
- Where are objects stored?

These topics form the foundation for understanding memory management and Garbage Collection later in the book.

---

## Senior Engineering Insight

Many engineers attempt to debug Java applications by following variables.

Experienced engineers debug by following object references.

That small shift in thinking makes complex object graphs, memory leaks, caching issues, and concurrency bugs much easier to understand.

---

## Engineering Takeaways

- Variables do not store objects; they store references.
- Multiple references can point to the same object.
- Reassigning a reference does not modify the original object.
- Every object has its own identity, even if its contents are identical.
- An object becomes eligible for Garbage Collection only when no live references remain.

---

## Quick Revision

✓ Objects and references are different concepts.

✓ Variables store references, not objects.

✓ Multiple variables can reference the same object.

✓ Reassigning a reference does not affect other references.

✓ Garbage Collection depends on object reachability, not object age.
