# 2.4 The `equals()` and `hashCode()` Contract

## Why This Matters

Almost every Java developer has overridden `equals()`.

Far fewer have implemented it correctly.

An incorrect implementation may not fail immediately.

Instead, it introduces subtle bugs that appear only under specific conditions—duplicate records in a `HashSet`, failed lookups in a `HashMap`, inconsistent cache behaviour, or unexpected results when working with ORM frameworks such as Hibernate.

For this reason, senior interviewers rarely stop after asking about `equals()`.

The next question is almost always:

> **"If you override `equals()`, should you also override `hashCode()`? Why?"**

Understanding the relationship between these two methods is essential for writing correct Java applications.

---

## Real Interview

An interviewer presents the following code.

```java
Set<Customer> customers = new HashSet<>();

customers.add(new Customer("C101"));

System.out.println(
    customers.contains(new Customer("C101"))
);
```

The candidate confidently answers:

```
true
```

The interviewer runs the program.

```
false
```

The candidate is surprised.

The interviewer smiles and asks:

> "Can you explain why?"

That single question often leads to discussions about hashing, object equality, collection internals, and immutable objects.

---

## Why Does Java Need `hashCode()`?

Imagine searching for a customer in a database containing ten million records.

One approach would be to compare the customer against every record.

```
Customer 1
Customer 2
Customer 3
...
Customer 10,000,000
```

This would be slow.

Instead, we create an index.

A similar idea is used by hash-based collections.

Rather than searching every object, Java calculates a numeric value called a **hash code**.

```
Customer
    │
    ▼
hashCode()
    │
    ▼
Hash Value
    │
    ▼
Bucket
```

Objects with similar hash codes are grouped into the same bucket.

Searching then becomes significantly faster.

Without hashing, collections such as `HashMap` and `HashSet` would lose much of their performance advantage.

---

## The Equality Contract

Java defines a simple but critical rule.

### Rule 1

If two objects are equal according to `equals()`,

their `hashCode()` values **must** also be equal.

```
a.equals(b) == true

↓

a.hashCode() == b.hashCode()
```

This rule is mandatory.

Violating it breaks hash-based collections.

---

### Rule 2

If two objects have the same hash code,

they are **not necessarily equal**.

```
hashCode()

      105
     /   \
Object A Object B

equals() ?

Maybe.
Maybe not.
```

Different objects can legitimately produce the same hash code.

This situation is called a **hash collision**.

HashMap handles collisions internally.

We'll study that implementation in Chapter 4.

## A Broken Implementation

Consider the following class.

```java
public class Customer {

    private String customerId;

    @Override
    public boolean equals(Object obj) {
        // compares customerId
    }

}
```

The class overrides `equals()`.

It does **not** override `hashCode()`.

Now consider:

```java
Set<Customer> customers = new HashSet<>();

customers.add(new Customer("C101"));

customers.contains(new Customer("C101"));
```

Even though both objects are logically equal,

the lookup may fail.

Why?

Because `HashSet` first calculates the hash code.

If the hash codes differ,

the objects are placed into different buckets.

`equals()` is never even invoked.

---

## The Correct Implementation

Whenever `equals()` is overridden,

`hashCode()` should also be overridden.

Modern Java makes this simple.

```java
@Override
public int hashCode() {
    return Objects.hash(customerId);
}
```

Now,

```java
Customer c1 = new Customer("C101");

Customer c2 = new Customer("C101");
```

Both objects

- produce identical hash codes
- land in the same bucket
- are compared using `equals()`

The collection behaves correctly.

---

## Visualizing HashMap Lookup

```
Customer("C101")
        │
        ▼
   hashCode()
        │
        ▼
    Bucket #17
        │
        ▼
equals()
        │
        ▼
Object Found
```

Notice the sequence.

`HashMap` does **not** call `equals()` first.

It first identifies the correct bucket using `hashCode()`.

Only then does it compare objects using `equals()`.

This design is what gives hash-based collections their excellent performance.

## Records and Equality

Java Records automatically generate both methods.

```java
public record Customer(
    String customerId,
    String name
) {}
```

The compiler generates:

- `equals()`
- `hashCode()`
- `toString()`

This eliminates one of the most common sources of bugs in immutable data objects.

For DTOs, event payloads, and value objects, records are often an excellent choice.

## Lombok and Equality

Many enterprise projects use Lombok.

```java
@Data
public class Customer {

    private String customerId;

}
```

Lombok automatically generates both methods.

While convenient, engineers should still understand what code is being generated.

During interviews, explaining *why* these methods exist is more important than remembering that Lombok creates them.

## JPA Entity Pitfall

One of the most common mistakes in enterprise applications involves JPA entities.

Consider this simplified entity.

```java
@Entity
public class Customer {

    @Id
    private Long id;

    private String name;

}
```

Should `equals()` compare:

- Database ID?
- Business key?
- Every field?

There is no universal answer.

Using generated IDs before persistence can produce inconsistent behaviour because the identifier may not yet exist.

Many production bugs involving Hibernate originate from incorrect equality implementations.

For this reason, equality strategies should always align with the domain model rather than being copied blindly from IDE-generated templates.

We'll revisit entity equality in the Persistence volume.

## Production Perspective

A distributed cache stores customer objects as keys.

During production, cache hit rates unexpectedly drop.

Investigation reveals that a recent code change modified a field used in `hashCode()` after the object had already been inserted into the map.

The object still exists.

Its bucket does not.

Subsequent lookups search a different bucket and fail.

The cache appears to "lose" objects even though they remain in memory.

The bug isn't in the cache.

It's in the mutable key.

> **Production Perspective**
>
> Objects used as keys in hash-based collections should be immutable whenever possible.

## Interview Focus

**Question**

Why must `equals()` and `hashCode()` be overridden together?

**Strong Senior-Level Answer**

> Hash-based collections use `hashCode()` to locate the correct bucket and `equals()` to identify the matching object within that bucket. If two logically equal objects produce different hash codes, they will be stored in different buckets and collections such as `HashMap` and `HashSet` will behave incorrectly. Therefore, whenever `equals()` is overridden, `hashCode()` must also be overridden to maintain the equality contract.

**Common Follow-up Questions**

- Can two unequal objects have the same hash code?
- What is a hash collision?
- Which method does `HashMap` invoke first?
- Why should hash map keys be immutable?
- How do Java Records implement equality?

---

## Engineering Takeaways

- `equals()` defines logical equality.
- `hashCode()` determines where an object is stored in hash-based collections.
- Equal objects must produce equal hash codes.
- Hash collisions are expected and handled internally.
- Records automatically generate correct implementations.
- Mutable objects make poor keys for `HashMap` and `HashSet`.

---

✓ Prefer immutable objects as keys in hash-based collections.
