# 2.10 Modern Java Language Features — Records, Sealed Classes, and Pattern Matching

## Why This Matters

Java has evolved significantly over the past few releases.

While enterprise applications continue to rely on familiar object-oriented principles, modern Java introduces language features that reduce boilerplate, improve readability, and make code safer.

Senior engineers are expected to understand these features—not because every project uses them today, but because they influence how new Java applications are designed.

---

## Real Interview

An interviewer asks:

> **"If you were starting a new Spring Boot project on Java 21, what language features would you use that weren't available in Java 8?"**

The interviewer isn't testing syntax.

They're evaluating whether you keep up with the evolution of the language.

---

## Records

Many classes exist only to hold data.

Traditionally, writing such a class required:

- Fields
- Constructor
- Getters
- `equals()`
- `hashCode()`
- `toString()`

Records eliminate this boilerplate.

```java
public record Trade(
    String tradeId,
    BigDecimal amount
) {}
```

The compiler automatically generates the common methods.

Records are ideal for:

- DTOs
- API responses
- Event payloads
- Immutable value objects

They are **not** suitable when objects require mutable state or complex inheritance.

---

## Sealed Classes

Sometimes you want to restrict who can extend a class.

Java 17 introduced **sealed classes**.

```java
public sealed class Payment
    permits CardPayment, UpiPayment {
}
```

Only the listed classes may extend `Payment`.

This improves type safety and makes APIs easier to reason about.

Sealed classes are particularly useful for modeling finite business domains.

---

## Pattern Matching

Traditional type checking often looks like this:

```java
if (obj instanceof Trade) {

    Trade trade = (Trade) obj;

}
```

Modern Java simplifies this.

```java
if (obj instanceof Trade trade) {

    // use trade directly

}
```

The explicit cast disappears, making the code cleaner and less error-prone.

Pattern matching continues to evolve and is increasingly used with `switch` expressions.

---

## Switch Expressions

Modern `switch` statements can return values directly.

```java
String status = switch (code) {
    case 200 -> "SUCCESS";
    case 404 -> "NOT_FOUND";
    default -> "ERROR";
};
```

Compared to the traditional syntax, switch expressions are:

- More concise
- Less error-prone
- Free from accidental fall-through

---

## Should You Use These Features?

For new applications targeting Java 17 or Java 21, the answer is generally **yes**.

However, many enterprise systems still run on older Java versions.

A senior engineer should be comfortable reading and writing both modern and traditional Java.

Writing code that matches the team's standards is usually more important than using the newest language feature.

---

## Production Perspective

A team upgrades from Java 8 to Java 21.

Instead of rewriting the application, they gradually adopt modern features:

- Records replace DTOs.
- Switch expressions simplify decision logic.
- Pattern matching removes repetitive casts.

The result isn't just fewer lines of code.

The codebase becomes easier to read, review, and maintain.

Incremental adoption often delivers more value than large-scale rewrites.

---

## Interview Focus

**Question**

What modern Java language features do you use most frequently?

**Strong Senior-Level Answer**

> In Java 17 and above, I commonly use records for immutable DTOs, switch expressions for cleaner branching, and pattern matching for safer type checks. I'm familiar with sealed classes, although their adoption depends on the application's domain and Java version.

---

## Common Mistakes

❌ Using records for mutable domain entities.

❌ Assuming sealed classes replace interfaces.

❌ Introducing modern features into projects that haven't upgraded their Java version.

❌ Refactoring purely to use new syntax without improving readability.

---

## Senior Engineering Insight

Modern language features should solve problems, not create them.

Good engineers don't adopt new syntax because it's fashionable.

They adopt it because it makes the code simpler, safer, and easier for the team to maintain.

---

## Engineering Takeaways

- Records reduce boilerplate for immutable data carriers.
- Sealed classes restrict inheritance intentionally.
- Pattern matching simplifies type checking.
- Switch expressions improve readability and safety.
- Modern language features should be adopted thoughtfully.

---
