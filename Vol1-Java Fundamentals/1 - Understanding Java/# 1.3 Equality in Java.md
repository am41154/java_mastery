# 2.3 Equality in Java — `==` vs `equals()`

## Why This Matters

If there is one Java interview question that has remained relevant for over two decades, it is this:

> **What is the difference between `==` and `equals()`?**

At first glance, the answer seems straightforward.

Most developers respond:

- `==` compares references.
- `equals()` compares objects.

Although this is directionally correct, it is incomplete.

A senior engineer is expected to understand **what is actually being compared, how equality is defined, why wrapper classes sometimes behave unexpectedly, and how incorrect equality implementations can break collections such as `HashMap` and `HashSet`.**

Understanding equality is not just an interview topic.

It directly affects caching, persistence, collections, distributed systems, and application correctness.

## What Does `==` Compare?

The meaning of `==` depends on the type of operands.

### Primitive Types

For primitive values, `==` compares the actual value.

```java
int x = 10;
int y = 10;

System.out.println(x == y);
```

Output:

```
true
```

Both variables directly contain the value `10`.

There are no objects involved.

---

### Reference Types

For objects, `==` compares **references**, not contents.

```java
Trade trade1 = new Trade("T1001");
Trade trade2 = new Trade("T1001");

System.out.println(trade1 == trade2);
```

Output:

```
false
```

Although both objects contain identical data, they are different instances in memory.

Their references are different.

```
trade1 ─────► Trade Object A

trade2 ─────► Trade Object B
```

Reference equality answers one question:

> **Are these two variables pointing to the exact same object?**

Nothing more.

---

## What Does `equals()` Compare?

Unlike `==`, the purpose of `equals()` is to compare **logical equality**.

The default implementation inherited from `Object` behaves exactly like `==`.

```java
public boolean equals(Object obj) {
    return (this == obj);
}
```

That means unless a class overrides `equals()`, calling it provides no additional benefit.

Business objects usually redefine equality based on meaningful attributes.

```java
Trade trade1 = new Trade("T1001");
Trade trade2 = new Trade("T1001");

System.out.println(trade1.equals(trade2));
```

If `Trade` overrides `equals()` correctly, the output becomes:

```
true
```

The objects are different.

The business meaning is the same.

---

## Defining Logical Equality

Suppose two trade objects represent the same transaction.

```java
Trade trade1 = new Trade("TR-1001");

Trade trade2 = new Trade("TR-1001");
```

Although they occupy different memory locations, they represent the same business entity.

A sensible implementation might compare only the trade identifier.

```java
@Override
public boolean equals(Object obj) {

    if (this == obj)
        return true;

    if (!(obj instanceof Trade))
        return false;

    Trade other = (Trade) obj;

    return Objects.equals(this.tradeId, other.tradeId);
}
```

This implementation changes the definition of equality from:

> Same object

to

> Same business identity

That distinction is fundamental in enterprise applications.

## The Integer Cache

Now let's revisit the interview question.

```java
Integer a = 127;
Integer b = 127;
```

Why does

```java
a == b
```

return `true`?

Java maintains an internal cache of commonly used `Integer` objects.

By default, values from **-128 to 127** are reused.

```
Integer Cache

-128 ... -2 -1 0 1 2 ... 126 127
              ▲
              │
       Shared Objects
```

Both variables reference the same cached object.

```
a ─────┐
       ▼
     Integer(127)
       ▲
b ─────┘
```

Now consider:

```java
Integer a = 128;
Integer b = 128;
```

The value `128` is not cached by default.

Two separate objects are created.

```
a ─────► Integer(128)

b ─────► Integer(128)
```

Now,

```java
a == b
```

returns

```
false
```

> **Key Insight**
>
> Never use `==` to compare wrapper classes.
>
> The result may depend on Integer caching rather than business logic.

## Why Strings Often Confuse Developers

Strings introduce another optimization.

```java
String s1 = "Java";
String s2 = "Java";
```

Both variables reference the same object from the **String Pool**.

```
s1 ─────┐
         ▼
     "Java"
         ▲
s2 ─────┘
```

Therefore,

```java
s1 == s2
```

returns

```
true
```

Now consider:

```java
String s3 = new String("Java");
```

A new object is created.

```
s1 ─────► String Pool

s3 ─────► Heap Object
```

Now,

```java
s1 == s3
```

returns

```
false
```

while

```java
s1.equals(s3)
```

returns

```
true
```

We'll study the String Pool in detail in Chapter 3.

## Production Perspective

A payment service stores processed transaction IDs inside a `HashSet`.

During production, duplicate transactions unexpectedly appear.

The root cause is simple.

The `Transaction` class overrides `equals()` but forgets to override `hashCode()`.

As a result, logically identical objects are placed into different hash buckets.

The collection behaves incorrectly.

The issue isn't with `HashSet`.

It's with the equality contract.

This is one of the most common production bugs involving Java collections.

## Interview Focus

**Question**

What is the difference between `==` and `equals()`?

**Strong Senior-Level Answer**

> For primitive types, `==` compares values. For reference types, it compares object references. The `equals()` method compares logical equality as defined by the class. By default, `Object.equals()` behaves like `==`, but business classes typically override it to compare meaningful attributes.

**Common Follow-up Questions**

- Why does `Integer.valueOf(127)` behave differently from `Integer.valueOf(128)`?
- Why should wrapper classes not be compared using `==`?
- Why must `equals()` and `hashCode()` be overridden together?
- Why does `"Java" == "Java"` return `true`?
- What is the String Pool?

---

## Senior Engineering Insight

In enterprise applications, equality is rarely about memory.

It is about identity.

Two `Customer` objects loaded from different services should still represent the same customer if they share the same business identifier.

Thinking in terms of **business identity** rather than **object identity** leads to cleaner domain models and fewer subtle bugs.

---

## Engineering Takeaways

- `==` compares values for primitives and references for objects.
- `equals()` compares logical equality.
- The default implementation of `equals()` behaves like `==`.
- Wrapper classes should not be compared using `==`.
- Integer caching explains why `127` and `128` behave differently.
- Correct equality is essential for collections and domain models.

---

## Quick Revision

✓ Primitive `==` compares values.

✓ Object `==` compares references.

✓ `equals()` compares logical equality.

✓ Integer Cache ranges from **-128 to 127** by default.

✓ Never use `==` for wrapper classes or business objects.

✓ A correct `equals()` implementation is incomplete without a corresponding `hashCode()` implementation.
