# 2.5 Object Creation — What Really Happens When You Use `new`

## Why This Matters

Creating an object is one of the most common operations in Java.

```java
Trade trade = new Trade();
```

Every Java developer writes statements like this countless times.

Yet very few can explain what actually happens after pressing **Run**.

Does Java simply allocate memory?

Who initializes the fields?

When is the constructor executed?

Where is the object stored?

How does the variable receive the reference?

Understanding object creation provides the foundation for several advanced topics, including:

- JVM memory management
- Garbage Collection
- Dependency Injection
- Reflection
- Object pooling
- Performance optimization

Senior interviewers often use object creation as a gateway into deeper JVM discussions.

---

## Real Interview

An interviewer asks:

> **"What happens internally when Java executes `new Trade()`?"**

The candidate replies:

> "It creates an object on the heap."

The interviewer nods.

Then asks:

> "What happens before the constructor executes?"

Silence.

The candidate knew *where* the object was created.

The interviewer wanted to know *how* it was created.

That distinction often separates senior engineers from experienced developers.

---

## Breaking Down the Statement

Consider the following code.

```java
Trade trade = new Trade();
```

Although it appears to be a single operation, several steps occur behind the scenes.

```
Trade trade = new Trade();

       │
       ▼

1. JVM checks whether the class is loaded.

2. Memory is allocated for the object.

3. Instance variables receive default values.

4. Constructor executes.

5. Object reference is returned.

6. Reference is assigned to the variable.
```

All of this happens in a fraction of a millisecond.

---

## Step 1 — Class Loading

Before Java can create an object, the JVM must understand what that object looks like.

If the `Trade` class has not already been loaded,

the JVM first performs:

- Class loading
- Verification
- Linking
- Initialization

Only then can object creation begin.

This is why class loading always precedes object creation.

---

## Step 2 — Memory Allocation

Once the class is available,

the JVM allocates memory for the new object.

Conceptually,

```
Heap

+----------------------+
|                      |
|   Trade Object       |
|                      |
+----------------------+
```

The exact allocation strategy depends on the JVM implementation,

but the goal remains the same:

Reserve sufficient memory to store every instance field.

---

## Step 3 — Default Initialization

Before your constructor executes,

every instance field receives its default value.

For example,

```java
public class Trade {

    private int quantity;

    private boolean active;

    private String tradeId;

}
```

Immediately after allocation,

the object looks like this.

```
Trade

quantity = 0

active = false

tradeId = null
```

This guarantees that every object begins life in a valid state.

Only after default initialization does Java execute field initializers and constructors.

---

## Step 4 — Field Initialization

If fields contain explicit initial values,

they are evaluated before the constructor body.

```java
private String status = "NEW";
```

The object now becomes

```
Trade

quantity = 0

active = false

status = "NEW"
```

Only after these assignments does the constructor begin.

---

## Step 5 — Constructor Execution

Now the constructor runs.

```java
public Trade() {

    this.tradeId = UUID.randomUUID().toString();

}
```

Constructors initialize object state.

They do **not** allocate memory.

Memory allocation has already occurred.

This is a subtle but important distinction.

---

## Step 6 — Reference Assignment

Finally,

the JVM returns the reference to the newly created object.

```
trade
   │
   ▼

+----------------------+
| Trade                |
|----------------------|
| tradeId = ...        |
| status = NEW         |
+----------------------+
```

The variable now points to the fully initialized object.

The statement is complete.

---

## Constructor Chaining

Consider the following class.

```java
public class Trade {

    public Trade() {
        this("NEW");
    }

    public Trade(String status) {
        this.status = status;
    }

}
```

The first constructor delegates to another constructor using `this()`.

Java allows constructor chaining,

but it enforces one important rule:

`this()` or `super()` must be the **first statement** in a constructor.

This ensures that object initialization follows a predictable order.

---

## Object Creation Is Expensive

Creating an object is relatively inexpensive,

but it is **not free**.

Every object requires:

- Memory allocation
- Constructor execution
- Future Garbage Collection

Now imagine the following loop.

```java
for (int i = 0; i < 1_000_000; i++) {

    Trade trade = new Trade();

}
```

One million objects are created.

Most become unreachable almost immediately.

The JVM must later identify and reclaim those objects.

This is one reason why unnecessary object creation can affect application performance.

---

## Production Perspective

A high-frequency trading application begins experiencing increased Garbage Collection pauses.

CPU utilization remains low.

Database latency is normal.

Profiling reveals that a utility method creates several temporary objects for every incoming request.

Each object survives only a few milliseconds.

Individually,

the allocations are insignificant.

Collectively,

they generate millions of unnecessary objects every minute.

After reducing object creation,

Garbage Collection frequency decreases,

and application latency improves.

The business logic never changed.

The object lifecycle did.

## Interview Focus

**Question**

What happens internally when Java executes `new`?

**Strong Senior-Level Answer**

> Before creating an object, the JVM ensures that the class is loaded and initialized. It then allocates memory on the heap, initializes instance fields with default values, executes field initializers, invokes the constructor, and finally returns a reference to the newly created object.

**Common Follow-up Questions**

- Does the constructor allocate memory?
- When are default values assigned?
- What executes first: field initializers or the constructor?
- Why must `this()` be the first statement in a constructor?
- Is object creation expensive?

---

## Senior Engineering Insight

Most enterprise applications don't fail because objects are created.

They fail because **too many unnecessary objects are created**.

Modern JVMs are exceptionally good at allocating objects.

However,

creating fewer temporary objects generally leads to:

- Lower memory usage
- Less Garbage Collection
- Better throughput
- More predictable latency

Understanding object creation helps engineers write software that scales gracefully under load.

---

## Engineering Takeaways

- Object creation involves multiple JVM steps.
- Class loading always occurs before object creation.
- Default field initialization happens before constructors execute.
- Constructors initialize objects; they do not allocate memory.
- Every object contributes to future Garbage Collection work.
- Reducing unnecessary object creation can improve performance in high-throughput systems.
