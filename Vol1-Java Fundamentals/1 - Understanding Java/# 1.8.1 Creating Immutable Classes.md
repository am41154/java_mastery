# 2.8.1 Creating Immutable Classes

## Why Immutability Matters

Immutability is one of the simplest ways to build reliable software.

An immutable object cannot change after it has been created.

This offers several advantages:

- Thread-safe by design
- Easier to reason about
- Safe to cache and share
- Suitable as keys in `HashMap` and `HashSet`
- Eliminates accidental state changes

For these reasons, immutable objects are widely used for:

- DTOs
- Configuration objects
- Value objects
- Money, Currency, Date Range, and Address models

---

## Characteristics of an Immutable Class

A well-designed immutable class follows these principles:

1. Make the class `final`.
2. Make all fields `private` and `final`.
3. Initialize all fields through the constructor.
4. Do not provide setter methods.
5. Return defensive copies for mutable objects.

Following these rules ensures that the object's state cannot change after construction.

---

## Example

```java
public final class Employee {

    private final Long id;
    private final String name;

    public Employee(Long id, String name) {
        this.id = id;
        this.name = name;
    }

    public Long getId() {
        return id;
    }

    public String getName() {
        return name;
    }
}
```

Usage:

```java
Employee employee = new Employee(101L, "John");

// No setter methods available
// Employee state can never change
```

Once created, the `Employee` object remains unchanged throughout its lifetime.

---

## Defensive Copying

Immutability becomes more challenging when a class contains mutable fields.

Consider:

```java
public final class Employee {

    private final Date joiningDate;

    public Employee(Date joiningDate) {
        this.joiningDate = joiningDate;
    }

    public Date getJoiningDate() {
        return joiningDate;
    }
}
```

Although the field is `final`, the object it references is mutable.

A caller can modify it:

```java
Date date = employee.getJoiningDate();

date.setTime(0);
```

The internal state of `Employee` has now changed.

To prevent this, create **defensive copies**.

```java
public Employee(Date joiningDate) {
    this.joiningDate = new Date(joiningDate.getTime());
}

public Date getJoiningDate() {
    return new Date(joiningDate.getTime());
}
```

The caller now receives a copy rather than the original object.

This preserves immutability.

> **Best Practice**
>
> Prefer immutable types such as `LocalDate`, `Instant`, and `Duration` from the Java Time API whenever possible, as they eliminate the need for defensive copying.

---

## Why Is `String` Immutable?

`String` is one of Java's most widely used immutable classes.

Its immutability provides several benefits:

- Safe sharing through the String Pool.
- Thread safety without synchronization.
- Reliable behavior as a `HashMap` key.
- Improved security (for example, class names and file paths cannot be modified after creation).

This design decision has had a significant positive impact on Java's performance and reliability.

---

## Production Perspective

A configuration object containing database URLs and API endpoints is shared across hundreds of threads.

Because the object is immutable, no thread can accidentally modify its values.

The object can be safely cached and reused without synchronization, reducing both complexity and runtime overhead.

---

## Interview Focus

**Question**

How do you design an immutable class in Java?

**Strong Senior-Level Answer**

> Make the class final, declare all fields as private and final, initialize them through the constructor, avoid setter methods, and defensively copy mutable objects in both the constructor and getters. If possible, prefer immutable field types such as `String` or `LocalDate`.

---

## Engineering Takeaways

- `final` references alone do not create immutable objects.
- Immutable classes cannot change state after construction.
- Mutable fields require defensive copying.
- Prefer immutable types from the Java Standard Library.
- Immutability simplifies concurrent programming and improves application reliability.

---
