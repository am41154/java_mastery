## 6.18 G1 Collection Cycle

Unlike older garbage collectors, **G1 (Garbage First)** does not stop the entire JVM for every major collection.

Instead, it divides garbage collection into multiple carefully planned phases, allowing the application to continue running for most of the collection cycle.

Think of G1 as cleaning a large city.

Instead of shutting down the entire city for cleaning, it:

- Cleans one neighborhood
- Determines which neighborhoods contain the most garbage
- Cleans those neighborhoods first
- Gradually cleans the entire city

This approach keeps GC pauses short and predictable.

---

## Complete G1 Collection Cycle

```text
Application Running
        │
        ▼
Young GC
        │
        ▼
Concurrent Marking Start
        │
        ▼
Concurrent Marking
        │
        ▼
Remark
        │
        ▼
Cleanup
        │
        ▼
Mixed Collections
        │
        ▼
Normal Young GCs Continue
```

---

## 6.18.1 Young Collection

Young Collection is the most frequent GC cycle.

It collects only the Young Generation regions:

```text
Eden
Survivor
```

Old regions remain untouched.

### Example

Before GC:

```text
Eden:
A B C D E

Survivor:
F G

Old:
H I J K
```

Suppose only the following objects are still alive:

```text
A
C
E
```

After GC:

```text
Eden:
(empty)

Survivor:
A C E

Old:
H I J K
```

Dead objects are discarded.

Typical pause time:

```text
5–50 ms
```

---

## 6.18.2 Evacuation

Instead of compacting objects within the same region, G1 copies live objects into new empty regions.

This process is known as **Evacuation**.

### Advantages

- Eliminates fragmentation
- Provides contiguous free space
- Makes future allocations faster
- Avoids expensive heap-wide compaction

---

## 6.18.3 Initial Mark

As the Old Generation grows, G1 periodically starts a concurrent marking cycle.

The first step is **Initial Mark**.

During this phase, G1 performs a short Stop-The-World pause to identify all **GC Roots**.

```text
GC Roots
```

Once the roots are identified, the application resumes almost immediately.

Typical pause:

```text
Very Short
```

---

## 6.18.4 Concurrent Marking

After Initial Mark, the application continues running while dedicated GC threads traverse the object graph.

Example:

```text
Root
 │
 ├── A
 │     ├── B
 │     └── C
 │
 └── D
```

Every object reachable from the GC Roots is marked as live.

Unlike traditional collectors, this phase executes concurrently with the application, making it almost invisible to users.

---

## 6.18.5 SATB (Snapshot At The Beginning)

While concurrent marking is running, the application continues modifying object references.

Example:

Initial state:

```text
Root

A → B
```

Later:

```text
Root

A → C
```

Without additional mechanisms, object **B** could mistakenly appear unreachable and be reclaimed.

To prevent this, G1 uses **Snapshot At The Beginning (SATB)**.

SATB treats the object graph as if a snapshot was taken when concurrent marking started.

```text
Snapshot

A → B
```

Even if references change later:

```text
A → C
```

Object **B** is still considered live until the current marking cycle completes.

This guarantees that no reachable object is accidentally reclaimed.

---

## 6.18.6 Remark

Concurrent marking may miss some final reference updates that occur while the application is running.

The **Remark** phase performs a short Stop-The-World pause to:

- Complete object marking
- Process SATB buffers
- Update Remembered Sets
- Finalize live object information

Typical pause:

```text
5–20 ms
```

---

## 6.18.7 Cleanup

Once marking completes, G1 knows exactly how much garbage exists in every region.

Example:

| Region | Live Data |
|---------|----------:|
| R1 | 95% |
| R2 | 80% |
| R3 | 10% |
| R4 | 15% |
| R5 | 90% |

The collector prioritizes regions with the least amount of live data.

Collection order:

```text
R3
R4
```

This strategy is the reason the collector is named **Garbage First**.

---

## 6.18.8 Mixed Collection

Unlike Young GC, **Mixed Collection** collects both:

```text
Young Regions

+

Selected Old Regions
```

Only Old regions containing a high percentage of garbage are selected.

Example:

```text
Young:
Eden1
Eden2
Survivor

Old:
Region7 (15% live)
Region12 (20% live)

Skipped:
Region3 (90% live)
```

Collecting only highly reclaimable regions minimizes pause time.

---

## 6.18.9 Multiple Mixed Collections

A single Mixed Collection rarely cleans the complete Old Generation.

Instead, G1 spreads the work across multiple GC cycles.

```text
Mixed GC #1
      │
      ▼
Mixed GC #2
      │
      ▼
Mixed GC #3
      │
      ▼
Mixed GC #4
```

This incremental approach keeps pauses short while gradually reclaiming old regions.

---

## 6.18.10 Region Selection Strategy

Consider the following heap:

| Region | Garbage |
|---------|---------:|
| R1 | 5% |
| R2 | 20% |
| R3 | 70% |
| R4 | 60% |
| R5 | 10% |
| R6 | 80% |

G1 prioritizes:

```text
R6
R3
R4
```

instead of:

```text
R1
R5
```

because reclaiming more garbage per pause maximizes efficiency.

---

## 6.18.11 Evacuation During Mixed Collection

Selected regions are evacuated.

Before evacuation:

```text
Old Region

A
B
Dead
Dead
C
Dead
```

After evacuation:

```text
New Region

A
B
C
```

The original region becomes:

```text
Completely Free
```

This naturally eliminates memory fragmentation.

---

## 6.18.12 Why Mixed Collections Are Better Than Full GC

Traditional collectors often compact the entire Old Generation in one long pause.

Example:

```text
Old Generation

16 GB
```

Entire heap is compacted.

Possible pause:

```text
Several Seconds
```

G1 instead cleans only selected regions.

Example:

```text
16 GB Heap

Collect:
300 MB
```

The result is significantly shorter and more predictable pauses.

---

## 6.18.13 Predictable Pause Goals

One of G1's primary design goals is predictable pause times.

A commonly used JVM option is:

```bash
-XX:MaxGCPauseMillis=200
```

This value represents a **target**, not a strict guarantee.

G1 estimates how much work it can safely complete within the configured pause time and dynamically adjusts the number of regions selected for collection.

---

## 6.18.14 Full GC (Fallback)

Although G1 is designed to avoid Full GC, it may still occur under exceptional circumstances.

Common causes include:

- Insufficient free regions available for evacuation
- Allocation rate exceeding reclamation rate
- Humongous object fragmentation
- Explicit invocation of `System.gc()` (unless disabled)

During Full GC:

- All application threads stop
- Entire heap is marked
- Entire heap is compacted

Depending on heap size, pause times may range from hundreds of milliseconds to several seconds.

A well-tuned G1 deployment should encounter Full GC only rarely.

---

## Interview Questions

### Q1. Why is G1 called Garbage First?

Because it prioritizes collecting the regions containing the highest percentage of garbage, reclaiming the maximum amount of memory with the minimum pause time.

---

### Q2. What is the difference between Young GC and Mixed GC?

| Young GC | Mixed GC |
|-----------|-----------|
| Collects only Eden and Survivor regions | Collects Young regions and selected Old regions |
| Runs frequently | Runs after concurrent marking |
| Very short pause | Slightly longer but still incremental |

---

### Q3. What is SATB?

SATB (Snapshot At The Beginning) records the object graph as it existed when concurrent marking began. Objects reachable in that snapshot remain live for the duration of the marking cycle, preventing accidental reclamation while references continue changing.

---

### Q4. Does G1 compact memory?

Yes.

G1 compacts memory by evacuating live objects into new regions during every collection cycle, eliminating fragmentation without requiring expensive heap-wide compaction.

---

## Key Takeaways

- G1 performs incremental garbage collection instead of monolithic Full GCs.
- Young GC collects only Eden and Survivor regions.
- Concurrent Marking identifies live objects while the application continues executing.
- SATB enables safe concurrent marking despite ongoing reference changes.
- Mixed Collections reclaim memory from both Young and selected Old regions.
- Evacuation naturally compacts memory and eliminates fragmentation.
- `-XX:MaxGCPauseMillis` defines a pause-time goal rather than a strict guarantee.
- Full GC is a fallback mechanism and should be rare in a properly tuned G1 configuration.
