# 6.23 JVM Memory Troubleshooting (OutOfMemoryError, Memory Leaks, Heap Dumps)

Knowing how the JVM manages memory is only half the story.

A senior backend engineer is also expected to **diagnose and troubleshoot memory-related issues** in production.

Some of the most common JVM memory problems include:

- `OutOfMemoryError`
- Java memory leaks
- High GC frequency
- Long GC pauses
- Excessive heap usage
- Native memory exhaustion

Understanding how to identify and resolve these issues is a critical production skill.

---

# What Is an OutOfMemoryError?

An **OutOfMemoryError (OOM)** occurs when the JVM cannot allocate memory for a new object.

Example:

```java
List<byte[]> list = new ArrayList<>();

while (true) {
    list.add(new byte[1024 * 1024]);
}
```

Eventually:

```text
Exception in thread "main"

java.lang.OutOfMemoryError: Java heap space
```

The JVM attempted to allocate memory but no sufficient space was available.

---

# Common Types of OutOfMemoryError

Several different OutOfMemoryError variants exist.

| Error | Cause |
|---------|------|
| Java heap space | Heap is full |
| GC overhead limit exceeded | GC spends almost all its time reclaiming very little memory |
| Metaspace | Class metadata area is exhausted |
| Direct buffer memory | Off-heap direct memory exhausted |
| Unable to create new native thread | OS cannot create additional threads |
| Requested array size exceeds VM limit | Attempt to allocate an excessively large array |

Each variant points to a different underlying problem.

---

# Java Heap Space

The most common error.

```text
java.lang.OutOfMemoryError:

Java heap space
```

Typical causes:

- Memory leak
- Heap too small
- Excessive object creation
- Large caches

Example:

```
Application

↓

Creates Millions of Objects

↓

Heap Full

↓

OOM
```

---

# GC Overhead Limit Exceeded

Example:

```text
java.lang.OutOfMemoryError:

GC overhead limit exceeded
```

Meaning:

- JVM spends almost all CPU time running GC.
- Very little memory is reclaimed.
- Application makes almost no progress.

Typical symptom:

```text
98% CPU

↓

GC

↓

Only 1% Memory Recovered
```

Eventually, JVM throws this error instead of continuously wasting CPU cycles.

---

# Metaspace OutOfMemoryError

Example:

```text
java.lang.OutOfMemoryError:

Metaspace
```

Typical causes:

- Excessive dynamic class generation
- ClassLoader leaks
- Proxy generation
- Bytecode generation libraries

Common in:

- Application servers
- Spring Boot DevTools reload loops
- ByteBuddy
- CGLIB
- Hibernate proxies

---

# Direct Buffer Memory

Example:

```text
java.lang.OutOfMemoryError:

Direct buffer memory
```

Occurs when off-heap memory allocated using:

```java
ByteBuffer.allocateDirect(...)
```

becomes exhausted.

This memory is outside the Java heap.

---

# Unable to Create New Native Thread

Example:

```text
java.lang.OutOfMemoryError:

Unable to create new native thread
```

Usually caused by:

- Creating too many threads
- OS thread limits
- Insufficient native memory

Example:

```java
while (true) {
    new Thread(() -> {}).start();
}
```

Eventually the operating system refuses additional threads.

---

# What Is a Memory Leak?

A memory leak occurs when objects are **no longer useful** but are **still reachable**, preventing the garbage collector from reclaiming them.

Example:

```
Application

↓

Object No Longer Needed

↓

Reference Still Exists

↓

GC Cannot Remove It
```

Unlike languages with manual memory management, Java memory leaks are caused by **unnecessary references**, not by forgetting to free memory.

---

# Example of a Memory Leak

```java
public class Cache {

    private static final List<Object> DATA = new ArrayList<>();

    public void add(Object object) {
        DATA.add(object);
    }
}
```

Every added object remains referenced forever.

The heap continuously grows until:

```text
OutOfMemoryError
```

---

# Common Causes of Memory Leaks

## Static Collections

```java
static Map<String, Object> cache;
```

Objects stored forever unless explicitly removed.

---

## Unbounded Caches

Example:

```java
cache.put(key, value);
```

without eviction.

Popular solutions:

- LRU Cache
- Caffeine
- Redis

---

## Listener Leaks

Example:

```java
button.addListener(listener);
```

If listeners are never removed, referenced objects remain alive.

---

## ThreadLocal Leaks

```java
ThreadLocal<User> userContext;
```

If `remove()` is never called in thread pools, values may survive much longer than expected.

---

## ClassLoader Leaks

Common in:

- Web servers
- Application servers
- Hot deployments

Old ClassLoaders remain reachable, preventing entire applications from being garbage collected.

---

# Symptoms of a Memory Leak

Typical observations:

```
Heap Usage

↓

Keeps Growing

↓

Never Returns To Baseline

↓

Eventually OOM
```

GC still runs successfully, but used memory gradually increases after each collection.

---

# Healthy Heap Pattern

```
Memory

↑
│        /\      /\      /\
│       /  \    /  \    /  \
│______/____\__/____\__/____\____
```

Heap usage rises and falls repeatedly.

GC successfully reclaims unused objects.

---

# Memory Leak Pattern

```
Memory

↑
│        /\ 
│       /  \
│      /    \__
│     /        \___
│____/_____________\____
```

Each GC frees less memory than before.

Eventually the heap is exhausted.

---

# Heap Dumps

A **Heap Dump** is a snapshot of all objects currently present in heap memory.

It includes:

- Every object
- Object sizes
- References
- Retained memory
- Class information

Heap dumps are the primary tool for investigating memory leaks.

---

# Generating a Heap Dump

Automatically on OOM:

```bash
-XX:+HeapDumpOnOutOfMemoryError
```

Specify output file:

```bash
-XX:HeapDumpPath=/tmp/heap.hprof
```

Example:

```bash
java \
-XX:+HeapDumpOnOutOfMemoryError \
-XX:HeapDumpPath=/tmp \
-jar app.jar
```

---

# Creating Heap Dumps Manually

Using `jcmd`:

```bash
jcmd <pid> GC.heap_dump heap.hprof
```

Using `jmap`:

```bash
jmap -dump:live,file=heap.hprof <pid>
```

These commands generate a `.hprof` file for analysis.

---

# Heap Dump Analysis Tools

Popular tools include:

| Tool | Purpose |
|------|----------|
| Eclipse Memory Analyzer (MAT) | Detailed heap analysis |
| VisualVM | Heap browsing and monitoring |
| Java Mission Control (JMC) | JVM diagnostics |
| IntelliJ IDEA Profiler | Heap inspection (Ultimate Edition) |

These tools help identify:

- Largest objects
- Dominator tree
- Retained heap
- Duplicate strings
- Leak suspects

---

# Dominator Tree

One of the most useful heap dump views is the **Dominator Tree**.

It answers:

> **Which objects keep the most memory alive?**

Example:

```
Application

↓

Cache

↓

100000 Objects
```

Removing the cache reference would immediately free all dependent objects.

---

# Retained Heap

Two important metrics appear in heap analysis.

| Metric | Meaning |
|---------|----------|
| Shallow Heap | Memory occupied by the object itself |
| Retained Heap | Total memory that would be freed if the object were removed |

Retained Heap is often much more valuable when investigating leaks.

---

# Production Investigation Workflow

A common troubleshooting process is:

```
High Memory Usage

↓

Enable GC Logs

↓

Observe Heap Growth

↓

Capture Heap Dump

↓

Analyze Heap Dump

↓

Find Dominator

↓

Fix Memory Leak

↓

Verify Improvement
```

This systematic approach is far more effective than guessing.

---

# Best Practices

- Enable heap dumps on OutOfMemoryError.
- Monitor heap usage continuously.
- Use bounded caches with eviction policies.
- Remove unused listeners.
- Clear `ThreadLocal` values in thread pools.
- Review GC logs before increasing heap size.
- Analyze heap dumps instead of relying on assumptions.

---

# Interview Questions

### Q1. What is the difference between an OutOfMemoryError and a memory leak?

An OutOfMemoryError is the JVM failing to allocate memory. A memory leak is one possible cause, where unnecessary references prevent garbage collection until memory is exhausted.

---

### Q2. What causes `GC overhead limit exceeded`?

The JVM spends nearly all of its time performing garbage collection while reclaiming very little memory, indicating severe memory pressure.

---

### Q3. What is a heap dump?

A heap dump is a snapshot of all objects currently stored in the Java heap, including their references and memory usage.

---

### Q4. Which tools are commonly used to analyze heap dumps?

Common tools include:

- Eclipse Memory Analyzer (MAT)
- VisualVM
- Java Mission Control (JMC)

---

### Q5. What is Retained Heap?

Retained Heap is the total amount of memory that would become reclaimable if a particular object and everything reachable only through it were garbage collected.

---

# Key Takeaways

- OutOfMemoryError has multiple variants, each indicating a different memory problem.
- Java memory leaks are caused by unnecessary object references rather than manual memory management errors.
- Heap dumps are the primary artifact for investigating memory leaks.
- Dominator Tree and Retained Heap are essential concepts in heap dump analysis.
- Effective memory troubleshooting combines GC logs, heap dumps, and systematic analysis rather than simply increasing heap size.
