# 6.13 Garbage Collection Algorithms Overview – Throughput, Latency, and Pause Times

### Why This Matters

One of the most common misconceptions among Java developers is:

> **"Garbage Collection is a single algorithm."**

It is not.

The HotSpot JVM provides multiple garbage collectors,

each designed with different goals.

Some prioritize:

- Maximum throughput

Others prioritize:

- Low pause times

Others focus on:

- Extremely large heaps
- Cloud-native applications
- Predictable latency

Choosing the wrong collector can significantly impact application performance.

Before studying each collector individually,

it's important to understand **what problems they are trying to solve.**

---

# Why Do We Need Multiple Garbage Collectors?

Imagine two applications.

Application A:

```
Nightly Batch Job
```

Requirements:

- Finish as fast as possible
- High CPU utilization is acceptable
- Several seconds of pause time is acceptable

---

Application B:

```
Online Trading Platform
```

Requirements:

- Millisecond response time
- Very short GC pauses
- Continuous request processing

Clearly,

both applications require different garbage collection strategies.

There is no single collector that is optimal for every workload.

---

# The Three Primary Goals

Every garbage collector attempts to balance three competing objectives.

```
Garbage Collector

↓

Throughput

↓

Latency

↓

Memory Usage
```

Improving one often affects the others.

---

# Throughput

Throughput measures:

> **How much useful work the application performs.**

Example:

```
Application Running

99%

GC

1%
```

Excellent throughput.

Now consider:

```
Application Running

70%

GC

30%
```

Poor throughput.

Batch processing systems usually prioritize throughput.

---

# Latency

Latency measures:

> **How long the application pauses during GC.**

Example:

```
Pause

20 ms
```

Very good.

```
Pause

3 seconds
```

Potentially unacceptable for:

- Banking
- Trading
- Payment systems
- Online gaming

Modern low-latency collectors primarily optimize pause time.

---

# Memory Footprint

Some collectors require:

- More CPU
- More memory
- Additional bookkeeping

Others use less memory but produce longer pauses.

Memory usage is another important trade-off,

especially in containerized environments.

---

# The Trade-Off Triangle

Improving one characteristic often impacts the others.

```
             Throughput
                 ▲
                / \
               /   \
              /     \
     Memory -------- Latency
```

There is no perfect collector.

Instead,

there is a collector best suited for a particular workload.

---

# Stop-the-World (STW)

Most garbage collection activities require application threads to pause.

```
Application Running

↓

Stop-the-World

↓

Garbage Collection

↓

Application Resumes
```

Every collector experiences some STW activity,

but the duration varies significantly.

Reducing pause time is a major design goal of modern collectors.

---

# Concurrent Collection

Older collectors performed most work while the application was paused.

Modern collectors increasingly perform GC work concurrently with application threads.

```
Application Running

↓

GC Running Concurrently

↓

Application Continues
```

Benefits:

- Lower pause times
- Better responsiveness

Trade-off:

- Higher CPU overhead
- More complex algorithms

---

# Compaction

Over time,

the Heap can become fragmented.

```
Object

Gap

Object

Gap

Object
```

Compaction moves live objects together.

```
Object

Object

Object

Free Space
```

Benefits:

- Faster allocation
- Reduced fragmentation

Trade-off:

- Additional GC work
- Potential pause time

Different collectors perform compaction differently.

---

# Collector Evolution

The HotSpot JVM has evolved significantly.

```
Serial

↓

Parallel

↓

CMS

↓

G1

↓

ZGC

↓

Shenandoah
```

Each generation addressed limitations of the previous one.

The evolution generally moved toward:

- Lower latency
- Better scalability
- Larger heap support
- More concurrent execution

---

# Collector Comparison

| Collector | Primary Goal | Typical Pause Time | Suitable For |
|------------|--------------|-------------------|--------------|
| Serial | Simplicity | Long | Small applications |
| Parallel | Maximum Throughput | Medium | Batch processing |
| CMS *(deprecated)* | Lower Latency | Short | Legacy enterprise systems |
| G1 | Balanced Performance | Low | General-purpose server applications |
| ZGC | Ultra-low Latency | Very Low | Large heaps, latency-sensitive systems |
| Shenandoah | Ultra-low Latency | Very Low | Large heaps, cloud-native workloads |

We'll study each collector in detail in the following sections.

---

# Default Garbage Collector

Modern Java versions use **G1 GC** by default.

Why?

Because it provides a strong balance between:

- Throughput
- Pause time
- Scalability

It performs well for most enterprise applications without extensive tuning.

# Interview Focus

### Question

Why does the JVM provide multiple garbage collectors?

### Strong Senior-Level Answer

> Different applications have different performance requirements. Some prioritize throughput, while others require minimal pause times or efficient operation with very large heaps. Each garbage collector represents a different balance between throughput, latency, CPU usage, and memory overhead.

---

### Question

What is the difference between throughput and latency?

### Strong Senior-Level Answer

> Throughput measures the percentage of time spent performing useful application work, while latency refers to the duration of application pauses, particularly Stop-the-World pauses caused by garbage collection. High-throughput collectors may tolerate longer pauses, whereas low-latency collectors minimize pause times even if they consume more CPU.

---

# Common Mistakes

❌ Assuming the newest garbage collector is always the best choice.

❌ Optimizing for throughput when latency is the real business requirement.

❌ Choosing a collector without measuring application behavior.

❌ Assuming all collectors perform the same type of garbage collection.

---

# Engineering Insight

Experienced JVM engineers rarely begin troubleshooting by changing the garbage collector.

Instead, they first ask:

- What is the application's latency requirement?
- What is the average and peak heap size?
- What are the observed GC pause times?
- Is CPU utilization acceptable?
- What service-level objectives (SLOs) must be met?

The garbage collector is selected to satisfy business requirements—not personal preference.

---

# Engineering Takeaways

- Multiple garbage collectors exist because workloads differ.
- Every collector balances throughput, latency, CPU usage, and memory footprint differently.
- Stop-the-World pauses cannot be eliminated entirely but can often be minimized.
- Modern collectors perform much of their work concurrently.
- G1 GC is the default choice because it provides a balanced compromise for most applications.

---

# Quick Revision

✓ GC algorithms optimize different goals.

✓ Throughput and latency are different metrics.

✓ Modern collectors emphasize concurrent execution.

✓ G1 is the default collector in modern HotSpot JVMs.

✓ Always choose a collector based on workload characteristics.

# Senior Interview Challenge

A team wants to replace G1 GC with ZGC because they read that "ZGC is the fastest collector."

### What questions would you ask before agreeing?

**Expected Discussion Points**

- What are the application's latency SLOs?
- What is the current GC pause time?
- How large is the Heap?
- Is GC actually the performance bottleneck?
- What is the CPU overhead budget?
- Have GC logs been analyzed?

Senior engineers avoid changing GC algorithms based on trends. They use **GC logs, application metrics, and business requirements** to make evidence-based decisions.
