# 6.19 Remembered Sets (RSet) and Cross-Region References

One of the biggest innovations in G1 GC is the **Remembered Set (RSet)**.

Without Remembered Sets, G1 would lose most of its performance advantages.

Remembered Sets allow G1 to collect individual regions **without scanning the entire heap**, making region-based garbage collection practical.

---

## Why Are Remembered Sets Needed?

Recall that G1 divides the heap into many independent regions.

```text
+---------+---------+---------+---------+
|Region 1 |Region 2 |Region 3 |Region 4 |
+---------+---------+---------+---------+
```

Suppose G1 wants to collect only **Region 3**.

The obvious question is:

> **Can an object in another region still reference an object inside Region 3?**

The answer is **Yes**.

Example:

```text
Region 1

Object A
   │
   ▼
Region 3

Object X
```

If G1 collects Region 3 without knowing about this reference, it may incorrectly delete **Object X**, even though it is still reachable.

Therefore, G1 must know about references coming **from other regions**.

---

## The Traditional Approach

Older collectors typically scan the entire Old Generation.

Example:

```text
Old Generation

Object A → Object X
Object B
Object C
Object D
...
Millions of objects
```

To determine whether Object X is reachable, the collector scans everything.

For a large heap, this is expensive.

Example:

```text
Heap Size

32 GB
```

Scanning the entire heap during every collection would significantly increase pause times.

---

## The G1 Approach

Instead of scanning every region, G1 maintains additional metadata.

Each region keeps track of:

> **Which other regions contain references into me?**

This metadata is called the **Remembered Set (RSet)**.

---

## What Is a Remembered Set?

Every region owns a Remembered Set.

Example:

```text
Region 1
RSet

Region 2
RSet

Region 3
RSet

Region 4
RSet
```

The Remembered Set for a region contains information about references from **other regions** into that region.

---

## Example

Consider the following heap.

```text
Region 1

A
│
▼

Region 3

X
```

Another reference:

```text
Region 2

B
│
▼

Region 3

Y
```

The Remembered Set of Region 3 stores:

```text
Region 3 RSet

Referenced From:
Region 1
Region 2
```

Now, when collecting Region 3, G1 only needs to examine Regions 1 and 2.

Every other region can safely be ignored.

---

## Why This Is Powerful

Imagine a heap with 2,000 regions.

Without Remembered Sets:

```text
Scan

Region 1
Region 2
Region 3
...
Region 2000
```

With Remembered Sets:

```text
Collect Region 350

Need To Scan Only

Region 12
Region 84
Region 500
```

This dramatically reduces GC work.

---

## Cross-Region References

Objects frequently reference objects located in different regions.

Example:

```text
Region 1

Customer
     │
     ▼

Region 5

Order
```

Or:

```text
Region 8

Cache
     │
     ▼

Region 17

Product
```

These are called **Cross-Region References**.

Remembered Sets maintain exactly this information.

---

## How Are Remembered Sets Updated?

Suppose the application executes:

```java
customer.setOrder(order);
```

where:

- `customer` is in Region 2
- `order` is in Region 8

A new cross-region reference has been created.

The JVM records this information automatically.

The application's execution continues normally.

---

## The Role of the Write Barrier

Every time a reference field is modified, the JVM executes a small piece of additional code called the **Write Barrier**.

Example:

```java
customer.order = order;
```

Internally, the JVM performs:

```text
Update Reference

↓

Execute Write Barrier

↓

Update Remembered Set
```

This ensures that Remembered Sets remain accurate throughout the application's lifetime.

---

## Write Barrier Is Not Garbage Collection

A common misconception is that the Write Barrier performs garbage collection.

It does not.

Its responsibility is only to record reference changes.

Actual garbage collection happens later.

---

## Card Tables

Recording every individual object reference would consume too much memory.

Instead, G1 tracks references at a coarser granularity using **Card Tables**.

The heap is divided into small fixed-size memory blocks called **Cards**.

Example:

```text
Heap

+----+----+----+----+----+
|Card|Card|Card|Card|Card|
+----+----+----+----+----+
```

Each card typically represents **512 bytes** of heap memory (implementation-dependent).

When an object reference changes, the corresponding card is marked as **Dirty**.

---

## Dirty Cards

Suppose an application modifies an object.

```text
Before

Card 25

Customer
```

Reference updated:

```java
customer.setOrder(order);
```

The JVM marks:

```text
Card 25

Dirty
```

Later, G1 examines only dirty cards instead of scanning the entire heap.

This further reduces GC overhead.

---

## Relationship Between Card Tables and Remembered Sets

The overall process looks like this:

```text
Application Updates Reference
            │
            ▼
      Write Barrier
            │
            ▼
      Card Marked Dirty
            │
            ▼
Concurrent Refinement Thread
            │
            ▼
Remembered Set Updated
```

This pipeline allows most of the bookkeeping work to happen concurrently, reducing Stop-The-World pauses.

---

## Concurrent Refinement Threads

Instead of updating Remembered Sets immediately inside the application thread, G1 often delegates part of the work to **Concurrent Refinement Threads**.

Their responsibilities include:

- Processing dirty cards
- Updating Remembered Sets
- Reducing application thread overhead

This improves overall throughput while maintaining accurate metadata.

---

## Why Remembered Sets Improve Performance

Suppose the heap contains:

```text
2000 Regions
```

A Young GC wants to collect:

```text
Region 145
```

Without Remembered Sets:

```text
Scan

2000 Regions
```

With Remembered Sets:

```text
Scan

12 Regions
```

The reduction in work is enormous, leading to much shorter pause times.

---

## Advantages of Remembered Sets

- Eliminates full heap scanning during regional collections
- Enables independent region collection
- Reduces Stop-The-World pause time
- Makes G1 scalable for very large heaps
- Works seamlessly with concurrent marking

---

## Limitations

Remembered Sets are not free.

They introduce:

- Additional memory overhead
- Write Barrier execution on reference updates
- Background processing by refinement threads

G1 accepts this overhead because the reduction in GC pause times is usually much greater than the additional cost.

---

## Interview Questions

### Q1. What is a Remembered Set?

A Remembered Set (RSet) is metadata maintained for each G1 region that records which other regions contain references into that region.

---

### Q2. Why are Remembered Sets important?

They allow G1 to collect individual regions without scanning the entire heap, significantly reducing garbage collection pause times.

---

### Q3. What is a Cross-Region Reference?

A Cross-Region Reference is a reference from an object in one region to an object located in another region.

---

### Q4. What is the purpose of the Write Barrier?

The Write Barrier records reference updates so that Card Tables and Remembered Sets remain accurate during garbage collection.

---

### Q5. What is a Dirty Card?

A Dirty Card is a memory block whose references have been modified since the last processing cycle. Dirty cards are later examined to update Remembered Sets.

---

## Key Takeaways

- Every G1 region maintains its own Remembered Set.
- Remembered Sets record incoming cross-region references.
- The Write Barrier captures reference updates during application execution.
- Card Tables efficiently track modified memory regions.
- Concurrent Refinement Threads process dirty cards and update Remembered Sets.
- Remembered Sets allow G1 to avoid expensive full-heap scans, enabling predictable low-pause garbage collection.
