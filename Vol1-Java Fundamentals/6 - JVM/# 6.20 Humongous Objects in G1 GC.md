# 6.20 Humongous Objects in G1 GC

One of the unique concepts introduced by G1 GC is the handling of **Humongous Objects**.

Unlike normal objects, humongous objects cannot always fit efficiently within a single region.

G1 treats these objects differently to avoid excessive copying and improve allocation performance.

---

## What Is a Humongous Object?

A **Humongous Object** is an object whose size is **greater than 50% of a G1 Region**.

Suppose the region size is:

```text
4 MB
```

Then:

| Object Size | Classification |
|-------------|----------------|
| 1 MB | Normal Object |
| 2 MB | Normal Object |
| 2.1 MB | Humongous Object |
| 5 MB | Humongous Object |
| 20 MB | Humongous Object |

The threshold is always:

```text
Object Size > Region Size / 2
```

---

## Why Does G1 Treat Them Differently?

Normal objects are copied during evacuation.

Example:

```text
Old Region

Object A
Object B
Object C
```

During GC:

```text
Old Region
        │
        ▼

New Region

Object A
Object B
Object C
```

This process is efficient for relatively small objects.

Now consider a 40 MB object.

Copying such a large object during every evacuation would be expensive.

It would:

- Increase pause times
- Consume memory bandwidth
- Require many contiguous destination regions

Therefore, G1 uses a special allocation strategy.

---

## Humongous Region Allocation

Instead of storing the object inside a normal region, G1 reserves one or more **contiguous regions** exclusively for the object.

Example:

Region size:

```text
4 MB
```

Object size:

```text
10 MB
```

Allocation:

```text
+---------+---------+---------+
|Region 10|Region 11|Region 12|
+---------+---------+---------+

Humongous Object
```

These regions are dedicated entirely to the humongous object.

---

## Example

Heap:

```text
+----+----+----+----+----+----+
| R1 | R2 | R3 | R4 | R5 | R6 |
+----+----+----+----+----+----+
```

A 9 MB object is allocated.

Assume each region is 4 MB.

Allocation becomes:

```text
+----+----+----+----+----+----+
| R1 | R2 | H1 | H2 | H3 | R6 |
+----+----+----+----+----+----+

H1-H3 store one humongous object.
```

---

## Humongous Objects Are Allocated Directly in the Old Generation

Unlike normal objects, humongous objects are generally allocated directly into **Old Generation regions**.

Reason:

- They are expensive to copy repeatedly.
- They are unlikely to benefit from multiple Young GC evacuations.

This reduces unnecessary copying.

---

## Why Are Contiguous Regions Required?

A humongous object occupies consecutive memory regions.

Example:

```text
Correct

Region 20
Region 21
Region 22
```

Not:

```text
Region 20
Region 35
Region 48
```

Contiguous regions simplify addressing and management.

---

## Internal Fragmentation

Suppose:

Region size:

```text
4 MB
```

Object size:

```text
9 MB
```

Allocation:

```text
Region 1 = 4 MB
Region 2 = 4 MB
Region 3 = 4 MB
```

Total reserved:

```text
12 MB
```

Actual object:

```text
9 MB
```

Unused space:

```text
3 MB
```

This unused memory is called **Internal Fragmentation**.

---

## Why Humongous Objects Can Cause Problems

Imagine repeated allocations:

```text
9 MB
10 MB
11 MB
8 MB
```

Eventually the heap may contain:

```text
+----+----+----+----+----+----+
| H1 | H1 | R3 | H2 | H2 | R6 |
+----+----+----+----+----+----+
```

Although plenty of memory remains free, there may not be enough **contiguous** regions for the next humongous object.

Allocation may fail.

---

## Fragmentation Example

```text
Free Regions

R5
R8
R12
R17
```

Need:

```text
Three Consecutive Regions
```

Available:

```text
No
```

Even though four free regions exist, none are contiguous.

This can eventually trigger a Full GC.

---

## Collection of Humongous Objects

Humongous objects are **not normally evacuated** during Young GC.

Instead, they are reclaimed during:

- Concurrent Marking
- Mixed Collections
- Full GC (if necessary)

This avoids repeatedly copying very large objects.

---

## Real-World Examples

Large objects often include:

- Large byte arrays
- Large image buffers
- Huge JSON payloads
- Large XML documents
- Video buffers
- Big cache entries

Example:

```java
byte[] buffer = new byte[20 * 1024 * 1024];
```

If the region size is 4 MB:

```text
20 MB > 2 MB

Humongous Object
```

---

## Performance Impact

Frequent humongous object allocation may cause:

- Increased memory fragmentation
- Longer concurrent marking cycles
- More Mixed Collections
- Occasional Full GCs
- Lower application throughput

---

## JVM Tuning Considerations

Large applications should monitor:

- Humongous object allocation frequency
- Region size
- Full GC occurrences
- Heap fragmentation

Sometimes increasing the region size reduces the number of humongous objects.

Example:

Current region size:

```text
2 MB
```

Threshold:

```text
1 MB
```

After increasing region size:

```text
8 MB
```

Threshold becomes:

```text
4 MB
```

Many previously humongous objects are now treated as normal objects.

---

## Monitoring Humongous Objects

GC logs often report humongous allocations.

Typical indicators include:

- Humongous regions
- Humongous allocation
- Humongous reclaim

A growing number of humongous regions may indicate that application object sizes should be reviewed.

---

## Best Practices

- Avoid unnecessarily large arrays.
- Process large files in chunks.
- Stream large JSON or XML payloads instead of loading them entirely into memory.
- Monitor GC logs for excessive humongous allocations.
- Choose an appropriate heap and region size for the application's workload.

---

## Interview Questions

### Q1. What is a Humongous Object in G1?

A humongous object is any object whose size is greater than half of a G1 region.

---

### Q2. Why are humongous objects allocated differently?

Because repeatedly copying very large objects during evacuation would significantly increase garbage collection pause times.

---

### Q3. Where are humongous objects allocated?

They are typically allocated directly into contiguous Old Generation regions.

---

### Q4. Why can humongous objects trigger Full GC?

They require contiguous free regions. Even when sufficient total free memory exists, fragmentation may prevent contiguous allocation, forcing G1 to perform a Full GC.

---

### Q5. How can applications reduce humongous object problems?

By avoiding unnecessarily large objects, processing data in chunks, streaming large payloads, and tuning the G1 region size when appropriate.

---

## Key Takeaways

- A humongous object is larger than **50% of a G1 region**.
- Humongous objects occupy one or more contiguous regions.
- They are generally allocated directly into Old Generation regions.
- They are not typically evacuated during Young GC.
- Excessive humongous allocations can lead to fragmentation and Full GCs.
- Monitoring GC logs and tuning region size are important for applications handling very large objects.
