# 6.5 Runtime Data Areas – Heap, Stack, Metaspace, PC Register, and Native Method Stack

### Why This Matters

One of the most common misconceptions among Java developers is:

> **"The JVM stores everything in the Heap."**

This is incorrect.

The JVM divides memory into several specialized regions, each with a different purpose.

Understanding these memory areas is essential for diagnosing:

- `OutOfMemoryError`
- `StackOverflowError`
- Memory leaks
- High memory usage
- Performance bottlenecks
- Garbage collection behavior

Every object, method call, class definition, and thread interacts with one or more of these memory regions.

---

# JVM Runtime Data Areas

When the JVM starts, it creates multiple runtime memory areas.

```
                JVM Memory
                     │
      ---------------------------------
      │               │              │
     Heap         Thread Stack   Metaspace
      │               │
      │          Program Counter
      │
 Native Method Stack
```

These areas can be broadly divided into:

### Shared Memory

- Heap
- Metaspace

Shared by all threads.

---

### Thread-Specific Memory

Each thread has its own:

- Java Stack
- Program Counter (PC Register)
- Native Method Stack

Whenever a new thread is created,

new thread-specific memory is also created.

---

# Memory Overview

| Memory Area | Shared? | Stores |
|--------------|----------|-----------------------------|
| Heap | ✅ Yes | Objects and arrays |
| Java Stack | ❌ No | Stack frames, local variables |
| Metaspace | ✅ Yes | Class metadata |
| PC Register | ❌ No | Current instruction |
| Native Method Stack | ❌ No | Native method execution |

Each region exists for a different reason.

---

# Heap Memory

The Heap stores:

- Objects
- Arrays
- Instance variables

Example:

```java
Employee employee =
        new Employee();
```

Memory:

```
Reference

↓

Stack
```

↓

```
Employee Object

↓

Heap
```

The Heap is:

- Shared
- Managed by the Garbage Collector
- Usually the largest JVM memory region

Most JVM tuning focuses on Heap memory.

---

# Java Stack

Every thread owns its own Java Stack.

Example:

```java
void process() {

    int count = 10;

}
```

Memory:

```
Stack Frame

↓

Method Parameters

↓

Local Variables

↓

Intermediate Results
```

When the method returns,

the stack frame is removed automatically.

Stack memory is **not** garbage collected.

---

# Metaspace

Metaspace stores:

- Class metadata
- Method metadata
- Runtime constant pool
- Reflection information

Example:

```
Employee.class

↓

Metaspace
```

Objects themselves are **not** stored here.

Unlike Heap memory,

Metaspace grows dynamically (subject to JVM limits).

Prior to Java 8, this area was known as the **Permanent Generation (PermGen)**.

Java 8 replaced PermGen with Metaspace to improve memory management.

---

# Program Counter (PC Register)

Every thread owns its own Program Counter.

It stores:

```
Current JVM Instruction
```

Example:

```
Instruction #10

↓

Instruction #11

↓

Instruction #12
```

As the JVM executes bytecode,

the PC Register tracks which instruction should execute next.

This enables threads to pause and resume independently.

---

# Native Method Stack

Not every method executes Java bytecode.

Example:

```java
System.arraycopy(...)
```

or certain file system and networking operations.

When Java invokes native code through JNI,

execution uses the Native Method Stack.

```
Java

↓

JNI

↓

Native Stack

↓

Operating System
```

This memory area supports interaction with platform-specific libraries.

---

# Shared vs Thread Memory

Suppose three threads execute simultaneously.

```
             Heap
              │
   -----------------------
   │         │          │
Thread 1  Thread 2  Thread 3
  Stack     Stack     Stack
```

Notice:

- All threads share the Heap.
- Each thread has its own Stack.
- Each thread has its own PC Register.
- Each thread has its own Native Method Stack.

This separation is fundamental to Java's threading model.

---

# Memory During Method Execution

Example:

```java
Employee employee =
        new Employee();

employee.calculateSalary();
```

Memory usage:

```
Stack

↓

employee Reference

↓

Heap

↓

Employee Object
```

When:

```java
calculateSalary()
```

is called,

a new stack frame is created.

When the method returns,

that frame disappears,

but the object remains in the Heap as long as it is still reachable.

# Common JVM Errors

Different memory regions produce different errors.

| Error | Memory Area |
|--------|-------------|
| `OutOfMemoryError: Java heap space` | Heap |
| `StackOverflowError` | Java Stack |
| `OutOfMemoryError: Metaspace` | Metaspace |
| Native memory exhaustion | Native Method Stack / OS |

Recognizing the affected memory area dramatically speeds up troubleshooting.

---

# Spring Boot Perspective

A typical Spring Boot application uses all runtime data areas.

- Beans and domain objects → Heap
- Controller and service method calls → Stack
- Loaded framework classes → Metaspace
- Request processing → PC Register
- Native SSL or file operations → Native Method Stack

As applications grow,

memory usage across these regions also grows.

Monitoring tools such as JFR, VisualVM, and `jcmd` expose these memory areas separately.

---

# Production Perspective

Imagine an application that creates thousands of objects per second.

Heap usage rises rapidly,

but stack usage remains stable.

This indicates:

- High object allocation rate
- Increased garbage collection activity
- Possible memory leak

Conversely,

an application with deep recursive calls may fail with:

```
StackOverflowError
```

even though plenty of heap memory is still available.

Understanding which memory region is affected prevents unnecessary JVM tuning.

# Interview Focus

### Question

What are the Runtime Data Areas in the JVM?

### Strong Senior-Level Answer

> The JVM divides runtime memory into several areas. The Heap stores objects and is shared across threads. Each thread has its own Java Stack for method execution, Program Counter for tracking the current instruction, and Native Method Stack for native code execution. Class metadata is stored in Metaspace. Each area serves a distinct purpose and contributes differently to application performance and memory management.

---

# Common Mistakes

❌ Assuming everything is stored in the Heap.

❌ Confusing objects with class metadata.

❌ Believing stack memory is garbage collected.

❌ Ignoring thread-specific memory when diagnosing concurrency issues.

---

# Engineering Insight

When experienced engineers encounter JVM errors,

their first question is not:

> "How much memory does the application use?"

Instead, they ask:

> **"Which memory area is failing?"**

The answer determines the investigation:

- Heap → Object allocation and GC.
- Stack → Recursion or large stack frames.
- Metaspace → Class loading.
- Native Memory → JNI, direct buffers, or OS resources.

This targeted approach saves significant debugging time.

---

# Engineering Takeaways

- JVM memory is divided into specialized runtime areas.
- Heap and Metaspace are shared.
- Stack, PC Register, and Native Method Stack are thread-specific.
- Different runtime errors originate from different memory regions.
- Understanding memory layout is the foundation for JVM tuning.

---

# Senior Interview Challenge

A production application reports the following:

- Heap usage remains below **40%**
- No Full GC events
- Application crashes with:

```
java.lang.StackOverflowError
```

### Would increasing the Heap size solve the problem? Why or why not?

**Expected Discussion Points**

- No.
- `StackOverflowError` originates from the Java Stack, not the Heap.
- Common causes include infinite recursion or excessively deep call chains.
- Increasing `-Xmx` affects Heap memory only.
- Investigation should focus on recursion depth, stack usage, and thread stack size (`-Xss`), not garbage collection.

Senior engineers diagnose JVM issues by identifying the affected memory region before making configuration changes.
