# 6.4 The Class Loading Lifecycle – Loading, Linking, and Initialization

### Why This Matters

When developers say:

> "The JVM loads a class."

it sounds like a single operation.

In reality, class loading is a **multi-stage lifecycle**.

Before a class can execute, the JVM performs several operations to ensure:

- The class exists.
- The bytecode is valid.
- Memory is allocated.
- References are resolved.
- Static initialization is complete.

Only then is the class ready for execution.

Understanding this lifecycle explains:

- Static initialization behavior
- Class loading order
- Startup performance
- `ExceptionInInitializerError`
- Class loading interview questions

---

# The Complete Lifecycle

A class passes through the following stages:

```
Loading

↓

Linking

↓

Initialization

↓

Ready for Use
```

The **Linking** phase itself consists of three sub-phases:

```
Verification

↓

Preparation

↓

Resolution
```

Overall flow:

```
Loading

↓

Verification

↓

Preparation

↓

Resolution

↓

Initialization
```

---

# Example Class

We'll use the following class throughout this section:

```java
class Employee {

    static int counter = 100;

    static {

        System.out.println("Static Block");

    }

}
```

Let's see how the JVM prepares this class.

---

# Stage 1 – Loading

During loading, the Class Loader:

- Locates the `.class` file.
- Reads its bytecode.
- Creates the runtime representation of the class.
- Stores class metadata in **Metaspace**.

At this point:

- Static blocks have **not** executed.
- Objects have **not** been created.
- Methods have **not** run.

Only the class definition is available to the JVM.

---

# Stage 2 – Verification

Verification ensures that the bytecode is:

- Valid
- Safe
- Structurally correct

The JVM checks things such as:

- Correct bytecode format
- Stack consistency
- Illegal memory access
- Type safety
- Invalid instructions

Without verification,

malicious or corrupted bytecode could compromise the JVM.

---

# Why Verification Matters

Suppose someone modifies a `.class` file manually.

Example:

```
Corrupted Bytecode

↓

Invalid Instruction

↓

JVM Rejects Class
```

Instead of crashing the operating system,

the JVM refuses to load the class.

Verification is one of Java's key security features.

---

# Stage 3 – Preparation

Preparation allocates memory for **static fields**.

However,

only **default values** are assigned.

Example:

```java
static int counter = 100;
```

During preparation:

```
counter = 0
```

because the explicit assignment has not yet occurred.

Likewise:

```java
static String name = "Java";
```

becomes:

```
name = null
```

during preparation.

---

# Stage 4 – Resolution

Resolution replaces **symbolic references** with **direct runtime references**.

Example:

```java
Employee employee =
        new Employee();
```

Initially, the bytecode contains symbolic references such as:

```
Employee
```

During resolution, the JVM locates the actual runtime class and links these references.

This enables efficient execution.

---

# Stage 5 – Initialization

Initialization executes:

- Static field assignments
- Static initialization blocks

Returning to our example:

```java
class Employee {

    static int counter = 100;

    static {

        System.out.println("Static Block");

    }

}
```

Initialization performs:

1. Assign:

```
counter = 100
```

2. Execute:

```
Static Block
```

The class is now fully initialized.

---

# Initialization Order

Consider:

```java
class Parent {

    static {

        System.out.println("Parent");

    }

}

class Child extends Parent {

    static {

        System.out.println("Child");

    }

}
```

Execution:

```java
new Child();
```

Output:

```
Parent

Child
```

The parent class is always initialized before the child.

---

# What Triggers Class Initialization?

A class is initialized when it is first actively used.

Examples include:

### Creating an object

```java
new Employee();
```

---

### Accessing a static field

```java
Employee.counter;
```

---

### Invoking a static method

```java
Employee.print();
```

---

### Reflection

```java
Class.forName("Employee");
```

Reflection explicitly requests the class, forcing initialization.

---

# What Does NOT Trigger Initialization?

Simply declaring a reference does **not** initialize the class.

```java
Employee employee;
```

No object is created.

No static block executes.

Similarly, referencing a compile-time constant may not trigger initialization because the compiler can inline the value.

Example:

```java
class Constants {

    static final int MAX = 100;

}
```

Using:

```java
System.out.println(Constants.MAX);
```

may not initialize `Constants`, depending on the nature of the constant.

---

# Static Block Example

```java
class Demo {

    static {

        System.out.println("Loaded");

    }

}
```

Execution:

```java
new Demo();

new Demo();
```

Output:

```
Loaded
```

Only once.

Static initialization occurs a single time for each class loader.

---

# `ExceptionInInitializerError`

Suppose a static block throws an exception:

```java
class Demo {

    static {

        int value = 10 / 0;

    }

}
```

During initialization:

```
ArithmeticException

↓

ExceptionInInitializerError
```

The class fails to initialize.

Subsequent attempts to use the class will also fail because initialization never completed successfully.

---

# Spring Boot Perspective

When Spring Boot starts,

thousands of classes are initialized.

Many of these classes contain:

- Static constants
- Logging initialization
- Framework configuration
- Driver registration
- Factory initialization

Keeping static initialization lightweight helps reduce application startup time.

Heavy work inside static blocks can noticeably slow large applications.

---

# Production Perspective

A production application fails immediately after deployment.

Logs show:

```
ExceptionInInitializerError
```

Investigation reveals:

```java
static {

    Database.connect();

}
```

The database is temporarily unavailable.

Because the connection attempt occurs during class initialization,

the class never loads successfully,

preventing the application from starting.

This is one reason experienced engineers avoid placing external resource initialization inside static blocks.

---

# Interview Focus

### Question

What happens during the class loading lifecycle?

### Strong Senior-Level Answer

> The JVM prepares a class in multiple stages. During Loading, it reads the class file and creates the runtime representation. During Linking, it verifies bytecode, allocates memory for static fields with default values, and resolves symbolic references. During Initialization, it executes static field assignments and static initialization blocks, making the class ready for use.

---

# Common Mistakes

❌ Assuming static fields receive explicit values during preparation.

❌ Performing expensive operations inside static blocks.

❌ Believing static blocks execute every time an object is created.

❌ Confusing class loading with object creation.

---

# Engineering Insight

Object creation and class initialization are different concepts.

```
Class Initialization

↓

One Time
```

```
Object Creation

↓

Many Times
```

A class may be initialized once,

while millions of objects are created from it during the application's lifetime.

Recognizing this distinction helps optimize startup performance and understand runtime behavior.

---

# Engineering Takeaways

- Class loading is a multi-stage process.
- Linking includes Verification, Preparation, and Resolution.
- Static fields receive default values during Preparation.
- Static initializers execute during Initialization.
- Class initialization occurs only once per class loader.

---

# Quick Revision

✓ Loading reads the `.class` file.

✓ Verification checks bytecode safety.

✓ Preparation allocates static fields with default values.

✓ Resolution links symbolic references.

✓ Initialization executes static assignments and static blocks.

# Senior Interview Challenge

Review the following code:

```java
class Config {

    static int timeout = loadTimeout();

    static int loadTimeout() {
        System.out.println("Loading...");
        return 30;
    }

}
```

Then execute:

```java
Config config1 = new Config();
Config config2 = new Config();
```

### How many times is `loadTimeout()` executed? Why?

**Expected Discussion Points**

- It executes **once**.
- The class is initialized only once by a given class loader.
- Both object creations use the already initialized class.
- Static initialization belongs to the class, not individual objects.

Senior engineers distinguish **class lifecycle** from **object lifecycle**, a concept that frequently appears in JVM and Spring Boot interview discussions.
