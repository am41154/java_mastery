# 6.6 Heap Memory – Object Allocation and Garbage Collection Fundamentals

### Why This Matters

Most JVM performance problems eventually involve the **Heap**.

If you've encountered errors such as:

```
java.lang.OutOfMemoryError: Java heap space
```

or noticed:

- Frequent Garbage Collection
- High memory usage
- Slow response times
- Long GC pauses

then Heap memory is often the first place to investigate.

Understanding how objects are allocated and managed in the Heap is fundamental to becoming an effective Java backend engineer.

---

# What Is Heap Memory?

The **Heap** is the runtime memory area where Java stores:

- Objects
- Arrays
- Instance variables

Example:

```java
Employee employee =
        new Employee();
```

Memory layout:

```
Stack
│
└── employee (reference)
        │
        ▼
Heap
│
└── Employee Object
```

Notice that:

- The **reference** is stored on the Stack.
- The **actual object** is stored in the Heap.

---

# Characteristics of the Heap

The Heap is:

- Shared by all threads
- Automatically managed by the JVM
- Managed by the Garbage Collector
- Usually the largest memory region

Unlike Stack memory,

objects remain in the Heap until they become unreachable.

---

# Why Doesn't Java Allocate Objects on the Stack?

Consider:

```java
Employee employee =
        new Employee();
```

Multiple methods,

or even multiple threads,

may reference the same object.

```
Method A

↓

Employee Object

↑

Method B
```

Since objects can outlive the method that created them,

the Heap provides shared, long-lived storage.

> **Note:** Modern JVMs may optimize some short-lived objects using **escape analysis** and **scalar replacement**, effectively avoiding Heap allocation in certain cases. We'll discuss these optimizations later in this chapter.

---

# Heap Structure

Modern JVMs divide the Heap into generations.

```
                Heap

      ------------------------
      |   Young Generation   |
      ------------------------
      |    Old Generation    |
      ------------------------
```

The Young Generation is further divided into:

```
Young Generation

↓

Eden

↓

Survivor 0

↓

Survivor 1
```

Visual representation:

```
+-----------------------------------+
|          Young Generation         |
| +-------+-------+---------------+ |
| | Eden  |  S0   |      S1       | |
| +-------+-------+---------------+ |
+-----------------------------------+
|         Old Generation            |
+-----------------------------------+
```

---

# Why Use Generations?

The JVM is designed around an important observation:

> **Most Java objects die young.**

Examples:

- HTTP request objects
- DTOs
- JSON objects
- Local collections
- Temporary strings

These objects are created,

used briefly,

and discarded.

Instead of scanning the entire Heap repeatedly,

the JVM focuses first on the Young Generation.

This makes garbage collection significantly more efficient.

---

# Object Allocation

New objects are usually allocated in **Eden**.

Example:

```java
for(int i = 0; i < 1000; i++) {

    new Employee();

}
```

Memory:

```
New Object

↓

Eden
```

As Eden fills,

a **Minor Garbage Collection** occurs.

---

# Minor Garbage Collection

When Eden becomes full:

```
Eden Full

↓

Minor GC

↓

Unused Objects Removed

↓

Live Objects Copied to Survivor Space
```

Objects that are no longer reachable are reclaimed immediately.

Objects that are still referenced survive.

---

# Survivor Spaces

The Young Generation contains two Survivor spaces:

```
Survivor 0 (S0)

Survivor 1 (S1)
```

During each Minor GC:

```
Eden

↓

S0

↓

S1

↓

S0

↓

S1
```

Live objects move between Survivor spaces.

Their **age** increases after each successful collection.

---

# Promotion to Old Generation

Objects that survive several Minor GCs are promoted.

```
Eden

↓

Survivor

↓

Survivor

↓

Old Generation
```

These are generally long-lived objects such as:

- Spring Beans
- Application caches
- Singleton objects
- Long-lived domain models

---

# Old Generation

The Old Generation stores objects that remain alive for a long time.

Examples:

```
Application Cache

Singleton

Spring Bean

Database Connection Pool

Configuration Objects
```

Garbage collection here is less frequent,

but more expensive.

Cleaning the Old Generation typically requires a **Major GC** or **Full GC**, depending on the collector and JVM implementation.

---

# Object Lifecycle

Consider:

```java
Employee employee =
        new Employee();
```

Lifecycle:

```
Object Created

↓

Allocated in Eden

↓

Minor GC

↓

Survivor

↓

Minor GC

↓

Old Generation

↓

Unreachable

↓

Garbage Collected
```

Most objects never reach the Old Generation because they become unreachable much earlier.

---

# Reachability

The Garbage Collector determines whether an object is still reachable.

Example:

```java
Employee employee =
        new Employee();

employee = null;
```

Memory:

```
No References

↓

Object Unreachable

↓

Eligible for GC
```

The JVM does **not** immediately destroy the object.

It simply becomes **eligible** for garbage collection.

---

# Heap and Multiple Threads

The Heap is shared.

```
Thread 1

↓

Heap

↑

Thread 2

↑

Thread 3
```

This allows multiple threads to work with the same objects.

However,

it also introduces the need for synchronization when mutable objects are shared.

---

# Spring Boot Perspective

In a Spring Boot application,

Heap memory stores:

- Controllers
- Services
- Repositories
- Request DTOs
- Response DTOs
- Hibernate entities
- Cached objects
- Session data
- JSON objects
- Collections

Large enterprise applications may create **millions of objects every minute**.

Efficient object allocation and garbage collection are therefore critical to maintaining throughput.

---

# Production Perspective

Suppose an API server receives **20,000 requests per second**.

Each request creates:

- Request DTO
- Response DTO
- Validation objects
- Temporary strings
- Collections

Most of these objects exist for only a few milliseconds.

This is exactly why the Young Generation exists.

Without generational garbage collection,

the JVM would waste enormous time scanning long-lived application objects for every request.

---

# Interview Focus

### Question

Why is the Heap divided into Young and Old Generations?

### Strong Senior-Level Answer

> The JVM uses generational memory because most objects in Java applications are short-lived. New objects are allocated in the Young Generation, where garbage collection is fast and efficient. Objects that survive multiple collection cycles are promoted to the Old Generation. This design minimizes garbage collection overhead and improves application performance.

---

# Common Mistakes

❌ Assuming every object survives until a Full GC.

❌ Believing objects are destroyed immediately after becoming unreachable.

❌ Confusing Heap memory with Stack memory.

❌ Treating Minor GC and Full GC as identical operations.

---

# Engineering Insight

Many developers assume that **large Heap sizes always improve performance**.

In reality:

- A larger Heap reduces GC frequency.
- But each major collection may take longer.
- Excessively large Heaps can increase pause times and memory footprint.

Effective JVM tuning balances allocation rate, GC frequency, pause times, and application throughput rather than simply maximizing Heap size.

---

# Engineering Takeaways

- Objects are generally allocated in Eden.
- Most objects die young.
- Survivor spaces hold objects that survive Minor GCs.
- Long-lived objects are promoted to the Old Generation.
- Heap design is optimized around real-world object lifecycles.

---

# Quick Revision

✓ Heap stores objects.

✓ New objects start in Eden.

✓ Minor GC cleans the Young Generation.

✓ Survivor spaces track object age.

✓ Long-lived objects move to the Old Generation.

# Senior Interview Challenge

A production microservice shows the following pattern:

- Minor GC occurs every **2 seconds**
- Old Generation usage remains low
- Throughput is acceptable
- Pause times are very short

### Would you consider this behavior a problem? Why or why not?

**Expected Discussion Points**

- Not necessarily.
- Frequent Minor GCs are expected in applications with high object allocation rates.
- Minor GCs are designed to be fast because they process only the Young Generation.
- If pause times are low and application throughput is healthy, frequent Minor GCs can be perfectly normal.
- JVM tuning should be driven by measurable performance issues rather than GC frequency alone.

Senior engineers evaluate **GC impact**, not simply **GC count**. A JVM performing frequent, short Minor GCs can often outperform one performing infrequent but lengthy Full GCs.
