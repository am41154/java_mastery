# 6.22 Garbage Collector Selection and JVM Tuning

Java provides multiple garbage collectors, each designed for different workloads.

Choosing the right collector and configuring it appropriately can significantly improve application performance.

There is **no single garbage collector that is best for every application**.

The choice depends on factors such as:

- Heap size
- Response time requirements
- Throughput requirements
- CPU availability
- Application workload
- Object allocation rate

---

# Evolution of Java Garbage Collectors

| Java Version | Default Collector |
|--------------|-------------------|
| Java 7 | Parallel GC |
| Java 8 | Parallel GC |
| Java 9+ | G1 GC |
| Java 21 | G1 GC |

Although G1 is the default collector in modern Java versions, other collectors remain available for specialized use cases.

---

# Available Garbage Collectors

The JVM currently supports:

- Serial GC
- Parallel GC
- G1 GC
- ZGC
- Shenandoah GC
- Epsilon GC (No-Op Collector)

Each collector has different design goals.

---

# 6.22.1 Serial GC

Serial GC uses a **single GC thread**.

During collection:

- Application stops
- One GC thread performs all work

```
Application
     │
     ▼
Stop-The-World
     │
     ▼
Single GC Thread
```

### Enable

```bash
-XX:+UseSerialGC
```

### Advantages

- Simple implementation
- Very low CPU usage
- Small memory footprint

### Disadvantages

- Long pauses on large heaps
- Cannot utilize multiple CPU cores

### Best Suited For

- Small desktop applications
- Development environments
- Tiny JVMs
- Embedded systems

---

# 6.22.2 Parallel GC

Parallel GC focuses on **maximum throughput**.

Multiple GC threads work simultaneously.

```
GC

Thread 1
Thread 2
Thread 3
Thread 4
```

### Enable

```bash
-XX:+UseParallelGC
```

### Advantages

- High throughput
- Excellent CPU utilization
- Good for batch processing

### Disadvantages

- Longer Stop-The-World pauses
- Less suitable for latency-sensitive applications

### Best Suited For

- Batch jobs
- Data processing
- Scientific computation
- Backend jobs where throughput matters more than latency

---

# 6.22.3 G1 GC

G1 divides the heap into many equally sized regions.

Instead of collecting the entire heap, it collects selected regions.

```
Heap

+----+----+----+----+----+
| R1 | R2 | R3 | R4 | R5 |
+----+----+----+----+----+
```

### Enable

```bash
-XX:+UseG1GC
```

### Advantages

- Predictable pause times
- Incremental collections
- Automatic compaction
- Excellent support for large heaps

### Disadvantages

- More complex than Parallel GC
- Additional metadata overhead (Remembered Sets)

### Best Suited For

- Microservices
- REST APIs
- Enterprise applications
- Cloud-native workloads
- Large production systems

---

# 6.22.4 ZGC

**Z Garbage Collector (ZGC)** is designed for extremely low latency.

Most GC work happens concurrently while the application continues running.

Typical pause times remain:

```text
Less than 1 ms
```

even for very large heaps.

### Enable

```bash
-XX:+UseZGC
```

### Advantages

- Extremely low pause times
- Excellent scalability
- Supports multi-terabyte heaps

### Disadvantages

- Slightly lower throughput than Parallel GC
- Higher implementation complexity

### Best Suited For

- Financial trading
- Real-time systems
- Large memory applications
- Latency-critical services

---

# 6.22.5 Shenandoah GC

Shenandoah is another low-pause collector.

Like ZGC, it performs most work concurrently.

### Enable

```bash
-XX:+UseShenandoahGC
```

### Advantages

- Very low pause times
- Concurrent compaction
- Suitable for large heaps

### Best Suited For

- Large enterprise systems
- Cloud services
- Applications requiring predictable latency

---

# 6.22.6 Epsilon GC

Epsilon is a **No-Op Garbage Collector**.

It performs **no garbage collection**.

Objects are allocated until the heap becomes full.

Eventually:

```
OutOfMemoryError
```

### Enable

```bash
-XX:+UseEpsilonGC
```

### Typical Uses

- Performance benchmarking
- GC research
- Short-lived applications

It is **not intended for production use**.

---

# Garbage Collector Comparison

| Collector | Pause Time | Throughput | Heap Size | Typical Use Case |
|------------|------------|------------|-----------|------------------|
| Serial | High | Low | Small | Embedded systems |
| Parallel | Medium | Very High | Medium-Large | Batch processing |
| G1 | Low | High | Large | Enterprise applications |
| ZGC | Very Low | High | Very Large | Low-latency systems |
| Shenandoah | Very Low | High | Large | Cloud applications |
| Epsilon | None | Maximum | Any | Benchmarking |

# Choosing the Right Collector

## Small Applications

```
Heap

512 MB
```

Recommended:

```
Serial GC
```

---

## Batch Processing

Requirements:

- Maximum throughput
- CPU utilization more important than latency

Recommended:

```
Parallel GC
```

---

## Enterprise Applications

Requirements:

- Stable latency
- Good throughput
- Predictable pauses

Recommended:

```
G1 GC
```

---

## Ultra Low Latency

Requirements:

```
Pause

< 1 ms
```

Recommended:

```
ZGC

or

Shenandoah
```

# G1 Tuning Options

Target pause time:

```bash
-XX:MaxGCPauseMillis=200
```

Desired pause:

```
200 ms
```

Remember, this is a goal rather than a strict guarantee.

---

Enable String Deduplication:

```bash
-XX:+UseStringDeduplication
```

Useful for applications with many duplicate `String` instances.

---

Enable Parallel Reference Processing:

```bash
-XX:+ParallelRefProcEnabled
```

Improves processing of soft, weak, phantom, and final references during GC.

# GC Tuning Strategy

A common tuning workflow is:

```
Observe

↓

Collect GC Logs

↓

Analyze Heap Usage

↓

Identify Bottlenecks

↓

Adjust JVM Options

↓

Measure Again
```

Avoid tuning blindly.

Always base changes on measurable evidence from production-like workloads.

---

# Common Tuning Mistakes

## Increasing Heap Without Analysis

Large heaps reduce GC frequency but may increase pause duration.

Bigger is not always better.

---

## Ignoring GC Logs

Changing JVM options without examining GC logs often hides the real issue.

Always analyze before tuning.

---

## Chasing Very Small Pause Targets

Example:

```bash
-XX:MaxGCPauseMillis=5
```

Such aggressive targets may reduce throughput because the collector performs more frequent work to meet the goal.

---

## Assuming the Default Collector Is Always Optimal

G1 is an excellent general-purpose default, but specialized workloads may benefit from Parallel GC, ZGC, or Shenandoah.

---

# Production Recommendations

For most modern Java backend applications:

- Java 17 or Java 21
- G1 GC
- Appropriate heap sizing
- GC logging enabled
- Continuous monitoring
- Tune only when measurements justify changes

---

# Interview Questions

### Q1. Which garbage collector is the default in Java 17 and Java 21?

**G1 GC** is the default garbage collector.

---

### Q2. Which collector provides the lowest pause times?

Both **ZGC** and **Shenandoah GC** are designed for extremely low pause times, often below one millisecond for many workloads.

---

### Q3. Which collector provides the highest throughput?

**Parallel GC** is optimized for maximum throughput.

---

### Q4. Which collector would you choose for a Spring Boot microservice?

In most cases, **G1 GC** is the preferred choice because it offers a good balance between throughput and predictable pause times.

---

### Q5. Should JVM tuning begin by changing GC parameters?

No.

The recommended approach is:

1. Enable GC logging.
2. Measure application behavior.
3. Analyze the collected data.
4. Tune only where evidence indicates a bottleneck.

---

# Key Takeaways

- Different garbage collectors optimize different performance characteristics.
- G1 is the default collector for modern Java versions and suits most enterprise applications.
- Parallel GC prioritizes throughput, while ZGC and Shenandoah prioritize ultra-low latency.
- Proper heap sizing and GC logging are fundamental before attempting JVM tuning.
- Effective JVM tuning is an iterative, measurement-driven process rather than trial and error.
