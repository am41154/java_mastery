# 6.14 Serial Garbage Collector – The Simplest Garbage Collector

### Why This Matters

The **Serial Garbage Collector** is the oldest and simplest garbage collector in the HotSpot JVM.

Although it is rarely used for modern enterprise applications,

understanding it is valuable because:

- It introduces fundamental GC concepts.
- Many modern collectors evolved from its design.
- It is frequently discussed in JVM interviews.
- It performs surprisingly well for small applications.

The Serial GC is the easiest collector to understand before moving on to more advanced algorithms.

---

# What Is the Serial Garbage Collector?

As its name suggests,

the Serial GC performs garbage collection using **a single thread**.

During garbage collection:

- Only one GC thread is active.
- All application threads are paused.
- Garbage collection completes.
- Application execution resumes.

```
Application Running

↓

Stop-the-World

↓

Single GC Thread

↓

Garbage Collection

↓

Application Resumes
```

---

# Why Is It Called "Serial"?

Because all GC work happens sequentially.

```
GC Task 1

↓

GC Task 2

↓

GC Task 3

↓

Completed
```

No parallelism.

No concurrency.

Everything executes using one GC thread.

---

# Memory Areas Managed

Serial GC manages:

```
Young Generation

↓

Minor GC
```

and

```
Old Generation

↓

Full GC
```

Both operations use a single GC thread.

---

# Object Allocation

Objects follow the familiar lifecycle.

```
New Object

↓

Eden

↓

Minor GC

↓

Survivor

↓

Old Generation
```

Nothing changes compared to the generational Heap model.

Only the implementation of garbage collection differs.

---

# Minor Garbage Collection

Suppose Eden becomes full.

```
Eden Full

↓

Stop Application

↓

Single GC Thread

↓

Remove Dead Objects

↓

Move Live Objects

↓

Resume Application
```

Because only one thread performs the work,

pause time increases as the Heap grows.

---

# Full Garbage Collection

When the Old Generation requires collection:

```
Application Running

↓

Stop-the-World

↓

Single GC Thread

↓

Entire Heap Collection

↓

Resume Application
```

For large Heaps,

this pause can become significant.

---

# Visual Example

Imagine a 500 MB Heap.

```
Heap

↓

Objects

↓

Serial GC

↓

One Thread Cleans Heap
```

Now imagine:

```
50 GB Heap
```

The same single GC thread must still process all live objects.

Naturally,

collection takes much longer.

---

# Performance Characteristics

Serial GC performs well when:

- Heap size is small
- Few CPU cores are available
- Pause time is not critical

It performs poorly when:

- Heap is large
- CPU cores are abundant
- Low latency is required

---

# Advantages

Simple implementation.

```
Easy

↓

Predictable

↓

Low Overhead
```

Additional benefits:

- Minimal GC bookkeeping
- Efficient for single-core environments
- Lower CPU consumption than more advanced collectors
- Good choice for tiny applications and command-line tools

---

# Disadvantages

Major limitations include:

- Long Stop-the-World pauses
- No parallel GC work
- No concurrent execution
- Poor scalability on multicore servers

As modern servers commonly have 8, 16, 32, or more CPU cores,

Serial GC leaves most processing power unused during garbage collection.

---

# Enabling Serial GC

Serial GC can be selected using:

```bash
java -XX:+UseSerialGC
```

Example:

```bash
java -XX:+UseSerialGC -jar app.jar
```

---

# Typical Use Cases

Serial GC is suitable for:

- Small desktop applications
- Educational examples
- Unit tests
- Lightweight command-line utilities
- Embedded JVM deployments
- Applications with very small Heaps

It is generally **not recommended** for high-throughput backend services.

---

# Spring Boot Perspective

Consider a Spring Boot microservice:

```
Heap

↓

4 GB
```

Traffic:

```
15,000 Requests/Second
```

Serial GC would likely produce noticeable GC pauses because every collection stops the application while a single thread processes the Heap.

Modern Spring Boot services typically benefit from collectors such as G1, ZGC, or Shenandoah depending on workload characteristics.

---

# Production Perspective

Suppose an enterprise application uses:

```
Heap

↓

32 GB
```

During Full GC:

```
Application Stops

↓

Single GC Thread

↓

Entire Heap Scanned

↓

Application Resumes
```

The pause may last several seconds,

making the collector unsuitable for latency-sensitive systems.

---

# Comparison with Parallel GC

| Feature | Serial GC | Parallel GC |
|----------|-----------|-------------|
| GC Threads | 1 | Multiple |
| CPU Utilization | Low | High |
| Throughput | Lower | Higher |
| Pause Time | Longer | Shorter (for the same Heap) |
| Best For | Small applications | Large batch workloads |

Parallel GC improves performance by utilizing multiple CPU cores,

which we'll study in the next section.

# Interview Focus

### Question

What is the Serial Garbage Collector?

### Strong Senior-Level Answer

> The Serial Garbage Collector performs all garbage collection work using a single GC thread. During both Minor and Full GC, all application threads are paused. It has minimal overhead and works well for small applications, but its long Stop-the-World pauses make it unsuitable for modern, large-scale server applications.

---

### Question

When would you choose Serial GC?

A good answer includes:

- Small Heap
- Single-core or resource-constrained environments
- Command-line tools
- Small desktop applications
- Testing and educational environments

---

# Common Mistakes

❌ Assuming Serial GC is obsolete and never used.

❌ Using Serial GC for large production servers.

❌ Expecting good scalability on multicore hardware.

❌ Confusing "serial" with sequential application execution—the application itself can still be multithreaded; the **GC work** is what uses a single thread.

---

# Engineering Insight

A common misconception is:

> "Serial GC is slow."

A more accurate statement is:

> **Serial GC is extremely efficient for small workloads but scales poorly as Heap size and CPU count increase.**

For a tiny command-line utility with a 128 MB Heap,

Serial GC may actually outperform more sophisticated collectors because it has very little management overhead.

---

# Engineering Takeaways

- Serial GC uses one GC thread.
- All garbage collection is Stop-the-World.
- It has minimal overhead.
- It works well for small applications.
- It scales poorly on modern multicore servers.

---

# Quick Revision

✓ Single GC thread.

✓ Stop-the-World for every collection.

✓ Simple implementation.

✓ Best suited for small Heaps.

✓ Rarely appropriate for enterprise backend systems.

---

# Senior Interview Challenge

A developer forces every JVM in the company to use:

```bash
-XX:+UseSerialGC
```

including:

- Spring Boot microservices
- Kafka consumers
- Payment services
- REST APIs

The production servers each have:

- 32 CPU cores
- 64 GB RAM

### Would you support this decision? Why or why not?

**Expected Discussion Points**

- No.
- Serial GC cannot take advantage of available CPU cores during garbage collection.
- Stop-the-World pauses become increasingly expensive as the Heap grows.
- Throughput and latency will likely degrade compared to collectors designed for multicore systems.
- Modern server workloads generally benefit from G1, ZGC, Shenandoah, or, for throughput-focused workloads, Parallel GC.

Senior engineers choose the garbage collector based on **workload characteristics, hardware resources, and latency requirements**, not because one collector is universally "best."
