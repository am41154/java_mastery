# 6.21 JVM Garbage Collection Logging and Analysis

Understanding **how the JVM performs Garbage Collection** is just as important as understanding how the collectors work.

In production environments, GC logs are one of the first things engineers analyze when applications experience:

- High latency
- Slow response times
- Frequent OutOfMemoryError
- CPU spikes
- Long Stop-The-World pauses
- Memory leaks

GC logs provide detailed information about every garbage collection event.

---

# Why GC Logs Matter

Imagine a production application that suddenly becomes slow.

Without GC logs, you might only observe:

- CPU utilization is high.
- Response times have increased.
- Memory usage appears unstable.

With GC logs, you can determine:

- Which collector is running
- How frequently GC occurs
- How long each pause lasts
- Whether Full GC is happening
- Heap usage before and after collection
- Allocation rate
- Promotion rate
- Whether the application is approaching OutOfMemoryError

GC logs are one of the most valuable JVM diagnostic tools.

---

# Enabling GC Logging

## Java 8

```bash
-XX:+PrintGCDetails
-XX:+PrintGCDateStamps
-Xloggc:gc.log
```

Example:

```bash
java \
-XX:+PrintGCDetails \
-XX:+PrintGCDateStamps \
-Xloggc:gc.log \
-jar app.jar
```

---

## Java 9+

Java 9 introduced the **Unified JVM Logging Framework**.

GC logging is now enabled using:

```bash
-Xlog:gc
```

Example:

```bash
java -Xlog:gc -jar app.jar
```

More detailed logging:

```bash
-Xlog:gc*
```

Include timestamps:

```bash
-Xlog:gc*:time
```

Write logs to a file:

```bash
-Xlog:gc*:file=gc.log
```

Example:

```bash
java \
-Xlog:gc*:file=gc.log:time \
-jar app.jar
```

---

# Sample GC Log

Example:

```text
[2.345s][info][gc]
GC(12) Pause Young (Normal) (G1 Evacuation Pause)

256M->82M(1024M)

12.45ms
```

Let's understand every part.

---

# GC Number

```text
GC(12)
```

This means:

```text
12th Garbage Collection
```

The counter increases after every GC cycle.

---

# GC Type

Example:

```text
Pause Young
```

Meaning:

```text
Young Generation Collection
```

Other examples:

```text
Pause Full

Pause Mixed

Concurrent Mark

Remark

Cleanup
```

Each represents a different phase of the collector.

---

# Heap Before and After GC

Example:

```text
256M->82M(1024M)
```

Interpretation:

| Value | Meaning |
|--------|---------|
| 256M | Heap used before GC |
| 82M | Heap used after GC |
| 1024M | Total heap size |

Memory reclaimed:

```text
174 MB
```

---

# Pause Time

Example:

```text
12.45ms
```

Meaning:

```text
Application stopped for

12.45 milliseconds
```

This is the **Stop-The-World pause**.

---

# Young GC Example

```text
GC(25)

Pause Young

400M->150M(2G)

18ms
```

Interpretation:

- Young GC occurred
- Heap usage reduced from 400 MB to 150 MB
- Total heap size is 2 GB
- Application paused for 18 ms

This is considered healthy.

---

# Mixed GC Example

```text
GC(82)

Pause Mixed

4G->2.6G(8G)

95ms
```

Interpretation:

- Mixed Collection executed
- Both Young and selected Old regions were collected
- 1.4 GB reclaimed
- Pause lasted 95 ms

---

# Full GC Example

```text
GC(110)

Pause Full

7.8G->3.5G(8G)

4.2s
```

Interpretation:

- Full heap collection occurred
- Heap was compacted
- Application stopped for 4.2 seconds

Frequent Full GCs usually indicate:

- Heap pressure
- Poor tuning
- Memory leak
- Excessive humongous objects

---

# Concurrent Marking Example

```text
GC(55)

Concurrent Mark Cycle
```

This phase usually does **not** stop the application.

Later:

```text
GC(55)

Remark

11 ms
```

Only the Remark phase causes a brief Stop-The-World pause.

---

# Reading a Sequence of Logs

Example:

```text
Young GC

↓

Young GC

↓

Young GC

↓

Concurrent Mark

↓

Remark

↓

Cleanup

↓

Mixed GC

↓

Young GC
```

This sequence represents a healthy G1 collection cycle.

---

# Signs of a Healthy JVM

Typical characteristics include:

- Mostly Young GCs
- Occasional Mixed GCs
- Very rare Full GCs
- Short pause times
- Stable heap occupancy

Example:

```text
Young GC

15 ms

20 ms

18 ms

22 ms

Mixed GC

45 ms

Young GC

19 ms
```

This pattern is expected for a well-tuned G1 application.

---

# Warning Signs

Frequent Full GCs:

```text
Full GC

↓

Full GC

↓

Full GC
```

Indicates:

- Heap too small
- Memory leak
- Poor GC tuning

---

Very Long Pauses:

```text
Young GC

2 seconds
```

Possible causes:

- Large heap
- Excessive object allocation
- CPU contention

---

Heap Usage Never Drops

Example:

```text
4G

↓

3.9G

↓

3.8G

↓

3.9G

↓

3.95G
```

Even after repeated GCs, memory usage remains high.

Possible causes:

- Memory leak
- Large in-memory cache
- Long-lived objects

---

Too Many Young GCs

Example:

```text
Young GC

↓

Young GC

↓

Young GC

↓

Young GC

↓

Young GC
```

Possible causes:

- Very high allocation rate
- Young Generation too small

# Useful JVM GC Options

Show GC information:

```bash
-Xlog:gc
```

Detailed logging:

```bash
-Xlog:gc*
```

Include timestamps:

```bash
-Xlog:gc*:time
```

Write to file:

```bash
-Xlog:gc*:file=gc.log
```

Rotate log files:

```bash
-Xlog:gc*:file=gc.log:filecount=10,filesize=20M
```

# GC Log Analysis Tools

Several tools help visualize GC behavior.

Popular choices include:

| Tool | Purpose |
|------|----------|
| GCViewer | Offline GC log analysis |
| GCeasy | Web-based GC analysis |
| Eclipse Memory Analyzer (MAT) | Heap dump analysis |
| Java Mission Control (JMC) | JVM monitoring and profiling |
| VisualVM | JVM monitoring and heap analysis |

These tools provide graphs showing:

- Heap usage
- Pause times
- Allocation rate
- GC frequency
- Throughput

# Real Production Example

Suppose an application reports:

```text
Response Time

100 ms

↓

5 seconds
```

GC logs show:

```text
Full GC

4.5 seconds
```

Root cause:

```text
Application paused during Full GC
```

The issue is not slow business logic—it is JVM memory management.

Without GC logs, identifying the real cause would be much more difficult.

# Interview Questions

### Q1. Why should GC logging be enabled in production?

GC logs help diagnose memory-related issues such as long pauses, Full GCs, memory leaks, allocation pressure, and OutOfMemoryError conditions.

---

### Q2. How do you enable GC logging in Java 17?

```bash
-Xlog:gc*
```

or

```bash
-Xlog:gc*:file=gc.log
```

using the Unified JVM Logging framework.

---

### Q3. What does `256M->82M(1024M)` mean?

- Heap before GC: **256 MB**
- Heap after GC: **82 MB**
- Total heap size: **1024 MB**

---

### Q4. What is the first thing you check in GC logs?

Typically:

- Presence of Full GCs
- GC pause duration
- GC frequency
- Heap usage before and after GC

These provide a quick indication of overall JVM health.

---

### Q5. Which tools are commonly used to analyze GC logs?

Common tools include:

- GCViewer
- GCeasy
- Java Mission Control (JMC)
- VisualVM
- Eclipse Memory Analyzer (MAT)

---

# Key Takeaways

- GC logs are essential for diagnosing JVM performance issues.
- Java 9+ uses the Unified JVM Logging framework (`-Xlog`).
- GC logs reveal pause times, heap usage, GC frequency, and collector phases.
- Frequent Full GCs or long pauses often indicate memory pressure or tuning issues.
- GC analysis tools make it easier to visualize and interpret GC behavior.
- Every senior Java engineer should be comfortable reading and analyzing GC logs from production systems.
