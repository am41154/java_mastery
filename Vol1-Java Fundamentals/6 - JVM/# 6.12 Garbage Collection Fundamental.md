# 6.12 Garbage Collection Fundamentals – Reachability, Object Lifecycle, and GC Generations

### Why This Matters

One of Java's greatest strengths is **automatic memory management**.

Unlike languages such as C or C++, Java developers do not manually allocate and free memory.

Instead, the JVM automatically identifies objects that are no longer needed and reclaims their memory through **Garbage Collection (GC)**.

This provides several advantages:

- Reduced memory leaks caused by manual deallocation
- Improved developer productivity
- Greater application stability
- Simpler memory management

However, Garbage Collection is **not free**.

Poor object allocation patterns, memory leaks, and incorrect JVM tuning can significantly affect application performance.

Understanding how GC works is essential for every senior Java backend engineer.

---

# What Is Garbage Collection?

Garbage Collection is the process by which the JVM:

1. Identifies objects that are no longer reachable.
2. Reclaims their memory.
3. Makes that memory available for new object allocations.

Example:

```java
Employee employee = new Employee();

employee = null;
```

Memory:

```
Heap

↓

Employee Object

↓

No References

↓

Eligible for GC
```

Notice the wording:

> **Eligible for Garbage Collection**

The object is **not immediately destroyed**.

The JVM decides **when** to reclaim the memory.

---

# Why Doesn't Java Delete Objects Immediately?

Imagine the JVM removed objects the instant they became unreachable.

Every assignment like:

```java
employee = null;
```

would require immediate memory cleanup,

creating significant overhead.

Instead,

the JVM batches cleanup operations,

making memory management far more efficient.

---

# Object Lifecycle

Every object follows a lifecycle.

```
Object Created

↓

Used

↓

No Longer Reachable

↓

Eligible for GC

↓

Memory Reclaimed
```

The time between becoming unreachable and being collected is determined by the Garbage Collector.

---

# Reachability

Garbage Collection is based on **reachability**, not variable scope alone.

Consider:

```java
Employee employee =
        new Employee();
```

Initially:

```
employee

↓

Employee Object
```

The object is reachable.

Later:

```java
employee = null;
```

```
No References

↓

Employee Object
```

Now the object is unreachable.

---

# Reachability Graph

The JVM begins from a set of **GC Roots**.

```
GC Roots

↓

Static Variables

↓

Thread Stacks

↓

JNI References

↓

Live Objects
```

Objects reachable from these roots remain alive.

Everything else becomes eligible for collection.

---

# What Are GC Roots?

Typical GC Roots include:

- Local variables in active Stack Frames
- Static fields
- Active threads
- JNI references
- Internal JVM references

The JVM starts from these roots and traverses the object graph.

---

# Example

```java
Employee employee =
        new Employee();

Department department =
        new Department();

employee.setDepartment(department);
```

Graph:

```
GC Root

↓

employee

↓

Employee

↓

Department
```

Both objects remain alive because they are reachable.

---

# Circular References

A common misconception is:

> "Circular references cause memory leaks."

Not in Java.

Example:

```java
Employee employee =
        new Employee();

Department department =
        new Department();

employee.department = department;

department.employee = employee;

employee = null;

department = null;
```

Memory:

```
Employee

↔

Department
```

Although the objects reference each other,

nothing references them from a GC Root.

Therefore,

the JVM can safely collect both objects.

---

# Generational Garbage Collection

Modern JVMs divide the Heap into generations.

```
Heap

↓

Young Generation

↓

Old Generation
```

Young Generation contains:

```
Eden

↓

Survivor 0

↓

Survivor 1
```

This design is based on a simple observation:

> **Most Java objects die young.**

---

# Young Generation

Examples of short-lived objects:

- Request DTOs
- Response DTOs
- Temporary Strings
- JSON objects
- Local collections

These objects typically disappear within milliseconds.

Minor Garbage Collection focuses on reclaiming these objects quickly.

---

# Old Generation

Objects that survive multiple Minor GCs are promoted.

Examples:

- Spring Beans
- Singleton services
- Configuration objects
- Long-lived caches
- Connection pools

These objects are collected less frequently.

---

# Minor vs Major vs Full GC

There are different types of garbage collection events.

| GC Type | Memory Area | Typical Purpose |
|----------|-------------|-----------------|
| Minor GC | Young Generation | Reclaim short-lived objects |
| Major GC | Old Generation | Reclaim long-lived objects (collector-dependent terminology) |
| Full GC | Entire Heap (and sometimes additional areas depending on the JVM) | Complete garbage collection cycle |

> **Note:** The terms *Major GC* and *Full GC* are often used differently depending on the JVM implementation and garbage collector. In practice, modern tools usually report the specific collector event rather than relying solely on these generic names.

---

# When Does GC Run?

Garbage Collection is **not** triggered every time an object becomes unreachable.

Instead,

the JVM decides when collection is appropriate.

Common triggers include:

- Eden becoming full
- Memory pressure
- Allocation failures
- Collector-specific heuristics

Applications cannot rely on a predictable collection schedule.

---

# Can Developers Force GC?

Java provides:

```java
System.gc();
```

However,

this is **only a request**.

The JVM is free to:

- Ignore it
- Delay it
- Perform a collection later

Production applications should not depend on `System.gc()` for memory management.

---

# Stop-the-World (STW)

Most garbage collection activities require application threads to pause.

```
Application Running

↓

GC Starts

↓

Application Paused

↓

Memory Reclaimed

↓

Application Resumes
```

This pause is called a:

> **Stop-the-World (STW) Pause**

Modern collectors strive to minimize these pauses,

but they cannot eliminate them entirely.

---

# Spring Boot Perspective

Consider a REST API handling:

```
20,000 Requests/Second
```

Each request creates:

- Request DTO
- Response DTO
- Validation objects
- JSON objects
- Temporary collections

Millions of objects may be created every minute.

Most become unreachable immediately after the request completes.

Minor GC efficiently reclaims these short-lived objects,

allowing the application to continue allocating new ones.

---

# Production Perspective

A microservice experiences:

- Increased response latency
- High CPU usage
- Frequent GC activity

Investigation reveals:

```
Millions of Temporary Objects

↓

Frequent Minor GC

↓

CPU Time Spent Collecting Garbage
```

The problem is **not the Garbage Collector itself**.

The root cause is excessive object allocation.

Reducing unnecessary object creation often improves performance more effectively than changing GC settings.

# Interview Focus

### Question

How does the JVM determine whether an object can be garbage collected?

### Strong Senior-Level Answer

> The JVM uses reachability analysis. It starts from GC Roots such as active thread stacks, static fields, and JNI references, then traverses the object graph. Objects that are not reachable from any GC Root become eligible for garbage collection.

---

### Question

Does Java use reference counting?

### Strong Senior-Level Answer

> No. Modern JVMs use reachability analysis rather than simple reference counting. This approach correctly handles circular references, which reference counting alone cannot reclaim.

---

# Common Mistakes

❌ Assuming objects are destroyed immediately after becoming unreachable.

❌ Believing `System.gc()` always performs a garbage collection.

❌ Confusing eligibility for GC with immediate memory reclamation.

❌ Assuming circular references always cause memory leaks.

---

# Engineering Insight

Many developers assume that frequent Garbage Collection is inherently bad.

In reality,

frequent **Minor GC** events can be perfectly healthy.

A well-tuned application may perform many fast Minor GCs while maintaining excellent throughput.

Performance problems arise when:

- GC pauses become long.
- Object allocation rate is excessive.
- The Old Generation fills too quickly.
- Memory leaks prevent reclamation.

Experienced engineers evaluate **GC impact**, not simply **GC frequency**.

---

# Engineering Takeaways

- Garbage Collection is based on reachability analysis.
- Unreachable objects become eligible for collection.
- Most Java objects are short-lived.
- The JVM manages memory automatically but not instantly.
- Efficient applications reduce unnecessary object allocation rather than relying solely on JVM tuning.

# Senior Interview Challenge

A production service shows:

- Heap utilization around **60%**
- Minor GC every **3–4 seconds**
- Full GC is extremely rare
- Application latency remains low
- Throughput is stable

### Should you tune the Garbage Collector?

**Expected Discussion Points**

- Not necessarily.
- Frequent Minor GCs are expected in applications with high allocation rates.
- Healthy throughput and low pause times indicate the JVM is functioning efficiently.
- GC tuning should be driven by measurable performance issues, not by GC frequency alone.
- First analyze allocation rate, pause times, and application behavior before modifying JVM parameters.

Senior engineers optimize **evidence-based performance bottlenecks**, not normal JVM behavior.
