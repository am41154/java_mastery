# 6.7 Stack Memory – Method Execution, Stack Frames, and `StackOverflowError`

### Why This Matters

When developers discuss JVM memory,

they often focus entirely on the Heap.

However, every Java method you write executes using another critical memory area:

> **The Java Stack**

The Stack is responsible for:

- Method execution
- Local variables
- Method parameters
- Intermediate calculations
- Return addresses

Unlike the Heap,

the Stack is **thread-specific** and is **not managed by the Garbage Collector**.

Understanding Stack memory is essential for diagnosing:

- `StackOverflowError`
- Deep recursion problems
- Thread memory usage
- Method execution flow

---

# What Is the Java Stack?

Every Java thread owns its own Stack.

```
Thread 1

↓

Stack
```

```
Thread 2

↓

Stack
```

```
Thread 3

↓

Stack
```

Stacks are **never shared** between threads.

This isolation allows each thread to execute independently.

---

# What Does the Stack Store?

The Stack stores:

- Method calls
- Local variables
- Method parameters
- Return values
- References to Heap objects

It **does not** store the objects themselves.

---

# Example

```java
public void process() {

    Employee employee =
            new Employee();

}
```

Memory layout:

```
Stack

↓

employee (reference)

↓

Heap

↓

Employee Object
```

The variable `employee` lives on the Stack.

The actual `Employee` object lives in the Heap.

---

# Stack Frames

Whenever a method is called,

the JVM creates a **Stack Frame**.

Example:

```java
public void process() {

    calculate();

}
```

Execution:

```
process()

↓

calculate()
```

Stack:

```
+------------------+
| calculate()      |
+------------------+
| process()        |
+------------------+
```

The most recently invoked method is always at the top of the Stack.

---

# Method Return

After:

```java
calculate();
```

finishes,

its Stack Frame is removed.

```
Before Return

+------------------+
| calculate()      |
+------------------+
| process()        |
+------------------+
```

↓

```
After Return

+------------------+
| process()        |
+------------------+
```

This automatic cleanup makes Stack memory extremely efficient.

---

# Nested Method Calls

Example:

```java
main()

↓

service()

↓

repository()

↓

database()
```

Stack:

```
+------------------+
| database()       |
+------------------+
| repository()     |
+------------------+
| service()        |
+------------------+
| main()           |
+------------------+
```

This is known as the **Call Stack**.

---

# Local Variables

Example:

```java
void calculate() {

    int salary = 100;

    int bonus = 20;

}
```

Both variables are stored inside the method's Stack Frame.

When the method exits,

these variables disappear automatically.

No garbage collection is required.

---

# Method Parameters

Parameters are also stored in the Stack Frame.

Example:

```java
calculate(int salary)
```

Stack Frame:

```
salary

↓

100
```

Each method invocation receives its own independent parameters.

---

# Stack vs Heap

Consider:

```java
Employee employee =
        new Employee();
```

Memory:

```
Stack

employee

↓

Heap

Employee Object
```

Important distinction:

| Stack | Heap |
|--------|------|
| Stores references | Stores objects |
| Thread-specific | Shared |
| Automatically cleaned | Garbage collected |
| Very fast | Slightly slower |

---

# Recursive Calls

Consider:

```java
void calculate() {

    calculate();

}
```

Execution:

```
calculate()

↓

calculate()

↓

calculate()

↓

calculate()

↓

...
```

Each call creates a new Stack Frame.

Eventually:

```
Stack Full

↓

StackOverflowError
```

---

# Why Does `StackOverflowError` Occur?

The Stack has a limited size.

Each recursive call consumes additional memory.

Eventually,

no space remains for another Stack Frame.

Example:

```java
public void recurse() {

    recurse();

}
```

Output:

```
Exception in thread "main"

java.lang.StackOverflowError
```

The problem is not object allocation—

it is **unbounded method invocation**.

# Thread Stack Size

Each thread receives its own Stack.

Typical stack size is configurable using:

```
-Xss
```

Example:

```bash
java -Xss1m
```

Increasing the stack size allows deeper call stacks,

but also increases memory consumed per thread.

Choosing an appropriate value depends on application characteristics.

---

# Heap vs Stack Lifecycle

```
Method Call

↓

Stack Frame Created

↓

Method Returns

↓

Stack Frame Removed
```

Object lifecycle:

```
Object Created

↓

Heap

↓

Garbage Collection

↓

Memory Reclaimed
```

Notice:

- Stack memory is released immediately.
- Heap memory is reclaimed later by the Garbage Collector.

---

# Spring Boot Perspective

A typical HTTP request executes through multiple layers:

```
Controller

↓

Service

↓

Repository

↓

JDBC Driver

↓

Database
```

Each layer adds a Stack Frame.

When the request completes,

all Stack Frames are removed automatically.

Only Heap objects that are still referenced remain.

---

# Production Perspective

A production application crashes with:

```
java.lang.StackOverflowError
```

Investigation reveals:

```java
class Node {

    void process() {

        process();

    }

}
```

The issue is an unintended recursive call.

Increasing Heap memory (`-Xmx`) has no effect because the failure occurs in the Stack.

The correct solution is to eliminate the recursion or redesign the algorithm.

# Interview Focus

### Question

What is stored in the Java Stack?

### Strong Senior-Level Answer

> The Java Stack is thread-specific and stores Stack Frames for active method invocations. Each Stack Frame contains local variables, method parameters, intermediate results, return information, and references to Heap objects. Objects themselves are allocated in the Heap, while the Stack manages the execution state of methods.

---

# Common Mistakes

❌ Believing objects are stored in the Stack.

❌ Confusing `StackOverflowError` with `OutOfMemoryError`.

❌ Increasing Heap size to fix stack-related problems.

❌ Assuming Stack memory is garbage collected.

---

# Engineering Insight

One of the easiest ways to distinguish Heap and Stack problems is by the exception:

```
StackOverflowError

↓

Deep or infinite recursion
```

```
OutOfMemoryError

↓

Heap or Metaspace exhaustion
```

These errors originate from different memory regions and require different troubleshooting strategies.

---

# Engineering Takeaways

- Every thread owns its own Stack.
- Method calls create Stack Frames.
- Local variables and parameters live in the Stack.
- Objects live in the Heap.
- `StackOverflowError` usually indicates excessive recursion or very deep call chains.

---

# Quick Revision

✓ Stack is thread-specific.

✓ Stack stores method execution data.

✓ Objects are not stored in the Stack.

✓ Stack Frames are created and destroyed automatically.

✓ Infinite recursion leads to `StackOverflowError`.

---

# Senior Interview Challenge

A recursive algorithm works correctly for small inputs but fails in production when processing a deeply nested data structure.

The application throws:

```
java.lang.StackOverflowError
```

### What solutions would you consider?

**Expected Discussion Points**

- Verify whether recursion depth is proportional to input size.
- Replace recursion with an iterative solution using an explicit stack (`Deque` or `Stack`) if appropriate.
- Consider increasing the thread stack size (`-Xss`) only after understanding the algorithm's behavior.
- Avoid treating the problem as a Heap issue.
- Profile the application to determine the maximum call depth and redesign if necessary.

Senior engineers recognize that increasing `-Xss` may delay the failure but does not address the underlying algorithmic issue if recursion is effectively unbounded.
