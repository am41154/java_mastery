# 6.3 Class Loader Subsystem – How the JVM Loads Classes

### Why This Matters

When you execute:

```bash
java -jar order-service.jar
```

your application does **not** immediately begin executing the `main()` method.

Before a single line of your code runs, the JVM must locate, load, verify, and initialize thousands of classes.

For a typical Spring Boot application, this may include:

- Java Runtime classes
- Spring Framework classes
- Third-party libraries
- Your application classes

A moderately sized microservice can load **10,000–30,000 classes** during startup.

Understanding how this process works helps explain:

- Slow application startup
- `ClassNotFoundException`
- `NoClassDefFoundError`
- Classpath issues
- Plugin architectures
- Spring Boot startup behavior

---

# What Is a Class Loader?

A **Class Loader** is a JVM component responsible for loading `.class` files into memory.

It performs tasks such as:

- Locating class files
- Reading bytecode
- Creating runtime class objects
- Ensuring each class is loaded only once

Without the Class Loader,

the JVM cannot execute Java code.

---

# Class Loading Process

Suppose the application starts with:

```java
public class Main {

    public static void main(String[] args) {

        Employee employee =
                new Employee();

    }

}
```

The JVM performs:

```
Load Main.class

↓

Load Employee.class

↓

Load Object.class

↓

Load String.class

↓

Continue Execution
```

Classes are loaded **on demand**.

If a class is never referenced,

it may never be loaded.

---

# The Three Built-in Class Loaders

Modern JVMs use three primary class loaders.

```
Bootstrap Class Loader

↓

Platform Class Loader

↓

Application Class Loader
```

Each has a different responsibility.

---

# Bootstrap Class Loader

The Bootstrap Class Loader is the parent of all class loaders.

It loads core Java runtime classes.

Examples:

```
java.lang.Object

java.lang.String

java.util.List

java.util.HashMap
```

These classes are required by almost every Java application.

Characteristics:

- Implemented by the JVM (native code)
- Loads core JDK classes
- Highest level in the hierarchy

---

# Platform Class Loader

The Platform Class Loader loads platform modules introduced with the Java Platform Module System (JPMS).

Examples include:

- `java.sql`
- `java.xml`
- `java.management`

These are standard Java platform APIs but not part of the minimal core runtime.

---

# Application Class Loader

The Application Class Loader loads classes from the application's classpath or module path.

Examples:

```
com.company.Employee

com.company.OrderService

com.company.Main
```

It is the loader developers interact with most frequently.

---

# Class Loader Hierarchy

```
          Bootstrap
               │
         Platform
               │
       Application
               │
      Your Application
```

Notice the hierarchy.

Each loader delegates to its parent before attempting to load a class itself.

---

# Parent Delegation Model

One of the most important JVM concepts is the **Parent Delegation Model**.

Suppose your application requests:

```java
String.class
```

The Application Class Loader asks:

```
Can my parent load this?
```

↓

Platform Class Loader asks:

```
Can my parent load this?
```

↓

Bootstrap Class Loader loads:

```
java.lang.String
```

The request then returns down the hierarchy.

---

# Why Delegation Exists

Imagine an application contains:

```java
package java.lang;

public class String {

}
```

Without delegation,

your fake `String` class could replace the real one.

This would be a major security risk.

Instead,

the Bootstrap Class Loader always loads trusted core classes first.

This prevents accidental or malicious replacement of JDK classes.

---

# Visualizing Delegation

```
Application Loader

↓

Platform Loader

↓

Bootstrap Loader

↓

Class Found?

↓

Yes → Return Class

↓

No → Child Attempts Loading
```

Parent loaders always receive the first opportunity.

# The Class Loading Lifecycle

Loading a class involves several stages.

```
Loading

↓

Linking

↓

Initialization
```

Let's examine each.

---

# Step 1 – Loading

The Class Loader:

- Locates the `.class` file
- Reads the bytecode
- Creates an internal runtime representation of the class

At this stage,

methods are **not** yet executed.

---

# Step 2 – Linking

Linking has three phases:

```
Verification

↓

Preparation

↓

Resolution
```

### Verification

Checks that the bytecode is valid and safe.

---

### Preparation

Allocates memory for static variables and assigns default values.

Example:

```java
static int counter;
```

At preparation:

```
counter = 0
```

---

### Resolution

Converts symbolic references into direct runtime references.

The JVM connects classes together.

---

# Step 3 – Initialization

Now static initializers execute.

Example:

```java
class Employee {

    static {

        System.out.println("Loaded");

    }

}
```

Output:

```
Loaded
```

Static fields receive their explicit values during this phase.

Example:

```java
static int count = 100;
```

Only after initialization is the class considered fully ready.

---

# Lazy Class Loading

Classes are loaded only when needed.

Example:

```java
class ReportGenerator {

}
```

If `ReportGenerator` is never referenced,

its class may never be loaded.

Lazy loading reduces startup overhead and memory consumption.

---

# Spring Boot Perspective

Spring Boot relies heavily on the Class Loader.

During startup it loads:

- Spring Core
- Spring Context
- Spring Beans
- Spring MVC
- Jackson
- Hibernate
- Logging frameworks
- Your application classes

The large number of classes explains why startup time increases as applications grow.

Technologies such as Spring Boot DevTools and custom class loaders also use class loading mechanisms to support features like hot reloading.

---

# Production Perspective

Suppose a developer removes a dependency from the build but forgets to remove its usage.

The application compiles successfully because another module still references the class.

At runtime:

```
java.lang.NoClassDefFoundError
```

appears.

The class loader cannot locate the required class during execution.

Understanding the difference between compile-time and runtime class loading is crucial when diagnosing these issues.

---

# Interview Focus

### Question

What is the Parent Delegation Model?

### Strong Senior-Level Answer

> The Parent Delegation Model ensures that a class loader first delegates class loading requests to its parent before attempting to load the class itself. This guarantees that core Java classes are loaded by trusted parent class loaders, prevents duplicate class loading, and improves both security and consistency across the JVM.

---

# Common Mistakes

❌ Confusing `ClassNotFoundException` with `NoClassDefFoundError`.

❌ Assuming every class is loaded during JVM startup.

❌ Ignoring the parent delegation mechanism.

❌ Attempting to override core JDK classes.

---

# Engineering Insight

Class loading is often invisible until something goes wrong.

When diagnosing startup failures, dependency conflicts, or plugin issues, experienced engineers ask questions such as:

- Which class loader loaded this class?
- Is the class present on the runtime classpath?
- Was the class successfully initialized?
- Is there more than one version of the same class?

These questions frequently lead to the root cause of complex production issues.

---

# Engineering Takeaways

- The Class Loader loads `.class` files into the JVM.
- Java uses Bootstrap, Platform, and Application Class Loaders.
- Parent Delegation improves security and avoids duplicate loading.
- Classes are loaded lazily when first needed.
- Loading, Linking, and Initialization are distinct phases.

---

# Quick Revision

✓ Classes are loaded on demand.

✓ Bootstrap loads core Java classes.

✓ Platform loads Java platform modules.

✓ Application Class Loader loads application classes.

✓ Parent Delegation is central to JVM class loading.

# Senior Interview Challenge

A Spring Boot application starts successfully on one server but fails on another with:

```
java.lang.NoClassDefFoundError:
com.company.PaymentService
```

The JAR file is identical on both servers.

### What JVM and deployment-related areas would you investigate?

**Expected Discussion Points**

- Runtime classpath differences
- Missing dependent JARs
- Corrupted deployment artifacts
- Different Java versions or module configurations
- Custom class loaders (application server, plugin framework)
- Class initialization failures caused by static blocks or missing native libraries

Senior engineers recognize that not every "class loading" problem is caused by a missing class. Sometimes the class is found but fails during initialization, resulting in similar runtime symptoms.
