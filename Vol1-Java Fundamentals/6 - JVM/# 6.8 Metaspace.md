# 6.8 Metaspace – Class Metadata, Dynamic Class Loading, and `OutOfMemoryError: Metaspace`

### Why This Matters

When developers encounter:

```text
java.lang.OutOfMemoryError: Metaspace
```

their first instinct is often to increase the Heap size.

Unfortunately,

that usually **does not solve the problem**.

Why?

Because **Metaspace is not part of the Java Heap**.

Metaspace stores **class metadata**, not application objects.

As modern frameworks like Spring Boot, Hibernate, ByteBuddy, CGLIB, and dynamic proxy libraries generate and load classes at runtime, understanding Metaspace has become increasingly important.

---

# What Is Metaspace?

Metaspace is the JVM memory area that stores information about loaded classes.

It contains:

- Class definitions
- Method metadata
- Field metadata
- Runtime Constant Pool
- Method bytecode information
- Reflection metadata
- Annotations
- Class Loader metadata

It **does not** store objects.

Example:

```java
class Employee {
}
```

Memory layout:

```
Metaspace

↓

Employee.class Metadata
```

```
Heap

↓

Employee Objects
```

---

# Heap vs Metaspace

| Heap | Metaspace |
|-------|-----------|
| Stores objects | Stores class metadata |
| Managed by GC | Metadata reclaimed when class loaders become unreachable |
| Contains arrays | Contains class structures |
| Shared across threads | Shared across threads |

One common misconception is:

> "Metaspace stores static variables."

Not exactly.

Static **field definitions** belong to the class metadata,

while the actual storage for static field values is managed by the JVM as part of the class's runtime data. From a practical perspective, think of Metaspace as storing **information about the class**, not ordinary object instances.

---

# Before Java 8 – PermGen

Older JVMs used:

```
Permanent Generation (PermGen)
```

Problems with PermGen included:

- Fixed maximum size
- Frequent tuning
- `OutOfMemoryError: PermGen space`
- Difficult sizing

Starting with **Java 8**,

PermGen was removed and replaced by Metaspace.

---

# Why Metaspace Is Better

Unlike PermGen,

Metaspace uses **native memory**.

Benefits:

- Dynamically grows as needed
- Better memory management
- Fewer tuning issues
- Reduced risk of premature failures

However,

it is **not unlimited**.

The JVM can still throw:

```text
OutOfMemoryError: Metaspace
```

if native memory or configured limits are exhausted.

---

# When Does Metaspace Grow?

Every time the JVM loads a new class,

Metaspace usage increases.

Example:

```
Employee.class

↓

Loaded

↓

Metaspace
```

Large enterprise applications may load:

- Spring Framework
- Hibernate
- Jackson
- Logging libraries
- JDBC drivers
- Thousands of application classes

Each contributes metadata to Metaspace.

---

# Dynamic Class Generation

Modern frameworks often generate classes dynamically.

Examples:

- Spring AOP proxies
- Hibernate proxies
- ByteBuddy
- CGLIB
- JDK Dynamic Proxies

Instead of:

```java
EmployeeService
```

the JVM may also create:

```
EmployeeService$$SpringProxy
```

Each generated class consumes additional Metaspace.

---

# Example

Suppose Spring creates:

```
OrderService

↓

OrderService$$Proxy
```

Both class definitions are loaded.

Both occupy Metaspace.

As more proxies are created,

Metaspace usage increases.

---

# Class Unloading

Can Metaspace shrink?

Yes—

but only when:

1. A Class Loader becomes unreachable.
2. The classes loaded by that Class Loader are no longer in use.
3. The JVM performs class unloading during garbage collection (collector-dependent).

Simply deleting object instances does **not** free class metadata.

---

# Class Loader Relationship

```
Application Class Loader

↓

Employee.class

↓

Metaspace
```

If the Class Loader remains alive,

its classes generally remain loaded.

This is why class loader leaks can eventually exhaust Metaspace.

---

# Class Loader Leaks

Imagine a web server repeatedly deploying an application.

```
Deploy

↓

New Class Loader

↓

Classes Loaded

↓

Old Class Loader Never Released

↓

Repeat...
```

Eventually:

```
Thousands of Loaded Classes

↓

Metaspace Full

↓

OutOfMemoryError
```

This type of leak was historically common in long-running application servers.

---

# Monitoring Metaspace

Useful tools include:

- `jcmd`
- `jmap`
- Java Flight Recorder (JFR)
- VisualVM
- Java Mission Control (JMC)

These tools help answer questions such as:

- How many classes are loaded?
- How much Metaspace is used?
- Are classes being unloaded?
- Is a Class Loader leak occurring?

We'll explore these tools later in this chapter.

---

# Spring Boot Perspective

Spring Boot applications commonly use:

- Dependency Injection
- AOP
- Dynamic Proxies
- Reflection
- Hibernate
- Jackson

These features increase Metaspace usage because they involve additional class metadata and, in some cases, runtime-generated classes.

Fortunately,

Metaspace requirements for typical Spring Boot services remain modest unless the application repeatedly loads new classes or leaks Class Loaders.

---

# Production Perspective

A server experiences:

```
java.lang.OutOfMemoryError:
Metaspace
```

Heap usage:

```
35%
```

Garbage Collection:

```
Healthy
```

Investigation reveals:

A plugin framework continuously creates new Class Loaders,

but never releases the old ones.

Heap tuning has no effect.

The actual problem is uncontrolled Metaspace growth caused by a Class Loader leak.

# Interview Focus

### Question

What is Metaspace?

### Strong Senior-Level Answer

> Metaspace is the JVM memory area that stores class metadata, including class definitions, method metadata, runtime constant pools, and reflection information. Introduced in Java 8 to replace PermGen, it uses native memory and can grow dynamically. Excessive class loading or Class Loader leaks can result in `OutOfMemoryError: Metaspace`.

---

# Common Mistakes

❌ Confusing Heap memory with Metaspace.

❌ Assuming increasing `-Xmx` fixes Metaspace errors.

❌ Believing objects are stored in Metaspace.

❌ Ignoring Class Loader leaks in long-running applications.

---

# Engineering Insight

When diagnosing JVM memory issues,

always identify **which memory region is exhausted**.

Consider these two errors:

```text
OutOfMemoryError: Java heap space
```

and

```text
OutOfMemoryError: Metaspace
```

Although both are `OutOfMemoryError`s,

they indicate **completely different root causes**.

Experienced engineers investigate:

- Heap dumps for Heap issues.
- Loaded classes and Class Loaders for Metaspace issues.

---

# Engineering Takeaways

- Metaspace stores class metadata.
- Objects are not stored in Metaspace.
- Java 8 replaced PermGen with Metaspace.
- Runtime-generated classes increase Metaspace usage.
- Class Loader leaks are a common cause of Metaspace exhaustion.

---

# Quick Revision

✓ Metaspace stores class metadata.

✓ Introduced in Java 8.

✓ Uses native memory.

✓ Dynamic proxies consume Metaspace.

✓ Class Loader leaks can cause `OutOfMemoryError: Metaspace`.

# Senior Interview Challenge

A Spring Boot application running for several weeks eventually fails with:

```text
java.lang.OutOfMemoryError: Metaspace
```

Observations:

- Heap usage remains stable.
- GC logs appear healthy.
- The application dynamically loads plugins at runtime.

### What would you investigate?

**Expected Discussion Points**

- Number of loaded classes over time.
- Whether old Class Loaders are being released.
- Plugin lifecycle and unloading.
- Dynamic proxy generation.
- Metaspace usage using JFR, `jcmd`, or VisualVM.
- JVM option `-XX:MaxMetaspaceSize` (only after understanding the root cause).

Senior engineers recognize that simply increasing the Metaspace limit may postpone the failure but does not resolve a Class Loader leak.
