# 6.15 Parallel Garbage Collector – Maximizing Throughput with Multiple GC Threads

### Why This Matters

As server hardware evolved from single-core processors to modern multicore CPUs, the limitations of the **Serial Garbage Collector** became increasingly apparent.

Imagine a production server with:

- 32 CPU cores
- 64 GB RAM

Yet during garbage collection,

only **one CPU core** performs GC work while the remaining 31 cores sit idle.

The **Parallel Garbage Collector (Parallel GC)** was designed to solve this problem by performing garbage collection using multiple threads.

Its primary goal is **maximum application throughput**, not minimal pause time.

---

# What Is the Parallel Garbage Collector?

The Parallel Garbage Collector performs Stop-the-World garbage collection,

just like the Serial GC.

The difference is:

Instead of one GC thread,

it uses **multiple GC threads**.

```
Application Running

↓

Stop-the-World

↓

Multiple GC Threads

↓

Garbage Collection

↓

Application Resumes
```

The application still pauses,

but the garbage collection work finishes faster.

---

# Serial vs Parallel

Serial GC:

```
Application

↓

Pause

↓

GC Thread 1

↓

Resume
```

Parallel GC:

```
Application

↓

Pause

↓

GC Thread 1

GC Thread 2

GC Thread 3

GC Thread 4

↓

Resume
```

The workload is divided across multiple processors.

---

# Why Is It Called "Throughput Collector"?

Parallel GC aims to maximize:

> **The percentage of total time spent doing useful application work.**

Example:

```
Application

97%

GC

3%
```

Excellent throughput.

If garbage collection completes faster,

the application spends more time processing business requests.

---

# Memory Areas Managed

Parallel GC manages the generational Heap.

```
Heap

↓

Young Generation

↓

Old Generation
```

Minor GC:

- Parallel

Full GC:

- Also parallel in modern HotSpot implementations

The application remains paused during both.

---

# Object Lifecycle

Objects continue following the same lifecycle.

```
New Object

↓

Eden

↓

Minor GC

↓

Survivor

↓

Old Generation
```

Only the garbage collection strategy changes.

---

# Minor Garbage Collection

Suppose Eden becomes full.

Instead of:

```
Single Thread

↓

Scan Eden
```

Parallel GC performs:

```
Thread 1

↓

Part of Eden
```

```
Thread 2

↓

Part of Eden
```

```
Thread 3

↓

Part of Eden
```

```
Thread 4

↓

Part of Eden
```

All threads work simultaneously,

reducing collection time.

---

# Full Garbage Collection

The same concept applies to the Old Generation.

```
Entire Heap

↓

Split Across

↓

Multiple GC Threads

↓

Collection Completes Faster
```

The pause still exists,

but it is typically shorter than with Serial GC for the same workload.

---

# CPU Utilization

Suppose a server has:

```
16 CPU Cores
```

Serial GC:

```
1 Core Busy

15 Idle
```

Parallel GC:

```
8–16 GC Threads

↓

Multiple Cores Active
```

This allows modern hardware to be used far more effectively.

---

# Advantages

Parallel GC offers:

- High throughput
- Better CPU utilization
- Faster garbage collection than Serial GC
- Good scalability for multicore servers
- Excellent performance for CPU-intensive workloads

It has been a popular choice for enterprise batch processing for many years.

---

# Disadvantages

Parallel GC still performs:

```
Stop-the-World
```

during garbage collection.

As Heap size increases,

pause durations can still become significant.

It is therefore less suitable for:

- Online trading
- Payment gateways
- Low-latency APIs
- Interactive applications

where predictable response time is more important than raw throughput.

---

# Enabling Parallel GC

Parallel GC can be selected using:

```bash
java -XX:+UseParallelGC
```

Example:

```bash
java -XX:+UseParallelGC -jar app.jar
```

---

# Typical Use Cases

Parallel GC is well suited for:

- Batch processing
- ETL pipelines
- Report generation
- Scientific computation
- Data analytics
- Long-running background jobs

These workloads value:

```
Maximum Work Completed

>

Minimal Pause Time
```

---

# Throughput vs Latency

Consider two applications.

### Batch Job

```
GC Pause

1 Second
```

Acceptable.

The job still completes quickly.

---

### Payment Service

```
GC Pause

1 Second
```

Potentially unacceptable.

Customers experience delayed transactions.

This illustrates why throughput and latency are different optimization goals.

Imagine two Spring Boot services.

```

Throughput matters most.

### REST API

Response Time Critical
```

# Production Perspective

A financial institution runs:

```
Overnight ETL

↓

500 Million Records
```

The process is expected to finish before business hours.

A few hundred milliseconds—or even seconds—of GC pause time has little business impact.

Maximizing throughput is the priority,

making Parallel GC an excellent choice.

# Comparison with Serial GC

| Feature | Serial GC | Parallel GC |
|----------|-----------|-------------|
| GC Threads | 1 | Multiple |
| Stop-the-World | Yes | Yes |
| CPU Utilization | Low | High |
| Throughput | Moderate | High |
| Scalability | Poor | Good |
| Best For | Small applications | Batch workloads |

---

# Comparison with G1

| Parallel GC | G1 GC |
|--------------|--------|
| Optimizes throughput | Balances throughput and latency |
| Longer pauses | Shorter, more predictable pauses |
| Simpler algorithm | More sophisticated collector |
| Excellent for batch processing | Excellent for general-purpose server applications |

# Interview Focus

### Question

How does Parallel GC differ from Serial GC?

### Strong Senior-Level Answer

> Both collectors perform Stop-the-World garbage collection. The key difference is that Serial GC uses a single GC thread, whereas Parallel GC performs garbage collection using multiple threads. This significantly improves throughput on multicore systems but does not eliminate application pauses.

---

### Question

When would you choose Parallel GC?

A strong answer includes:

- Batch processing
- ETL jobs
- Analytics workloads
- CPU-intensive background processing
- Applications where throughput is more important than latency

---

# Common Mistakes

❌ Assuming Parallel GC performs concurrent garbage collection.

❌ Believing Parallel GC eliminates Stop-the-World pauses.

❌ Using Parallel GC for latency-sensitive REST APIs without measuring pause times.

❌ Assuming more GC threads always improve performance regardless of workload.

---

# Engineering Insight

A common misconception is:

> "Parallel GC is outdated because G1 is the default."

Not necessarily.

For applications where:

- User-facing latency is unimportant
- CPU utilization is acceptable
- Maximum throughput is the primary objective

Parallel GC can still outperform more sophisticated collectors due to its simpler design and lower runtime overhead.

This is why it remains a relevant choice for many enterprise batch workloads.

---

# Engineering Takeaways

- Parallel GC uses multiple GC threads.
- It still performs Stop-the-World collection.
- It is optimized for throughput rather than latency.
- It scales effectively on multicore hardware.
- It remains a strong option for batch and analytics workloads.

# Senior Interview Challenge

A team runs a nightly data-processing job that:

- Processes **800 million records**
- Runs on a **32-core server**
- Has a **48 GB Heap**
- Has no interactive users
- Must finish before **6:00 AM**

They are considering switching from Parallel GC to ZGC because they heard it has lower pause times.

### Would you recommend the change?

**Expected Discussion Points**

- Not automatically.
- Pause time is not the primary business concern.
- Throughput and completion time are more important than ultra-low latency.
- Parallel GC may provide equal or better throughput with lower complexity.
- Benchmark both collectors using production-like workloads before making a decision.

Senior engineers optimize for **business objectives**, not collector popularity. A low-latency collector is not necessarily the fastest choice for throughput-oriented workloads.
