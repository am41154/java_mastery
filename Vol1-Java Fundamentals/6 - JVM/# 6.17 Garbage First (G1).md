# 6.17 Garbage First (G1) Garbage Collector – Architecture, Heap Regions, and Design Goals

### Why This Matters

Beginning with **Java 9**, the **Garbage First (G1) Garbage Collector** became the default garbage collector in the HotSpot JVM.

Today, most enterprise Java applications—including Spring Boot microservices running on Java 17 and Java 21—use **G1 GC** unless another collector is explicitly configured.

G1 was designed to solve several limitations of earlier collectors:

- Reduce long Stop-the-World pauses.
- Eliminate heap fragmentation through compaction.
- Scale efficiently to multi-core servers.
- Handle large heaps.
- Deliver predictable pause times.

Understanding G1 is essential for every senior Java backend engineer.

---

# Why Was G1 Introduced?

Previous collectors each had strengths and weaknesses.

| Collector | Strength | Limitation |
|-----------|----------|------------|
| Serial | Simple | Long pauses |
| Parallel | High throughput | Long pauses |
| CMS | Lower pauses | Fragmentation, Concurrent Mode Failure |

G1 attempts to combine the advantages of these collectors while minimizing their drawbacks.

---

# Why "Garbage First"?

Traditional collectors often viewed the Heap as two large continuous areas:

```
Young Generation

↓

Old Generation
```

G1 takes a different approach.

Instead of collecting memory strictly by generation,

it prioritizes regions that are expected to reclaim the **most garbage first**.

Hence the name:

> **Garbage First**

---

# Region-Based Heap

Unlike earlier collectors,

G1 divides the Heap into many equally sized regions.

```
Heap

↓

+----+----+----+----+
| R1 | R2 | R3 | R4 |
+----+----+----+----+
| R5 | R6 | R7 | R8 |
+----+----+----+----+
| R9 |R10 |R11 |R12 |
+----+----+----+----+
```

Typical region sizes range from **1 MB to 32 MB**, depending on the overall Heap size.

The JVM automatically selects an appropriate region size.

---

# Dynamic Regions

Unlike traditional collectors,

regions are **not permanently assigned**.

A region may become:

- Eden
- Survivor
- Old Generation
- Free

during different phases of application execution.

Example:

```
Region 1

↓

Eden
```

Later:

```
Region 1

↓

Free
```

The JVM dynamically manages region roles.

---

# Visualizing the Heap

Example:

```
+-----+-----+-----+-----+
| Eden| Eden| Old | Old |
+-----+-----+-----+-----+
|Surv.| Eden| Free| Old |
+-----+-----+-----+-----+
| Free|Surv.| Old | Eden|
+-----+-----+-----+-----+
```

Unlike previous collectors,

Young and Old regions are interleaved throughout the Heap.

There is no requirement that all Young regions be physically adjacent.

---

# Why Regions Improve Performance

Suppose the Heap contains:

```
100 Regions
```

Only:

```
12 Regions
```

contain large amounts of garbage.

Instead of scanning the entire Heap,

G1 focuses on those regions that provide the greatest memory recovery for the least amount of work.

This improves efficiency and helps control pause times.

---

# Predictable Pause Times

One of G1's major innovations is its pause-time target.

Developers can specify:

```text
MaxGCPauseMillis = 200
```

This is a **goal**, not a guarantee.

The JVM attempts to choose enough work to complete within approximately the requested pause time.

If more work remains,

it is deferred to future collections.

---

# Concurrent and Stop-the-World Phases

Like CMS,

G1 performs much of its work concurrently.

High-level lifecycle:

```
Young Collection

↓

Concurrent Mark

↓

Remark

↓

Cleanup

↓

Mixed Collections
```

Some phases pause the application,

while others execute concurrently.

We'll examine each phase in detail in the next chapter.

---

# Compaction

Unlike CMS,

G1 always evacuates live objects during collection.

```
Before

Object

Gap

Object

Gap

Object
```

↓

```
After

Object

Object

Object

Free Space
```

Benefits:

- No fragmentation.
- Faster allocations.
- Reduced Full GC frequency.

This is one of G1's biggest improvements over CMS.

---

# Incremental Collection

Rather than collecting the entire Heap,

G1 typically collects only selected regions.

```
100 Regions

↓

Collect 8 Regions

↓

Resume Application

↓

Collect Another Set Later
```

This incremental approach helps keep pauses shorter and more predictable.

---

# Multi-Core Optimization

Modern production servers commonly have:

- 8 cores
- 16 cores
- 32 cores
- 64 cores

G1 uses multiple GC threads where appropriate while also performing significant work concurrently with application threads.

This allows it to scale effectively on modern hardware.

---

# Enabling G1

G1 became the default collector beginning with Java 9.

It can also be enabled explicitly:

```bash
java -XX:+UseG1GC
```

Example:

```bash
java -XX:+UseG1GC -jar app.jar
```

# Production Perspective

Imagine an e-commerce platform during a flash sale.

Traffic increases dramatically,

creating millions of temporary request objects.

Instead of pausing to scan the entire Heap,

G1 identifies regions with the highest proportion of reclaimable memory and collects those first.

This targeted approach allows the application to continue serving requests with relatively stable pause times.

# Comparison with CMS

| CMS | G1 |
|-----|----|
| Mark-Sweep | Region-Based Collection |
| Heap Fragmentation | Automatic Compaction |
| Concurrent Mode Failure | Greatly Reduced Risk |
| Complex Tuning | Better Default Behavior |
| Removed | Current Default Collector |

---

# Comparison with Parallel GC

| Parallel GC | G1 |
|-------------|----|
| Throughput Focus | Balanced Throughput + Latency |
| Whole-Generation Collection | Region-Based Collection |
| Longer Pauses | More Predictable Pauses |
| Simpler Algorithm | More Adaptive Algorithm |

# Interview Focus

### Question

Why is G1 called "Garbage First"?

### Strong Senior-Level Answer

> G1 divides the Heap into regions and prioritizes collecting the regions expected to reclaim the most memory first. Instead of scanning the entire Young or Old Generation, it selects high-value regions to meet its pause-time goals efficiently.

---

### Question

What are the main advantages of G1 over CMS?

### Strong Senior-Level Answer

> G1 provides automatic heap compaction, reducing fragmentation. It uses a region-based Heap instead of a contiguous generational layout, supports predictable pause-time goals, and avoids many of the tuning challenges and fragmentation issues associated with CMS.

---

# Common Mistakes

❌ Assuming G1 completely eliminates Stop-the-World pauses.

❌ Believing `MaxGCPauseMillis` is a guaranteed maximum pause.

❌ Thinking regions are permanently assigned to Eden or Old Generation.

❌ Assuming G1 always outperforms every other collector regardless of workload.

---

# Engineering Insight

A common misconception is:

> "G1 is a low-latency collector."

A more accurate statement is:

> **G1 is a balanced collector.**

Its primary design objective is **predictable pause times while maintaining strong throughput**.

For applications requiring ultra-low latency on very large heaps, collectors such as ZGC or Shenandoah may be more appropriate.

---

# Engineering Takeaways

- G1 is the default HotSpot collector in modern Java.
- It divides the Heap into equal-sized regions.
- Regions dynamically change roles during execution.
- G1 prioritizes regions with the highest garbage density.
- Automatic compaction reduces fragmentation and improves allocation efficiency.

# Senior Interview Challenge

A Spring Boot service running on Java 21 uses the default G1 GC.

The team notices:

- Heap size: **12 GB**
- Stable throughput
- Average GC pause: **65 ms**
- Occasional pause: **180 ms**
- `MaxGCPauseMillis=100`

### Should the team conclude that G1 is malfunctioning?

**Expected Discussion Points**

- No.
- `MaxGCPauseMillis` is a tuning target, not a strict guarantee.
- Actual pause times depend on allocation rate, live data size, CPU availability, remembered-set processing, and concurrent marking progress.
- Occasional pauses above the target are expected and do not necessarily indicate a problem.
- Evaluate GC logs, allocation patterns, and application latency before making tuning decisions.

Senior engineers understand that G1 optimizes toward pause-time goals while balancing throughput and overall system efficiency—it does not guarantee hard real-time behavior.
