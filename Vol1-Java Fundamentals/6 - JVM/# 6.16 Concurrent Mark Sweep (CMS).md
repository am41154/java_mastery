# 6.16 Concurrent Mark Sweep (CMS) – The First Low-Latency Garbage Collector (Historical)

### Why This Matters

As Java applications became increasingly interactive,

developers realized that **throughput alone was no longer enough**.

Applications such as:

- Online banking
- E-commerce websites
- Trading platforms
- Enterprise web applications

required **consistent response times**.

A garbage collector that paused the application for several seconds—even if it delivered excellent throughput—was no longer acceptable.

To address this challenge, the JVM introduced the **Concurrent Mark Sweep (CMS)** collector.

CMS was revolutionary because it performed **most of its work while the application continued running**.

Although CMS has been removed from modern HotSpot JVMs, it introduced concepts that influenced today's collectors, including **G1, ZGC, and Shenandoah**.

---

# Why Was CMS Introduced?

Consider a web application:

```
Users

↓

REST API

↓

Database
```

Suppose the JVM pauses for:

```
5 Seconds
```

During this pause:

- No requests are processed.
- Response times spike.
- Users experience delays or timeouts.

For interactive applications,

long Stop-the-World pauses are often more harmful than slightly lower throughput.

---

# CMS Design Goal

Unlike the Parallel GC,

whose primary objective is:

```
Maximum Throughput
```

CMS focuses on:

```
Low Pause Time
```

It achieves this by performing much of the garbage collection concurrently with application threads.

---

# High-Level Workflow

CMS divides garbage collection into several phases.

```
Initial Mark

↓

Concurrent Mark

↓

Remark

↓

Concurrent Sweep
```

Some phases pause the application.

Others execute while the application continues processing requests.

---

# Phase 1 – Initial Mark

```
Stop-the-World
```

Purpose:

- Identify objects directly reachable from GC Roots.

```
GC Roots

↓

Direct References
```

This phase is typically very short because it examines only the initial set of root references.

---

# Phase 2 – Concurrent Mark

```
Application Running

+

GC Running
```

The collector traverses the object graph,

discovering reachable objects while the application continues executing.

This significantly reduces pause time compared to collectors that perform the entire marking phase while the application is stopped.

---

# Phase 3 – Remark

During concurrent marking,

the application may modify object references.

Therefore,

CMS performs a second Stop-the-World phase.

Purpose:

- Correct any changes that occurred during concurrent marking.
- Produce an accurate view of live objects.

```
Concurrent Changes

↓

Remark

↓

Consistent Object Graph
```

This pause is usually much shorter than a Full GC pause.

---

# Phase 4 – Concurrent Sweep

After identifying unreachable objects,

CMS reclaims their memory.

```
Application Running

+

Sweep Dead Objects
```

Again,

this work happens concurrently with the application.

---

# CMS Timeline

```
Application Running

↓

Initial Mark (STW)

↓

Concurrent Mark

↓

Remark (STW)

↓

Concurrent Sweep

↓

Application Continues
```

Notice that only two phases require Stop-the-World pauses.

---

# Major Advantage

Compared with Parallel GC,

CMS dramatically reduced pause times because most GC work occurred concurrently.

This made it attractive for:

- Web servers
- Application servers
- Enterprise Java applications

where responsiveness mattered more than maximum throughput.

---

# The Fragmentation Problem

CMS used a **mark-sweep** algorithm.

```
Object

Gap

Object

Gap

Object
```

After sweeping,

dead objects disappear,

but free space may remain fragmented.

```
Object

Free

Object

Free

Object
```

Unlike compacting collectors,

CMS did **not** normally move live objects together.

---

# Why Fragmentation Is a Problem

Imagine the Heap contains:

```
10 MB Free

5 MB Used

10 MB Free
```

Total free memory:

```
20 MB
```

Now suppose the application needs:

```
15 MB
```

Although sufficient free memory exists overall,

there is **no single contiguous block** large enough.

Allocation may fail,

forcing a more expensive Full GC.

---

# Concurrent Mode Failure

One of CMS's biggest weaknesses was:

```
Concurrent Mode Failure
```

This occurred when:

- CMS could not reclaim memory quickly enough.
- The Old Generation filled before concurrent collection completed.

When this happened,

the JVM fell back to a traditional Stop-the-World Full GC.

```
CMS Running

↓

Old Generation Fills

↓

Concurrent Mode Failure

↓

Full GC

↓

Long Pause
```

Ironically,

the collector designed to reduce pauses could occasionally trigger very long pauses.

---

# CPU Overhead

Because CMS runs concurrently,

it consumes CPU resources while the application is also executing.

```
Application Threads

+

CMS Threads
```

Benefits:

- Lower pause times.

Trade-offs:

- Higher CPU utilization.
- Reduced peak throughput.

---

# Enabling CMS (Historical)

Historically, CMS could be enabled using:

```bash
java -XX:+UseConcMarkSweepGC
```

This option is no longer supported in modern HotSpot JVMs because CMS has been removed.

---

# Why Was CMS Removed?

CMS solved many latency problems,

but it also had important limitations:

- Heap fragmentation.
- Concurrent Mode Failure.
- Complex tuning requirements.
- High maintenance cost within the JVM.
- Difficulty scaling to very large heaps.

As a result,

CMS was deprecated in Java 9 and removed in Java 14.

It was replaced by more advanced collectors,

particularly **G1 GC**.

---

# Spring Boot Perspective

Many older enterprise Spring applications running on Java 8 used CMS because:

- REST APIs required predictable response times.
- Heap sizes were moderate.
- Pause-time reduction was more important than maximum throughput.

Modern Spring Boot applications running on Java 17 or later generally use G1 GC by default instead.

---

# Production Perspective

Imagine an e-commerce application running on Java 8.

Traffic suddenly doubles during a festival sale.

CMS begins concurrent collection,

but object allocation outpaces memory reclamation.

Eventually:

```
Concurrent Mode Failure

↓

Full GC

↓

Application Pauses

↓

Checkout Requests Time Out
```

The application experiences latency spikes precisely when demand is highest.

This limitation motivated the development of newer collectors.

---

# Comparison with Parallel GC

| Feature | Parallel GC | CMS |
|----------|-------------|-----|
| Primary Goal | Throughput | Low Latency |
| Stop-the-World | Entire GC | Only selected phases |
| Concurrent Execution | No | Yes |
| Heap Compaction | Yes | Generally No |
| Fragmentation Risk | Low | High |

---

# Comparison with G1

| CMS | G1 |
|-----|----|
| Concurrent marking | Concurrent marking |
| Mark-Sweep | Region-based collection |
| Fragmentation possible | Compaction integrated |
| Removed from modern JDKs | Default collector today |
| Complex tuning | Easier default behavior |

# Interview Focus

### Question

Why was CMS introduced?

### Strong Senior-Level Answer

> CMS was designed to reduce application pause times by performing most garbage collection work concurrently with application threads. It targeted interactive applications where predictable response times were more important than maximum throughput.

---

### Question

Why was CMS eventually removed?

### Strong Senior-Level Answer

> Although CMS significantly reduced Stop-the-World pauses, it suffered from heap fragmentation, Concurrent Mode Failures, complex tuning, and increasing maintenance complexity. G1 GC addressed many of these issues while providing more predictable performance and built-in compaction, leading to CMS being removed from modern HotSpot JVMs.

---

# Common Mistakes

❌ Assuming CMS eliminated Stop-the-World pauses completely.

❌ Believing CMS compacted the Heap during normal operation.

❌ Confusing concurrent execution with parallel execution.

❌ Attempting to use CMS on modern JDK versions where it is no longer available.

---

# Engineering Insight

CMS represented a major milestone in JVM evolution.

For the first time,

the JVM demonstrated that significant portions of garbage collection could execute concurrently with application threads.

Many concepts pioneered by CMS—including concurrent marking and reduced pause-time goals—directly influenced the design of G1, ZGC, and Shenandoah.

Understanding CMS helps explain why modern collectors are structured the way they are, even though CMS itself is now obsolete.

---

# Engineering Takeaways

- CMS prioritized low pause times over maximum throughput.
- Most GC work executed concurrently with the application.
- Heap fragmentation was one of its major weaknesses.
- Concurrent Mode Failure could trigger expensive Full GCs.
- CMS has been removed and replaced by more advanced collectors.

---

# Quick Revision

✓ First major concurrent collector.

✓ Reduced Stop-the-World pauses.

✓ Used mark-sweep without regular compaction.

✓ Susceptible to fragmentation.

✓ Replaced by G1 GC.

# Senior Interview Challenge

A legacy Java 8 application still runs with CMS.

The team reports:

- Increasing Full GC frequency.
- Occasional pauses of several seconds.
- Heap utilization remains below 80%.
- GC logs show repeated **Concurrent Mode Failure** events.

### What is the likely explanation?

**Expected Discussion Points**

- CMS cannot reclaim memory quickly enough.
- The Old Generation fills before concurrent collection completes.
- Fragmentation may prevent efficient allocation.
- The JVM falls back to Stop-the-World Full GC.
- Upgrading to a modern JDK and evaluating G1 (or another current collector) is likely preferable to extensive CMS tuning.

Senior engineers recognize CMS as an important historical collector but generally recommend modern alternatives for current production systems.
