# 7.10 Local Variable Type Inference (`var`)

One of the most noticeable language enhancements introduced in **Java 10** is **Local Variable Type Inference**, commonly known as **`var`**.

Unlike languages such as JavaScript or Python, Java remains a **statically typed language**.

The `var` keyword **does not make Java dynamically typed**.

Instead, it allows the compiler to infer the variable's type at compile time, reducing boilerplate while preserving type safety.

---

# Why Was `var` Introduced?

Consider the following code.

```java
Map<String, List<Employee>> employees =
        new HashMap<String, List<Employee>>();
```

The type information is repeated multiple times.

With Java 7:

```java
Map<String, List<Employee>> employees =
        new HashMap<>();
```

Java 10 simplifies it further.

```java
var employees = new HashMap<String, List<Employee>>();
```

The compiler already knows the type.

---

# What is `var`?

`var` is **not a data type**.

It is a reserved type name that tells the compiler:

> "Infer the variable's type from the initializer."

Example:

```java
var message = "Hello";
```

Compiler infers:

```java
String message = "Hello";
```

Similarly,

```java
var number = 100;
```

becomes

```java
int number = 100;
```

Type inference happens at **compile time**, not runtime.

---

# Type Inference

Example:

```java
var list = List.of("Java", "Spring");
```

Compiler infers:

```java
List<String>
```

Another example:

```java
var employee =
        new Employee();
```

Compiler infers:

```java
Employee
```

---

# Primitive Types

```java
var age = 30;
```

Type:

```java
int
```

```java
var salary = 50000L;
```

Type:

```java
long
```

```java
var percentage = 99.5;
```

Type:

```java
double
```

The compiler follows normal Java type inference rules.

---

# Objects

```java
var map =
        new HashMap<String, Integer>();
```

Type:

```java
HashMap<String, Integer>
```

Even though the variable is declared with `var`, it remains strongly typed.

---

# Generic Types

Without `var`:

```java
Map<String,
    List<Employee>> employees =
        new HashMap<>();
```

With `var`:

```java
var employees =
        new HashMap<String, List<Employee>>();
```

The inferred type is exactly the same.

---

# `var` with Streams

Without `var`:

```java
Stream<String> stream =
        names.stream();
```

With `var`:

```java
var stream =
        names.stream();
```

The inferred type:

```java
Stream<String>
```

---

# `var` in Loops

Traditional loop:

```java
for (Employee employee : employees) {

}
```

Using `var`:

```java
for (var employee : employees) {

}
```

Enhanced for-loops become cleaner, especially when working with generic types.

---

# `var` in Traditional for Loop

```java
for (var i = 0; i < 10; i++) {

}
```

Compiler infers:

```java
int
```

---

# `var` in try-with-resources

Java 11 extended `var` usage.

Example:

```java
try (var reader =
         Files.newBufferedReader(path)) {

}
```

The compiler infers the exact resource type.

---

# `var` with Lambda Parameters (Java 11)

Before Java 11:

```java
(a, b) -> a + b
```

Java 11:

```java
(var a, var b) -> a + b
```

Why?

To support annotations.

Example:

```java
(@NotNull var name) ->
        name.toUpperCase()
```

All lambda parameters must consistently use either `var` or inferred types.

---

# Where `var` Can Be Used

✔ Local variables

```java
var name = "Java";
```

✔ Enhanced for-loops

```java
for (var employee : employees)
```

✔ Traditional loops

```java
for (var i = 0; i < 10; i++)
```

✔ try-with-resources

```java
try (var reader = ...)
```

✔ Lambda parameters (Java 11)

```java
(var x, var y) -> x + y
```

---

# Where `var` Cannot Be Used

## Instance Variables

Invalid:

```java
class Employee {

    var name = "John";
}
```

Compilation error.

---

## Method Parameters

Invalid:

```java
void process(var employee) {

}
```

---

## Return Type

Invalid:

```java
public var findEmployee() {

}
```

---

## Without Initialization

Invalid:

```java
var value;
```

Compiler cannot infer the type.

---

## Initialized with null

Invalid:

```java
var value = null;
```

Compiler error:

```text
Cannot infer type
```

`null` has no standalone type.

---

# `var` is Still Strongly Typed

Example:

```java
var age = 25;

age = "Java";
```

Compilation error:

```text
Required:

int

Found:

String
```

This demonstrates that Java remains statically typed.

---

# Readability Considerations

Good example:

```java
var employees =
        repository.findAll();
```

The variable name and method clearly indicate the type.

Poor example:

```java
var result = process(data);
```

The type is not obvious.

Readers must inspect the method definition.

---

# When Should `var` Be Used?

Excellent use cases:

```java
var list =
        List.of("A", "B");
```

```java
var stream =
        employees.stream();
```

```java
var iterator =
        map.entrySet().iterator();
```

These declarations reduce verbosity without sacrificing readability.

---

# When Should `var` Be Avoided?

Avoid:

```java
var x = calculate();
```

```java
var y = service.execute();
```

The inferred type is unclear.

Explicit typing improves readability in such cases.

---

# Performance

A common misconception:

> "`var` makes Java faster."

False.

The compiler replaces `var` with the inferred type during compilation.

Generated bytecode is effectively identical.

Example:

```java
var name = "Java";
```

and

```java
String name = "Java";
```

produce equivalent bytecode.

---

# Best Practices

- Use `var` when the type is obvious from the initializer.
- Avoid `var` when it hides important type information.
- Prefer meaningful variable names.
- Do not overuse `var` simply because it is available.
- Remember that readability is more important than reducing a few keystrokes.

---

# Interview Questions

### Q1. Is `var` a keyword or a data type?

`var` is a reserved type name used for local variable type inference. It is not a new data type.

---

### Q2. Does `var` make Java dynamically typed?

No.

The compiler infers the type at compile time, and the variable remains strongly and statically typed.

---

### Q3. Where can `var` be used?

`var` can be used for:

- Local variables
- Enhanced `for` loops
- Traditional `for` loops
- `try-with-resources`
- Lambda parameters (Java 11)

It cannot be used for fields, method parameters, or return types.

---

### Q4. Can `var` be initialized with `null`?

No.

The compiler cannot infer a type from `null`, so the declaration is invalid.

---

### Q5. Does using `var` improve performance?

No.

`var` is a compile-time feature only. The generated bytecode is effectively the same as if the explicit type had been written.

---

# Real-World Usage in Spring Boot

Consider a service method:

Without `var`:

```java
List<EmployeeDto> employees =
        employeeService.findActiveEmployees();
```

With `var`:

```java
var employees =
        employeeService.findActiveEmployees();
```

This is acceptable because the variable name and method make the type obvious.

However:

```java
var result =
        execute();
```

is discouraged because it obscures the type and intent.

---

# Key Takeaways

- `var` reduces boilerplate by allowing the compiler to infer local variable types.
- Java remains statically typed; `var` does not introduce dynamic typing.
- `var` is limited to local contexts such as variables, loops, and `try-with-resources`.
- Use `var` when it improves readability, not merely to shorten code.
- Well-chosen variable names and clear initializers are essential when using `var`.
- Senior engineers prioritize code readability and maintainability over minimal syntax.
