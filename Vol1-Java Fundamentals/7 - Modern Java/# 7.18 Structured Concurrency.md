# 7.18 Structured Concurrency & Scoped Values (Java 21)

Java applications have traditionally used threads as independent units of execution.

While this model works, it often leads to code that is difficult to manage when multiple concurrent tasks belong to a **single logical operation**.

Consider an HTTP request that simultaneously:

- Calls a User Service
- Calls an Order Service
- Calls a Payment Service

These three tasks belong to one business operation.

Traditional concurrency APIs do not explicitly model this relationship.

Java 21 introduces **Structured Concurrency** to solve this problem.

Together with **Scoped Values**, it provides a modern approach to writing concurrent applications that are:

- Easier to understand
- Easier to cancel
- Easier to debug
- Safer than ThreadLocal
- Ideal for Virtual Threads

> **Note:** Structured Concurrency and Scoped Values are **Preview Features** in Java 21. They require enabling preview features during compilation and execution.

---

# The Problem with Traditional Concurrency

Suppose an API needs information from three microservices.

Traditional approach:

```java
Future<User> user =
        executor.submit(...);

Future<Order> order =
        executor.submit(...);

Future<Payment> payment =
        executor.submit(...);
```

Later:

```java
user.get();

order.get();

payment.get();
```

Problems:

- Independent lifecycles
- Difficult cancellation
- Error propagation is manual
- Resource cleanup is complex
- Parent-child relationships are unclear

---

# What is Structured Concurrency?

Structured Concurrency treats multiple concurrent tasks as **one unit of work**.

Instead of:

```text
Parent

↓

Creates Threads

↓

Threads Continue Independently
```

It becomes:

```text
Parent Task

├── Child Task 1

├── Child Task 2

└── Child Task 3

↓

Wait

↓

Complete Together
```

The parent controls the entire lifecycle.

---

# Why Was It Introduced?

Goals:

- Simplify concurrent programming
- Improve error handling
- Automatic cancellation
- Better observability
- Easier debugging
- Better Virtual Thread integration

---

# StructuredTaskScope

The central API is:

```java
StructuredTaskScope
```

Example:

```java
try (var scope =
        new StructuredTaskScope.ShutdownOnFailure()) {

    ...
}
```

The scope owns every child task.

When the scope exits:

- Tasks complete
- Resources are released
- Cancellation occurs automatically when necessary

---

# Forking Tasks

Example:

```java
try (var scope =
        new StructuredTaskScope.ShutdownOnFailure()) {

    var user =
            scope.fork(
                    () -> userService.findUser()
            );

    var order =
            scope.fork(
                    () -> orderService.findOrder()
            );

    scope.join();

    scope.throwIfFailed();

}
```

Each `fork()` creates a child task, typically executed using a Virtual Thread.

---

# join()

```java
scope.join();
```

Waits until:

- All tasks complete
- Or the scope is cancelled

Unlike calling `Future.get()` repeatedly, a single call synchronizes the entire operation.

---

# throwIfFailed()

```java
scope.throwIfFailed();
```

If any child task fails:

- Exception is propagated
- Remaining tasks are cancelled (depending on the scope type)

This greatly simplifies error handling.

---

# ShutdownOnFailure

Most common scope.

Behavior:

```text
Task A

↓

Success

Task B

↓

Exception

↓

Cancel Task C

↓

Throw Exception
```

This avoids unnecessary work after a failure.

---

# ShutdownOnSuccess

Another useful scope.

Example:

```java
StructuredTaskScope.ShutdownOnSuccess
```

Behavior:

```text
Task A

↓

Success

↓

Cancel Remaining Tasks
```

Useful when multiple services provide equivalent results.

For example:

- Primary cache
- Secondary cache
- Remote API

Return the first successful result.

---

# Example: Aggregating Services

Suppose an endpoint needs:

- User
- Orders
- Recommendations

```java
try (var scope =
        new StructuredTaskScope.ShutdownOnFailure()) {

    var user =
            scope.fork(() ->
                    userService.get());

    var orders =
            scope.fork(() ->
                    orderService.get());

    var recommendations =
            scope.fork(() ->
                    recommendationService.get());

    scope.join();

    scope.throwIfFailed();

    return new Dashboard(
            user.get(),
            orders.get(),
            recommendations.get()
    );
}
```

Simple.

Readable.

Safe.

# Benefits over CompletableFuture

Traditional:

```text
thenApply()

↓

thenCompose()

↓

exceptionally()

↓

handle()

↓

whenComplete()
```

Large pipelines become difficult to read.

Structured Concurrency:

```text
fork()

↓

join()

↓

Result
```

Much closer to sequential programming.

---

# Cancellation

Suppose:

```text
Task A

↓

Success

Task B

↓

Failure

↓

Task C Running
```

Structured Concurrency automatically cancels Task C.

Traditional `ExecutorService` code requires explicit cancellation.

---

# Relationship with Virtual Threads

Virtual Threads make creating child tasks inexpensive.

Structured Concurrency provides a way to manage those tasks.

Think of them as complementary features.

```text
Virtual Threads

↓

Cheap Concurrency

+

Structured Concurrency

↓

Organized Concurrency
```

# What are Scoped Values?

`ThreadLocal` has long been used to share contextual information.

Example:

```java
ThreadLocal<UserContext>
```

Problems:

- Mutable
- Memory leaks
- Difficult cleanup
- Poor compatibility with large numbers of threads

Scoped Values provide a safer alternative.

---

# ThreadLocal Example

```java
ThreadLocal<String> USER =
        new ThreadLocal<>();
```

Set:

```java
USER.set("Alice");
```

Later:

```java
USER.get();
```

Cleanup:

```java
USER.remove();
```

Forgetting `remove()` may cause memory leaks in thread pools.

---

# Scoped Values

Create:

```java
static final ScopedValue<String>
        USER =
        ScopedValue.newInstance();
```

Bind:

```java
ScopedValue.where(
        USER,
        "Alice"
).run(() -> {

    service.process();
});
```

Read:

```java
USER.get();
```

No explicit cleanup required.

---

# Why Scoped Values?

Unlike `ThreadLocal`:

- Immutable
- Lexically scoped
- Automatically cleaned up
- Better suited for Virtual Threads
- No accidental leakage across unrelated tasks

---

# Real-World Example

Suppose every request has:

- User ID
- Correlation ID
- Tenant ID

Instead of:

```java
ThreadLocal
```

Use:

```java
ScopedValue
```

Every child task automatically inherits the value within the scope.

Perfect for:

- Logging
- Tracing
- Security
- Multi-tenancy

---

# Comparison

| ThreadLocal | ScopedValue |
|--------------|-------------|
| Mutable | Immutable |
| Manual cleanup | Automatic |
| Thread lifetime | Lexical scope |
| Easy to leak | Safer |
| Designed before Virtual Threads | Designed with Virtual Threads in mind |

---

# Best Practices

- Use Structured Concurrency for related concurrent tasks.
- Prefer `ShutdownOnFailure` for service aggregation.
- Use `ShutdownOnSuccess` for racing equivalent tasks.
- Use Scoped Values instead of `ThreadLocal` where appropriate.
- Keep child tasks independent and side-effect free.
- Combine Structured Concurrency with Virtual Threads for maximum scalability.

---

# Common Mistakes

## Using Structured Concurrency for Independent Tasks

If tasks are unrelated, there may be no benefit in grouping them under a single scope.

---

## Treating Scoped Values as Mutable State

Scoped Values are intended for immutable contextual information, not shared mutable data.

---

## Ignoring Preview Status

Both APIs are preview features in Java 21 and require enabling preview features during compilation and execution.

Their APIs may evolve in future Java releases.

# Interview Questions

### Q1. What problem does Structured Concurrency solve?

It groups related concurrent tasks into a single logical unit, simplifying lifecycle management, cancellation, and error handling.

---

### Q2. What is the purpose of `StructuredTaskScope`?

It manages child tasks created within a scope and coordinates their execution, completion, cancellation, and failure propagation.

---

### Q3. What is the difference between `ShutdownOnFailure` and `ShutdownOnSuccess`?

- `ShutdownOnFailure` cancels remaining tasks when one task fails.
- `ShutdownOnSuccess` cancels remaining tasks when the first successful result is obtained.

---

### Q4. Why were Scoped Values introduced?

To provide a safer, immutable, and automatically managed alternative to `ThreadLocal`, particularly for applications using Virtual Threads.

---

### Q5. Are Structured Concurrency and Scoped Values standard features in Java 21?

No.

They are preview features in Java 21 and require preview support to be enabled.

# Modern Java 21 Example

```java
try (var scope =
        new StructuredTaskScope.ShutdownOnFailure()) {

    var user =
            scope.fork(userService::getUser);

    var orders =
            scope.fork(orderService::getOrders);

    var payment =
            scope.fork(paymentService::getPayment);

    scope.join();

    scope.throwIfFailed();

    return new Dashboard(

            user.get(),

            orders.get(),

            payment.get()
    );
}
```

This example demonstrates:

- Virtual Threads
- Structured Concurrency
- Automatic cancellation
- Simplified error handling
- Improved readability

---

# Key Takeaways

- Structured Concurrency organizes related concurrent tasks into a single managed scope.
- `StructuredTaskScope` simplifies task creation, synchronization, cancellation, and failure handling.
- Virtual Threads provide inexpensive concurrency, while Structured Concurrency provides structured coordination.
- Scoped Values offer an immutable, automatically managed alternative to `ThreadLocal`.
- Together, these features represent the future direction of concurrent programming in Java and are increasingly relevant for modern enterprise applications.
