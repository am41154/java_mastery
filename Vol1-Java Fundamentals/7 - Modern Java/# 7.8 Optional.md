# 7.8 Optional

One of the most controversial yet widely used features introduced in **Java 8** is the `Optional` class.

Its primary goal is to **reduce the occurrence of `NullPointerException` (NPE)** and make the absence of a value **explicit**.

`Optional` encourages developers to think about missing values instead of assuming an object is always present.

Today, `Optional` is commonly used in:

- Spring Data JPA repositories
- Stream API
- REST APIs
- Domain modeling
- Functional programming

---

# Why Was Optional Introduced?

Before Java 8, methods commonly returned `null` to indicate the absence of a value.

Example:

```java
public Employee findEmployee(int id) {

    if (employeeExists(id)) {
        return employee;
    }

    return null;
}
```

Usage:

```java
Employee employee = service.findEmployee(1);

System.out.println(employee.getName());
```

If the employee is not found:

```text
NullPointerException
```

The caller must always remember to perform a null check.

```java
if (employee != null) {
    System.out.println(employee.getName());
}
```

This often leads to repetitive and error-prone code.

---

# What Is Optional?

`Optional<T>` is a container object that may or may not contain a non-null value.

Think of it as:

```text
Optional

↓

Value Present ?

↓

Yes / No
```

Instead of returning `null`, a method returns:

- A value wrapped inside an `Optional`
- An empty `Optional`

---

# Creating Optional Objects

## Optional.of()

Used when the value is guaranteed to be non-null.

```java
Optional<String> optional =
        Optional.of("Java");
```

Access:

```java
System.out.println(optional.get());
```

Output:

```text
Java
```

Passing `null` to `Optional.of()` results in a `NullPointerException`.

---

# Optional.ofNullable()

Allows the value to be `null`.

```java
String value = null;

Optional<String> optional =
        Optional.ofNullable(value);
```

Result:

```text
Optional.empty
```

This is the most commonly used factory method.

---

# Optional.empty()

Creates an empty Optional.

```java
Optional<String> optional =
        Optional.empty();
```

---

# Checking for Values

## isPresent()

```java
Optional<String> optional =
        Optional.of("Java");

if (optional.isPresent()) {
    System.out.println(optional.get());
}
```

Output:

```text
Java
```

While valid, this style is often discouraged because it resembles traditional null checks.

---

# isEmpty()

Introduced in Java 11.

```java
if (optional.isEmpty()) {
    System.out.println("No Value");
}
```

Equivalent to:

```java
!optional.isPresent()
```

but more readable.

---

# get()

Returns the wrapped value.

```java
Optional<String> optional =
        Optional.of("Spring");

String value = optional.get();
```

If the Optional is empty:

```text
NoSuchElementException
```

Therefore, `get()` should generally be avoided unless the presence of a value is guaranteed.

---

# orElse()

Returns the wrapped value if present; otherwise, returns a default value.

```java
String name =
        optional.orElse("Unknown");
```

Example:

```java
Optional<String> optional =
        Optional.empty();

System.out.println(
        optional.orElse("Default")
);
```

Output:

```text
Default
```

---

# orElseGet()

Uses a `Supplier` to lazily create the default value.

```java
String value =
        optional.orElseGet(
            () -> fetchFromDatabase()
        );
```

Difference:

```java
orElse(value)
```

The value is created eagerly.

```java
orElseGet(supplier)
```

The supplier executes only if needed.

---

# orElse() vs orElseGet()

Example:

```java
Optional<String> optional =
        Optional.of("Java");
```

```java
optional.orElse(expensiveOperation());
```

`expensiveOperation()` executes even though the Optional already contains a value.

With:

```java
optional.orElseGet(
        () -> expensiveOperation()
);
```

The expensive operation is **not executed**.

This makes `orElseGet()` preferable when computing the default value is costly.

---

# orElseThrow()

Throws an exception if no value is present.

```java
Employee employee =
repository.findById(id)
          .orElseThrow();
```

Custom exception:

```java
Employee employee =
repository.findById(id)
          .orElseThrow(
              () -> new EmployeeNotFoundException()
          );
```

Very common in Spring Boot services.

---

# ifPresent()

Executes an action only when a value exists.

```java
optional.ifPresent(
        System.out::println
);
```

Equivalent to:

```java
if (optional.isPresent()) {
    System.out.println(optional.get());
}
```

but more concise.

---

# ifPresentOrElse() (Java 9)

```java
optional.ifPresentOrElse(
    System.out::println,
    () -> System.out.println("No Value")
);
```

Output:

```text
No Value
```

if the Optional is empty.

---

# map()

Transforms the contained value while keeping it wrapped in an `Optional`.

Example:

```java
Optional<String> name =
        Optional.of("Alice");

Optional<String> upper =
        name.map(String::toUpperCase);
```

Output:

```text
Optional[ALICE]
```

---

# flatMap()

Useful when the mapping function already returns an `Optional`.

Without `flatMap()`:

```java
Optional<Optional<Employee>>
```

With `flatMap()`:

```java
Optional<Employee>
```

Example:

```java
optional.flatMap(
        repository::findManager
);
```

---

# filter()

Keeps the value only if it matches a condition.

```java
Optional<String> result =
Optional.of("Java")
        .filter(s -> s.length() > 3);
```

Output:

```text
Optional[Java]
```

Another example:

```java
Optional.of("Go")
        .filter(s -> s.length() > 3);
```

Output:

```text
Optional.empty
```

---

# Chaining Optional Operations

Example:

```java
String city =
repository.findById(id)
          .map(Employee::getAddress)
          .map(Address::getCity)
          .orElse("Unknown");
```

Pipeline:

```text
Employee

↓

Address

↓

City

↓

Default
```

This eliminates multiple nested null checks.

---

# Optional in Spring Data JPA

Repository:

```java
Optional<Employee> findById(Long id);
```

Usage:

```java
Employee employee =
repository.findById(id)
          .orElseThrow(
              EmployeeNotFoundException::new
          );
```

This is the recommended pattern for service-layer code.

---

# Common Mistakes

## Using Optional as a Field

Avoid:

```java
class Employee {

    Optional<String> name;
}
```

Fields should typically use normal types.

Optional is primarily intended for **method return types**.

---

## Using Optional as a Method Parameter

Avoid:

```java
void process(Optional<Employee> employee)
```

Prefer:

```java
void process(Employee employee)
```

or use method overloading if optional behavior is required.

---

## Calling get()

Avoid:

```java
optional.get();
```

without first ensuring a value is present.

Prefer:

- `orElse()`
- `orElseGet()`
- `orElseThrow()`
- `ifPresent()`

---

## Returning null from an Optional Method

Incorrect:

```java
Optional<Employee> find() {

    return null;
}
```

Correct:

```java
return Optional.empty();
```

Returning `null` defeats the entire purpose of `Optional`.

---

# Performance Considerations

`Optional` is a lightweight wrapper, but it still introduces object allocation.

Therefore:

Good use cases:

- Public APIs
- Repository methods
- Service methods

Avoid using `Optional`:

- For every getter
- As entity fields
- Inside performance-critical loops

---

# Best Practices

- Use `Optional` as a return type when a value may legitimately be absent.
- Prefer `orElseThrow()` for mandatory values.
- Use `orElseGet()` when creating the default value is expensive.
- Avoid `get()` unless the value is guaranteed to exist.
- Do not use `Optional` for fields, setters, or method parameters.
- Never return `null` from a method that returns `Optional`.

---

# Interview Questions

### Q1. Why was `Optional` introduced?

To make the absence of a value explicit and reduce `NullPointerException` by encouraging safer handling of missing values.

---

### Q2. What is the difference between `Optional.of()` and `Optional.ofNullable()`?

- `Optional.of()` requires a non-null value and throws a `NullPointerException` if given `null`.
- `Optional.ofNullable()` accepts both null and non-null values, returning an empty `Optional` when the value is null.

---

### Q3. What is the difference between `orElse()` and `orElseGet()`?

- `orElse()` evaluates its argument eagerly, even if the Optional already contains a value.
- `orElseGet()` invokes the supplied `Supplier` only when the Optional is empty, making it more efficient for expensive fallback computations.

---

### Q4. Why is `Optional.get()` discouraged?

Because it throws a `NoSuchElementException` when the Optional is empty. Safer alternatives include `orElse()`, `orElseThrow()`, and `ifPresent()`.

---

### Q5. Should `Optional` be used as an entity field or method parameter?

Generally no. `Optional` is intended primarily for method return types, not for fields, setters, or method parameters.

---

# Key Takeaways

- `Optional` represents the possible absence of a value in a type-safe way.
- It helps reduce `NullPointerException` and encourages explicit handling of missing values.
- Use `Optional.ofNullable()` when the value may be null.
- Prefer `orElseThrow()`, `orElseGet()`, and `ifPresent()` over `get()`.
- Use `Optional` as a return type rather than for fields or method parameters.
- When used appropriately, `Optional` improves API clarity and leads to more robust Java applications.
