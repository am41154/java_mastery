# 7.12 Records (Java 16)

**Records** were officially introduced in **Java 16** to simplify the creation of **immutable data carrier classes**.

Prior to Records, Java developers had to write a large amount of boilerplate code for simple objects whose primary purpose was to hold data.

Records eliminate this boilerplate while preserving Java's strong typing and object-oriented principles.

They are now widely used in:

- REST API DTOs
- Request/Response models
- Configuration objects
- Event messages
- Immutable domain models
- Kafka messages

---

# Why Were Records Introduced?

Consider a simple DTO.

Before Java 16:

```java
public class Employee {

    private final int id;
    private final String name;

    public Employee(int id, String name) {
        this.id = id;
        this.name = name;
    }

    public int getId() {
        return id;
    }

    public String getName() {
        return name;
    }

    @Override
    public boolean equals(Object o) {
        ...
    }

    @Override
    public int hashCode() {
        ...
    }

    @Override
    public String toString() {
        ...
    }
}
```

Most of this code is boilerplate.

With Records:

```java
public record Employee(
        int id,
        String name
) {
}
```

That's all.

The compiler generates the remaining code.

---

# What Does the Compiler Generate?

For the record:

```java
public record Employee(
        int id,
        String name
) {
}
```

The compiler automatically generates:

- Private final fields
- Canonical constructor
- Accessor methods
- `equals()`
- `hashCode()`
- `toString()`

Equivalent to writing all of that manually.

---

# Record Components

```java
public record Employee(
        int id,
        String name,
        double salary
) {
}
```

The parameters inside the record declaration are called **record components**.

Each component becomes:

- A private final field
- A constructor parameter
- An accessor method

Example:

```java
Employee employee =
        new Employee(
                1,
                "Alice",
                50000
        );
```

Access values:

```java
employee.id();

employee.name();

employee.salary();
```

Notice:

There are **no getters**.

The accessor method has the same name as the component.

---

# Immutability

Records are immutable.

Example:

```java
Employee employee =
        new Employee(
                1,
                "Alice"
        );
```

This is not allowed:

```java
employee.name = "Bob";
```

Compilation error.

Fields are implicitly:

```java
private final
```

---

# toString()

Automatically generated.

```java
System.out.println(employee);
```

Output:

```text
Employee[id=1, name=Alice]
```

---

# equals()

Records implement value-based equality.

Example:

```java
Employee e1 =
        new Employee(1, "Alice");

Employee e2 =
        new Employee(1, "Alice");
```

```java
System.out.println(
        e1.equals(e2)
);
```

Output:

```text
true
```

Unlike ordinary classes, no manual implementation is required.

---

# hashCode()

Automatically generated using all record components.

Useful when records are stored in:

- HashMap
- HashSet
- Cache keys

---

# Canonical Constructor

The compiler generates a constructor automatically.

Equivalent to:

```java
public Employee(
        int id,
        String name
) {
    this.id = id;
    this.name = name;
}
```

You may explicitly define it.

```java
public record Employee(
        int id,
        String name
) {

    public Employee(
            int id,
            String name
    ) {
        this.id = id;
        this.name = name;
    }
}
```

---

# Compact Constructor

Often validation is required.

Example:

```java
public record Employee(
        int id,
        String name
) {

    public Employee {

        if (id <= 0) {
            throw new IllegalArgumentException(
                    "Invalid ID"
            );
        }

        Objects.requireNonNull(name);
    }
}
```

Notice:

There is **no parameter list**.

The compiler supplies it automatically.

---

# Additional Methods

Records can contain normal methods.

Example:

```java
public record Rectangle(
        int width,
        int height
) {

    public int area() {
        return width * height;
    }
}
```

Usage:

```java
Rectangle rectangle =
        new Rectangle(10,20);

rectangle.area();
```

Output:

```text
200
```

---

# Static Members

Records can contain static fields and methods.

```java
public record Employee(
        int id,
        String name
) {

    public static Employee empty() {
        return new Employee(0, "");
    }
}
```

# Nested Records

Records may be nested.

```java
public class Order {

    public record Item(
            String name,
            int quantity
    ) {

    }

```

# Records and Interfaces

Records cannot extend classes because they already extend `java.lang.Record`.

However, they can implement interfaces.

Example:

```java
public record Employee(
        int id,
        String name
) implements Serializable {

}
```

---

# Records Cannot Extend Classes

Invalid:

```java
public record Employee(...)
        extends Person
```

Compilation error.

Reason:

Every record implicitly extends:

```java
java.lang.Record
```

---

# Records vs POJO

| POJO | Record |
|------|--------|
| Boilerplate | Minimal |
| Mutable (usually) | Immutable |
| Manual equals/hashCode | Generated |
| Manual constructor | Generated |
| Manual toString | Generated |
| Getters | Component accessors |

---

# Records vs Lombok @Data

Lombok:

```java
@Data
class Employee {

}
```

Generates:

- Getters
- Setters
- equals()
- hashCode()
- toString()

Records:

```java
record Employee(...)
```

Advantages:

- Language feature
- No annotation processor
- Truly immutable
- Cleaner bytecode

# Records with JPA

Records are **not recommended** for JPA entities.

Reasons:

JPA requires:

- Mutable state
- No-argument constructor
- Proxy generation
- Lazy loading

Records violate these assumptions.

Instead:

Use Records for:

- DTOs
- API models
- Projections

Use classes for:

- JPA Entities

---

# Performance

Records are not inherently faster than classes.

Benefits come from:

- Less code
- Fewer mistakes
- Better readability
- Immutable design

The JVM optimizes records similarly to normal classes.

---

# Reflection

Records can be detected at runtime.

```java
Employee.class.isRecord();
```

Returns:

```text
true
```

Retrieve components:

```java
Employee.class.getRecordComponents();
```

Useful for frameworks.

---

# Best Practices

- Use Records for immutable data carriers.
- Prefer Records for DTOs and API models.
- Keep business logic minimal inside Records.
- Perform validation in compact constructors.
- Avoid using Records as JPA entities.
- Use meaningful component names since they become the public API.

---

# Common Mistakes

## Expecting Setters

```java
employee.setName("Bob");
```

Compilation error.

Records are immutable.

---

## Adding Mutable Fields

```java
record Employee(List<String> skills)
```

The reference is immutable, but the list itself may still be mutable.

Use defensive copies if necessary:

```java
public record Employee(List<String> skills) {

    public Employee {
        skills = List.copyOf(skills);
    }
}
```

---

## Using Records for Rich Domain Models

Records are ideal for data carriers, not for entities with extensive mutable state and behavior.

# Interview Questions

### Q1. Why were Records introduced?

To reduce boilerplate for immutable data carrier classes by automatically generating constructors, accessors, `equals()`, `hashCode()`, and `toString()`.

---

### Q2. Are Records immutable?

Yes.

Their components become private final fields, and there are no setters.

---

### Q3. Can Records extend another class?

No.

Every Record implicitly extends `java.lang.Record`, so it cannot extend any other class.

However, Records can implement interfaces.

---

### Q4. Should Records be used as JPA entities?

Generally no.

JPA relies on mutable state, no-argument constructors, proxies, and lazy loading, all of which conflict with the design of Records.

---

### Q5. What is a Compact Constructor?

A compact constructor allows validation and normalization of record components without explicitly declaring the parameter list. The compiler automatically supplies the parameters.

---

# Key Takeaways

- Records are a language feature introduced in Java 16 for immutable data carriers.
- They dramatically reduce boilerplate by generating constructors, accessors, `equals()`, `hashCode()`, and `toString()`.
- Records are ideal for DTOs, REST models, configuration objects, and event payloads.
- Use compact constructors for validation and defensive copying.
- Avoid using Records for JPA entities; prefer them for immutable application boundaries.
- Records improve readability, maintainability, and align well with modern Java and Spring Boot development.
