# 7.20 Modern Java Best Practices & Interview Summary

The Java language has evolved significantly from Java 8 to Java 21.

Modern Java is no longer just about writing object-oriented code—it emphasizes:

- Simplicity
- Immutability
- Expressiveness
- Safety
- Scalability
- Performance
- Maintainability

A Senior Backend Engineer is expected not only to know these features but also to understand **when to use them and when not to use them**.

This chapter summarizes the evolution of modern Java and provides practical recommendations for enterprise development.

---

| Version | Major Features |

# Modern Java Design Principles

Prefer writing code that is:

- Immutable
- Readable
- Thread-safe
- Functional where appropriate
- Object-oriented where appropriate
- Easy to test
- Easy to maintain

The goal is **clarity**, not cleverness.

---

# Prefer Immutability

Instead of:

```java
class Employee {

    private String name;

    public void setName(String name) {

        this.name = name;
    }
}
```

Prefer:

```java
public record Employee(

        Long id,

        String name

) {
}
```

Benefits:

- Thread-safe by default
- Easier to reason about
- Fewer bugs
- Better support for caching and concurrency

---

# Prefer Records for DTOs

Good candidates:

- REST Requests
- REST Responses
- Kafka Events
- Configuration Objects
- Value Objects
- Projection Models

Avoid Records for:

- JPA Entities
- Mutable domain models

---

# Prefer Sealed Classes for Closed Hierarchies

Instead of allowing unrestricted inheritance:

```java
class Payment
```

Prefer:

```java
sealed interface Payment
```

Benefits:

- Better compiler checks
- Stronger domain modeling
- Safer pattern matching

---

# Prefer Pattern Matching

Instead of:

```java
if (obj instanceof Employee) {

    Employee e = (Employee) obj;
}
```

Prefer:

```java
if (obj instanceof Employee employee) {

}
```

Cleaner.

Safer.

Less boilerplate.

---

# Prefer Switch Expressions

Instead of:

```java
switch (...)
```

using temporary variables and `break`, prefer:

```java
return switch (...) {

    ...
};
```

Benefits:

- No fall-through
- Better readability
- Exhaustive handling

---

# Use Text Blocks for Structured Content

Instead of:

```java
String sql =
    "SELECT ...";
```

Prefer:

```java
String sql = """
SELECT ...
FROM ...
""";
```

Ideal for:

- SQL
- JSON
- XML
- HTML
- GraphQL

---

# Use Virtual Threads for I/O

Good candidates:

- REST APIs
- Database access
- Message brokers
- File operations
- Network communication

Avoid expecting significant gains for CPU-intensive computations.

---

# Structured Concurrency

Prefer grouping related concurrent tasks.

Instead of:

```java
Future

Future

Future
```

Prefer:

```java
StructuredTaskScope
```

when using modern Java.

---

# Replace ThreadLocal When Appropriate

Legacy:

```java
ThreadLocal
```

Modern:

```java
ScopedValue
```

for immutable contextual information (once the feature becomes standard).

---

# Functional Programming Guidelines

Use Streams when they improve readability.

Good:

```java
employees.stream()

         .filter(Employee::isActive)

         .map(Employee::name)

         .toList();
```

Avoid deeply nested pipelines.

Bad:

```java
stream()

.filter()

.map()

.flatMap()

.peek()

.collect(...)

...
```

Readability always wins.

---

# Optional Best Practices

Good:

```java
Optional<Employee>
```

Bad:

```java
Optional<List<Employee>>
```

Bad:

```java
class Employee {

    Optional<String> name;
}
```

`Optional` is intended primarily for **method return types**, not fields, parameters, or collections.

---

# Stream Best Practices

Use Streams for:

- Filtering
- Mapping
- Aggregation
- Grouping
- Partitioning

Avoid Streams for:

- Complex stateful algorithms
- Heavy side effects
- Performance-critical nested loops without profiling

---

# Exception Handling

Prefer:

```java
Specific Exceptions
```

Avoid:

```java
catch(Exception e)
```

Log meaningful context and propagate exceptions appropriately.

---

# Modern Collection APIs

Prefer:

```java
List.of()

Set.of()

Map.of()
```

for immutable collections.

Instead of:

```java
new ArrayList<>();
```

when mutation is unnecessary.

---

# Date & Time API

Always prefer:

```java
java.time
```

Avoid:

```java
java.util.Date

java.util.Calendar
```

unless interacting with legacy APIs.

---

# Concurrency Recommendations

Use:

- `ExecutorService`
- `CompletableFuture`
- Virtual Threads (Java 21)
- Structured Concurrency (where appropriate)

Avoid:

- Creating threads manually for every task
- Shared mutable state
- Excessive synchronization

---

# Spring Boot Recommendations

Modern Spring Boot applications should:

- Use constructor injection.
- Prefer immutable configuration classes.
- Use Records for DTOs.
- Use `java.time`.
- Use Virtual Threads (when appropriate and supported).
- Leverage Spring Boot's auto-configuration instead of unnecessary custom configuration.

# Coding Style Recommendations

A Senior Backend Engineer should write code that is:

- Self-explanatory
- Consistent
- Small
- Testable
- Immutable where possible
- Easy to review

Avoid overly clever implementations.

Readable code scales better than clever code.

# Common Interview Traps

## Overusing Streams

Streams are not always the best solution.

Sometimes a simple loop is more readable.

---

## Misusing Optional

Avoid:

```java
Optional.get()
```

without checking presence.

Prefer:

```java
orElse()

orElseThrow()

ifPresent()
```

---

## Using Records as JPA Entities

Not recommended.

JPA expects mutable entities and supports lazy-loading through proxies.

---

## Assuming Virtual Threads Replace Reactive Programming

Virtual Threads simplify many blocking workloads but do not eliminate the need for reactive programming in streaming and backpressure scenarios.

---

## Ignoring Immutability

Immutable objects are generally easier to test, cache, and use safely in concurrent applications.
