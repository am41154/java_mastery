# 7.17 Virtual Threads (Project Loom)

One of the most revolutionary features introduced in **Java 21 (LTS)** is **Virtual Threads**, developed under **Project Loom**.

For more than 25 years, Java applications relied on **Operating System (OS) threads**, also called **Platform Threads**.

Although powerful, platform threads are expensive to create and limit application scalability.

Virtual Threads dramatically reduce the cost of thread creation, allowing Java applications to handle **millions of concurrent tasks** while preserving Java's familiar **thread-per-request** programming model.

This is arguably the biggest improvement to Java concurrency since the introduction of the `ExecutorService`.

---

# Why Were Virtual Threads Introduced?

Traditional Java servers work like this:

```text
HTTP Request

↓

Platform Thread

↓

Business Logic

↓

Database Call

↓

Response
```

Each request consumes an OS thread.

When thousands of requests arrive simultaneously:

- Thousands of OS threads are required.
- Threads consume significant memory.
- Context switching becomes expensive.
- CPU spends time scheduling threads instead of executing application code.

As concurrency grows, throughput eventually degrades.

---

# The Scalability Problem

Suppose:

- 20,000 concurrent HTTP requests
- Each request waits 95% of the time for:
  - Database
  - Network
  - REST APIs
  - File I/O

Platform threads remain blocked during these waits.

```text
Thread

↓

Waiting...

↓

Waiting...

↓

Waiting...
```

CPU cores remain mostly idle while memory is consumed by blocked threads.

This is highly inefficient.

---

# Traditional Solution: Thread Pools

Java applications commonly use thread pools.

```java
ExecutorService executor =
        Executors.newFixedThreadPool(200);
```

Benefits:

- Limits thread creation
- Reuses worker threads

However:

- Pool size is fixed.
- Requests may queue.
- Choosing the correct pool size is difficult.
- Blocking operations still occupy threads.

---

# Reactive Programming

Frameworks such as:

- Spring WebFlux
- Netty
- Vert.x

avoid blocking by using asynchronous event loops.

Advantages:

- Excellent scalability

Disadvantages:

- Complex programming model
- Callback chains
- Reactive operators
- Steeper learning curve
- Harder debugging

Many enterprise teams found reactive programming difficult to adopt.

---

# The Goal of Project Loom

Project Loom asks a simple question:

> Can we keep the simple thread-per-request programming model while scaling to millions of concurrent tasks?

The answer is:

**Virtual Threads.**

---

# What is a Virtual Thread?

A Virtual Thread is a lightweight thread managed primarily by the **JVM**, not directly by the operating system.

Unlike platform threads:

- Creation is inexpensive.
- Memory footprint is much smaller.
- Blocking operations no longer monopolize an OS thread.

Example:

```java
Thread thread =
        Thread.ofVirtual()
              .start(() ->
                      processRequest());
```

This creates a Virtual Thread instead of a Platform Thread.

---

# Platform Thread vs Virtual Thread

| Platform Thread | Virtual Thread |
|-----------------|----------------|
| Managed by OS | Managed by JVM |
| Expensive creation | Very cheap creation |
| Large stack | Small, growable stack |
| Limited count | Millions possible |
| High context-switch cost | Much lower scheduling overhead |
| Tied to OS thread | Mounted only when running |

---

# Creating Virtual Threads

### Individual Thread

```java
Thread.startVirtualThread(() -> {

    System.out.println("Running");
});
```

---

### Builder API

```java
Thread thread =
Thread.ofVirtual()
      .name("worker")
      .start(() -> {

          process();
      });
```

---

### ExecutorService

Recommended for applications:

```java
try (ExecutorService executor =
        Executors.newVirtualThreadPerTaskExecutor()) {

    executor.submit(() ->
            processRequest());
}
```

Each submitted task receives its own Virtual Thread.

No thread pool sizing is required.

# Blocking is No Longer Expensive

Example:

```java
Thread.sleep(
        Duration.ofSeconds(5)
);
```

Platform thread:

```text
OS Thread

↓

Blocked
```

Virtual thread:

```text
Virtual Thread

↓

Unmounted

↓

Carrier Thread Released
```

The carrier thread becomes available to execute other virtual threads.

This is the key innovation behind Project Loom.

---

# Carrier Threads

Virtual Threads do not execute directly on the CPU.

Instead:

```text
Virtual Thread

↓

Carrier Thread

↓

Operating System Thread

↓

CPU
```

Carrier threads are ordinary platform threads managed by the JVM.

When a Virtual Thread blocks on supported operations, it is **unmounted**, freeing the carrier thread for other work.

---

# Mounting and Unmounting

Execution cycle:

```text
Virtual Thread

↓

Mounted

↓

Running

↓

Blocking I/O

↓

Unmounted

↓

Carrier Released

↓

I/O Completes

↓

Mounted Again

↓

Continue Execution
```

This transparent scheduling enables high scalability without changing programming style.

---

# Continuations (Conceptual Overview)

Internally, Virtual Threads are built on **continuations**, which allow the JVM to suspend and resume execution efficiently.

Think of a continuation as a snapshot of a thread's execution state.

```text
Execute

↓

Pause

↓

Save State

↓

Resume Later
```

Developers do not work with continuations directly; the JVM handles them behind the scenes.

---

# Memory Consumption

Typical platform thread stack:

```text
≈ 1 MB
```

100,000 threads would require roughly:

```text
≈ 100 GB
```

Not practical.

Virtual Threads allocate stack frames dynamically and grow as needed, dramatically reducing memory usage.

This enables millions of concurrent threads on commodity hardware.

# Blocking I/O Example

Traditional code:

```java
String response =
        httpClient.send(request);
```

With Virtual Threads:

- The code remains blocking and easy to read.
- The JVM parks the Virtual Thread while waiting.
- Carrier threads continue executing other tasks.

No callbacks.

No reactive operators.

No `CompletableFuture` chains for simple cases.

# Spring Boot 3.2 Support

Spring Boot 3.2 provides built-in support for Virtual Threads.

Enable them with:

```properties
spring.threads.virtual.enabled=true
```

Once enabled:

- Web request handling uses Virtual Threads.
- `@Async` can leverage Virtual Threads.
- Blocking controllers become highly scalable without code changes (assuming compatible libraries).

This makes adoption straightforward for many applications.

# CPU-Bound vs I/O-Bound Work

Virtual Threads excel at **I/O-bound** workloads.

Examples:

- Database queries
- REST API calls
- File I/O
- Message brokers
- Network communication

They provide little benefit for **CPU-bound** workloads where threads spend most of their time performing computation rather than waiting.

---

# Pinning

Some operations can prevent a Virtual Thread from unmounting, a phenomenon known as **pinning**.

Common causes include:

- Long-running synchronized blocks
- Native method calls
- Certain legacy libraries that block at the OS level

Example:

```java
synchronized(lock) {

    Thread.sleep(5000);
}
```

The Virtual Thread may remain pinned to its carrier thread, reducing scalability.

Prefer higher-level concurrency utilities such as `ReentrantLock` when appropriate and avoid holding locks across blocking operations.

---

# Best Practices

- Use Virtual Threads primarily for I/O-bound tasks.
- Continue writing straightforward blocking code where appropriate.
- Avoid long synchronized sections around blocking operations.
- Use `Executors.newVirtualThreadPerTaskExecutor()` for task execution.
- Test third-party libraries for Virtual Thread compatibility before large-scale adoption.

---

# Common Misconceptions

## "Virtual Threads Replace Platform Threads"

No.

Platform Threads still exist and are used as carrier threads by the JVM.

---

## "Virtual Threads Make Code Faster"

Not necessarily.

They improve **scalability** by allowing far more concurrent blocking tasks, but they do not accelerate CPU-intensive algorithms.

---

## "Reactive Programming Is Obsolete"

No.

Reactive programming remains valuable for streaming, backpressure, and event-driven pipelines.

Virtual Threads simply provide another excellent option for many server-side applications.

# Interview Questions

### Q1. Why were Virtual Threads introduced?

To enable millions of concurrent tasks while preserving Java's familiar thread-per-request programming model.

---

### Q2. What is the difference between a Platform Thread and a Virtual Thread?

Platform Threads are managed by the operating system and are relatively expensive. Virtual Threads are lightweight, managed by the JVM, and can scale to much higher levels of concurrency.

---

### Q3. What is a Carrier Thread?

A Carrier Thread is a platform thread that temporarily executes one or more Virtual Threads. When a Virtual Thread blocks on supported operations, it is unmounted and the Carrier Thread is released for other work.

---

### Q4. Are Virtual Threads useful for CPU-bound workloads?

Generally no. They primarily benefit I/O-bound applications where threads spend significant time waiting on external resources.

---

### Q5. What is Thread Pinning?

Thread Pinning occurs when a Virtual Thread cannot be unmounted from its Carrier Thread, typically due to operations such as blocking inside long-running synchronized sections or certain native calls, reducing scalability.

---

# Key Takeaways

- Virtual Threads are the flagship feature of Java 21 and Project Loom.
- They enable highly scalable thread-per-request applications without requiring reactive programming.
- Virtual Threads are lightweight, inexpensive to create, and managed by the JVM.
- Blocking operations no longer monopolize platform threads because Virtual Threads can be unmounted while waiting.
- They are best suited for I/O-bound workloads such as web servers, database access, and network communication.
- Understanding Carrier Threads, mounting/unmounting, and thread pinning is essential for senior Java backend engineers working with modern Java applications.
