# 7.13 Sealed Classes (Java 17)

**Sealed Classes**, introduced in **Java 17 (LTS)**, allow developers to **explicitly control which classes or interfaces are permitted to extend or implement a type**.

Before Java 17, any non-final public class could generally be extended by anyone.

This made it difficult to model **closed hierarchies**, where only a known and limited set of subclasses should exist.

Sealed Classes solve this problem by giving the author of a class complete control over its inheritance hierarchy.

They are particularly useful in:

- Domain modeling
- State machines
- Financial systems
- Payment processing
- AST (Abstract Syntax Tree)
- Pattern Matching
- Compiler design

---

# Why Were Sealed Classes Introduced?

Suppose we model different payment types.

Before Java 17:

```java
public class Payment {

}
```

Anyone can create:

```java
class CryptoPayment extends Payment {

}

class GiftCardPayment extends Payment {

}

class UnknownPayment extends Payment {

}
```

The hierarchy is open.

Sometimes this flexibility is undesirable.

We may want to allow **only** specific payment types.

---

# What is a Sealed Class?

A sealed class explicitly specifies which classes are allowed to extend it.

Example:

```java
public sealed class Payment
        permits CreditCardPayment,
                UpiPayment,
                BankTransferPayment {

}
```

Now only these classes may extend `Payment`.

Any other subclass causes a compilation error.

---

# The `permits` Clause

The `permits` clause lists all permitted subclasses.

```java
public sealed class Shape
        permits Circle,
                Rectangle,
                Triangle {

}
```

Compiler knows the complete hierarchy.

This enables better type checking and future language features such as exhaustive pattern matching.

---

# Rules for Permitted Classes

Every permitted subclass must declare exactly one of the following modifiers:

- `final`
- `sealed`
- `non-sealed`

---

# final Subclass

A final subclass ends the inheritance chain.

Example:

```java
public final class Circle
        extends Shape {

}
```

No further inheritance is allowed.

---

# sealed Subclass

A sealed subclass continues restricting inheritance.

```java
public sealed class Rectangle
        extends Shape
        permits Square {

}
```

Only `Square` can extend `Rectangle`.

---

# non-sealed Subclass

A non-sealed subclass reopens the hierarchy.

```java
public non-sealed class Triangle
        extends Shape {

}
```

Now anyone can extend `Triangle`.

Example:

```java
class RightTriangle
        extends Triangle {

}
```

---

# Complete Example

```java
public sealed interface Vehicle
        permits Car,
                Bike,
                Truck {

}
```

```java
public final class Car
        implements Vehicle {

}
```

```java
public sealed class Truck
        implements Vehicle
        permits MiniTruck {

}
```

```java
public final class MiniTruck
        extends Truck {

}
```

```java
public non-sealed class Bike
        implements Vehicle {

}
```

Hierarchy:

```text
Vehicle (sealed)

├── Car (final)

├── Truck (sealed)

│     └── MiniTruck (final)

└── Bike (non-sealed)

      └── Any subclass
```

---

# Sealed Interfaces

Interfaces can also be sealed.

Example:

```java
public sealed interface Animal
        permits Dog,
                Cat {

}
```

Implementations:

```java
public final class Dog
        implements Animal {

}
```

```java
public final class Cat
        implements Animal {

}
```

---

# Same Module / Package Rule

Permitted subclasses must be:

- In the same module (if using the Java Module System), or
- In the same package (when not using modules)

This allows the compiler to verify the complete hierarchy.

---

# Reflection Support

The JVM provides APIs to inspect sealed classes.

Example:

```java
Shape.class.isSealed();
```

Returns:

```text
true
```

Retrieve permitted subclasses:

```java
Shape.class.getPermittedSubclasses();
```

Useful for frameworks and tooling.

---

# Sealed Classes and Pattern Matching

One of the biggest motivations for sealed classes is **pattern matching**.

Example:

```java
sealed interface Shape
        permits Circle,
                Rectangle {

}
```

When switching over a `Shape`, the compiler knows all possible implementations.

```java
switch (shape) {

    case Circle c -> ...

    case Rectangle r -> ...
}
```

Since all permitted subclasses are covered, no default branch may be required (depending on the Java version and language feature).

This improves safety and maintainability.

Banking system:

```java
sealed interface Account

```

Benefits:

- Prevents unauthorized account types.

# Comparison with final

```java
public final class Employee {

}
```

No inheritance at all.

---

```java
public sealed class Employee
        permits Manager,
                Developer {

}
```

Limited inheritance.

---

```java
public class Employee {

}
```

Unlimited inheritance.

---

# Comparison Table

| Feature | Normal Class | final Class | Sealed Class |
|----------|--------------|-------------|--------------|
| Extendable | Yes | No | Only permitted subclasses |
| Restricts hierarchy | No | Completely | Selectively |
| Compiler knows subclasses | No | Yes (none) | Yes |
| Supports closed hierarchies | No | No | Yes |

---

# Sealed Classes vs Abstract Classes

| Abstract Class | Sealed Class |
|----------------|--------------|
| May have abstract methods | May be abstract or concrete |
| Controls behavior | Controls inheritance |
| Any subclass allowed | Only permitted subclasses |
| Focus on abstraction | Focus on hierarchy restriction |

A class can be both `abstract` and `sealed`.

Example:

```java
public abstract sealed class Payment
        permits CardPayment,
                UpiPayment {

}
```

# Performance

Sealed Classes do **not** significantly improve runtime performance.

Their benefits are primarily:

- Better type safety
- Stronger domain modeling
- Improved compiler analysis
- Easier maintenance
- Better support for pattern matching

---

# Best Practices

- Use Sealed Classes when the hierarchy is intentionally closed.
- Prefer `final` subclasses unless further extension is required.
- Use `non-sealed` only when reopening the hierarchy is intentional.
- Combine Sealed Classes with Records for immutable domain models.
- Model business rules directly in the type system where possible.

---

# Common Mistakes

## Forgetting a Required Modifier

Invalid:

```java
class Circle extends Shape {

}
```

Compilation error.

Every permitted subclass must be declared as:

- `final`
- `sealed`
- `non-sealed`

---

## Using Sealed Classes for Every Hierarchy

Not every class hierarchy needs to be sealed.

If extension by third parties is a valid use case, a normal class or interface is often more appropriate.

---

## Confusing `sealed` with `final`

`final` prevents **all** inheritance.

`sealed` allows **controlled** inheritance.

# Interview Questions

### Q1. Why were Sealed Classes introduced?

To allow developers to define a closed inheritance hierarchy by explicitly specifying which classes are permitted to extend or implement a type.

---

### Q2. What are the three modifiers that permitted subclasses can use?

- `final`
- `sealed`
- `non-sealed`

---

### Q3. What is the purpose of the `permits` clause?

The `permits` clause lists all classes that are allowed to extend a sealed class or implement a sealed interface.

---

### Q4. What is the difference between `final` and `sealed`?

- `final` completely prohibits inheritance.
- `sealed` allows inheritance, but only by explicitly permitted subclasses.

---

### Q5. Why do Sealed Classes work well with Pattern Matching?

Because the compiler knows the complete set of permitted subclasses, it can verify that all possible cases are handled, enabling exhaustive pattern matching.

# Records + Sealed Classes

A common modern Java design combines Records and Sealed Interfaces.

```java
public sealed interface Payment
        permits CardPayment,
                UpiPayment,
                CashPayment {
}
```

```java
public record CardPayment(
        String cardNumber,
        double amount
) implements Payment {
}
```

```java
public record UpiPayment(
        String upiId,
        double amount
) implements Payment {
}
```

```java
public record CashPayment(
        double amount
) implements Payment {
}
```

Benefits:

- Immutable data (`record`)
- Controlled hierarchy (`sealed`)
- Better compiler support
- Cleaner domain model
- Excellent fit for REST APIs, event-driven systems, and modern enterprise applications

---

# Key Takeaways

- Sealed Classes provide controlled inheritance by explicitly declaring permitted subclasses.
- Every permitted subclass must be `final`, `sealed`, or `non-sealed`.
- Sealed Interfaces extend the same concept to interface hierarchies.
- They improve domain modeling, type safety, and maintainability.
- Sealed Classes work particularly well with Records and Pattern Matching, making them a cornerstone of modern Java 17+ application design.
