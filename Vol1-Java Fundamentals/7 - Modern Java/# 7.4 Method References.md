# 7.4 Method References

Method References were introduced in **Java 8** as a more concise alternative to certain lambda expressions.

They improve readability by referring directly to an existing method instead of writing a lambda that simply calls that method.

Method References are widely used with:

- Stream API
- Collections Framework
- Optional
- CompletableFuture
- Spring Framework
- Event-driven programming

---

# Why Method References?

Consider the following lambda:

```java
List<String> names = List.of("Alice", "Bob", "Charlie");

names.forEach(name -> System.out.println(name));
```

The lambda simply invokes an existing method.

It does not contain any additional logic.

This can be simplified using a Method Reference.

```java
names.forEach(System.out::println);
```

The code is shorter and easier to read.

---

# What Is a Method Reference?

A Method Reference is a shorthand notation for a lambda expression that directly invokes an existing method.

General syntax:

```java
ClassName::methodName
```

or

```java
objectReference::methodName
```

Instead of writing:

```java
x -> someMethod(x)
```

we can write:

```java
ClassName::someMethod
```

provided the method signatures are compatible.

---

# Lambda vs Method Reference

Lambda:

```java
String::length
```

is equivalent to:

```java
str -> str.length()
```

Another example:

Lambda:

```java
name -> System.out.println(name)
```

Method Reference:

```java
System.out::println
```

Both produce identical behavior.

---

# Types of Method References

Java supports four kinds of Method References.

| Type | Example |
|------|----------|
| Static Method | `ClassName::staticMethod` |
| Instance Method of a Particular Object | `object::instanceMethod` |
| Instance Method of an Arbitrary Object of a Particular Type | `ClassName::instanceMethod` |
| Constructor Reference | `ClassName::new` |

Let's examine each one.

---

# 1. Static Method Reference

Suppose we have:

```java
public class MathUtil {

    public static int square(int x) {
        return x * x;
    }
}
```

Lambda:

```java
Function<Integer, Integer> f =
        x -> MathUtil.square(x);
```

Method Reference:

```java
Function<Integer, Integer> f =
        MathUtil::square;
```

Usage:

```java
System.out.println(f.apply(5));
```

Output:

```text
25
```

---

# 2. Instance Method Reference of a Particular Object

Suppose:

```java
Printer printer = new Printer();
```

Method:

```java
public void print(String message) {
    System.out.println(message);
}
```

Lambda:

```java
Consumer<String> consumer =
        message -> printer.print(message);
```

Method Reference:

```java
Consumer<String> consumer =
        printer::print;
```

---

# 3. Instance Method Reference of an Arbitrary Object

This is the most confusing type for beginners.

Example:

```java
List<String> names =
        List.of("Bob", "Alice", "Charlie");
```

Sorting:

Lambda:

```java
names.sort((a, b) -> a.compareTo(b));
```

Method Reference:

```java
names.sort(String::compareTo);
```

How does this work?

Internally:

```text
(a, b)

↓

a.compareTo(b)
```

The first parameter becomes the object on which the method is invoked.

---

# Another Example

Lambda:

```java
Function<String, Integer> length =
        str -> str.length();
```

Method Reference:

```java
Function<String, Integer> length =
        String::length;
```

Both behave identically.

---

# 4. Constructor Reference

Constructors can also be referenced.

Suppose:

```java
public class Employee {

    public Employee() {

    }
}
```

Lambda:

```java
Supplier<Employee> supplier =
        () -> new Employee();
```

Method Reference:

```java
Supplier<Employee> supplier =
        Employee::new;
```

Usage:

```java
Employee employee = supplier.get();
```

---

# Constructor with Parameters

Suppose:

```java
public Employee(String name) {
    this.name = name;
}
```

Lambda:

```java
Function<String, Employee> f =
        name -> new Employee(name);
```

Method Reference:

```java
Function<String, Employee> f =
        Employee::new;
```

The compiler automatically matches the constructor signature with the functional interface.

---

# Array Constructor Reference

Java also supports array creation.

Lambda:

```java
Function<Integer, String[]> f =
        size -> new String[size];
```

Method Reference:

```java
Function<Integer, String[]> f =
        String[]::new;
```

Usage:

```java
String[] array = f.apply(5);
```

Creates:

```text
String[5]
```

---

# Method References with Stream API

One of the most common use cases.

Lambda:

```java
List<String> names =
        List.of("Alice", "Bob");

names.stream()
     .forEach(name -> System.out.println(name));
```

Method Reference:

```java
names.stream()
     .forEach(System.out::println);
```

Much cleaner.

---

# Another Stream Example

Lambda:

```java
List<String> names =
        List.of("alice", "bob");

List<String> result =
        names.stream()
             .map(name -> name.toUpperCase())
             .toList();
```

Method Reference:

```java
List<String> result =
        names.stream()
             .map(String::toUpperCase)
             .toList();
```

---

# Method References and Constructors in Streams

Suppose:

```java
record Person(String name) {}
```

Lambda:

```java
List<Person> people =
        names.stream()
             .map(name -> new Person(name))
             .toList();
```

Method Reference:

```java
List<Person> people =
        names.stream()
             .map(Person::new)
             .toList();
```

---

# When Should Method References Be Used?

Use Method References when the lambda:

- Calls exactly one existing method.
- Does not contain additional business logic.
- Improves readability.

Example:

Good:

```java
list.forEach(System.out::println);
```

Not ideal:

```java
list.forEach(name -> {
    validate(name);
    log(name);
    save(name);
});
```

This contains multiple operations and should remain a lambda.

---

# Method Reference vs Lambda

| Lambda | Method Reference |
|----------|------------------|
| More flexible | More concise |
| Supports complex logic | Suitable only for existing methods |
| Easier for multi-step operations | Best for simple delegation |
| Can contain multiple statements | Represents a single method invocation |

---

# Performance

Method References are **not faster** than lambda expressions.

The compiler converts both into similar bytecode using:

- `invokedynamic`
- `LambdaMetafactory`

The choice between them should be based on readability rather than performance.

---

# Best Practices

- Prefer Method References when they make the code simpler.
- Avoid forcing Method References if they reduce clarity.
- Keep complex business logic inside lambdas or regular methods.
- Use constructor references when creating objects in streams or factories.
- Use method references consistently within a codebase for better readability.

---

# Interview Questions

### Q1. What is a Method Reference?

A Method Reference is a shorthand syntax for a lambda expression that directly invokes an existing method or constructor.

---

### Q2. How many types of Method References exist in Java?

There are four:

1. Static Method Reference
2. Instance Method Reference of a Particular Object
3. Instance Method Reference of an Arbitrary Object of a Particular Type
4. Constructor Reference

---

### Q3. What is the difference between a lambda and a Method Reference?

A lambda can contain arbitrary logic, while a Method Reference simply delegates to an existing method or constructor.

---

### Q4. Are Method References faster than lambdas?

No.

Both are implemented using the same underlying JVM mechanisms (`invokedynamic` and `LambdaMetafactory`). The difference is primarily in readability.

---

### Q5. When should you prefer a Method Reference?

When the lambda only forwards its arguments to an existing method or constructor and the resulting code becomes more readable.

---

# Key Takeaways

- Method References provide a concise alternative to simple lambda expressions.
- Java supports static, instance, constructor, and array constructor references.
- They are heavily used with Streams, Optional, CompletableFuture, and other functional APIs.
- Method References improve readability but offer no inherent performance advantage over lambdas.
- Use Method References only when they make the code clearer; otherwise, prefer explicit lambdas.
