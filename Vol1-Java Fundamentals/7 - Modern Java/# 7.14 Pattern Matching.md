# 7.14 Pattern Matching (`instanceof` & `switch`)

Pattern Matching is one of the most significant language improvements introduced in recent Java releases.

It aims to make type checking, casting, and branching **safer**, **shorter**, and **more readable**.

The evolution happened over multiple Java versions:

| Java Version | Feature |
|--------------|---------|
| Java 16 | Pattern Matching for `instanceof` |
| Java 17 | Sealed Classes (foundation) |
| Java 21 | Pattern Matching for `switch` (final) |

Together, these features significantly reduce boilerplate while improving compiler-assisted type safety.

---

# Why Was Pattern Matching Introduced?

Consider a traditional `instanceof` check.

```java
Object obj = "Hello";
```

Before Java 16:

```java
if (obj instanceof String) {

    String value = (String) obj;

    System.out.println(value.toUpperCase());
}
```

Problems:

- Explicit cast
- Duplicate type name
- Boilerplate
- Easy to make mistakes

Java 16 simplifies this.

---

# Pattern Matching for `instanceof`

```java
Object obj = "Hello";
```

```java
if (obj instanceof String value) {

    System.out.println(value.toUpperCase());
}
```

Notice:

- No explicit cast
- Compiler creates the variable
- Type-safe
- More readable

---

# What Happens Internally?

Traditional approach:

```text
instanceof

↓

Cast

↓

Use Variable
```

Pattern Matching:

```text
instanceof

↓

Compiler Performs Cast

↓

Pattern Variable
```

The compiler guarantees that `value` is a `String` inside the `if` block.

---

# Scope of Pattern Variables

Valid:

```java
if (obj instanceof String value) {

    System.out.println(value);
}
```

Invalid:

```java
System.out.println(value);
```

Outside the `if` block, the variable no longer exists.

---

# Another Example

Without Pattern Matching:

```java
if (shape instanceof Circle) {

    Circle circle = (Circle) shape;

    System.out.println(circle.radius());
}
```

With Pattern Matching:

```java
if (shape instanceof Circle circle) {

    System.out.println(circle.radius());
}
```

Much cleaner.

---

# Pattern Matching with Additional Conditions

Example:

```java
if (obj instanceof String value &&
        value.length() > 5) {

    System.out.println(value);
}
```

The pattern variable can be used immediately in the condition.

---

# Pattern Matching in Loops

```java
for (Object obj : objects) {

    if (obj instanceof Employee employee) {

        System.out.println(employee.name());
    }
}
```

No explicit cast is required.

# Pattern Matching with Sealed Classes

Suppose:

```java
sealed interface Shape
        permits Circle,
                Rectangle {
}
```

Records:

```java
record Circle(double radius)
        implements Shape {
}
```

```java
record Rectangle(
        double width,
        double height
) implements Shape {
}
```

Now type checks become straightforward.

```java
if (shape instanceof Circle circle) {

    System.out.println(circle.radius());
}
```

---

# Pattern Matching for `switch`

Java 21 extends Pattern Matching to `switch`.

Traditional approach:

```java
if (shape instanceof Circle) {

}

else if (shape instanceof Rectangle) {

}
```

Modern approach:

```java
switch (shape) {

    case Circle circle ->
            System.out.println(circle.radius());

    case Rectangle rectangle ->
            System.out.println(
                    rectangle.width()
            );
}
```

Cleaner.

More readable.

Compiler-assisted.

---

# Why Is `switch` Better?

Traditional:

```text
if

↓

else if

↓

else if

↓

else
```

Modern:

```text
switch

↓

Pattern

↓

Action
```

The compiler can verify completeness.

---

# Exhaustive Switching

Suppose:

```java
sealed interface Payment
        permits CardPayment,
                CashPayment {
}
```

Switch:

```java
switch (payment) {

    case CardPayment card -> ...

    case CashPayment cash -> ...
}
```

Every permitted implementation is handled.

No default branch is required.

Compiler verifies completeness.

This is one of the biggest advantages of combining Sealed Classes with Pattern Matching.

---

# Record Patterns (Java 21)

Records integrate naturally.

Suppose:

```java
record Employee(
        int id,
        String name
) {
}
```

Pattern:

```java
if (obj instanceof Employee(
        int id,
        String name
)) {

    System.out.println(name);
}
```

The record components are extracted automatically.

This feature is called a **Record Pattern**.

---

# Nested Record Patterns

Example:

```java
record Address(
        String city
) {
}
```

```java
record Employee(
        String name,
        Address address
) {
}
```

Pattern:

```java
if (obj instanceof Employee(

        String name,

        Address(String city)

)) {

    System.out.println(city);
}
```

Nested objects can be destructured directly.

---

# Guarded Patterns

Pattern matching supports additional conditions.

Example:

```java
switch (shape) {

    case Circle c
            when c.radius() > 100 ->
            System.out.println("Large Circle");

    case Circle c ->
            System.out.println("Small Circle");

    default ->
            System.out.println("Other");
}
```

The `when` clause acts as a guard.

---

Modern:

REST response handling.

# Performance

Pattern Matching is primarily a language feature.

Benefits include:

- Fewer explicit casts
- Better compiler optimization opportunities
- Cleaner bytecode generation
- Improved readability

Runtime performance is generally comparable to traditional type checks.

---

# Best Practices

- Prefer Pattern Matching over explicit casting.
- Use Pattern Matching with Sealed Classes whenever possible.
- Prefer `switch` over long `if-else` chains for closed hierarchies.
- Keep `switch` cases simple and focused.
- Use Record Patterns for immutable DTOs and domain models.

---

# Common Mistakes

## Using Pattern Variables Outside Scope

Invalid:

```java
if (obj instanceof String value) {

}

System.out.println(value);
```

The pattern variable exists only where the compiler can guarantee the type.

---

## Mixing Pattern Matching with Manual Casts

Avoid:

```java
if (obj instanceof Employee employee) {

    Employee e =
            (Employee) obj;
}
```

The cast is redundant.

---

## Forgetting Exhaustiveness

When switching over non-sealed types, a `default` branch is typically required because the compiler cannot know all possible implementations.

# Interview Questions

### Q1. What problem does Pattern Matching solve?

It eliminates repetitive type checks and explicit casts, making code shorter, safer, and easier to read.

---

### Q2. What is Pattern Matching for `instanceof`?

It combines type checking and casting into a single operation by introducing a pattern variable that is automatically cast to the matched type.

---

### Q3. Why does Pattern Matching work particularly well with Sealed Classes?

Because the compiler knows the complete set of permitted subclasses, it can verify that all cases are handled in a `switch`, enabling exhaustive pattern matching.

---

### Q4. What are Record Patterns?

Record Patterns allow the components of a record to be destructured directly in pattern matching expressions, avoiding manual accessor calls.

---

### Q5. What is a Guarded Pattern?

A Guarded Pattern uses a `when` clause to apply additional conditions after a successful type match, allowing more precise pattern selection.

# Modern Java Example

Combining multiple language features:

```java
public sealed interface Payment
        permits CardPayment,
                CashPayment,
                UpiPayment {
}
```

```java
public record CardPayment(
        String cardNumber,
        double amount
) implements Payment {
}
```

Processing:

```java
public String process(Payment payment) {

    return switch (payment) {

        case CardPayment card
                when card.amount() > 10000 ->
                "Large Card Payment";

        case CardPayment card ->
                "Card Payment";

        case CashPayment cash ->
                "Cash Payment";

        case UpiPayment upi ->
                "UPI Payment";
    };
}
```

This example combines:

- Records
- Sealed Interfaces
- Pattern Matching
- Switch Expressions
- Guarded Patterns

It represents the modern style of Java programming introduced in Java 21.

---

# Key Takeaways

- Pattern Matching removes the need for explicit casting after `instanceof`.
- Pattern Matching for `switch` provides a cleaner and safer alternative to long `if-else` chains.
- Sealed Classes enable exhaustive pattern matching by defining a closed hierarchy.
- Record Patterns simplify the extraction of data from immutable records.
- Guarded Patterns add conditional logic to pattern matching.
- Together, these features significantly improve readability, maintainability, and type safety in modern Java applications.
