# 5.9 Callable, Future, and CompletableFuture

### Why This Matters

Imagine an e-commerce application displaying a product page.

To render the page, it needs to fetch:

- Product details
- Inventory status
- Customer reviews
- Pricing information
- Recommended products

A straightforward implementation performs these operations one after another.

```
Product

↓

Inventory

↓

Reviews

↓

Pricing

↓

Recommendations
```

If each service takes **300 ms**, the total response time becomes approximately:

```
300 + 300 + 300 + 300 + 300

= 1500 ms
```

However, these operations are independent.

Why wait for one to finish before starting the next?

This is where **asynchronous programming** becomes valuable.

---

# From Runnable to Callable

Earlier, we learned that `Runnable` cannot return a value.

```java
Runnable task = () -> {

    // Work

};
```

If a task needs to produce a result, use `Callable`.

```java
Callable<Product> task = () -> {

    return fetchProduct();

};
```

A `Callable`:

- Returns a value
- Can throw checked exceptions
- Is typically submitted to an `ExecutorService`

---

# Future

Submitting a `Callable` returns a `Future`.

```java
Future<Product> future =
    executor.submit(task);
```

Think of a `Future` as:

> "A placeholder for a result that will be available later."

```
Submit Task

↓

Task Running

↓

Future

↓

Result Ready
```

The caller can continue doing other work while the task executes.

---

# Getting the Result

Eventually,

retrieve the result.

```java
Product product = future.get();
```

If the task has not finished,

the current thread waits.

```
Main Thread

↓

future.get()

↓

Waiting...

↓

Task Completes

↓

Continue
```

---

# The Problem with Future

Suppose we need:

- Product
- Inventory
- Reviews

With `Future`:

```java
Future<Product>

Future<Inventory>

Future<Reviews>
```

Eventually:

```java
productFuture.get();

inventoryFuture.get();

reviewFuture.get();
```

Managing multiple futures quickly becomes cumbersome.

Additional limitations include:

- Blocking `get()`
- Difficult error handling
- No easy way to combine results
- Poor support for task composition

Java introduced a better solution:

```
CompletableFuture
```

---

# CompletableFuture

`CompletableFuture` supports asynchronous programming with task composition.

Instead of waiting after every operation,

you describe a workflow.

Conceptually:

```
Task A

↓

Task B

↓

Task C
```

without manually managing threads.

---

# Running Asynchronous Tasks

```java
CompletableFuture<Product> future =
    CompletableFuture.supplyAsync(
        () -> fetchProduct()
    );
```

The task runs asynchronously.

The current thread continues immediately.

---

# Chaining Operations

Suppose we fetch a product and then calculate its price.

```java
CompletableFuture
    .supplyAsync(() -> fetchProduct())
    .thenApply(product ->
        calculatePrice(product)
    );
```

Flow:

```
Fetch Product

↓

Calculate Price

↓

Return Result
```

Each step begins automatically after the previous one completes.

---

# Combining Independent Tasks

Suppose we need:

- Product
- Inventory

These are independent.

```
Fetch Product

        ||

Fetch Inventory

        ||

Combine Results
```

`CompletableFuture` supports this naturally.

```java
productFuture.thenCombine(
    inventoryFuture,
    (product, inventory) -> ...
);
```

Both tasks execute concurrently.

Only after both complete are the results combined.

---

# Waiting for Multiple Tasks

Sometimes,

you only need to know when **all** tasks finish.

```java
CompletableFuture.allOf(
    future1,
    future2,
    future3
);
```

Execution:

```
Task A

Task B

Task C

↓

All Complete

↓

Continue
```

Similarly,

```java
CompletableFuture.anyOf(...)
```

continues when the **first** task completes.

---

# Exception Handling

Asynchronous tasks may fail.

Instead of surrounding every call with `try-catch`,

`CompletableFuture` provides built-in handling.

Example:

```java
future.exceptionally(ex -> {

    return defaultValue;

});
```

This allows graceful recovery while keeping the asynchronous pipeline readable.

---

# Future vs CompletableFuture

| Feature | Future | CompletableFuture |
|----------|--------|-------------------|
| Return result | ✓ | ✓ |
| Blocking `get()` | ✓ | Optional |
| Chain operations | ✗ | ✓ |
| Combine tasks | ✗ | ✓ |
| Exception handling | Limited | Rich API |
| Asynchronous pipelines | ✗ | ✓ |

---

# Spring Boot Perspective

Suppose an endpoint returns a customer dashboard.

Instead of:

```
Customer

↓

Orders

↓

Payments

↓

Recommendations
```

A service can execute them concurrently.

```
Customer      Orders

Payments      Recommendations

        ||

Combine Results

↓

Return Response
```

This reduces overall latency because independent I/O operations proceed simultaneously.

---

# Production Perspective

A travel booking application gathers information from four external services:

- Flights
- Hotels
- Weather
- Currency exchange

Originally,

the services were called sequentially.

Average response time:

```
2.4 seconds
```

After switching to `CompletableFuture`,

the calls execute concurrently.

Average response time drops to:

```
700 ms
```

No individual service became faster.

The application simply stopped waiting unnecessarily.

# Interview Focus

### Question

Why is `CompletableFuture` preferred over `Future`?

### Strong Senior-Level Answer

> `Future` represents the result of an asynchronous computation but provides only basic operations and typically requires blocking with `get()`. `CompletableFuture` extends this model by supporting non-blocking composition, chaining, combination of multiple asynchronous tasks, and built-in exception handling, making it far more suitable for modern asynchronous applications.

---

# Common Mistakes

❌ Calling `future.get()` immediately after submission.

❌ Using `CompletableFuture` for CPU-bound work without an appropriate executor.

❌ Creating long chains without handling exceptions.

❌ Assuming asynchronous code is automatically faster.

---

# Engineering Insight

Asynchronous programming is most beneficial when tasks spend time **waiting**.

Examples include:

- Database queries
- REST API calls
- Kafka communication
- File systems
- Cloud services

If your task is purely CPU-intensive,

running many asynchronous computations on the same limited set of CPU cores may not improve throughput.

Always consider the nature of the workload before introducing asynchronous execution.

---

# Engineering Takeaways

- `Callable` returns a value.
- `Future` represents a pending result.
- `Future.get()` blocks the caller.
- `CompletableFuture` enables non-blocking task composition.
- Independent I/O operations are excellent candidates for asynchronous execution.

---

# Quick Revision

✓ Use `Callable` when a result is required.

✓ `Future` is a placeholder for a future result.

✓ Avoid blocking with `get()` unless necessary.

✓ `CompletableFuture` supports chaining and combining tasks.

✓ Parallelize independent operations to reduce latency.

---

# Senior Interview Challenge

A dashboard endpoint performs four independent REST API calls sequentially.

Average latency:

```
1.8 seconds
```

### How would you redesign it?

**Expected Discussion Points**

- Execute independent calls concurrently using `CompletableFuture`.
- Combine results with `thenCombine()` or `allOf()`.
- Handle failures gracefully using `exceptionally()` or `handle()`.
- Use an appropriate executor instead of relying on the common pool for production workloads.
- Measure improvements and monitor thread pool utilization.

A senior engineer views `CompletableFuture` not as a replacement for threads, but as a tool for building maintainable asynchronous workflows.
