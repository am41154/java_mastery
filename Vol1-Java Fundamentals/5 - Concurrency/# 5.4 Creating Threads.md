# 5.4 Creating Threads – Thread, Runnable, Callable, and Future

### Why This Matters

When developers first learn Java, they are often introduced to thread creation like this:

```java
new Thread(() -> {
    System.out.println("Hello");
}).start();
```

While this works, **you will almost never create threads this way in a production Spring Boot application.**

Modern applications use:

- `Runnable`
- `Callable`
- `ExecutorService`
- `CompletableFuture`

Understanding **when** to use each is far more important than memorizing their APIs.

---

# Option 1 – Extending Thread

The simplest way to create a thread is by extending the `Thread` class.

```java
class Worker extends Thread {

    @Override
    public void run() {
        System.out.println("Processing...");
    }
}
```

Execution:

```java
Worker worker = new Worker();

worker.start();
```

Execution flow:

```
Main Thread

↓

start()

↓

New Thread

↓

run()
```

---

## Why This Approach Is Rarely Used

Although simple,

it has several drawbacks.

### Single Inheritance

Java allows only one parent class.

```java
class Worker extends Thread
```

The class can no longer extend another useful base class.

---

### Task and Thread Are Tightly Coupled

The object represents both:

- The work
- The thread executing the work

This reduces flexibility and reusability.

---

# Option 2 – Implementing Runnable

A better approach is to separate the task from the thread.

```java
class Worker implements Runnable {

    @Override
    public void run() {
        System.out.println("Processing...");
    }
}
```

Execution:

```java
Runnable task = new Worker();

Thread thread = new Thread(task);

thread.start();
```

Architecture:

```
Task (Runnable)

↓

Thread

↓

CPU
```

The same task can be executed by different threads.

---

## Lambda Expression

Since `Runnable` is a functional interface,

Java 8 allows:

```java
Runnable task = () ->
    System.out.println("Processing");
```

Cleaner.

More concise.

Preferred in modern Java.

---

# Runnable Limitations

A `Runnable`:

- Does not return a result.
- Cannot throw checked exceptions directly.

Example:

```java
Runnable task = () -> {

    return 10;
};
```

Compilation error.

---

# Option 3 – Callable

`Callable` addresses these limitations.

```java
Callable<Integer> task = () -> {

    return 10;
};
```

Unlike `Runnable`,

a `Callable` can:

- Return a value.
- Throw checked exceptions.

---

# Runnable vs Callable

| Feature | Runnable | Callable |
|----------|-----------|-----------|
| Returns value | ✗ No | ✓ Yes |
| Throws checked exceptions | ✗ No | ✓ Yes |
| Introduced | Java 1.0 | Java 5 |
| Method | `run()` | `call()` |

---

# How Do We Execute a Callable?

A `Callable` cannot be passed directly to a `Thread`.

Instead,

it is submitted to an `ExecutorService`.

```java
ExecutorService executor =
    Executors.newSingleThreadExecutor();

Callable<Integer> task = () -> 42;

Future<Integer> future =
    executor.submit(task);
```

The returned `Future` represents the pending result.

---

# Understanding Future

Think of a `Future` as a promise.

```
Submit Task

↓

Task Running

↓

Future

↓

Result Available Later
```

Instead of waiting immediately,

your application can continue doing other work.

Later,

retrieve the result.

```java
Integer value = future.get();
```

---

# Future.get()

Calling:

```java
future.get();
```

blocks the current thread until the computation finishes.

```
Current Thread

↓

future.get()

↓

Waiting...

↓

Task Completes

↓

Continue
```

> **Interview Tip**
>
> Although `Future` enables asynchronous execution, calling `get()` immediately often removes the benefit because it blocks the caller.

We'll see better alternatives with `CompletableFuture` later in this chapter.

---

# Which Approach Should You Choose?

| Approach | Recommended? |
|----------|---------------|
| Extend `Thread` | Rarely |
| Implement `Runnable` | Yes |
| `Callable` + `ExecutorService` | Yes |
| `CompletableFuture` | Preferred for asynchronous workflows |

---

# Production Perspective

A payment service must:

- Validate an order.
- Call an external payment gateway.
- Generate an invoice.

A junior developer creates a new `Thread` for every payment request.

Under heavy load, thousands of threads are created, increasing memory usage and context switching.

The service is refactored to submit `Callable` tasks to a fixed thread pool.

Benefits include:

- Controlled thread creation
- Better resource utilization
- Improved throughput
- Easier monitoring and management

---

# Interview Focus

### Question

Why is implementing `Runnable` generally preferred over extending `Thread`?

### Strong Senior-Level Answer

> Implementing `Runnable` separates the task from the thread that executes it. This promotes better design, allows the class to extend another superclass if needed, and integrates naturally with thread pools and the Executor Framework. Extending `Thread` tightly couples the task with the execution mechanism and is rarely used in modern applications.

---

# Common Mistakes

❌ Extending `Thread` for every background task.

❌ Creating a new thread for each request.

❌ Calling `Future.get()` immediately after submission.

❌ Using `Thread` directly instead of a thread pool.

---

# Engineering Insight

Creating a thread is an **expensive operating system operation**.

Every thread requires:

- Memory for its stack
- Scheduling by the operating system
- Context switching
- Lifecycle management

Production applications therefore **reuse threads** instead of creating new ones repeatedly.

This is the primary reason the **Executor Framework** exists—and why direct thread creation is uncommon in enterprise Java.

---

# Engineering Takeaways

- Extending `Thread` is simple but inflexible.
- `Runnable` separates work from execution.
- `Callable` supports return values and checked exceptions.
- `Future` represents the result of an asynchronous computation.
- Thread pools are preferred over manually creating threads.

---

# Quick Revision

✓ Extend `Thread` only in exceptional cases.

✓ Prefer `Runnable` for fire-and-forget tasks.

✓ Use `Callable` when a result is required.

✓ `Future.get()` blocks until completion.

✓ Production systems execute tasks through thread pools.

---

# Senior Interview Challenge

A REST API receives **10,000 requests per minute**.

Each request creates:

```java
new Thread(task).start();
```

The application gradually slows down and eventually runs out of memory.

### How would you redesign the solution?

**Expected Discussion Points**

- Avoid creating a thread per request.
- Use an `ExecutorService` with an appropriately sized thread pool.
- Reuse worker threads instead of constantly allocating new ones.
- Evaluate whether the workload is CPU-bound or I/O-bound when sizing the pool.
- Consider `CompletableFuture` for non-blocking asynchronous workflows.

A senior engineer focuses on **managing tasks**, not **creating threads**.
