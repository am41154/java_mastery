# 5.6 Synchronization – `synchronized`, Intrinsic Locks, and `ReentrantLock`

### Why This Matters

In the previous section, we saw how multiple threads can corrupt shared data.

Consider this code again:

```java
class Counter {

    private int count = 0;

    public void increment() {
        count++;
    }
}
```

If several threads execute:

```java
counter.increment();
```

simultaneously,

the final value of `count` may be incorrect.

The solution is to ensure that **only one thread can execute the critical section at a time.**

This process is called **synchronization**.

---

# What Is Synchronization?

Synchronization is a mechanism that coordinates access to shared mutable data.

Instead of allowing multiple threads to modify the same object simultaneously,

only one thread is permitted to enter the protected code.

Conceptually:

```
Without Synchronization

Thread A ─────┐
              ├── Shared Data
Thread B ─────┘

Both execute together
```

```
With Synchronization

Thread A

↓

LOCK

↓

Critical Section

↓

UNLOCK

↓

Thread B
```

Only one thread executes the critical section at a time.

---

# Critical Section

A **critical section** is the part of the code that accesses shared mutable data.

Example:

```java
count++;
```

Although it is only one line of Java,

it performs:

```
Read

↓

Modify

↓

Write
```

The entire sequence must execute as a single protected operation.

---

# Using `synchronized`

Java provides the `synchronized` keyword.

```java
public synchronized void increment() {

    count++;
}
```

Now,

before executing the method,

the JVM acquires the object's intrinsic lock.

```
Thread A

↓

Acquire Lock

↓

increment()

↓

Release Lock
```

If another thread calls:

```java
increment();
```

it must wait until the lock is released.

---

# Synchronized Block

Sometimes,

only a small portion of the method requires protection.

```java
public void process() {

    validate();

    synchronized (this) {

        updateSharedState();

    }

    log();
}
```

Only the critical section is synchronized.

Advantages:

- Smaller lock scope
- Better concurrency
- Improved throughput

> **Best Practice**
>
> Synchronize only the code that actually needs protection.

---

# What Object Is Locked?

Consider:

```java
synchronized (lock) {

    // Critical Section

}
```

The thread acquires the monitor associated with:

```
lock
```

Not necessarily:

```
this
```

Example:

```java
private final Object lock = new Object();

public void increment() {

    synchronized (lock) {

        count++;

    }
}
```

Using a dedicated lock object avoids exposing your synchronization strategy to external code.

---

# Static Synchronization

Instance methods lock:

```
this
```

Static synchronized methods lock:

```
Class Object
```

Example:

```java
public static synchronized void updateConfig() {

}
```

Conceptually:

```
MyService.class

↓

Monitor Lock

↓

Static Method
```

This protects shared static data.

---

# Drawbacks of `synchronized`

Although simple,

it has limitations.

### Blocking

If one thread holds the lock,

every other thread must wait.

---

### Reduced Throughput

Large synchronized sections reduce parallelism.

---

### No Timeout

A waiting thread cannot specify:

> "Wait for only two seconds."

---

### No Fairness

The JVM does not guarantee that waiting threads acquire the lock in the order they arrived.

---

# Enter `ReentrantLock`

Java provides a more flexible locking mechanism.

```java
Lock lock = new ReentrantLock();
```

Usage:

```java
lock.lock();

try {

    count++;

} finally {

    lock.unlock();

}
```

Always release the lock inside a `finally` block.

Otherwise,

an exception may leave the lock permanently held.

---

# Why Is It Called Reentrant?

Suppose a thread already owns the lock.

```
Thread A

↓

methodA()

↓

methodB()
```

If both methods require the same lock,

the thread can acquire it again without deadlocking itself.

This property is called **reentrancy**.

The lock maintains an internal hold count.

```
First lock()  → Hold Count = 1

Second lock() → Hold Count = 2

unlock()      → Hold Count = 1

unlock()      → Hold Count = 0
```

The lock is released only when the hold count reaches zero.

---

# Advantages of `ReentrantLock`

### Try to Acquire a Lock

```java
if (lock.tryLock()) {

    try {

        process();

    } finally {

        lock.unlock();

    }

}
```

The thread continues only if the lock is immediately available.

---

### Timed Lock Acquisition

```java
lock.tryLock(2, TimeUnit.SECONDS);
```

Instead of waiting forever,

the thread can give up after a timeout.

---

### Interruptible Locking

```java
lock.lockInterruptibly();
```

A waiting thread can respond to interruption requests.

---

### Optional Fairness

```java
new ReentrantLock(true);
```

Fair locks attempt to grant access in FIFO order.

Keep in mind that fairness can reduce throughput because it limits scheduling flexibility.

---

# `synchronized` vs `ReentrantLock`

| Feature | `synchronized` | `ReentrantLock` |
|----------|----------------|-----------------|
| Simple syntax | ✓ | ✗ |
| Automatic lock release | ✓ | ✗ (must unlock manually) |
| Timeout support | ✗ | ✓ |
| Interruptible lock acquisition | ✗ | ✓ |
| Fair locking option | ✗ | ✓ |
| Reentrant | ✓ | ✓ |

---

# Which Should You Use?

For most business applications,

start with:

```java
synchronized
```

Choose:

```java
ReentrantLock
```

only when you require advanced capabilities such as:

- Timeouts
- Interruptible waits
- Fairness
- Non-blocking lock attempts (`tryLock()`)

> **Interview Tip**
>
> Don't choose `ReentrantLock` simply because it has more features. Choose it when your requirements justify the additional complexity.

---

# Production Perspective

An order processing service synchronizes the entire payment workflow.

```
Validate

↓

Call Payment Gateway

↓

Write Database

↓

Send Email
```

Since the external payment call takes several seconds,

all other threads remain blocked waiting for the same lock.

The implementation is redesigned so that only the shared state update is synchronized.

This significantly improves throughput while maintaining correctness.

The lesson:

**Protect the critical section—not the entire workflow.**

# Interview Focus

### Question

What is the difference between `synchronized` and `ReentrantLock`?

### Strong Senior-Level Answer

> Both provide mutual exclusion and are reentrant. `synchronized` is built into the Java language, automatically acquires and releases the intrinsic monitor, and is suitable for most use cases. `ReentrantLock` offers additional features such as timed lock acquisition, interruptible waiting, optional fairness, and `tryLock()`, making it appropriate for more advanced concurrency scenarios.

---

# Common Mistakes

❌ Synchronizing an entire method when only a few lines require protection.

❌ Forgetting to release a `ReentrantLock`.

❌ Holding a lock while performing slow I/O or network calls.

❌ Assuming fair locks always improve performance.

---

# Engineering Insight

A frequent production issue is **locking around slow operations**.

For example:

```java
synchronized (lock) {

    callExternalService();

}
```

If the external service responds slowly,

every waiting thread is blocked unnecessarily.

A better design is to:

1. Perform the slow operation outside the lock.
2. Synchronize only the update to shared state.

Reducing lock duration is often more effective than changing the lock implementation.

---

# Engineering Takeaways

- Synchronization protects shared mutable state.
- `synchronized` uses intrinsic monitor locks.
- Synchronize the smallest possible critical section.
- `ReentrantLock` provides advanced locking features.
- Short lock duration is essential for scalable applications.

---

# Quick Revision

✓ Synchronization prevents concurrent execution of critical sections.

✓ `synchronized` automatically releases locks.

✓ `ReentrantLock` must always be unlocked in a `finally` block.

✓ Both locking mechanisms are reentrant.

✓ Minimize lock scope to improve scalability.

# Senior Interview Challenge

A service synchronizes an entire method:

```java
public synchronized void processOrder(Order order) {

    validate(order);

    paymentGateway.charge(order);

    inventory.update(order);

    notification.send(order);

}
```

During peak traffic, response times increase dramatically.

### How would you redesign this implementation?

**Expected Discussion Points**

- Identify the actual shared mutable state.
- Avoid holding locks during network calls or I/O.
- Synchronize only the inventory update if that's the shared resource.
- Consider finer-grained locking or concurrent data structures.
- Evaluate whether the shared state can be eliminated entirely.

Senior engineers optimize **lock scope**, not just lock type.
