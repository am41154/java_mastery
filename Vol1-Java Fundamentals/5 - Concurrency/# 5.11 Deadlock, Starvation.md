# 5.11 Deadlock, Starvation, and Livelock

### Why This Matters

Concurrency bugs are often difficult to reproduce.

Unlike syntax errors or compilation failures, they may appear only under heavy production traffic.

Three of the most common concurrency failures are:

- Deadlock
- Starvation
- Livelock

A senior backend engineer should be able to recognize the symptoms, explain why they occur, and design systems to avoid them.

---

# Deadlock

A **deadlock** occurs when two or more threads wait indefinitely for each other to release resources.

None of the threads can proceed.

Imagine two locks:

```
Lock A

Lock B
```

Two threads execute in opposite order.

```
Thread A

↓

Acquire Lock A

↓

Waiting for Lock B
```

```
Thread B

↓

Acquire Lock B

↓

Waiting for Lock A
```

Result:

```
Nobody moves.
```

Both threads wait forever.

---

# Visualizing a Deadlock

```
          Thread A
              │
     Holds Lock A
              │
     Waiting for Lock B
              ▲
              │
              │
     Waiting for Lock A
              │
     Holds Lock B
              │
          Thread B
```

A circular dependency has formed.

---

# Conditions Required for Deadlock

A deadlock generally requires four conditions:

1. **Mutual Exclusion**
   - A resource can be held by only one thread at a time.

2. **Hold and Wait**
   - A thread holds one resource while waiting for another.

3. **No Preemption**
   - Resources cannot be forcibly taken away.

4. **Circular Wait**
   - Threads form a circular chain of dependencies.

Removing any one of these conditions prevents deadlock.

---

# Avoiding Deadlocks

### Acquire Locks in a Consistent Order

Bad:

```
Thread A

Lock A → Lock B
```

```
Thread B

Lock B → Lock A
```

Good:

```
All Threads

Lock A

↓

Lock B
```

Using a consistent lock ordering is one of the simplest and most effective deadlock prevention techniques.

---

### Keep Lock Scope Small

Avoid holding locks while:

- Calling external APIs
- Reading large files
- Performing slow database operations
- Waiting for user input

The longer a lock is held,

the greater the chance of contention.

---

### Use `tryLock()`

With `ReentrantLock`:

```java
if (lock.tryLock()) {

    try {

        // Work

    } finally {

        lock.unlock();

    }

}
```

Instead of waiting forever,

the thread can fail gracefully or retry later.

---

# Starvation

A thread experiences **starvation** when it waits indefinitely because other threads continuously receive access to a shared resource.

Example:

```
High Priority Threads

↓

High Priority Threads

↓

High Priority Threads

↓

Low Priority Thread

(Still Waiting)
```

The low-priority thread may never execute.

Unlike deadlock,

the application is still making progress.

The problem is that **one specific thread is not**.

---

# Causes of Starvation

- Excessive thread priorities
- Long-running synchronized blocks
- Unfair lock acquisition
- Resource monopolization

---

# Preventing Starvation

Good practices include:

- Keep critical sections short.
- Avoid unnecessary thread priorities.
- Consider fair locks when fairness is more important than maximum throughput.
- Balance workloads across worker threads.

---

# Livelock

A **livelock** occurs when threads remain active but make no useful progress.

Unlike deadlock,

threads continue running.

Example:

Imagine two people walking toward each other in a hallway.

```
Person A

Moves Left

↓

Person B

Moves Right

↓

Still Blocking

↓

Both Move Again

↓

Still Blocking
```

Both remain polite,

but neither gets past the other.

The system is busy,

yet no useful work is completed.

---

# Deadlock vs Starvation vs Livelock

| Problem | Threads Running? | Progress? |
|----------|------------------|-----------|
| Deadlock | ✗ No | ✗ No |
| Starvation | Some | Partial |
| Livelock | ✓ Yes | ✗ No |

---

# Detecting Deadlocks

Production JVMs provide several diagnostic tools.

Common approaches include:

- Thread dumps (`jstack`)
- JVM monitoring tools
- Java Flight Recorder (JFR)
- VisualVM
- Mission Control

A thread dump often shows:

```
Found one Java-level deadlock
```

along with the threads and locks involved.

Learning to read thread dumps is an important production skill.

---

# Spring Boot Perspective

Suppose two services synchronize on different resources.

```
InventoryService

↓

Lock Inventory

↓

Call PaymentService
```

Meanwhile:

```
PaymentService

↓

Lock Payment

↓

Call InventoryService
```

Under concurrent requests,

a deadlock becomes possible.

This is why service-layer synchronization should be designed carefully and external calls should generally occur outside synchronized sections.

---

# Production Perspective

An order processing system occasionally stops processing requests.

CPU usage remains low.

Memory usage is stable.

Thread dumps reveal:

```
Thread-21

Waiting for Lock A
```

```
Thread-34

Waiting for Lock B
```

Each thread already owns the lock the other needs.

The issue is diagnosed as a deadlock caused by inconsistent lock ordering.

After enforcing a consistent lock acquisition strategy,

the problem disappears.

# Interview Focus

### Question

What is the difference between deadlock, starvation, and livelock?

### Strong Senior-Level Answer

> A deadlock occurs when threads wait indefinitely for resources held by one another, preventing all progress. Starvation occurs when a thread is continually denied access to a resource while other threads continue to execute. Livelock occurs when threads remain active and keep responding to each other, but no useful work is completed.

---

# Common Mistakes

❌ Acquiring multiple locks in different orders.

❌ Holding locks during slow I/O operations.

❌ Assuming deadlocks occur only in large systems.

❌ Ignoring thread dumps during production debugging.

---

# Engineering Insight

Concurrency bugs rarely appear during development because:

- Few threads are active.
- Timing differences are small.
- Test environments have low contention.

Production systems expose these problems because hundreds or thousands of threads interact simultaneously.

This is why concurrency design should prioritize **correctness first, performance second**.

---

# Engineering Takeaways

- Deadlocks result from circular waiting.
- Starvation prevents individual threads from making progress.
- Livelock keeps threads busy without useful work.
- Consistent lock ordering prevents many deadlocks.
- Thread dumps are invaluable for diagnosing concurrency issues.

---

# Quick Revision

✓ Deadlock = waiting forever.

✓ Starvation = one thread never gets a chance.

✓ Livelock = everyone is moving, but nothing completes.

✓ Acquire locks in a consistent order.

✓ Keep synchronized sections as short as possible.

---

# Senior Interview Challenge

Two services synchronize on shared resources.

Occasionally, production requests stop responding.

Thread dumps show:

```
Thread A

Waiting for Lock B
```

```
Thread B

Waiting for Lock A
```

### How would you investigate and resolve the issue?

**Expected Discussion Points**

- Confirm the deadlock using thread dumps or JFR.
- Identify the lock acquisition order.
- Refactor code to acquire locks consistently.
- Reduce lock scope where possible.
- Consider replacing explicit locking with higher-level concurrent utilities if appropriate.

Senior engineers diagnose concurrency failures using evidence—such as thread dumps and lock ownership—not by guesswork.
