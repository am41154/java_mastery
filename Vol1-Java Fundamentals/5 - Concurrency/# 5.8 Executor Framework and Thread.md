# 5.8 Executor Framework and Thread Pools

### Why This Matters

Consider a REST API that receives:

```
10,000 requests per minute
```

Suppose every request executes:

```java
new Thread(() -> process()).start();
```

Initially, everything appears to work.

As traffic increases:

- Memory consumption rises.
- CPU utilization increases.
- Context switching becomes excessive.
- The JVM spends more time managing threads than executing business logic.

The application slows down dramatically.

The problem isn't concurrency.

The problem is **thread management**.

This is exactly why Java introduced the **Executor Framework**.

---

# Why Not Create Threads Directly?

Creating a thread is an expensive operation.

Each thread requires:

- Stack memory
- JVM resources
- Operating system scheduling
- Context switching

Imagine receiving 5,000 requests.

```
Request 1

↓

New Thread

-------------------

Request 2

↓

New Thread

-------------------

Request 3

↓

New Thread
```

Thousands of threads are created.

Many spend most of their time waiting.

This approach does not scale.

---

# The Idea Behind Thread Pools

Instead of creating a new thread for every task,

reuse a fixed set of worker threads.

```
Incoming Tasks

      │

      ▼

Task Queue

      │

      ▼

 ┌───────────────┐
 │ Thread Pool   │
 ├───────────────┤
 │ Worker-1      │
 │ Worker-2      │
 │ Worker-3      │
 │ Worker-4      │
 └───────────────┘
```

Tasks are submitted.

Threads are reused.

This dramatically reduces overhead.

---

# What Is Executor?

Java provides the `Executor` abstraction.

Instead of saying:

> "Create a thread."

You say:

> "Execute this task."

```java
Executor executor =
    Executors.newSingleThreadExecutor();

executor.execute(task);
```

The caller no longer manages threads directly.

It delegates execution to the framework.

---

# ExecutorService

The most commonly used implementation is:

```java
ExecutorService executor =
    Executors.newFixedThreadPool(4);
```

Now submit tasks:

```java
executor.submit(task);
```

Execution flow:

```
Task

↓

ExecutorService

↓

Thread Pool

↓

Worker Thread
```

Notice that no new thread is created for each request.

Existing workers perform the work.

---

# Lifecycle of a Task

Suppose the pool contains:

```
4 Threads
```

Eight tasks arrive.

```
Task 1

↓

Worker 1

-----------------

Task 2

↓

Worker 2

-----------------

Task 3

↓

Worker 3

-----------------

Task 4

↓

Worker 4
```

Remaining tasks wait in the queue.

```
Task Queue

↓

Task 5

Task 6

Task 7

Task 8
```

As workers finish,

they automatically pick up the next task.

# Creating Different Thread Pools

## Fixed Thread Pool

```java
ExecutorService executor =
    Executors.newFixedThreadPool(8);
```

Characteristics:

- Fixed number of workers
- Reuses threads
- Suitable for most server applications

---

## Single Thread Executor

```java
Executors.newSingleThreadExecutor();
```

Characteristics:

- Only one worker thread
- Tasks execute sequentially
- Preserves submission order

Useful for background jobs that must not run concurrently.

---

## Cached Thread Pool

```java
Executors.newCachedThreadPool();
```

Characteristics:

- Creates threads as needed
- Reuses idle threads
- Can grow very large

Useful for short-lived asynchronous tasks, but it can create an unbounded number of threads if tasks arrive faster than they complete.

---

## Scheduled Thread Pool

```java
Executors.newScheduledThreadPool(2);
```

Supports delayed and periodic execution.

Example:

```java
scheduler.schedule(
    task,
    5,
    TimeUnit.SECONDS
);
```

Or execute repeatedly:

```java
scheduler.scheduleAtFixedRate(...);
```

Ideal for:

- Cleanup jobs
- Cache refresh
- Health checks
- Scheduled reports

---

# Shutting Down a Thread Pool

Always shut down an `ExecutorService` when it is no longer needed.

```java
executor.shutdown();
```

This:

- Stops accepting new tasks.
- Allows existing tasks to finish.

To stop immediately:

```java
executor.shutdownNow();
```

Be aware that `shutdownNow()` attempts to interrupt running tasks and returns tasks that never started. It does **not** guarantee immediate termination.

> **Best Practice**
>
> Prefer `shutdown()` for graceful application shutdown.

# Thread Pool Sizing

A common interview question:

> **How many threads should a thread pool have?**

The answer depends on the workload.

---

### CPU-Bound Tasks

Examples:

- Image processing
- Encryption
- Compression
- Mathematical calculations

Recommended pool size:

```
Approximately

Number of CPU cores
```

Reason:

Additional threads mostly increase context switching.

---

### I/O-Bound Tasks

Examples:

- Database calls
- REST APIs
- File systems
- Kafka
- Redis

Threads spend significant time waiting.

More threads than CPU cores are often appropriate because while some threads wait for I/O, others can continue executing.

> **Interview Tip**
>
> Thread pool sizing is driven by **workload characteristics**, not by a fixed formula.

---

# ExecutorService vs Thread

| Thread | ExecutorService |
|---------|-----------------|
| Creates a new thread | Reuses worker threads |
| Manual lifecycle management | Framework-managed |
| Poor scalability | High scalability |
| No task queue | Built-in queue |
| Rarely used directly in enterprise apps | Standard approach |

# Spring Boot Perspective

In a typical Spring Boot application:

```
HTTP Request

↓

Tomcat Worker Thread

↓

Controller

↓

Service

↓

Repository
```

The request is already executing inside a thread pool managed by the web server.

Creating additional threads inside business logic is often unnecessary and can reduce performance.

Instead,

use asynchronous execution only when the workload genuinely benefits from it.

---

# Production Perspective

A notification service sends emails.

Initially:

```java
new Thread(() -> sendEmail()).start();
```

Under heavy traffic,

the JVM creates thousands of threads.

Memory usage increases dramatically.

The implementation is replaced with:

```java
ExecutorService executor =
    Executors.newFixedThreadPool(20);
```

Email tasks are submitted to the pool.

The service now:

- Uses a predictable number of threads.
- Maintains stable memory usage.
- Handles higher throughput.
- Becomes easier to monitor and tune.

# Interview Focus

### Question

Why is `ExecutorService` preferred over manually creating threads?

### Strong Senior-Level Answer

> Creating threads is expensive. `ExecutorService` separates task submission from thread management by reusing worker threads through a thread pool. This reduces creation overhead, limits resource consumption, improves scalability, and provides features such as task queues, graceful shutdown, and lifecycle management.

---

# Common Mistakes

❌ Creating a new thread for every request.

❌ Forgetting to shut down an `ExecutorService`.

❌ Using `newCachedThreadPool()` without understanding its growth characteristics.

❌ Choosing thread pool sizes without considering whether the workload is CPU-bound or I/O-bound.

---

# Engineering Insight

The **Executor Framework** shifts your thinking.

Instead of asking:

> "How do I create more threads?"

Ask:

> "How should I schedule and manage my tasks?"

This distinction is fundamental.

Modern Java applications manage **tasks**, while the framework manages **threads**.

---

# Engineering Takeaways

- Thread creation is expensive.
- Thread pools improve scalability by reusing workers.
- `ExecutorService` is the standard API for task execution.
- Size thread pools according to workload characteristics.
- Always shut down thread pools gracefully.

---

# Quick Revision

✓ Prefer `ExecutorService` over `Thread`.

✓ Fixed thread pools are suitable for many server applications.

✓ Cached thread pools can grow without bound.

✓ Scheduled thread pools support delayed and periodic execution.

✓ Design around tasks, not threads.

# Senior Interview Challenge

A Spring Boot application processes uploaded documents.

Each upload creates:

```java
new Thread(() -> processDocument()).start();
```

At peak traffic, the application experiences:

- High memory usage
- Increased GC activity
- Thousands of active threads
- Slower response times

### How would you redesign the solution?

**Expected Discussion Points**

- Replace manual thread creation with an `ExecutorService`.
- Choose an appropriate thread pool based on whether processing is CPU-bound or I/O-bound.
- Introduce bounded queues where appropriate to prevent overload.
- Implement graceful shutdown.
- Monitor queue length, active thread count, and task execution time.

A senior engineer treats thread pools as a capacity-planning and resource-management problem, not just a concurrency API.
