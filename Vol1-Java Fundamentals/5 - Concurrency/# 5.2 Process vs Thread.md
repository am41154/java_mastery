# 5.2 Process vs Thread

### Why This Matters

One of the first questions asked in Java concurrency interviews is:

> **"What is the difference between a process and a thread?"**

Many candidates answer:

> "A process is heavier than a thread."

While correct, that answer barely scratches the surface.

As a backend engineer, you should understand:

- How the operating system manages processes.
- How threads relate to a process.
- Why Java applications use multiple threads instead of multiple processes.
- How this impacts performance and memory.

---

# What Is a Process?

A **process** is an independent running program.

Examples:

- Google Chrome
- IntelliJ IDEA
- Spotify
- A Spring Boot application

Each process has its own:

- Memory space
- Heap
- Stack
- Open files
- Network connections
- Operating system resources

Conceptually:

```
Operating System

│

├── Chrome Process

├── IntelliJ Process

├── Spotify Process

└── Spring Boot Process
```

Each process is isolated from the others.

A failure in one process normally does not affect another.

---

# Process Isolation

Imagine two applications.

```
Calculator

Memory

────────────

Spring Boot

Memory
```

The Calculator application cannot directly access the Spring Boot application's memory.

The operating system enforces this isolation.

This provides:

- Stability
- Security
- Fault isolation

Communication between processes requires mechanisms such as:

- Sockets
- REST APIs
- Message brokers
- Shared memory (OS-specific)
- Pipes

Inter-process communication (IPC) is relatively expensive compared to communication between threads.

---

# What Is a Thread?

A **thread** is the smallest unit of execution within a process.

A single process can contain multiple threads.

```
Spring Boot Process

│

├── Thread 1

├── Thread 2

├── Thread 3

└── Thread 4
```

Unlike processes, threads share most resources belonging to the process.

Each thread has its own:

- Program counter
- Stack
- Execution state

But threads share:

- Heap
- Objects
- Static variables
- Open files
- Database connections (through pools)
- Other process-level resources

This sharing makes communication between threads fast—but it also introduces concurrency problems.

---

# Visualizing Shared Memory

```
                Spring Boot Process

        ┌────────────────────────────┐
        │           Heap             │
        │                            │
        │   Shared Java Objects      │
        └────────────────────────────┘

          ▲           ▲          ▲

      Thread-1    Thread-2   Thread-3

       Stack       Stack      Stack
```

Every thread can access the same objects stored in the heap.

This enables efficient collaboration.

It also creates the possibility of race conditions.

---

# Why Use Threads Instead of Processes?

Suppose an application receives 500 HTTP requests.

Option 1:

Create 500 separate processes.

Problems:

- Huge memory usage
- Slow process creation
- Expensive context switching
- High operating system overhead

Option 2:

Use multiple threads within one process.

Benefits:

- Lower memory consumption
- Faster creation
- Shared resources
- Better performance

This is why Java application servers use threads to process requests.

---

# Threads in a Spring Boot Application

Consider a REST API.

```
GET /orders
```

Five users send requests simultaneously.

```
Spring Boot Process

│

├── Thread-17

├── Thread-18

├── Thread-19

├── Thread-20

└── Thread-21
```

Each thread executes the same application code but processes a different request.

Although the threads are independent, they all access the same:

- Service objects
- Singleton beans
- Cache
- Database connection pool
- Shared configuration

Proper synchronization is therefore essential.

# Context Switching

The CPU can execute only a limited number of threads simultaneously.

When more threads exist than available CPU cores,

the operating system switches execution between them.

```
CPU

↓

Thread A

↓

Thread B

↓

Thread C

↓

Thread A
```

This is called **context switching**.

Although context switching enables concurrency, it has a cost.

Too many threads can actually reduce performance.

> **Interview Tip**
>
> Creating more threads than the workload requires does not automatically make an application faster. Excessive context switching can become a bottleneck.

---

# Production Perspective

A team builds an image-processing service.

Initially, each image is processed by launching a new operating system process.

As traffic increases, memory usage and startup time become unacceptable.

The application is redesigned to use a fixed thread pool within a single JVM.

Processing throughput improves significantly because threads are much lighter than processes and share application resources efficiently.

# Interview Focus

### Question

What is the difference between a process and a thread?

### Strong Senior-Level Answer

> A process is an independent execution environment with its own memory and operating system resources. A thread is a lightweight unit of execution within a process. Threads share the process's heap and resources but maintain their own stacks and execution state. This makes thread creation and communication much more efficient, while also introducing synchronization challenges.

---

# Common Misconceptions

❌ Every thread has its own heap.

❌ Threads cannot access the same object.

❌ More threads always improve performance.

❌ Processes communicate as efficiently as threads.

---

# Engineering Insight

A common misconception is that thread safety only matters for static variables.

In reality, any object shared across threads can become a source of concurrency bugs.

In a typical Spring Boot application, singleton beans are shared by all request-processing threads.

If these beans maintain mutable state without proper synchronization, subtle race conditions can occur even under moderate traffic.

Designing stateless services whenever possible greatly simplifies concurrent programming.

---

# Engineering Takeaways

- A process is an independent running program.
- A process can contain multiple threads.
- Threads share the process heap but have separate stacks.
- Thread communication is faster than inter-process communication.
- Shared memory improves efficiency but requires careful synchronization.

---

# Quick Revision

✓ Processes are isolated from each other.

✓ Threads belong to a process.

✓ Threads share heap memory.

✓ Each thread has its own stack.

✓ Too many threads increase context-switching overhead.

# Senior Interview Challenge

Your application performs CPU-intensive image processing.

A developer proposes creating **1,000 threads** on an **8-core machine** to maximize performance.

### How would you evaluate this proposal?

**Expected Discussion Points**

- Only a limited number of threads can execute simultaneously.
- The operating system will spend significant time context switching.
- Thread creation consumes memory.
- Thread pools should generally be sized according to workload characteristics (CPU-bound vs I/O-bound).
- More threads do not necessarily result in better throughput.

Understanding this trade-off is a key indicator of production-level concurrency knowledge.
