# 5.5 Thread Safety and Race Conditions

### Why This Matters

Concurrency becomes difficult **not because multiple threads exist**, but because they **share mutable data**.

A piece of code may work perfectly during development with a single user.

Under production load, the exact same code may:

- Produce incorrect results
- Lose updates
- Corrupt data
- Fail intermittently

These bugs are among the hardest to diagnose because they depend on timing.

Understanding **thread safety** and **race conditions** is one of the most important skills for a backend engineer.

---

# What Is Thread Safety?

A class is **thread-safe** if it behaves correctly when accessed by multiple threads simultaneously.

Consider a simple service:

```java
class Counter {

    private int count = 0;

    public void increment() {
        count++;
    }
}
```

Looks harmless.

Now imagine hundreds of requests calling:

```java
counter.increment();
```

at the same time.

Is the result guaranteed to be correct?

**No.**

---

# Understanding Race Conditions

A **race condition** occurs when multiple threads access shared data concurrently, and the final result depends on the order in which the threads execute.

Example:

```
Initial Count = 0
```

Two threads execute:

```java
count++;
```

At first glance,

you might expect:

```
Final Count = 2
```

But that's not guaranteed.

---

# Why?

The statement:

```java
count++;
```

is **not** a single CPU instruction.

Conceptually, it consists of three steps:

```
1. Read value

2. Add 1

3. Write value back
```

---

# The Race

Suppose two threads execute simultaneously.

```
Count = 0

Thread A reads 0

Thread B reads 0

Thread A increments to 1

Thread B increments to 1

Thread A writes 1

Thread B writes 1
```

Expected:

```
2
```

Actual:

```
1
```

One update is lost.

This is called a **lost update**, one of the most common race conditions.

---

# Visualizing the Problem

```
Shared Counter

Count = 0

          ▲
          │

Thread A      Thread B

Read 0        Read 0

+1            +1

Write 1       Write 1

Final = 1
```

Neither thread is "wrong."

The issue is that they operate on shared state without coordination.

---

# Real-World Example

Consider an inventory service.

```
Available Stock = 1
```

Two customers purchase the last item.

```
Thread A

↓

Read Stock = 1

------------------

Thread B

↓

Read Stock = 1

------------------

Thread A

↓

Update Stock = 0

------------------

Thread B

↓

Update Stock = 0
```

The system has now sold:

```
2 products

while

Only 1 existed.
```

This bug may occur only under heavy traffic, making it difficult to reproduce.

---

# What Makes Code Thread-Safe?

Generally, code is thread-safe if it avoids unsafe shared mutable state or coordinates access correctly.

Common approaches include:

- Synchronization (`synchronized`)
- Explicit locks (`ReentrantLock`)
- Atomic classes (`AtomicInteger`)
- Immutable objects
- Concurrent collections
- Thread confinement (each thread owns its own data)

We'll explore each of these later in the chapter.

---

# Stateless Objects Are Naturally Thread-Safe

Consider:

```java
class Calculator {

    public int add(int a, int b) {
        return a + b;
    }
}
```

No shared fields.

No mutable state.

Every invocation depends only on its inputs.

This class is inherently thread-safe.

---

# Mutable Shared State Is the Real Problem

Now compare:

```java
class UserService {

    private User currentUser;

}
```

If multiple threads modify:

```java
currentUser
```

unexpected behavior becomes possible.

The problem is **not the object itself**.

The problem is **shared mutable state**.

---

# Spring Boot Perspective

By default,

Spring creates **singleton beans**.

```
Application

↓

Singleton Service

↓

Thread 1

Thread 2

Thread 3

Thread 4
```

Every request accesses the same service instance.

If the service stores mutable request-specific data in instance variables,

the application is no longer thread-safe.

Good Spring services are typically:

- Stateless
- Immutable where possible
- Free of shared mutable request state

# Interview Focus

### Question

What is a race condition?

### Strong Senior-Level Answer

> A race condition occurs when multiple threads access and modify shared mutable data concurrently, and the correctness of the result depends on the order in which those threads execute. Without proper synchronization, operations such as incrementing a counter or updating shared objects can produce inconsistent or lost updates.

---

# Common Misconceptions

❌ `count++` is an atomic operation.

❌ Local variables require synchronization.

❌ Every object shared between threads is unsafe.

❌ Singleton beans are automatically thread-safe.

---

# Engineering Insight

The simplest way to make code thread-safe is often **not** to add synchronization.

Instead, redesign the code to eliminate shared mutable state.

For example:

- Keep data local to the request.
- Use immutable value objects.
- Pass state through method parameters instead of storing it in fields.

Reducing shared state typically results in code that is both simpler and more scalable.

---

# Engineering Takeaways

- Thread safety is about correct behavior under concurrent access.
- Race conditions arise from unsynchronized access to shared mutable state.
- `count++` is not atomic.
- Stateless classes are naturally thread-safe.
- Eliminating shared mutable state is often better than synchronizing it.

---

# Quick Revision

✓ Race conditions depend on execution timing.

✓ Shared mutable state is the primary source of concurrency bugs.

✓ `count++` performs read-modify-write operations.

✓ Singleton Spring beans should remain stateless.

✓ Thread safety begins with good design, not locks.

# Senior Interview Challenge

A Spring Boot service contains:

```java
@Service
public class OrderService {

    private int processedOrders = 0;

    public void process(Order order) {
        processedOrders++;
    }
}
```

The service receives **500 concurrent requests per second**.

### What problems do you see, and how would you redesign this implementation?

**Expected Discussion Points**

- `processedOrders` is shared mutable state.
- `processedOrders++` is not atomic and can lose updates.
- Options include:
  - Using `AtomicInteger` if only a counter is needed.
  - Synchronizing access if appropriate.
  - Moving metrics to a monitoring framework (e.g., Micrometer) instead of maintaining them manually.
  - Keeping the service stateless whenever possible.

A senior engineer first questions **whether the shared state is necessary at all**, before deciding how to synchronize it.
