# 5.1 Why Concurrency Matters

### Why This Matters

Imagine an e-commerce application during a flash sale.

Thousands of users are simultaneously:

- Searching for products
- Adding items to their carts
- Making payments
- Tracking orders

If the application could process only **one request at a time**, every other user would have to wait.

```
Request 1

↓

Request 2

↓

Request 3

↓

Request 4

↓

Request 5
```

The application would quickly become unusable.

Modern backend systems solve this problem using **concurrency**, allowing multiple tasks to make progress at the same time.

Understanding concurrency is essential for building scalable, responsive, and reliable Java applications.

---

# What Is Concurrency?

Concurrency is the ability of a program to manage multiple tasks during the same period of time.

It does **not** necessarily mean that tasks execute at the exact same instant.

Instead, the operating system and JVM schedule work so that multiple tasks appear to progress together.

Conceptually:

```
Without Concurrency

Task A

↓

Task B

↓

Task C
```

```
With Concurrency

Task A

↓

Task B

↓

Task C

(All progressing during the same time period)
```

---

# Concurrency vs Parallelism

These terms are often used interchangeably, but they are not the same.

### Concurrency

Multiple tasks are **in progress** during the same time period.

A single CPU core can rapidly switch between tasks.

```
CPU

↓

Task A

↓

Task B

↓

Task A

↓

Task C

↓

Task B
```

Although only one task executes at any instant, all tasks continue making progress.

---

### Parallelism

Multiple tasks execute **simultaneously**.

This requires multiple CPU cores.

```
Core 1      Core 2      Core 3

Task A      Task B      Task C
```

All three tasks truly execute at the same time.

> **Interview Tip**
>
> Concurrency is about **managing multiple tasks efficiently**.
>
> Parallelism is about **executing multiple tasks simultaneously**.

Parallelism is often built on top of concurrency.

---

# Why Modern Applications Need Concurrency

Almost every enterprise application performs several independent operations.

Consider an online banking request.

```
Receive HTTP Request

↓

Authenticate User

↓

Validate Input

↓

Read Database

↓

Update Account

↓

Publish Kafka Event

↓

Write Audit Log

↓

Return Response
```

Some of these operations involve waiting:

- Database calls
- Network communication
- File access
- External APIs

While one thread waits for I/O, another thread can execute useful work.

This improves overall system throughput.

---

# A Spring Boot Example

Suppose your application exposes:

```
GET /customers
```

At 10:00 AM,

five users send requests simultaneously.

```
Client 1

↓

Client 2

↓

Client 3

↓

Client 4

↓

Client 5
```

The embedded web server assigns different worker threads.

```
Thread-1

↓

Request 1

----------------

Thread-2

↓

Request 2

----------------

Thread-3

↓

Request 3

----------------

Thread-4

↓

Request 4

----------------

Thread-5

↓

Request 5
```

Each request proceeds independently.

Without concurrency, only one request could be processed while the others remained blocked.

---

# Benefits of Concurrency

Properly designed concurrent applications provide several advantages.

### Better Responsiveness

One slow operation does not block unrelated requests.

---

### Higher Throughput

More work is completed during the same period.

---

### Improved Resource Utilization

While one thread waits for I/O, another thread can use the CPU.

---

### Scalability

Applications can serve many users concurrently without requiring a separate process for each request.

---

# Concurrency Is Not Free

Concurrency introduces significant complexity.

Multiple threads may access the same data simultaneously.

```
Thread A

↓

Shared Object

↑

Thread B
```

Without proper synchronization, problems such as:

- Race conditions
- Lost updates
- Data corruption
- Deadlocks
- Memory visibility issues

can occur.

These issues are often intermittent and difficult to reproduce.

---

# Production Perspective

Consider an inventory management system.

Two customers purchase the last remaining item at nearly the same time.

Both requests read:

```
Stock = 1
```

Both decrease the value.

Both save:

```
Stock = 0
```

The system has now sold **two products** even though only one was available.

The application behaved correctly for a single user, but failed under concurrent access.

This is one of the most common examples of a **race condition**, which we'll explore later in this chapter.

# Interview Focus

### Question

Why is concurrency important in backend applications?

### Strong Senior-Level Answer

> Backend applications serve many users simultaneously. Concurrency allows multiple requests to make progress at the same time, improving responsiveness, throughput, and resource utilization. However, it also introduces challenges such as race conditions, synchronization, and memory visibility, which must be handled carefully.

# Common Misconceptions

❌ Concurrency always means parallel execution.

❌ More threads always improve performance.

❌ Thread safety is only required for high-traffic systems.

❌ Synchronization solves every concurrency problem.

---

# Engineering Insight

Concurrency should not be introduced simply because multiple threads are available.

It is beneficial only when:

- Independent tasks can execute concurrently.
- Shared state is managed safely.
- The overhead of thread management is justified.

Creating excessive threads increases:

- Context switching
- Memory consumption
- CPU overhead

In many cases, a well-designed thread pool outperforms creating hundreds of individual threads.

---

# Engineering Takeaways

- Concurrency enables multiple tasks to make progress together.
- Parallelism requires multiple CPU cores.
- Backend applications rely heavily on concurrency.
- Shared mutable state introduces synchronization challenges.
- Correctness is more important than raw speed.

# Senior Interview Challenge

Your Spring Boot service receives 2,000 requests per second.

A new developer suggests creating a brand-new thread for every incoming request to maximize performance.

### How would you respond?

**Expected Discussion Points**

- Creating a thread per request does not scale.
- Threads consume memory and CPU resources.
- Excessive threads increase context switching.
- Modern servers use thread pools (`ExecutorService`, Tomcat worker threads, etc.).
- Efficient concurrency is about managing threads wisely, not maximizing their number.

This mindset distinguishes engineers who understand concurrency from those who simply know the API.
