# 5.12 Choosing the Right Concurrency Tool

### Why This Matters

By now, you've learned numerous concurrency concepts and APIs:

- `Thread`
- `Runnable`
- `Callable`
- `Future`
- `CompletableFuture`
- `synchronized`
- `ReentrantLock`
- `volatile`
- `AtomicInteger`
- `ExecutorService`
- `ConcurrentHashMap`
- `BlockingQueue`
- `CountDownLatch`
- `Semaphore`

A common question remains:

> **Which one should I use?**

There is no single "best" concurrency tool.

The right choice depends on the problem you're solving.

This section provides a practical decision guide for real-world backend applications.

---

# Decision Flow

Start by asking one question:

> **What problem am I trying to solve?**

```
Need concurrent execution?

        │

        ▼

Need a return value?

      │        │

     No       Yes

      │        │

 Runnable   Callable

                │

                ▼

      ExecutorService
```

Now consider whether threads need to share data.

```
Shared Mutable State?

      │

     Yes

      │

Synchronization Required
```

The solution depends on the type of shared state.

---

# Scenario 1 – Simple Background Task

Example:

- Send email
- Write logs
- Generate thumbnails

Use:

```
Runnable
```

Submit it to:

```
ExecutorService
```

Avoid:

```
new Thread()
```

---

# Scenario 2 – Task Returns a Result

Example:

- Fetch customer details
- Calculate invoice
- Generate report

Use:

```
Callable
```

or

```
CompletableFuture
```

if asynchronous composition is required.

---

# Scenario 3 – Multiple Independent API Calls

Example:

```
Orders

Payments

Inventory

Recommendations
```

These operations are independent.

Best choice:

```
CompletableFuture
```

Benefits:

- Parallel execution
- Lower latency
- Better readability

---

# Scenario 4 – Shared Counter

Bad:

```java
int count;
```

Better:

```java
AtomicInteger
```

Use an atomic class when the requirement is limited to atomic operations such as incrementing, decrementing, or compare-and-set.

Avoid synchronizing an entire method for a simple counter.

---

# Scenario 5 – Shared Map

Bad:

```java
HashMap
```

Better:

```java
ConcurrentHashMap
```

Designed for concurrent access with significantly better scalability than synchronizing a standard map.

---

# Scenario 6 – Producer-Consumer Pattern

Example:

```
Orders

↓

Queue

↓

Workers
```

Use:

```
BlockingQueue
```

The queue manages waiting and coordination internally.

---

# Scenario 7 – One Thread Signals Another

Example:

```java
running = false;
```

Use:

```java
volatile
```

Only if:

- Visibility is required.
- No compound operations are performed.

Do **not** use `volatile` for counters or complex updates.

---

# Scenario 8 – Protecting Shared Business State

Example:

```
Bank Account

↓

Balance Update
```

Use:

```
synchronized
```

or

```
ReentrantLock
```

Choose `ReentrantLock` only when advanced features such as timeouts, interruptible locking, or `tryLock()` are required.

---

# Scenario 9 – Waiting for Multiple Tasks

Example:

```
Database

Cache

REST API

↓

Continue
```

Use:

```
CountDownLatch
```

If the synchronization is one-time.

For repeated synchronization phases,

consider:

```
CyclicBarrier
```

or

```
Phaser
```

depending on the use case.

---

# Scenario 10 – Limiting Resource Usage

Example:

```
Payment Gateway

Maximum

20 Requests
```

Use:

```
Semaphore
```

It limits the number of concurrent operations without manually managing queues or locks.

---

# Concurrency Decision Matrix

| Requirement | Recommended Tool |
|-------------|------------------|
| Execute background work | `ExecutorService` + `Runnable` |
| Return a result | `Callable` |
| Compose asynchronous tasks | `CompletableFuture` |
| Shared counter | `AtomicInteger` |
| Shared map | `ConcurrentHashMap` |
| Producer-consumer workflow | `BlockingQueue` |
| Visibility only | `volatile` |
| Mutual exclusion | `synchronized` |
| Advanced locking | `ReentrantLock` |
| Wait for tasks | `CountDownLatch` |
| Repeated synchronization | `CyclicBarrier` / `Phaser` |
| Limit concurrent access | `Semaphore` |

# Spring Boot Perspective

Most Spring Boot applications already use:

- Thread pools (embedded server, async executors)
- Concurrent collections
- Database connection pools
- Scheduled executors

As a developer,

your responsibility is usually **choosing the right abstraction**, not implementing low-level concurrency primitives from scratch.

---

# Production Best Practices

Before introducing concurrency, ask:

1. Can the design remain stateless?
2. Can shared mutable state be eliminated?
3. Can an existing concurrent collection solve the problem?
4. Is a thread pool already available?
5. Is asynchronous execution actually beneficial?

Often,

the simplest design is the most reliable.

---

# Interview Focus

### Question

When would you choose `AtomicInteger` instead of `synchronized`?

### Strong Senior-Level Answer

> `AtomicInteger` is ideal for simple atomic operations such as incrementing, decrementing, or compare-and-set because it provides lock-free thread safety with excellent performance. If multiple variables or a larger critical section must be updated consistently, `synchronized` or another locking mechanism is more appropriate.

---

# Common Mistakes

❌ Using `volatile` for counters.

❌ Creating threads manually instead of using a thread pool.

❌ Synchronizing entire methods unnecessarily.

❌ Using `HashMap` in shared concurrent code.

❌ Choosing concurrency primitives before questioning whether shared state is needed at all.

---

# Engineering Insight

One of the biggest differences between junior and senior engineers is **where they solve concurrency problems**.

A junior engineer often asks:

> "Which lock should I use?"

A senior engineer first asks:

> "Can I redesign this so that I don't need a lock?"

Reducing shared mutable state almost always leads to simpler, safer, and more scalable systems.

Concurrency should be introduced deliberately—not by default.

---

# Engineering Takeaways

- Choose concurrency tools based on the problem, not familiarity.
- Prefer high-level abstractions over manual thread management.
- Eliminate shared mutable state whenever possible.
- Use thread pools instead of creating threads directly.
- Optimize for correctness first, then performance.

# Senior Interview Challenge

You are designing a high-throughput Order Management System.

Requirements:

- Process thousands of orders per minute.
- Call multiple external services concurrently.
- Maintain an in-memory cache of active orders.
- Limit payment gateway requests to 50 concurrent calls.
- Execute background notifications.
- Keep request-processing services stateless.

### Design the concurrency approach.

**Expected Discussion Points**

- Use an `ExecutorService` for background task execution.
- Use `CompletableFuture` for parallel external service calls.
- Store active orders in a `ConcurrentHashMap`.
- Protect simple counters with `AtomicInteger` where appropriate.
- Use a `Semaphore` to enforce payment gateway limits.
- Keep Spring singleton services stateless.
- Avoid manual thread creation and unnecessary synchronization.
- Monitor thread pools, queue sizes, and contention in production.

A senior engineer doesn't rely on a single concurrency primitive—they combine the right tools to build systems that are correct, scalable, and maintainable.
