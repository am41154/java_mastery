# 5.10 Concurrent Collections and Synchronization Utilities

### Why This Matters

Suppose your application stores active user sessions in a `HashMap`.

```java
Map<String, Session> sessions =
    new HashMap<>();
```

One thread adds a session.

Another removes a session.

A third iterates over the map.

Under heavy load, the application begins throwing unexpected exceptions or producing inconsistent results.

The issue isn't the business logic.

The issue is that **most standard Java collections are not thread-safe**.

Java provides specialized concurrent collections and synchronization utilities to solve these problems efficiently.

---

# Why Normal Collections Are Unsafe

Consider:

```java
Map<Integer, String> map =
    new HashMap<>();
```

Multiple threads execute:

```java
map.put(...);

map.remove(...);

map.get(...);
```

simultaneously.

Since `HashMap` does not synchronize access,

its internal structure can become inconsistent.

Similarly,

iterating while another thread modifies the collection may result in:

```
ConcurrentModificationException
```

or undefined behavior.

---

# ConcurrentHashMap

The most commonly used concurrent collection is:

```java
ConcurrentHashMap<K, V>
```

It allows:

- Multiple concurrent readers.
- Concurrent updates with minimal contention.
- Better scalability than synchronizing an entire `HashMap`.

Conceptually:

```
Thread A

↓

ConcurrentHashMap

↑

Thread B

↑

Thread C
```

Instead of locking the whole map for every operation,

it uses fine-grained synchronization internally.

> **Best Practice**
>
> Prefer `ConcurrentHashMap` over `Collections.synchronizedMap()` for highly concurrent applications.

---

# CopyOnWriteArrayList

Suppose an application maintains a list of listeners.

Reads happen frequently.

Writes are rare.

```
Read

Read

Read

Read

Write
```

`CopyOnWriteArrayList` is designed for this pattern.

When modified,

it creates a new copy of the underlying array.

Advantages:

- Iteration requires no synchronization.
- Readers never block.

Disadvantage:

- Writes are expensive because the entire array is copied.

Ideal for:

- Event listeners
- Configuration lists
- Observer patterns

---

# BlockingQueue

Producer-consumer systems are common in backend applications.

```
Producer

↓

Queue

↓

Consumer
```

Examples:

- Logging
- Email processing
- Background jobs
- Message processing

`BlockingQueue` handles synchronization internally.

Producers insert:

```java
queue.put(task);
```

Consumers retrieve:

```java
queue.take();
```

If the queue is empty,

the consumer waits automatically.

No manual synchronization is required.

Common implementations include:

- `ArrayBlockingQueue`
- `LinkedBlockingQueue`
- `PriorityBlockingQueue`

---

# Concurrent Collection Summary

| Collection | Best Use Case |
|------------|---------------|
| `ConcurrentHashMap` | Shared maps |
| `CopyOnWriteArrayList` | Many reads, few writes |
| `BlockingQueue` | Producer-consumer patterns |
| `ConcurrentLinkedQueue` | High-throughput non-blocking queues |

---

# Synchronization Utilities

In addition to collections,

Java provides utilities for coordinating multiple threads.

These reside in:

```java
java.util.concurrent
```

---

# CountDownLatch

Imagine a reporting service that must wait for three independent tasks.

```
Database

API

Cache

↓

Generate Report
```

A `CountDownLatch` starts with a count.

Each completed task decrements it.

```
Count = 3

↓

Task 1 Complete

Count = 2

↓

Task 2 Complete

Count = 1

↓

Task 3 Complete

Count = 0

↓

Continue
```

The waiting thread proceeds only when the count reaches zero.

A `CountDownLatch` is a **one-time synchronization aid**.

Once the count reaches zero,

it cannot be reset.

---

# CyclicBarrier

Suppose five worker threads process different parts of a large dataset.

```
Worker 1

Worker 2

Worker 3

Worker 4

Worker 5
```

Each thread must finish its phase before any proceed to the next.

```
Phase 1

↓

Barrier

↓

Phase 2
```

A `CyclicBarrier` blocks all participating threads until every one reaches the barrier.

Unlike `CountDownLatch`,

it can be reused for multiple cycles.

---

# Semaphore

A semaphore controls access to a limited number of resources.

Imagine a database connection pool with:

```
10 Connections
```

At most ten threads should use connections simultaneously.

```
Connection Pool

↓

Semaphore

↓

Thread Access
```

If all permits are in use,

additional threads wait.

Semaphores are useful for:

- Limiting concurrent API calls
- Database connection management
- Resource throttling

---

# Phaser

`Phaser` is a more flexible synchronization utility for multi-phase workflows.

Example:

```
Phase 1

↓

Phase 2

↓

Phase 3
```

Unlike `CountDownLatch`,

threads can register and deregister dynamically.

It is commonly used in complex parallel algorithms.

---

# Choosing the Right Utility

| Requirement | Recommended Utility |
|-------------|---------------------|
| Shared map | `ConcurrentHashMap` |
| Read-heavy list | `CopyOnWriteArrayList` |
| Producer-consumer queue | `BlockingQueue` |
| Wait for fixed number of tasks | `CountDownLatch` |
| Synchronize repeated phases | `CyclicBarrier` |
| Limit concurrent access | `Semaphore` |
| Multi-phase coordination | `Phaser` |

Understanding these abstractions helps explain how enterprise frameworks achieve scalability without requiring developers to write low-level synchronization code.

# Production Perspective

A notification service sends emails.

Instead of creating hundreds of worker threads,

tasks are placed into a `BlockingQueue`.

A fixed thread pool consumes tasks from the queue.

Benefits:

- Stable memory usage
- Controlled concurrency
- Natural backpressure
- Simpler code

The queue absorbs traffic spikes while workers process tasks at a sustainable rate.

# Interview Focus

### Question

When would you use `ConcurrentHashMap` instead of `HashMap`?

### Strong Senior-Level Answer

> `HashMap` is not thread-safe and should not be modified concurrently without external synchronization. `ConcurrentHashMap` is designed for concurrent access, allowing multiple threads to read and update the map efficiently using fine-grained synchronization, making it the preferred choice for shared maps in multithreaded applications.

---

# Common Mistakes

❌ Using `HashMap` in shared mutable state.

❌ Choosing `CopyOnWriteArrayList` for write-heavy workloads.

❌ Replacing every queue with a `BlockingQueue` without understanding throughput requirements.

❌ Using `CountDownLatch` when repeated synchronization cycles are required.

---

# Engineering Insight

Many developers reach for explicit locks whenever multiple threads are involved.

In reality,

the best solution is often to use the appropriate concurrent collection or synchronization utility.

Well-designed concurrent abstractions are:

- Easier to understand
- Less error-prone
- Better tested
- More scalable

The less manual synchronization you write,

the fewer concurrency bugs you'll introduce.

---

# Engineering Takeaways

- Standard collections are generally not thread-safe.
- Use concurrent collections when data is shared across threads.
- `BlockingQueue` simplifies producer-consumer systems.
- `CountDownLatch`, `CyclicBarrier`, `Semaphore`, and `Phaser` solve different coordination problems.
- Prefer high-level concurrency abstractions over manual locking where appropriate.

---

# Quick Revision

✓ `ConcurrentHashMap` is the standard concurrent map.

✓ `CopyOnWriteArrayList` is optimized for read-heavy workloads.

✓ `BlockingQueue` enables safe producer-consumer patterns.

✓ `CountDownLatch` is one-time; `CyclicBarrier` is reusable.

✓ `Semaphore` limits concurrent access to resources.

# Senior Interview Challenge

Your application integrates with a third-party payment gateway.

The provider allows **only 20 concurrent API requests**.

Traffic regularly exceeds this limit.

### How would you enforce the restriction?

**Expected Discussion Points**

- Use a `Semaphore` initialized with 20 permits.
- Acquire a permit before invoking the API.
- Release the permit in a `finally` block.
- Prevent resource leaks if exceptions occur.
- Monitor waiting times and throughput to tune capacity.

Senior engineers recognize that not every concurrency problem requires locks—often the right concurrency utility is the simplest and most robust solution.
