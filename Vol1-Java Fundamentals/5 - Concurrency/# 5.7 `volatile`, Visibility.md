# 5.7 `volatile`, Visibility, and the Java Memory Model (JMM)

### Why This Matters

Many developers believe that concurrency problems occur only when multiple threads modify the same variable.

Unfortunately, that's only part of the story.

Consider:

```java
class TaskRunner {

    private boolean running = true;

    public void stop() {
        running = false;
    }

    public void run() {

        while (running) {

            // Do work

        }
    }
}
```

Looks perfectly reasonable.

One thread executes:

```java
run();
```

Another executes:

```java
stop();
```

You might expect:

```
running = false

↓

Loop Stops
```

But in reality,

the loop may continue forever.

Understanding why requires understanding **memory visibility** and the **Java Memory Model (JMM)**.

---

# The Hidden Problem

Modern CPUs are incredibly fast.

Accessing main memory is comparatively slow.

To improve performance,

processors use caches.

```
Main Memory

       ▲
       │

CPU Cache A

CPU Cache B
```

Each CPU core may maintain its own cached copy of data.

---

# What Can Go Wrong?

Imagine:

```
running = true
```

Thread A reads:

```java
running
```

and stores it in its local cache.

```
CPU Cache A

running = true
```

Later,

Thread B executes:

```java
running = false;
```

```
Main Memory

running = false
```

However,

Thread A may continue using its cached copy.

```
CPU Cache A

running = true
```

The loop never exits.

---

# Visibility vs Atomicity

These are different concepts.

### Visibility

Can one thread see updates made by another thread?

---

### Atomicity

Can an operation execute as a single indivisible unit?

---

Example:

```java
count++;
```

Problem:

```
Atomicity
```

Example:

```java
running = false;
```

Problem:

```
Visibility
```

A senior engineer understands the distinction.

---

# Enter `volatile`

Java provides:

```java
volatile
```

Example:

```java
private volatile boolean running = true;
```

Now:

```java
running = false;
```

becomes visible to other threads immediately.

The JVM ensures:

- Reads observe the latest value.
- Writes are flushed to main memory.
- Threads do not rely solely on stale cached values.

---

# What Does `volatile` Guarantee?

### Visibility

✓ Yes

---

### Ordering Guarantees

✓ Yes

---

### Atomicity

✗ No

This distinction is critical.

---

# Common Interview Trap

Consider:

```java
private volatile int count = 0;

public void increment() {

    count++;

}
```

Is this thread-safe?

**No.**

Why?

Because:

```java
count++;
```

still performs:

```
Read

↓

Increment

↓

Write
```

Multiple threads can still interfere.

`volatile` guarantees visibility, not atomicity.

---

# Visualizing `volatile`

Without `volatile`:

```
Thread A

↓

CPU Cache

↓

Old Value
```

```
Thread B

↓

Main Memory

↓

New Value
```

Possible inconsistency.

---

With `volatile`:

```
Thread A

↓

Main Memory

↑

Thread B
```

Changes become visible across threads.

# What Is the Java Memory Model (JMM)?

The Java Memory Model defines:

- How threads interact through memory.
- When writes become visible.
- What optimizations the JVM may perform.
- Which operations establish ordering guarantees.

The JMM exists because Java applications run on:

- Different CPUs
- Different operating systems
- Different JVM implementations

Without a common memory model,

concurrent code could behave differently across platforms.

---

# Happens-Before Relationship

One of the most important JMM concepts is:

```
Happens-Before
```

If operation A happens-before operation B,

then:

- Changes made by A are visible to B.
- Ordering is guaranteed.

---

# Examples of Happens-Before

### Thread Start

```java
thread.start();
```

All actions before:

```java
start()
```

are visible to the new thread.

---

### Thread Join

```java
thread.join();
```

After `join()` returns,

the joining thread sees everything completed by the joined thread.

---

### Volatile Writes

```java
running = false;
```

A volatile write happens-before subsequent volatile reads of the same variable.

---

### Lock Release and Acquisition

```java
synchronized(lock) {

}
```

Unlocking happens-before a later lock acquisition on the same monitor.

This is one reason synchronization also provides visibility guarantees.

---

# Reordering

The JVM and CPU may reorder instructions to improve performance.

Example:

```java
a = 1;

b = 2;
```

The processor may execute these operations differently internally if it can prove the result is unchanged.

This optimization is usually harmless.

However,

in concurrent programs,

reordering can create unexpected behavior.

The JMM defines rules that restrict unsafe reordering.

`volatile` participates in these rules.

---

# When Should You Use `volatile`?

Good candidates:

### Status Flags

```java
volatile boolean running;
```

---

### Shutdown Signals

```java
volatile boolean shutdownRequested;
```

---

### Configuration Reload Indicators

```java
volatile boolean refreshRequired;
```

---

### Read-Mostly Variables

Variables that are updated infrequently but read by many threads.

---

# When Should You NOT Use `volatile`?

Avoid it for compound operations.

Example:

```java
count++;
```

```java
balance += amount;
```

```java
if (count == 0) {
    count++;
}
```

These require synchronization or atomic classes.

---

# Production Perspective

A batch processing application uses:

```java
private boolean stopped;
```

A monitoring thread sets:

```java
stopped = true;
```

to request shutdown.

Occasionally,

the worker thread never terminates.

The issue is not a race condition.

The issue is visibility.

Changing the field to:

```java
private volatile boolean stopped;
```

immediately resolves the problem.

# Interview Focus

### Question

What is the difference between `volatile` and `synchronized`?

### Strong Senior-Level Answer

> `volatile` guarantees visibility and ordering of updates across threads but does not provide atomicity. `synchronized` provides both mutual exclusion and visibility guarantees by ensuring only one thread executes a critical section at a time while also establishing happens-before relationships.

---

# Common Misconceptions

❌ `volatile` makes code thread-safe.

❌ `volatile` prevents race conditions.

❌ `volatile` replaces synchronization.

❌ Visibility and atomicity are the same thing.

---

# Engineering Insight

Many concurrency bugs are not caused by incorrect calculations.

They are caused by **threads observing stale data**.

This is why:

```java
volatile
```

exists.

However,

if multiple threads modify a value,

you typically need:

- Synchronization
- Atomic classes
- Concurrent data structures

rather than `volatile` alone.

A useful rule:

> Use `volatile` for communication.
>
> Use synchronization for coordination.

---

# Engineering Takeaways

- Visibility and atomicity are different concerns.
- CPU caches can hide updates from other threads.
- `volatile` guarantees visibility but not atomicity.
- The Java Memory Model defines how threads interact through memory.
- Happens-before relationships are fundamental to concurrent correctness.

---

# Quick Revision

✓ `volatile` ensures visibility.

✓ `volatile` does not make `count++` atomic.

✓ The Java Memory Model governs memory interactions.

✓ Synchronization provides visibility and mutual exclusion.

✓ Happens-before relationships guarantee ordering and visibility.

# Senior Interview Challenge

A service contains:

```java
private volatile int processedCount = 0;

public void process() {

    processedCount++;

}
```

The reported count is lower than expected under heavy load.

### Why does this happen, even though the variable is volatile?

**Expected Discussion Points**

- `volatile` guarantees visibility only.
- `processedCount++` is a read-modify-write operation.
- Multiple threads can overwrite each other's updates.
- Use `AtomicInteger`, synchronization, or other concurrency controls.
- Distinguish clearly between visibility and atomicity.

Engineers who understand this distinction typically have a strong grasp of the Java Memory Model.
