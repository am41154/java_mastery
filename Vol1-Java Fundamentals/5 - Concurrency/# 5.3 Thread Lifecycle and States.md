# 5.3 Thread Lifecycle and States

### Why This Matters

Almost every Java developer has written code like this:

```java
Thread thread = new Thread(task);

thread.start();
```

But what actually happens after `start()`?

- When does the thread begin execution?
- Can it be started again?
- Why does a thread sometimes appear to "pause"?
- What happens when `join()` is called?

Understanding the lifecycle of a thread helps explain many real-world debugging scenarios and is a common interview topic.

# NEW

A thread begins in the **NEW** state.

```java
Thread thread = new Thread(task);
```

At this point:

- Memory has been allocated.
- The thread object exists.
- Execution has **not** started.

```
NEW

â†“

Not yet started
```

---

# RUNNABLE

Calling:

```java
thread.start();
```

moves the thread into the **RUNNABLE** state.

```
NEW

â†“

start()

â†“

RUNNABLE
```

The thread is now eligible to execute.

However,

this does **not** guarantee immediate execution.

The operating system scheduler decides when the CPU will actually execute it.

> **Interview Tip**
>
> `start()` requests execution. It does **not** mean the thread starts running immediately.

# Can a Thread Be Started Twice?

Consider:

```java
Thread thread = new Thread(task);

thread.start();

thread.start();
```

Result:

```
IllegalThreadStateException
```

A thread represents a single execution.

Once it completes,

its lifecycle is over.

To execute the same task again,

create a new thread.

# Common Thread Methods

| Method | Purpose |
|---------|----------|
| `start()` | Starts a new thread |
| `run()` | Contains the task executed by the thread |
| `sleep()` | Pauses the current thread for a specified duration |
| `join()` | Waits for another thread to complete |
| `interrupt()` | Requests interruption of a thread |
| `isAlive()` | Checks whether the thread is still running |

# `start()` vs `run()`

This is one of the most frequently asked interview questions.

Consider:

```java
Thread thread = new Thread(task);

thread.run();
```

This does **not** create a new thread.

The `run()` method executes like a normal method call on the current thread.

```
Main Thread

â†“

run()

â†“

Task
```

Now compare:

```java
thread.start();
```

```
Main Thread

â†“

Creates New Thread

â†“

New Thread Executes run()
```

The difference is fundamental.

Always call `start()` when you want concurrent execution.

# Production Perspective

A backend developer writes:

```java
new Thread(task).run();
```

expecting asynchronous execution.

During testing, everything appears correct.

Under production load, requests become slow because the task executes on the request-handling thread instead of a new thread.

Replacing `run()` with `start()` (or, better yet, using an `ExecutorService`) restores asynchronous behavior and improves response times.

# Interview Focus

### Question

What is the difference between `start()` and `run()`?

### Strong Senior-Level Answer

> `run()` is simply a normal method. Calling it executes the task on the current thread. `start()` creates a new thread, registers it with the JVM and operating system scheduler, and eventually invokes `run()` on that new thread. Calling `start()` twice on the same thread results in an `IllegalThreadStateException`.

# Engineering Insight

In modern Java applications, developers rarely create threads directly.

Instead, frameworks such as Spring Boot, Tomcat, and `ExecutorService` manage thread lifecycles through thread pools.

Even so, understanding the underlying lifecycle is essential for debugging blocked threads, analyzing thread dumps, and reasoning about concurrent behavior.

---

# Engineering Takeaways

- Every thread progresses through well-defined lifecycle states.
- `start()` schedules a new thread; `run()` does not.
- A thread can be started only once.
- `BLOCKED`, `WAITING`, and `TIMED_WAITING` represent different waiting conditions.
- Most production applications rely on thread pools rather than manually creating threads.

# Senior Interview Challenge

A production thread dump shows hundreds of threads in the `BLOCKED` state.

### How would you investigate the issue?

**Expected Discussion Points**

- Identify the monitor or lock all threads are waiting for.
- Determine which thread currently owns the lock.
- Look for long-running synchronized blocks.
- Check for unnecessary lock contention.
- Consider reducing lock scope or using alternative concurrency constructs (`ReentrantLock`, concurrent collections, etc.).

A senior engineer uses thread states as diagnostic information rather than simply recognizing their names.
