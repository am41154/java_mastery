# 2.14 `Comparable` vs `Comparator` – Natural Ordering and Custom Sorting

### Why This Matters

Imagine an HR application.

Sometimes employees need to be sorted by:

- Employee ID
- Name
- Salary
- Date of Joining
- Performance Rating

Should the `Employee` class contain sorting logic for all of these?

No.

Some ordering naturally belongs to the object,

while other ordering depends on the business scenario.

Java solves this using two interfaces:

- `Comparable`
- `Comparator`

Understanding the difference is essential for working with collections, streams, and sorting APIs.

---

# Ordering Objects

Primitive values already know how to compare themselves.

Example:

```java
10 < 20
```

Strings also have a natural order.

```java
"Apple".compareTo("Banana");
```

But what about:

```java
Employee
```

Java has no idea how two employees should be ordered.

You must define the comparison.

---

# `Comparable`

`Comparable` defines an object's **natural ordering**.

Example:

```java
public class Employee
        implements Comparable<Employee> {

    private int employeeId;

    @Override
    public int compareTo(Employee other) {

        return Integer.compare(
                this.employeeId,
                other.employeeId
        );
    }

}
```

Now:

```java
Collections.sort(employeeList);
```

works automatically.

---

# How `compareTo()` Works

The method returns:

| Return Value | Meaning |
|--------------|---------|
| Negative | Current object comes first |
| Zero | Objects are equal for ordering |
| Positive | Other object comes first |

Example:

```java
5.compareTo(10)
```

Result:

```
Negative
```

Meaning:

```
5

↓

10
```

---

# Natural Ordering

Examples of natural ordering:

```
Employee

↓

Employee ID
```

```
Product

↓

Product Code
```

```
Student

↓

Roll Number
```

There should generally be **one obvious default ordering**.

---

# `Comparator`

Sometimes one ordering is not enough.

Example:

Sort employees by:

- Name
- Salary
- Joining Date

Instead of modifying the `Employee` class,

Java provides the `Comparator` interface.

---

# Comparator Example

```java
Comparator<Employee> salaryComparator =
    Comparator.comparing(Employee::getSalary);
```

Usage:

```java
employees.sort(salaryComparator);
```

No changes are required to the `Employee` class.

---

# Another Example

Sort by name:

```java
Comparator<Employee> nameComparator =
    Comparator.comparing(Employee::getName);
```

Or:

```java
employees.sort(nameComparator);
```

Different sorting strategies can coexist.

---

# Multiple Comparisons

Java allows chained comparisons.

Example:

```java
Comparator<Employee> comparator =
    Comparator
        .comparing(Employee::getDepartment)
        .thenComparing(Employee::getSalary)
        .thenComparing(Employee::getName);
```

This produces:

```
Department

↓

Salary

↓

Name
```

A common requirement in enterprise applications.

---

# Reverse Sorting

Ascending:

```java
employees.sort(
    Comparator.comparing(Employee::getSalary)
);
```

Descending:

```java
employees.sort(
    Comparator.comparing(Employee::getSalary)
              .reversed()
);
```

No custom comparison logic is required.

---

# Comparable vs Comparator

| Comparable | Comparator |
|------------|------------|
| Defines natural ordering | Defines custom ordering |
| Implemented by the class itself | Implemented separately |
| One default ordering | Multiple orderings possible |
| Uses `compareTo()` | Uses `compare()` |
| Modifies the class | Keeps the class unchanged |

---

# When to Use Which?

Use **Comparable** when:

- There is one obvious natural ordering.
- The ordering belongs to the object itself.
- The ordering rarely changes.

Use **Comparator** when:

- Multiple sorting rules exist.
- Sorting depends on business requirements.
- The class cannot be modified.
- Temporary sorting is needed.

---

# Java 8 Improvements

Before Java 8:

```java
Collections.sort(list,
    new Comparator<Employee>() {

        @Override
        public int compare(
                Employee a,
                Employee b) {

            return a.getSalary()
                    - b.getSalary();

        }

    });
```

Modern Java:

```java
employees.sort(
    Comparator.comparing(Employee::getSalary)
);
```

Cleaner,

more readable,

and less error-prone.

# Production Perspective

Suppose an online shopping platform displays products.

Users may sort by:

- Price
- Rating
- Popularity
- Newest Arrival

Each option represents a different comparison strategy.

The product class should not contain all of these rules.

Instead,

separate comparators keep the design clean and maintainable.

---

# Interview Focus

### Question

What is the difference between `Comparable` and `Comparator`?

### Strong Senior-Level Answer

> `Comparable` defines the natural ordering of a class by implementing the `compareTo()` method. `Comparator` defines external comparison strategies through the `compare()` method and supports multiple sorting criteria without modifying the original class. In modern Java, `Comparator` is often preferred because it provides greater flexibility and supports fluent composition.

---

# Common Mistakes

❌ Returning `a - b` for large integers (risk of overflow).

❌ Using `Comparable` when multiple sorting rules are required.

❌ Writing verbose anonymous comparator classes instead of using `Comparator.comparing()`.

❌ Forgetting that `compareTo()` should be consistent with `equals()` when objects are used in sorted collections like `TreeSet` or `TreeMap`.

---

# Engineering Insight

Equality and ordering are related,

but they solve different problems.

```
equals()

↓

Are these objects logically the same?
```

```
compareTo()

↓

Which object should come first?
```

For sorted collections,

an inconsistent comparison implementation can cause elements to disappear or appear duplicated.

Consistency matters.

---

# Engineering Takeaways

- `Comparable` defines natural ordering.
- `Comparator` defines custom ordering.
- Prefer `Comparator` when multiple sorting rules exist.
- Java 8 introduced fluent comparator creation.
- Keep comparison logic separate from business logic whenever possible.

---

# Quick Revision

✓ `Comparable` → `compareTo()`.

✓ `Comparator` → `compare()`.

✓ One natural order → Comparable.

✓ Multiple orders → Comparator.

✓ Java 8 provides fluent comparator APIs.

# Senior Interview Challenge

A `Product` class implements:

```java
compareTo()
```

based on:

```
Price
```

Later,

the business requests sorting by:

- Category
- Discount
- Rating
- Sales Count

### Would you continue modifying `compareTo()`?

**Expected Discussion Points**

- No.
- Keep `compareTo()` for the natural ordering only.
- Implement additional `Comparator` instances for business-specific sorting.
- Avoid changing the natural ordering for every new requirement.
- Use comparator composition for complex ordering rules.

Senior engineers separate **object identity**, **object equality**, and **object ordering**, treating each as an independent design concern.
