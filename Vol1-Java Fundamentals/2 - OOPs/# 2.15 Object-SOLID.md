# 2.15 Object-Oriented Design Principles – A SOLID Overview

### Why This Matters

Knowing Java syntax is not enough to build maintainable software.

Consider two applications:

- Both compile successfully.
- Both pass all unit tests.
- Both produce the correct output.

Yet after a year:

- One is easy to extend.
- The other is difficult to change without breaking existing functionality.

The difference is rarely the programming language.

It is usually the **design**.

This is where the **SOLID principles** become valuable.

---

# What Is SOLID?

SOLID is a collection of five object-oriented design principles that help developers build software that is:

- Maintainable
- Extensible
- Testable
- Loosely coupled
- Easier to understand

The acronym stands for:

| Principle | Meaning |
|------------|---------|
| **S** | Single Responsibility Principle |
| **O** | Open/Closed Principle |
| **L** | Liskov Substitution Principle |
| **I** | Interface Segregation Principle |
| **D** | Dependency Inversion Principle |

These principles are guidelines,

not rigid rules.

Good engineers understand when to apply them.

---

# 1. Single Responsibility Principle (SRP)

> **A class should have only one reason to change.**

Consider:

```java
class EmployeeService {

    void calculateSalary() { }

    void generatePdfReport() { }

    void sendEmail() { }

    void saveEmployee() { }

}
```

This class performs multiple unrelated responsibilities.

A better design:

```
EmployeeService

↓

Business Logic
```

```
ReportService

↓

Report Generation
```

```
EmailService

↓

Email Delivery
```

Each class has a focused responsibility.

---

# 2. Open/Closed Principle (OCP)

> **Software should be open for extension but closed for modification.**

Poor design:

```java
if(paymentType.equals("CARD")) {

}
else if(paymentType.equals("UPI")) {

}
else if(paymentType.equals("WALLET")) {

}
```

Every new payment method requires changing existing code.

Better design:

```
PaymentService

↓

CardPayment

UPIPayment

WalletPayment
```

New implementations can be added without modifying existing business logic.

---

# 3. Liskov Substitution Principle (LSP)

> **A subclass should be usable wherever its parent class is expected.**

Example:

```
Bird

↓

Sparrow

↓

Penguin
```

Suppose:

```java
Bird.fly();
```

A penguin cannot fly.

This hierarchy violates LSP.

The inheritance model is incorrect.

A better design separates flying behavior from bird classification.

---

# 4. Interface Segregation Principle (ISP)

> **Clients should not depend on methods they do not use.**

Poor design:

```java
interface Worker {

    work();

    eat();

    sleep();

}
```

A robot worker cannot eat or sleep.

Better design:

```
Workable

↓

work()
```

```
Eatable

↓

eat()
```

Small, focused interfaces reduce unnecessary dependencies.

---

# 5. Dependency Inversion Principle (DIP)

> **Depend on abstractions, not concrete implementations.**

Instead of:

```java
OrderService service =
    new OrderService(
        new StripePaymentService()
    );
```

Prefer:

```java
OrderService service =
    new OrderService(
        paymentService
    );
```

Where:

```java
paymentService
```

is an interface.

The implementation can now change independently.

This principle forms the basis of Spring's Dependency Injection.

---

# SOLID in Spring Boot

Spring applications naturally encourage SOLID.

Example:

```
Controller

↓

Service Interface

↓

Service Implementation

↓

Repository Interface
```

Benefits:

- Loose coupling
- Easy testing
- Flexible implementations
- Better separation of concerns

Most Spring applications follow these principles without developers explicitly thinking about them.

---

# How the Principles Work Together

Consider an order processing system.

```
OrderController

↓

OrderService

↓

PaymentService Interface

↓

CardPayment

UPIPayment

WalletPayment
```

This design demonstrates:

- SRP → Each class has one responsibility.
- OCP → New payment methods are added without modifying existing code.
- LSP → Every payment implementation can replace another.
- ISP → Small, focused interfaces.
- DIP → Business logic depends on abstractions.

The principles reinforce one another.

---

# Production Perspective

Suppose a shipping company initially supports only one courier.

Poor design:

```java
ShippingService

↓

Hardcoded Courier Logic
```

Six months later,

the business adds:

- DHL
- FedEx
- Blue Dart
- India Post

Developers now modify the same class repeatedly.

Merge conflicts increase.

Testing becomes difficult.

A SOLID design would introduce:

```
ShippingProvider

↓

DHL

FedEx

Blue Dart

India Post
```

Each provider becomes an independent implementation.

# Interview Focus

### Question

Why are the SOLID principles important?

### Strong Senior-Level Answer

> The SOLID principles provide design guidelines for building maintainable and extensible object-oriented software. They promote separation of concerns, loose coupling, and programming against abstractions. Together, they reduce the impact of change and make systems easier to test, understand, and extend over time.

---

# Common Mistakes

❌ Treating SOLID as rigid rules instead of design guidelines.

❌ Creating excessive abstractions for very small applications.

❌ Introducing interfaces without a meaningful purpose.

❌ Violating SRP by building "God Classes."

---

# Engineering Insight

Many developers believe SOLID means writing more classes.

In reality,

SOLID is about **reducing the cost of future change**.

A useful question during design is:

> **If the business introduces a new requirement tomorrow, how many existing classes must change?**

The fewer classes that require modification,

the better the design.

---

# Engineering Takeaways

- SOLID improves maintainability.
- Prefer focused classes with clear responsibilities.
- Extend systems through abstractions instead of modifying existing code.
- Keep interfaces small and meaningful.
- Depend on interfaces rather than implementations.

---

# Quick Revision

✓ SRP → One responsibility.

✓ OCP → Extend without modifying.

✓ LSP → Subclasses must remain substitutable.

✓ ISP → Small interfaces.

✓ DIP → Depend on abstractions.

# Senior Interview Challenge

You're reviewing the following design:

```
OrderService

↓

calculatePrice()

↓

saveOrder()

↓

generateInvoice()

↓

sendEmail()

↓

processPayment()

↓

publishKafkaEvent()

↓

updateInventory()
```

The class has grown to nearly **2,000 lines**.

### How would you redesign it using SOLID principles?

**Expected Discussion Points**

- Apply SRP by separating pricing, inventory, payment, notification, and invoicing into dedicated services.
- Introduce interfaces for payment and notification providers (DIP).
- Use polymorphism to support multiple payment methods (OCP).
- Keep service interfaces focused (ISP).
- Ensure implementations remain interchangeable (LSP).
- Coordinate the workflow through an orchestration service instead of a monolithic class.

Senior engineers use SOLID not as a checklist, but as a way to build systems that remain easy to change as business requirements evolve.
