# 2.5 Encapsulation – Protecting Object State

### Why This Matters

Imagine a banking application.

The `BankAccount` class contains:

```java
class BankAccount {

    double balance;

}
```

Since `balance` is publicly accessible,

any code can execute:

```java
account.balance = -1000000;
```

The object has entered an invalid state.

The class has lost control over its own data.

This is exactly the problem that **encapsulation** solves.

---

# What Is Encapsulation?

Encapsulation is the practice of:

- Keeping an object's data private.
- Controlling access through well-defined methods.

Instead of exposing internal state,

the object decides **how** its data can be modified.

Conceptually:

```
Without Encapsulation

Application

↓

Direct Field Access

↓

Object
```

```
With Encapsulation

Application

↓

Public Methods

↓

Object

↓

Private Data
```

The object's internal state is protected.

---

# Private Fields

Example:

```java
class BankAccount {

    private double balance;

}
```

Now:

```java
account.balance = 100;
```

Results in:

```
Compilation Error
```

External code cannot modify the field directly.

---

# Controlled Access

Instead,

the class provides methods.

```java
class BankAccount {

    private double balance;

    public void deposit(double amount) {

        balance += amount;

    }

    public double getBalance() {

        return balance;

    }

}
```

Now every update passes through business logic.

The object controls its own state.

---

# Why Not Make Everything Public?

Suppose:

```java
public class Employee {

    public int age;

}
```

Anyone can write:

```java
employee.age = -5;
```

Or:

```java
employee.age = 1000;
```

Clearly,

these values are invalid.

A better design:

```java
public void setAge(int age) {

    if (age < 0) {

        throw new IllegalArgumentException();

    }

    this.age = age;

}
```

Validation happens inside the class.

---

# Getters and Setters

Encapsulation is often associated with getters and setters.

Example:

```java
private String name;

public String getName() {

    return name;

}

public void setName(String name) {

    this.name = name;

}
```

However,

simply generating getters and setters for every field is **not** true encapsulation.

Good encapsulation protects invariants.

For example:

```java
public void withdraw(double amount) {

    if (amount > balance) {

        throw new IllegalArgumentException();

    }

    balance -= amount;

}
```

Notice that there is **no**:

```java
setBalance()
```

The balance changes only through valid business operations.

---

# Hiding Implementation Details

Encapsulation also hides *how* an object works internally.

Example:

```
ATM

↓

withdraw(500)
```

The customer doesn't know:

- Which database is used.
- How transactions are recorded.
- Which locks are acquired.
- How balances are stored.

Only the public behavior matters.

This separation allows internal implementation to evolve without affecting callers.

---

# Encapsulation in Enterprise Applications

Consider:

```
OrderService

↓

placeOrder()
```

The caller does not know whether the service:

- Writes to one database or many.
- Sends Kafka messages.
- Calls external REST APIs.
- Updates a cache.

The implementation is hidden behind a clear interface.

This is encapsulation at the service level.

---

# Benefits of Encapsulation

### Data Integrity

Objects remain in valid states.

---

### Flexibility

Internal implementation can change without affecting external code.

---

### Easier Maintenance

Business rules are centralized.

---

### Better Testing

Validation logic exists in one place.

---

### Reduced Coupling

Callers depend only on public behavior.

---

# Encapsulation vs Data Hiding

These terms are often used interchangeably,

but there is a subtle distinction.

### Data Hiding

Prevents direct access to implementation details.

Example:

```java
private int balance;
```

---

### Encapsulation

Bundles data and behavior together while controlling access.

Encapsulation uses data hiding,

but also includes validation and behavior.

---

# Spring Boot Perspective

Consider a service:

```java
public class PaymentService {

    public void processPayment() {

    }

}
```

Clients invoke:

```java
paymentService.processPayment();
```

They don't know:

- How transactions are managed.
- Which repositories are called.
- Whether retries occur.
- How auditing is implemented.

The implementation can change without changing the public API.

This is encapsulation applied to application architecture.

---

# Production Perspective

A shopping application initially exposes:

```java
public double price;
```

Later,

the business introduces:

- Discount validation
- Tax calculation
- Currency conversion

If callers modify `price` directly,

none of these rules execute.

The design is changed so that price updates occur only through dedicated methods.

This protects business invariants and keeps the object consistent.

# Interview Focus

### Question

What is encapsulation?

### Strong Senior-Level Answer

> Encapsulation is the practice of combining data and behavior within an object while restricting direct access to its internal state. It protects object integrity by exposing controlled operations instead of allowing unrestricted field modification. This improves maintainability, reduces coupling, and enables internal implementations to change without affecting external code.

---

# Common Mistakes

❌ Making every field `public`.

❌ Generating setters for all fields without considering business rules.

❌ Exposing mutable internal collections directly.

❌ Confusing encapsulation with simply using getters and setters.

---

# Engineering Insight

A useful guideline is:

> **Expose behavior, not data.**

Instead of:

```java
account.setBalance(5000);
```

prefer:

```java
account.deposit(5000);
```

The second approach communicates intent and ensures business rules are always enforced.

This is one reason domain-driven designs often avoid unnecessary setters.

---

# Engineering Takeaways

- Encapsulation protects object state.
- Keep fields private whenever possible.
- Expose meaningful business operations instead of unrestricted setters.
- Validation belongs inside the object.
- Well-encapsulated classes are easier to maintain and evolve.

---

# Quick Revision

✓ Encapsulation combines data and behavior.

✓ Private fields protect internal state.

✓ Public methods provide controlled access.

✓ Getters and setters alone do not guarantee good encapsulation.

✓ Expose behavior rather than raw data.

# Senior Interview Challenge

A `User` class exposes the following API:

```java
public class User {

    public String username;

    public String password;

    public boolean active;

}
```

Developers throughout the application modify these fields directly.

### How would you redesign this class?

**Expected Discussion Points**

- Make fields private.
- Require mandatory values through the constructor.
- Expose business methods such as `activate()`, `deactivate()`, and `changePassword()`.
- Validate password rules inside the class.
- Avoid setters that bypass business invariants.

Senior engineers use encapsulation to ensure that an object cannot enter an invalid state through normal usage.
