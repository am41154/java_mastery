# 2.8 Abstraction – Exposing What Matters, Hiding What Doesn't

### Why This Matters

Consider driving a car.

You interact with:

- Steering wheel
- Accelerator
- Brake
- Gear

You don't need to know:

- Fuel injection algorithms
- Engine timing
- Transmission mechanics
- ABS control logic

The car exposes only what is necessary.

Everything else is hidden.

This is the essence of **abstraction**.

---

# What Is Abstraction?

Abstraction is the process of exposing only the essential behavior of an object while hiding unnecessary implementation details.

In simple terms:

> **Focus on what an object does, not how it does it.**

Users interact with a clear interface.

The internal implementation remains hidden.

---

# Abstraction vs Encapsulation

These concepts are closely related but solve different problems.

### Encapsulation

Protects an object's internal state.

Question:

> **Who can access the data?**

---

### Abstraction

Hides implementation complexity.

Question:

> **What should users know about the implementation?**

Think of it this way:

```
Encapsulation

↓

Protect Data
```

```
Abstraction

↓

Hide Complexity
```

# Abstraction in Java

Java provides abstraction primarily through:

- Interfaces
- Abstract classes

Example:

```java
interface PaymentGateway {

    void processPayment();

}
```

Implementations:

```java
class CreditCardGateway
        implements PaymentGateway {

    @Override
    public void processPayment() {

        System.out.println("Processing card payment");

    }

}
```

Client code:

```java
PaymentGateway gateway =
        new CreditCardGateway();

gateway.processPayment();
```

The caller knows **what** to invoke,

not **how** it is implemented.

---

# Abstract Classes

An abstract class represents an incomplete implementation.

Example:

```java
abstract class Animal {

    abstract void sound();

}
```

Subclass:

```java
class Dog extends Animal {

    @Override
    void sound() {

        System.out.println("Bark");

    }

}
```

Abstract classes cannot be instantiated directly.

The following is invalid:

```java
Animal animal =
        new Animal();
```

---

# Interfaces

Interfaces define a contract.

Example:

```java
public interface NotificationService {

    void send();

}
```

Implementations:

```java
EmailNotification

SMSNotification

PushNotification
```

Every implementation follows the same contract while providing different behavior.

---

# Abstract Class vs Interface

| Abstract Class | Interface |
|----------------|-----------|
| Can have state | No instance state |
| Can have constructors | No constructors |
| Supports implemented and abstract methods | Primarily defines behavior |
| Extended using `extends` | Implemented using `implements` |
| Single inheritance | Multiple interfaces allowed |

Modern Java interfaces can also contain:

- Default methods
- Static methods
- Private helper methods

However,

their primary purpose remains defining behavior contracts.

---

# Why Abstraction Matters

Imagine writing:

```java
payment.processPayment();
```

The caller doesn't care whether the implementation uses:

- Stripe
- Razorpay
- PayPal
- UPI
- A mock implementation for testing

The abstraction isolates the business logic from implementation details.

# Spring Boot Perspective

Spring applications are built around abstractions.

Example:

```java
public interface UserRepository {

}
```

The application depends on the interface.

Spring Data JPA generates the implementation automatically.

Similarly,

services usually depend on interfaces:

```java
UserService

↓

UserServiceImpl
```

Controllers interact with the abstraction,

not the implementation.

This makes applications easier to test, replace, and extend.

# Interview Focus

### Question

What is abstraction?

### Strong Senior-Level Answer

> Abstraction is the process of exposing essential behavior while hiding implementation details. In Java, abstraction is achieved primarily through interfaces and abstract classes. It allows software to depend on contracts rather than concrete implementations, improving flexibility, maintainability, and testability.

---

# Common Mistakes

❌ Confusing abstraction with encapsulation.

❌ Creating interfaces with only one implementation "just because."

❌ Exposing implementation details through public APIs.

❌ Using abstract classes when an interface is sufficient.

---

# Engineering Insight

A useful guideline is:

> **Depend on behavior, not implementation.**

For example:

Instead of:

```java
StripePaymentService payment =
        new StripePaymentService();
```

Prefer:

```java
PaymentService payment =
        new StripePaymentService();
```

Tomorrow,

the implementation can change to:

```java
PaymentService payment =
        new RazorpayPaymentService();
```

without changing the calling code.

This principle is fundamental to Spring's Dependency Injection and clean architecture.

---

# Engineering Takeaways

- Abstraction hides implementation complexity.
- Interfaces define behavior contracts.
- Abstract classes provide partial implementations.
- Program against abstractions whenever possible.
- Abstraction improves flexibility and testability.

# Senior Interview Challenge

A service directly depends on:

```java
MySqlPaymentRepository repository =
        new MySqlPaymentRepository();
```

The company plans to support PostgreSQL and MongoDB in the future.

### How would you redesign this using abstraction?

**Expected Discussion Points**

- Introduce a `PaymentRepository` interface.
- Create database-specific implementations.
- Inject the required implementation using Dependency Injection.
- Keep business logic independent of the persistence technology.
- Improve testing by allowing mock implementations.

Senior engineers use abstraction to isolate business logic from changing technologies, making systems easier to evolve over time.

# The Four Pillars of OOP

At this point, you've covered the four foundational principles of Object-Oriented Programming:

| Pillar | Purpose |
|---------|----------|
| **Encapsulation** | Protect object state and enforce business rules |
| **Inheritance** | Reuse and specialize behavior |
| **Polymorphism** | Support multiple implementations through a common contract |
| **Abstraction** | Hide implementation details behind clear interfaces |

These principles work together rather than independently. Well-designed enterprise applications use all four to build systems that are maintainable, extensible, and easy to reason about.
