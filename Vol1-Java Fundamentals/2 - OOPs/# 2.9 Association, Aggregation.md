# 2.9 Association, Aggregation, and Composition

### Why This Matters

Objects rarely exist in isolation.

Consider an e-commerce application.

```
Customer

↓

Order

↓

Product

↓

Address

↓

Payment
```

Each object interacts with other objects.

The way these relationships are designed has a significant impact on:

- Maintainability
- Reusability
- Testability
- Coupling

Understanding object relationships is an essential part of object-oriented design.

---

# Association

Association represents a relationship between two independent objects.

Neither object owns the other.

They simply know about each other.

Example:

```
Teacher

↔

Student
```

A teacher teaches many students.

A student may learn from many teachers.

Both can exist independently.

---

## Java Example

```java
class Teacher {

    private String name;

}

class Student {

    private Teacher teacher;

}
```

The `Student` knows about a `Teacher`.

If the teacher object is removed,

the student can still exist.

---

# Types of Association

Association can be:

### One-to-One

```
Person

↓

Passport
```

---

### One-to-Many

```
Department

↓

Employees
```

---

### Many-to-Many

```
Student

↔

Course
```

These relationships are common in database modeling and JPA entity design.

---

# Aggregation

Aggregation is a specialized form of association.

It represents a **has-a** relationship,

but the contained object can exist independently.

Example:

```
Department

↓

Employees
```

If a department closes,

employees still exist.

They may simply move to another department.

---

## Java Example

```java
class Employee {

    private String name;

}

class Department {

    private List<Employee> employees;

}
```

The department contains employees,

but does not own their lifecycle.

---

# Visualizing Aggregation

```
Department

◇──────── Employee
```

The hollow diamond (◇) indicates aggregation.

The contained object has an independent lifecycle.

---

# Composition

Composition is the strongest form of association.

The parent object owns the child object completely.

If the parent is destroyed,

the child no longer exists.

Example:

```
House

↓

Room
```

A room cannot meaningfully exist without its house.

Similarly:

```
Car

↓

Engine
```

If the car is destroyed,

its engine (in this model) is considered part of the same lifecycle.

---

## Java Example

```java
class Engine {

}

class Car {

    private final Engine engine =
            new Engine();

}
```

The `Car` creates and owns the `Engine`.

External code cannot replace or manage it independently.

---

# Visualizing Composition

```
Car

◆──────── Engine
```

The filled diamond (◆) indicates composition.

The child object depends on the parent.

---

# Comparing the Relationships

| Relationship | Ownership | Lifecycle |
|--------------|-----------|-----------|
| Association | None | Independent |
| Aggregation | Weak | Independent |
| Composition | Strong | Dependent |

Think of it as a spectrum:

```
Association

↓

Aggregation

↓

Composition
```

As you move downward,

ownership becomes stronger.

---

# Real-World Examples

| Relationship | Example |
|--------------|----------|
| Association | Doctor ↔ Patient |
| Aggregation | Team → Player |
| Composition | House → Room |
| Composition | Order → OrderLine |
| Aggregation | Library → Books |
| Association | Customer ↔ Bank |

Notice that in aggregation,

the child object can often move to another parent.

In composition,

it cannot.

---

# Composition vs Aggregation

Consider:

```
Company

↓

Employees
```

Employees can resign and join another company.

This is aggregation.

Now consider:

```
Invoice

↓

InvoiceLine
```

Deleting the invoice should also remove its invoice lines.

This is composition.

Ownership determines the relationship.

---

# Spring Boot Perspective

Consider:

```
OrderService

↓

PaymentGateway
```

The service collaborates with the gateway,

but does not own its lifecycle.

This is association through dependency injection.

Now consider:

```
Order

↓

OrderItems
```

An `OrderItem` has no meaning without its parent `Order`.

This is typically modeled as composition.

JPA mappings often reflect these relationships using cascading operations.

---

# Production Perspective

Suppose you're designing an HR system.

Poor design:

```
Employee created inside Department
```

Deleting the department also deletes employees.

Clearly incorrect.

Better design:

```
Department

↓

Employee References
```

Employees continue to exist even if departments change.

Correctly identifying ownership prevents serious production bugs.

# Interview Focus

### Question

What is the difference between association, aggregation, and composition?

### Strong Senior-Level Answer

> Association represents a general relationship between independent objects. Aggregation is a weak ownership relationship where the child can exist independently of the parent. Composition is a strong ownership relationship where the child object's lifecycle is controlled by the parent. Choosing the correct relationship depends primarily on lifecycle and ownership semantics.

---

# Common Mistakes

❌ Treating every relationship as inheritance.

❌ Confusing aggregation with composition.

❌ Ignoring object lifecycle when designing relationships.

❌ Creating child objects that should actually be shared across multiple parents.

---

# Engineering Insight

A practical question to ask during design is:

> **Can the child object exist without the parent?**

If the answer is:

**Yes**

→ Association or Aggregation.

If the answer is:

**No**

→ Composition.

This simple rule resolves most design decisions correctly.

---

# Engineering Takeaways

- Association represents collaboration between objects.
- Aggregation represents weak ownership.
- Composition represents strong ownership.
- Ownership and lifecycle determine the appropriate relationship.
- Composition generally produces more cohesive domain models.

---

# Quick Revision

✓ Association = knows about.

✓ Aggregation = has-a (independent lifecycle).

✓ Composition = owns-a (dependent lifecycle).

✓ Composition is stronger than aggregation.

✓ Design relationships based on ownership, not convenience.

# Senior Interview Challenge

You're designing an online shopping platform.

Should the following relationships be modeled as **association**, **aggregation**, or **composition**?

- `Order` → `OrderItem`
- `Company` → `Employee`
- `Customer` → `Address`
- `Library` → `Book`
- `University` → `Professor`

**Expected Discussion Points**

- `Order` → `OrderItem` → Composition
- `Company` → `Employee` → Aggregation
- `Customer` → `Address` → Depends on the domain (often Composition if addresses belong exclusively to a customer, Association/Aggregation if shared)
- `Library` → `Book` → Aggregation
- `University` → `Professor` → Aggregation

Senior engineers choose relationships based on **business semantics**, not simply on class structure.
