# 2.13 `equals()` and `hashCode()` – The Most Important Contract in Java

### Why This Matters

Imagine you're storing users in a `HashSet`.

```java
Set<Employee> employees = new HashSet<>();

employees.add(new Employee(101, "Rahul"));
employees.add(new Employee(101, "Rahul"));
```

Business expectation:

```
1 Employee
```

Actual result (default implementation):

```
2 Employees
```

Why?

Because Java doesn't know that both objects represent the same employee.

It only knows they are two different objects in memory.

Understanding `equals()` and `hashCode()` is essential for using collections correctly and for designing reliable domain models.

---

# Object Identity vs Logical Equality

Consider:

```java
Employee e1 =
        new Employee(101, "Rahul");

Employee e2 =
        new Employee(101, "Rahul");
```

Memory:

```
e1 ─────► Object A

e2 ─────► Object B
```

Although both objects contain identical data,

they occupy different memory locations.

By default:

```java
e1.equals(e2)
```

returns:

```
false
```

because `Object.equals()` compares **identity**, not business data.

---

# What Should Equality Mean?

For many business objects,

equality is determined by data,

not memory.

Example:

```
Employee

↓

employeeId
```

If two employees have the same employee ID,

the business may consider them equal.

This is called **logical equality**.

---

# Overriding `equals()`

Example:

```java
public class Employee {

    private int employeeId;

    private String name;

    @Override
    public boolean equals(Object obj) {

        if (this == obj) {
            return true;
        }

        if (!(obj instanceof Employee)) {
            return false;
        }

        Employee other = (Employee) obj;

        return employeeId == other.employeeId;
    }

}
```

Now:

```java
e1.equals(e2);
```

returns:

```
true
```

---

# Why `hashCode()` Also Matters

Many developers correctly override `equals()`,

but forget `hashCode()`.

This breaks hash-based collections.

Consider:

```java
HashSet<Employee>
```

Internally:

```
hashCode()

↓

Bucket

↓

equals()
```

The collection first uses `hashCode()` to locate a bucket.

Only then does it invoke `equals()`.

If equal objects produce different hash codes,

they may be stored in different buckets,

causing duplicates and failed lookups.

---

# The Contract

Java defines a strict contract.

### Rule 1

If:

```java
a.equals(b)
```

is:

```
true
```

then:

```java
a.hashCode() == b.hashCode()
```

**must** also be true.

---

### Rule 2

If hash codes differ,

objects are definitely not equal.

---

### Rule 3

Equal objects must always produce the same hash code during their lifetime.

---

# Overriding `hashCode()`

Example:

```java
@Override
public int hashCode() {

    return Objects.hash(employeeId);

}
```

Now both methods use the same business key.

The contract is preserved.

---

# Visualizing a `HashSet`

```
Employee

↓

hashCode()

↓

Bucket #7

↓

equals()

↓

Already Exists?
```

Both methods work together.

Neither should be implemented independently.

# A Common Production Bug

Suppose:

```java
@Override
public boolean equals(...) {

    return employeeId == other.employeeId;

}
```

But:

```java
hashCode()
```

is not overridden.

Result:

```
HashSet

↓

Duplicate Employees
```

Even though:

```java
equals()
```

returns:

```
true
```

the collection behaves incorrectly because hashing is inconsistent.

---

# Mutable Fields and Equality

Consider:

```java
Employee employee =
        new Employee(101);

HashSet<Employee> set =
        new HashSet<>();

set.add(employee);
```

Later:

```java
employee.setEmployeeId(999);
```

The hash code changes.

Now:

```java
set.contains(employee);
```

may return:

```
false
```

The object still exists in the set,

but the hash bucket no longer matches.

This is why immutable business keys are strongly preferred.

---

# Identity vs Business Keys

Not every field should participate in equality.

Bad example:

```java
Employee

↓

Name

↓

Salary

↓

Address

↓

Phone

↓

Department
```

Changing the salary should not usually change object identity.

A better choice is:

```
Employee

↓

Employee ID
```

Select stable fields that uniquely identify the object.

---

# Spring Boot & Hibernate Perspective

Entity equality deserves special care.

Example:

```java
@Entity
class Employee {

    @Id
    private Long id;

}
```

Hibernate may create proxy objects.

Using every field in `equals()` can produce unexpected behavior.

A common approach is to base equality on the entity's primary key (once assigned) or on an immutable natural business key, depending on the application's domain model.

Understanding the persistence lifecycle is important when implementing equality for JPA entities.

---

# Production Perspective

A financial application stores transactions in a `HashMap`.

Developers override `equals()`,

but not `hashCode()`.

Under heavy load,

duplicate transaction records appear,

causing incorrect reporting.

The issue is eventually traced back to an incorrect equality implementation.

This illustrates how a small mistake can have significant business consequences.

---

# Interview Focus

### Question

Why must `equals()` and `hashCode()` be overridden together?

### Strong Senior-Level Answer

> Hash-based collections such as `HashMap` and `HashSet` first use `hashCode()` to locate candidate objects and then use `equals()` to verify equality. If two logically equal objects return different hash codes, the collections cannot function correctly. Therefore, any class that overrides `equals()` must also override `hashCode()` while maintaining the contract defined by the Java specification.

---

# Common Mistakes

❌ Overriding `equals()` but not `hashCode()`.

❌ Using mutable fields in equality.

❌ Comparing every field unnecessarily.

❌ Confusing object identity with logical equality.

---

# Engineering Insight

A simple rule to remember is:

> **`equals()` decides whether two objects are equal. `hashCode()` decides where Java should look for them.**

Both methods solve different problems,

but they must always agree.

This is why they're often referred to as a **contract**, not just two independent methods.

---

# Engineering Takeaways

- Default `equals()` compares object identity.
- Override `equals()` for logical equality.
- Always override `hashCode()` together with `equals()`.
- Use stable, immutable fields as equality keys.
- Incorrect implementations cause subtle bugs in collections.

---

# Quick Revision

✓ `equals()` checks logical equality.

✓ `hashCode()` supports hashing.

✓ Equal objects must have equal hash codes.

✓ Hash collections rely on both methods.

✓ Prefer immutable business keys.

# Senior Interview Challenge

You're reviewing this implementation:

```java
class Customer {

    private Long id;

    private String email;

    @Override
    public boolean equals(Object obj) {

        return Objects.equals(email, ((Customer) obj).email);

    }

    @Override
    public int hashCode() {

        return Objects.hash(id);

    }

}
```

### What is wrong with this implementation?

**Expected Discussion Points**

- `equals()` uses `email`.
- `hashCode()` uses `id`.
- Two customers with the same email but different IDs could be equal while producing different hash codes.
- This violates the `equals()`–`hashCode()` contract.
- Both methods must use the same logical identity fields.

Senior engineers recognize this issue immediately because it can lead to data corruption in hash-based collections and inconsistent application behavior.
