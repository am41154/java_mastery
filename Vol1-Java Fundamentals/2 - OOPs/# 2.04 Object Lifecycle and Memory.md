# 2.4 Object Lifecycle and Memory Representation

### Why This Matters

Every Java object follows a lifecycle.

It is:

- Created
- Initialized
- Used
- Becomes unreachable
- Removed by the Garbage Collector

Although Java hides memory management from developers, understanding what happens internally helps explain:

- Object references
- Garbage Collection
- Memory leaks
- Performance issues
- `NullPointerException`
- Object sharing

Senior engineers don't manually manage memory, but they understand how the JVM manages it on their behalf.

# Garbage Collection Is Automatic

Unlike C or C++,

Java developers never write:

```cpp
delete object;
```

The JVM automatically determines when memory can be reclaimed.

Benefits:

- Reduced memory management errors.
- Fewer dangling pointers.
- Improved application safety.

However,

automatic memory management does **not** eliminate memory leaks.

We'll revisit this in the JVM chapter.

# Production Perspective

A service stores user sessions in a static collection:

```java
static List<Session> sessions =
    new ArrayList<>();
```

Even after users log out,

old sessions are never removed.

Although the application no longer uses them,

the static list still references every session.

```
Static List

‚Üì

Session 1

‚Üì

Session 2

‚Üì

```

Because the objects remain reachable,

the Garbage Collector cannot reclaim them.

This is a classic memory leak caused by lingering references.

# Interview Focus

### Question

When does an object become eligible for garbage collection?

### Strong Senior-Level Answer

> An object becomes eligible for garbage collection when it is no longer reachable from any GC root, such as local variables, static fields, active threads, or JNI references. Eligibility does not mean immediate collection‚Äîthe JVM decides when to reclaim the memory.

# Common Mistakes

‚ùå Assuming `null` deletes an object.

‚ùå Confusing references with objects.

‚ùå Believing garbage collection happens immediately.

‚ùå Holding unnecessary references in static collections or caches.

---

# Engineering Insight

One of the most common production memory issues is **unintentionally keeping references alive**.

Examples include:

- Static collections
- Improper caches
- Event listeners that are never removed
- Long-lived singleton objects

The Garbage Collector can reclaim only what is unreachable.

If your application continues referencing an object,

the JVM has no choice but to keep it in memory.

---

# Engineering Takeaways

- Objects live on the heap; references point to them.
- Assignment copies references, not objects.
- Objects become eligible for GC when unreachable.
- Garbage collection is automatic but not immediate.
- Reachability‚Äînot `null`‚Äîdetermines object lifetime.

# Senior Interview Challenge

A Spring Boot application gradually consumes more memory over several days.

Heap dumps reveal thousands of `UserSession` objects.

Users have already logged out.

### How would you investigate the issue?

**Expected Discussion Points**

- Determine why the objects remain reachable.
- Look for static collections, caches, singleton beans, or listener registrations.
- Analyze heap dumps to identify GC roots.
- Remove stale references when sessions expire.
- Verify the fix using memory profiling tools.

Senior engineers debug memory problems by tracing **object reachability**, not by forcing garbage collection.
