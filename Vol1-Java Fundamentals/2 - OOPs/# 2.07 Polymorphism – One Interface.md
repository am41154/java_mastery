# 2.7 Polymorphism – One Interface, Many Implementations

### Why This Matters

Imagine you're building an e-commerce application.

Customers can pay using:

- Credit Card
- UPI
- Net Banking
- Wallet

Although each payment method works differently,

the checkout process shouldn't need separate logic for every provider.

Instead, it should simply say:

```
payment.process();
```

The actual implementation depends on the object being used.

This ability is called **polymorphism**.

---

# What Is Polymorphism?

The word *polymorphism* comes from Greek:

```
Poly

↓

Many

Morph

↓

Forms
```

In Java,

polymorphism means:

> **One interface or reference type can represent many different object implementations.**

This allows software to become flexible and extensible.

---

# A Simple Example

Consider:

```java
class Animal {

    void speak() {
        System.out.println("Animal Sound");
    }

}

class Dog extends Animal {

    @Override
    void speak() {
        System.out.println("Bark");
    }

}

class Cat extends Animal {

    @Override
    void speak() {
        System.out.println("Meow");
    }

}
```

Usage:

```java
Animal animal = new Dog();

animal.speak();
```

Output:

```
Bark
```

Although the reference type is `Animal`,

the JVM invokes the `Dog` implementation.

---

# Runtime Method Dispatch

Consider:

```java
Animal animal = new Cat();

animal.speak();
```

Execution:

```
Reference Type

↓

Animal

↓

Actual Object

↓

Cat

↓

Method Executed

↓

Cat.speak()
```

Method selection occurs **at runtime**.

This is known as **dynamic method dispatch**.

---

# Why Is This Powerful?

Without polymorphism:

```java
if(type.equals("CARD")) {

}

else if(type.equals("UPI")) {

}

else if(type.equals("WALLET")) {

}
```

Every new payment method requires modifying existing code.

With polymorphism:

```java
payment.process();
```

Adding a new payment provider requires only a new implementation.

Existing code remains unchanged.

---

# Real-World Analogy

Think about a TV remote.

```
Power Button

↓

Sony TV

Samsung TV

LG TV
```

The button is the same.

Each television performs the operation differently.

The remote doesn't need to know the internal implementation.

That's polymorphism.

---

# Compile-Time vs Runtime Polymorphism

Java supports two forms.

### Compile-Time Polymorphism

Achieved through **method overloading**.

Example:

```java
class Calculator {

    int add(int a, int b) {

        return a + b;

    }

    double add(double a, double b) {

        return a + b;

    }

}
```

The compiler determines which method to call.

---

### Runtime Polymorphism

Achieved through **method overriding**.

Example:

```java
Animal animal = new Dog();

animal.speak();
```

The JVM determines which implementation to execute.

---

# Method Overriding

Rules:

- Same method name.
- Same parameter list.
- Same (or covariant) return type.
- Child provides a different implementation.

Example:

```java
class Vehicle {

    void start() {

    }

}

class Car extends Vehicle {

    @Override
    void start() {

    }

}
```

The `@Override` annotation helps the compiler detect mistakes.

---

# Method Overloading

Example:

```java
void print(String value)

void print(int value)

void print(double value)
```

Same method name,

different parameter list.

Selection occurs at compile time.

---

# Overloading vs Overriding

| Overloading | Overriding |
|-------------|------------|
| Same class | Parent and child classes |
| Different parameters | Same parameters |
| Compile-time resolution | Runtime resolution |
| Increases method flexibility | Enables polymorphism |

---

# Programming to an Interface

One of the most important principles in enterprise Java is:

> **Program to an interface, not an implementation.**

Example:

```java
PaymentGateway gateway =
    new UPIGateway();
```

Instead of:

```java
UPIGateway gateway =
    new UPIGateway();
```

Now the implementation can change easily:

```java
PaymentGateway gateway =
    new CreditCardGateway();
```

The rest of the application remains unchanged.

---

# Polymorphism in Spring Boot

Spring relies heavily on polymorphism.

Example:

```java
public interface PaymentService {

    void process();

}
```

Implementations:

```java
CardPaymentService

UPIPaymentService

WalletPaymentService
```

The controller depends only on:

```java
PaymentService
```

Spring injects the appropriate implementation.

This is the foundation of **Dependency Injection**.

---

# Production Perspective

Consider a notification system.

```
NotificationService

↓

EmailNotification

SMSNotification

PushNotification
```

Business logic simply calls:

```java
notification.send();
```

Each implementation performs its own work.

Adding WhatsApp notifications requires only another implementation.

Existing code remains untouched.

This follows the **Open/Closed Principle**, which we'll discuss later in this chapter.

# Interview Focus

### Question

What is polymorphism?

### Strong Senior-Level Answer

> Polymorphism allows a single reference type to represent multiple object implementations. In Java, compile-time polymorphism is achieved through method overloading, while runtime polymorphism is achieved through method overriding and dynamic method dispatch. It promotes extensibility by allowing code to depend on abstractions rather than concrete implementations.

---

# Common Mistakes

❌ Confusing overloading with overriding.

❌ Forgetting the `@Override` annotation.

❌ Programming directly against concrete classes.

❌ Using large `if-else` chains where polymorphism would be more appropriate.

---

# Engineering Insight

One of the easiest ways to recognize good object-oriented design is this question:

> **Can I introduce a new implementation without modifying existing business logic?**

If the answer is yes,

the system is probably using polymorphism effectively.

This is why frameworks such as Spring, Hibernate, and the Java Collections Framework rely heavily on interfaces and polymorphic behavior.

---

# Engineering Takeaways

- Polymorphism enables one interface to support many implementations.
- Method overriding provides runtime polymorphism.
- Method overloading provides compile-time polymorphism.
- Program against interfaces rather than concrete classes.
- Polymorphism reduces coupling and improves extensibility.

---

# Quick Revision

✓ Overloading = compile time.

✓ Overriding = runtime.

✓ Runtime method selection is dynamic.

✓ Prefer interfaces over implementations.

✓ Polymorphism eliminates large conditional logic.

# Senior Interview Challenge

A reporting module contains the following code:

```java
if(reportType.equals("PDF")) {
    ...
}
else if(reportType.equals("EXCEL")) {
    ...
}
else if(reportType.equals("CSV")) {
    ...
}
```

A new report format is added almost every month.

### How would you redesign this using polymorphism?

**Expected Discussion Points**

- Define a `ReportGenerator` interface.
- Create implementations such as `PdfReportGenerator`, `ExcelReportGenerator`, and `CsvReportGenerator`.
- Use dependency injection or a factory to select the appropriate implementation.
- Eliminate repeated conditional logic.
- Allow new report types to be added without modifying existing processing code.

Senior engineers use polymorphism to make systems **open for extension but closed for modification**, enabling applications to evolve with minimal impact on existing code.
