# 2.6 Inheritance – Reusing and Extending Behavior

### Why This Matters

Software systems often contain objects that share common characteristics.

For example:

- A `Car` is a `Vehicle`.
- A `Bike` is a `Vehicle`.
- A `Truck` is a `Vehicle`.

Instead of duplicating common code in every class, Java allows one class to inherit the properties and behavior of another.

This mechanism is called **inheritance**.

Inheritance promotes code reuse and models **"is-a"** relationships between objects.

---

# What Is Inheritance?

Inheritance allows a class to acquire the fields and methods of another class.

The existing class is called the:

```
Superclass

Parent Class

Base Class
```

The new class is called the:

```
Subclass

Child Class

Derived Class
```

Java uses the `extends` keyword.

Example:

```java
class Vehicle {

    void start() {
        System.out.println("Vehicle Started");
    }

}

class Car extends Vehicle {

}
```

Now:

```java
Car car = new Car();

car.start();
```

Output:

```
Vehicle Started
```

The `Car` class inherits the `start()` method from `Vehicle`.

---

# Visualizing Inheritance

```
          Vehicle
              │
      ----------------
      │              │
     Car           Bike
```

Both `Car` and `Bike` inherit common functionality from `Vehicle`.

---

# Extending Behavior

A subclass can add its own fields and methods.

```java
class Vehicle {

    void start() {

    }

}

class Car extends Vehicle {

    void openSunroof() {

    }

}
```

The `Car` object now supports both:

```
Inherited Behavior

+

Car-Specific Behavior
```

---

# Accessing Parent Members

The child class automatically inherits accessible members.

```java
class Animal {

    void eat() {
        System.out.println("Eating");
    }

}

class Dog extends Animal {

}
```

Usage:

```java
Dog dog = new Dog();

dog.eat();
```

The method belongs to the parent,

but can be used by the child.

---

# The `super` Keyword

Sometimes a child class needs to access its parent.

Example:

```java
class Vehicle {

    void start() {
        System.out.println("Vehicle Started");
    }

}

class Car extends Vehicle {

    void startCar() {

        super.start();

        System.out.println("Car Ready");

    }

}
```

`super` refers to the immediate parent object.

It is commonly used to:

- Invoke parent constructors.
- Invoke overridden methods.
- Access hidden fields (although this is less common).

---

# Constructor Inheritance

Constructors are **not inherited**.

However,

when creating a child object,

the parent constructor executes first.

Example:

```java
class Vehicle {

    Vehicle() {

        System.out.println("Vehicle");

    }

}

class Car extends Vehicle {

    Car() {

        System.out.println("Car");

    }

}
```

Output:

```
Vehicle

Car
```

Execution Order:

```
Parent Constructor

↓

Child Constructor
```

This ensures the parent portion of the object is initialized before the child.

---

# The `extends` Relationship

Inheritance should model an **"is-a"** relationship.

Examples:

```
Car

IS A

Vehicle
```

```
SavingsAccount

IS A

BankAccount
```

Bad example:

```
Engine

IS A

Car
```

Clearly incorrect.

An engine is **part of** a car,

not a type of car.

We'll revisit this distinction when discussing composition.

---

# Java Supports Single Inheritance

A class can extend only **one** class.

Valid:

```java
class Car extends Vehicle {

}
```

Invalid:

```java
class Car extends Vehicle, Machine {

}
```

Java avoids multiple class inheritance to prevent ambiguity.

Multiple inheritance of behavior is achieved through interfaces, which we'll cover later.

---

# Benefits of Inheritance

### Code Reuse

Shared functionality exists in one place.

---

### Reduced Duplication

Common logic doesn't need to be rewritten.

---

### Easier Maintenance

Fixes in the parent class benefit all subclasses.

---

### Polymorphism

Inheritance enables runtime method overriding, one of the key features of Java.

We'll study polymorphism in the next section.

---

# Risks of Inheritance

Inheritance is powerful,

but it also increases coupling.

Changes in the parent class can affect every subclass.

Poor inheritance hierarchies become difficult to understand and maintain.

Example:

```
Animal

↓

Mammal

↓

Pet

↓

DomesticPet

↓

IndoorPet

↓

LuxuryIndoorPet
```

Deep inheritance trees often indicate poor design.

Modern Java applications generally prefer **shallow hierarchies**.

---

# Spring Boot Perspective

Most Spring applications use surprisingly little class inheritance.

Instead, they rely heavily on:

- Interfaces
- Dependency Injection
- Composition

For example:

```
OrderService

↓

PaymentGateway
```

The service depends on an interface,

not a parent implementation.

This makes applications easier to test and extend.

---

# Production Perspective

Suppose several payment providers share common behavior:

```
PaymentGateway

↓

CreditCardGateway

↓

UPIGateway

↓

NetBankingGateway
```

Common validation logic belongs in the parent.

Provider-specific logic belongs in each child class.

However,

if subclasses differ significantly,

composition or strategy patterns may provide a cleaner design.

---

# Interview Focus

### Question

What is inheritance?

### Strong Senior-Level Answer

> Inheritance is an object-oriented mechanism that allows one class to reuse and extend the behavior of another. It models an "is-a" relationship, enabling code reuse and polymorphism. However, inheritance should be used only when the relationship is truly hierarchical, as excessive inheritance can increase coupling and reduce flexibility.

---

# Common Mistakes

❌ Using inheritance for code reuse without an "is-a" relationship.

❌ Creating deep inheritance hierarchies.

❌ Assuming constructors are inherited.

❌ Forgetting that parent constructors execute before child constructors.

---

# Engineering Insight

Many junior developers reach for inheritance whenever two classes share code.

Experienced engineers ask a different question:

> **Do these classes truly represent an "is-a" relationship?**

If the answer is no,

inheritance is usually the wrong choice.

This is why modern frameworks such as Spring encourage **composition over inheritance**.

Inheritance should model a hierarchy,

not simply eliminate duplicate code.

---

# Engineering Takeaways

- Inheritance models an "is-a" relationship.
- Java supports single class inheritance.
- Child classes inherit accessible fields and methods.
- Parent constructors execute before child constructors.
- Use inheritance for true specialization, not merely for sharing code.

---

# Quick Revision

✓ Use `extends` to inherit from a class.

✓ Constructors are not inherited.

✓ Parent constructors execute first.

✓ Inheritance enables code reuse.

✓ Avoid deep inheritance hierarchies.

# Senior Interview Challenge

You're reviewing the following design:

```
Employee

↓

Developer

↓

BackendDeveloper

↓

SpringBootDeveloper

↓

Java17SpringBootDeveloper
```

Each subclass adds only a few utility methods.

### Would you keep this hierarchy? Why or why not?

**Expected Discussion Points**

- The hierarchy is unnecessarily deep.
- The relationship becomes rigid and difficult to maintain.
- Small behavioral differences can often be modeled using composition or strategies.
- Prefer interfaces and delegation for evolving business requirements.
- Use inheritance only when subclasses genuinely represent specialized forms of the parent.

Senior engineers recognize that **every inheritance relationship is a long-term design commitment**. They introduce it only when the domain truly requires it.
