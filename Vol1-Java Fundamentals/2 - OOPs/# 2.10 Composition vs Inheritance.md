# 2.10 Composition vs Inheritance – Choosing the Right Design

### Why This Matters

One of the most common mistakes in object-oriented design is using inheritance simply to reuse code.

While inheritance can reduce duplication, it also creates a strong dependency between classes.

Modern Java applications generally prefer **composition** because it produces systems that are more flexible, easier to test, and simpler to maintain.

This is why you'll often hear the guideline:

> **Favor Composition Over Inheritance**

---

# Understanding the Difference

Although both techniques enable code reuse, they model different relationships.

### Inheritance

Represents an **is-a** relationship.

```
Car

↓

Vehicle
```

A car **is a** vehicle.

---

### Composition

Represents a **has-a** relationship.

```
Car

↓

Engine
```

A car **has an** engine.

The distinction may seem small,

but it has major design implications.

---

# Inheritance Example

```java
class Vehicle {

    void start() {
        System.out.println("Starting...");
    }

}

class Car extends Vehicle {

}
```

Usage:

```java
Car car = new Car();

car.start();
```

The `Car` automatically inherits the `start()` method.

---

# Composition Example

```java
class Engine {

    void start() {
        System.out.println("Engine Started");
    }

}

class Car {

    private final Engine engine =
            new Engine();

    void start() {

        engine.start();

    }

}
```

Here,

the `Car` delegates work to the `Engine`.

The car **contains** an engine instead of **being** an engine.

# Why Composition Is Preferred

Composition provides several advantages.

### Loose Coupling

Objects communicate through well-defined interfaces.

Changes in one class are less likely to affect others.

---

### Better Flexibility

Suppose a car can use different engines.

```
Car

↓

PetrolEngine

DieselEngine

ElectricMotor
```

The engine can be replaced without changing the `Car` class.

With inheritance,

this flexibility is much harder to achieve.

---

### Easier Testing

Testing becomes straightforward.

```java
Car car =
    new Car(mockEngine);
```

The real engine can be replaced with a mock implementation.

This is one of the foundations of unit testing.

---

### Better Reusability

The same `Engine` object can be reused in different types of vehicles.

Composition encourages reusable building blocks.

---

# When Inheritance Is Appropriate

Inheritance should be used when:

- The relationship is genuinely **is-a**.
- Subclasses extend the parent's behavior naturally.
- The hierarchy is stable.
- Shared behavior belongs in the parent.

Examples:

```
Vehicle

↓

Car

Truck

Bike
```

```
Employee

↓

Manager

Developer
```

These are natural hierarchies.

---

# When Composition Is Better

Choose composition when:

- Objects collaborate.
- Behavior may change.
- Multiple implementations exist.
- Features should be interchangeable.
- Future requirements are uncertain.

Examples:

```
OrderService

↓

PaymentGateway
```

```
NotificationService

↓

EmailSender

SmsSender

PushSender
```

The service **uses** another object.

It is not a specialized version of it.

---

# A Common Design Mistake

Suppose you write:

```java
class Car extends Engine {

}
```

Does this make sense?

No.

A car is **not** an engine.

It **has** an engine.

The correct design is composition.

This illustrates an important rule:

> **Never use inheritance solely to reuse code.**

Use it only when the domain truly represents an "is-a" relationship.

---

# Composition in Spring Boot

Spring applications are built almost entirely on composition.

Example:

```java
@Service
public class OrderService {

    private final PaymentService paymentService;

    public OrderService(PaymentService paymentService) {
        this.paymentService = paymentService;
    }

}
```

The `OrderService` does not extend `PaymentService`.

Instead,

it collaborates with it through composition.

This design:

- Reduces coupling.
- Improves testability.
- Supports dependency injection.

---

# Production Perspective

Imagine a report generation system.

Poor design:

```
PdfReportGenerator

↓

ExcelReportGenerator

↓

CsvReportGenerator
```

Each new format extends the previous one.

The hierarchy quickly becomes confusing.

A better approach:

```
ReportService

↓

ReportGenerator Interface

↓

PDF

Excel

CSV
```

The service depends on an abstraction,

and new formats can be added independently.

# Interview Focus

### Question

Why is composition generally preferred over inheritance?

### Strong Senior-Level Answer

> Composition promotes loose coupling by allowing objects to collaborate rather than forming rigid hierarchies. It improves flexibility, testability, and maintainability because implementations can be replaced without affecting dependent classes. Inheritance should be reserved for genuine "is-a" relationships, whereas composition is appropriate for "has-a" relationships and changing behavior.

---

# Common Mistakes

❌ Using inheritance to eliminate duplicate code.

❌ Creating deep inheritance hierarchies.

❌ Ignoring object ownership and relationships.

❌ Extending classes that were not designed for inheritance.

---

# Engineering Insight

Modern Java frameworks consistently demonstrate this principle.

Examples include:

- Spring Dependency Injection
- Strategy Pattern
- Decorator Pattern
- Java Collections Framework

They all encourage object collaboration rather than deep inheritance trees.

A useful question to ask during design is:

> **Will this relationship still make sense if the implementation changes next year?**

If not,

composition is usually the safer choice.

---

# Engineering Takeaways

- Inheritance models "is-a" relationships.
- Composition models "has-a" relationships.
- Composition provides greater flexibility.
- Modern enterprise applications rely heavily on composition.
- Favor composition unless inheritance clearly represents the business domain.

# Senior Interview Challenge

A legacy application contains the following hierarchy:

```
Notification

↓

EmailNotification

↓

HtmlEmailNotification

↓

MarketingHtmlEmailNotification
```

Each subclass overrides only a small portion of the behavior.

### How would you redesign this system?

**Expected Discussion Points**

- Replace the deep hierarchy with a `NotificationSender` interface.
- Use composition to assemble features (formatting, delivery, templates).
- Delegate responsibilities instead of inheriting them.
- Apply the Strategy pattern for interchangeable behaviors.
- Keep each class focused on a single responsibility.

Senior engineers know that inheritance is easy to introduce but difficult to remove. Composition keeps systems adaptable as business requirements evolve.
