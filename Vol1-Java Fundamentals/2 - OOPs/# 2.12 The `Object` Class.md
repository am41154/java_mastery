# 2.12 The `Object` Class – The Root of the Java Class Hierarchy

### Why This Matters

Every class in Java ultimately inherits from a single class:

```java
java.lang.Object
```

Whether you explicitly extend another class or not,

your class automatically inherits from `Object`.

This means every Java object has access to methods such as:

- `equals()`
- `hashCode()`
- `toString()`
- `getClass()`
- `clone()`
- `wait()`
- `notify()`
- `notifyAll()`

Understanding the `Object` class is essential because these methods influence collections, logging, concurrency, persistence frameworks, and API design.

---

# The Root of Every Class

Consider:

```java
class Employee {

}
```

The compiler interprets this as:

```java
class Employee extends Object {

}
```

Every class eventually forms part of the same inheritance tree.

```
            Object
               │
        ----------------
        │              │
    Employee        Customer
```

Even classes like `String`, `Integer`, and your own domain classes inherit from `Object`.

---

# Common Methods in `Object`

| Method | Purpose |
|----------|---------|
| `equals()` | Compares objects |
| `hashCode()` | Generates hash value |
| `toString()` | Returns string representation |
| `getClass()` | Returns runtime class information |
| `clone()` | Creates a copy (if supported) |
| `wait()` | Thread coordination |
| `notify()` | Wakes a waiting thread |
| `notifyAll()` | Wakes all waiting threads |

Some of these methods are overridden frequently,

while others are rarely used directly.

---

# `toString()`

By default:

```java
Employee employee =
        new Employee();

System.out.println(employee);
```

Output:

```
Employee@4e25154f
```

This output contains:

```
Class Name

+

Hash Code
```

It is usually not meaningful to users.

---

# Overriding `toString()`

Example:

```java
class Employee {

    private String name;

    @Override
    public String toString() {

        return "Employee{name='" + name + "'}";

    }

}
```

Now:

```java
System.out.println(employee);
```

Output:

```
Employee{name='Rahul'}
```

Meaningful `toString()` implementations improve logging and debugging.

---

# `getClass()`

Every object knows its runtime type.

Example:

```java
Employee employee =
        new Employee();

System.out.println(employee.getClass());
```

Output:

```
class Employee
```

Frameworks such as Spring, Hibernate, and Jackson use runtime type information extensively.

---

# `equals()`

The default implementation compares **object references**.

Example:

```java
Employee e1 =
        new Employee();

Employee e2 =
        new Employee();

System.out.println(e1.equals(e2));
```

Output:

```
false
```

Although both objects are identical,

they occupy different memory locations.

We'll study `equals()` in detail in the next section.

---

# `hashCode()`

Each object has a hash code.

Example:

```java
Employee employee =
        new Employee();

System.out.println(employee.hashCode());
```

Hash codes are heavily used by:

- `HashMap`
- `HashSet`
- `ConcurrentHashMap`

A correct implementation of `hashCode()` is essential whenever `equals()` is overridden.

---

# `clone()`

The `clone()` method creates a copy of an object.

However,

it has several design limitations.

Modern Java applications generally prefer:

- Copy constructors
- Factory methods
- Builder patterns

Over `clone()`.

We'll discuss object copying later in the book.

---

# `wait()`, `notify()`, and `notifyAll()`

These methods support thread communication.

Example:

```
Thread A

↓

wait()
```

Later:

```
Thread B

↓

notify()
```

These methods are foundational to Java concurrency,

although modern applications often use higher-level concurrency utilities instead.

We'll revisit them in **Chapter 5 – Concurrency**.

---

# Why `Object` Matters

Because every object inherits these methods,

frameworks rely on them extensively.

Examples:

```
HashMap

↓

equals()

hashCode()
```

```
Logger

↓

toString()
```

```
Reflection

↓

getClass()
```

Poor implementations can introduce subtle bugs throughout an application.

---

# Spring Boot Perspective

Suppose a REST API logs domain objects:

```java
log.info("User: {}", user);
```

Logging frameworks internally invoke:

```java
user.toString();
```

If `toString()` is not overridden,

logs become difficult to read.

Similarly,

Hibernate and Spring often rely on `equals()` and `hashCode()` when managing entities and collections.

---

# Production Perspective

Imagine a `Customer` object stored in a `HashSet`.

If `equals()` is overridden,

but `hashCode()` is not,

duplicate customers may appear unexpectedly,

or lookups may fail.

The application compiles successfully,

but behaves incorrectly.

This is one of the most common production bugs related to object equality.

---

# Interview Focus

### Question

Why is the `Object` class important?

### Strong Senior-Level Answer

> The `Object` class is the root of the Java class hierarchy. Every Java class inherits from it, gaining methods such as `equals()`, `hashCode()`, `toString()`, and `getClass()`. These methods form the foundation of object comparison, hashing, logging, reflection, and concurrency, making them critical to both the language and enterprise frameworks.

---

# Common Mistakes

❌ Assuming `equals()` compares object contents by default.

❌ Forgetting to override `hashCode()` when overriding `equals()`.

❌ Ignoring `toString()` and producing unreadable logs.

❌ Using `clone()` without understanding its limitations.

---

# Engineering Insight

Although the `Object` class contains only a handful of methods,

they influence almost every Java application.

For example:

- Collections depend on `equals()` and `hashCode()`.
- Logging depends on `toString()`.
- Reflection depends on `getClass()`.
- Thread coordination depends on `wait()` and `notify()`.

Mastering these methods is far more important than memorizing their signatures.

---

# Engineering Takeaways

- Every Java class inherits from `Object`.
- `Object` provides fundamental behavior shared by all objects.
- Overriding `toString()` improves debugging.
- `equals()` and `hashCode()` must work together.
- Many Java frameworks rely on these inherited methods.

# Senior Interview Challenge

You're reviewing the following code:

```java
Employee e1 =
        new Employee(101, "Rahul");

Employee e2 =
        new Employee(101, "Rahul");
```

Business rules state that employees with the same employee ID should be considered equal.

### Is the default implementation of `equals()` sufficient?

**Expected Discussion Points**

- No.
- The default implementation compares object identity, not business data.
- Override `equals()` to compare employee IDs (or the chosen business key).
- Override `hashCode()` consistently.
- Verify behavior in collections such as `HashSet` and `HashMap`.

This discussion naturally leads to one of the most frequently asked Java interview topics.
