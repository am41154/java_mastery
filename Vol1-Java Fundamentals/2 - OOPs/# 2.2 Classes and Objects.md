# 2.2 Classes and Objects

### Why This Matters

Every Java application, regardless of its size, is built using **classes** and **objects**.

Whether you're creating a simple console application or a distributed microservices platform with Spring Boot, you're constantly working with objects.

Understanding the relationship between classes and objects is fundamental to writing maintainable, object-oriented software.

---

# What Is a Class?

A **class** is a blueprint that defines:

- What data an object holds.
- What operations an object can perform.

Think of a class as an architectural blueprint.

A blueprint describes a house but is **not** the house itself.

Similarly,

a Java class describes an object but is **not** the object.

Example:

```java
class Employee {

    String name;
    int age;

    void work() {
        System.out.println("Working...");
    }
}
```

This class defines:

State:

- `name`
- `age`

Behavior:

- `work()`

No object exists yet.

---

# What Is an Object?

An **object** is a runtime instance of a class.

Creating an object:

```java
Employee employee = new Employee();
```

Memory is allocated,

the object is initialized,

and a reference is returned.

```
Class

↓

Employee

↓

Object

↓

Employee@6d06d69c
```

Each object has its own independent state.

---

# Multiple Objects

Suppose:

```java
Employee emp1 = new Employee();

Employee emp2 = new Employee();
```

Memory:

```
Heap

┌──────────────┐
│ Employee #1  │
│ Name = Rahul │
└──────────────┘

┌──────────────┐
│ Employee #2  │
│ Name = Priya │
└──────────────┘
```

Although both objects come from the same class,

they store different data.

# State and Behavior

Every object consists of:

```
State

+

Behavior
```

Example:

```
BankAccount

State

Account Number

Balance

Owner

↓

Behavior

deposit()

withdraw()

transfer()
```

State is represented by fields.

Behavior is represented by methods.

# Object Identity

Consider:

```java
Employee e1 = new Employee();

Employee e2 = new Employee();
```

Even if:

```java
e1.name = "John";

e2.name = "John";
```

These are still different objects.

```
e1

↓

Object A
```

```
e2

↓

Object B
```

Identity is different even when values are equal.

---

# Object References

This is one of the most misunderstood concepts in Java.

```java
Employee e1 = new Employee();

Employee e2 = e1;
```

Many beginners assume two objects are created.

Actually:

```
e1 ───────┐
          │
          ▼
     Employee Object
          ▲
          │
e2 ───────┘
```

Both variables point to the **same object**.

Changing through one reference affects the other.

Example:

```java
e2.name = "Alice";
```

Now:

```java
System.out.println(e1.name);
```

Output:

```
Alice
```

because both references point to the same object.

---

# Reference vs Object

A reference is **not** the object.

Think of it like:

```
House Address

↓

Actual House
```

Changing the address variable does not create another house.

Similarly,

assigning another reference does not create another object.

---

# Creating Objects with Constructors

Objects are created using constructors.

```java
class Employee {

    Employee() {

        System.out.println("Created");

    }

}
```

Creating an object:

```java
Employee employee =
    new Employee();
```

Execution:

```
new

↓

Memory Allocation

↓

Constructor

↓

Reference Returned
```

Constructors initialize newly created objects.

We'll study constructors in detail in the next section.

# Object Lifecycle

A simplified lifecycle:

```
new

↓

Constructor

↓

Object Used

↓

No More References

↓

Eligible for Garbage Collection
```

The JVM automatically reclaims memory when an object is no longer reachable.

Consider a Spring Boot application.


# Interview Focus

### Question

What is the difference between a class and an object?

### Strong Senior-Level Answer

> A class is a blueprint that defines the state and behavior of an object. An object is a runtime instance of that class, created when memory is allocated using the `new` operator or through framework-managed instantiation. Multiple objects can be created from the same class, each maintaining its own independent state.

---

# Common Mistakes

❌ Assuming a class occupies memory for object data.

❌ Confusing references with objects.

❌ Believing assignment creates a new object.

❌ Comparing object references when logical equality is required.

---

# Engineering Insight

A surprising number of production bugs arise from misunderstanding object references.

Example:

```java
Order order1 = service.getOrder();

Order order2 = order1;
```

Updating:

```java
order2.setStatus("SHIPPED");
```

also changes:

```java
order1
```

because there is only **one object**.

Senior engineers always ask:

> "Am I copying the object or merely copying the reference?"

Understanding this distinction becomes especially important when working with mutable objects, caching, and multithreaded applications.

---

# Engineering Takeaways

- A class is a blueprint; an object is its runtime instance.
- Objects contain state and behavior.
- Multiple references can point to the same object.
- Object assignment copies references, not objects.
- Objects become eligible for garbage collection when no reachable references remain.

---

# Quick Revision

✓ Classes define objects.

✓ Objects are created using `new`.

✓ References point to objects.

✓ Assignment copies the reference.

✓ Objects live on the heap and are managed by the JVM.

# Senior Interview Challenge

Consider the following code:

```java
Employee e1 = new Employee();

Employee e2 = e1;

e2.setName("David");
```

### What will `e1.getName()` return, and why?

**Expected Discussion Points**

- Only one `Employee` object exists.
- `e1` and `e2` reference the same object.
- Updating the object through one reference is visible through the other.
- Assignment copies the reference, not the object.
- Creating an independent copy requires explicit copying (copy constructor, clone, or another copying strategy).

Understanding references is fundamental to mastering Java object behavior.
