# 2.3 Fields, Methods, and Constructors

### Why This Matters

An object is useful only when it can:

- Store information
- Perform actions
- Be initialized correctly

These responsibilities are handled by:

- Fields
- Methods
- Constructors

Together, they form the core building blocks of every Java class.

---

# Fields

A **field** (also called an instance variable) represents the state of an object.

Example:

```java
class Employee {

    String name;
    int age;
    double salary;

}
```

Here:

- `name`
- `age`
- `salary`

represent the object's data.

Each object maintains its own copy of these fields.

---

# Instance Fields

Consider:

```java
Employee e1 = new Employee();

Employee e2 = new Employee();
```

```
Heap

Employee #1

name = Rahul

age = 30

---------------

Employee #2

name = Priya

age = 28
```

Although both objects belong to the same class,

their fields are independent.

---

# Default Values

If fields are not initialized,

Java assigns default values.

| Type | Default Value |
|------|---------------|
| `int` | `0` |
| `long` | `0L` |
| `double` | `0.0` |
| `boolean` | `false` |
| Object Reference | `null` |

Example:

```java
class Student {

    int age;
    boolean active;
    String name;

}
```

A new object contains:

```
age = 0

active = false

name = null
```

Local variables do **not** receive default values and must be initialized before use.

---

# Methods

Methods define the behavior of an object.

Example:

```java
class Account {

    double balance;

    void deposit(double amount) {

        balance += amount;

    }

}
```

Methods operate on the object's state.

```
Object

↓

State

↓

Method

↓

Updated State
```

---

# Parameters and Return Values

Methods may accept input.

```java
public int square(int number) {

    return number * number;

}
```

Or return objects.

```java
public Customer findCustomer() {

    return customer;

}
```

Methods should have a single, well-defined responsibility.

---

# Constructors

A constructor initializes a newly created object.

Example:

```java
class Employee {

    Employee() {

        System.out.println("Employee Created");

    }

}
```

Creating an object:

```java
Employee employee =
    new Employee();
```

The constructor executes automatically.

---

# Characteristics of Constructors

Constructors:

- Have the same name as the class.
- Do not specify a return type.
- Execute automatically during object creation.
- Can be overloaded.

---

# Default Constructor

If no constructor is defined,

the compiler provides one automatically.

Example:

```java
class Employee {

}
```

Equivalent to:

```java
Employee() {

}
```

However,

once you define any constructor,

the compiler no longer generates the default constructor.

---

# Parameterized Constructor

Example:

```java
class Employee {

    String name;

    Employee(String name) {

        this.name = name;

    }

}
```

Usage:

```java
Employee emp =
    new Employee("Rahul");
```

Result:

```
name = Rahul
```

Parameterized constructors help create fully initialized objects.

---

# Constructor Overloading

A class may define multiple constructors.

```java
class Employee {

    Employee() {

    }

    Employee(String name) {

    }

    Employee(String name, int age) {

    }

}
```

The compiler selects the constructor based on the arguments provided.

---

# The `this` Keyword

Inside a constructor,

parameter names often match field names.

```java
class Employee {

    String name;

    Employee(String name) {

        this.name = name;

    }

}
```

Here:

```
this.name

↓

Object Field
```

```
name

↓

Constructor Parameter
```

`this` refers to the current object.

---

# Constructor Chaining

One constructor can invoke another using:

```java
this(...)
```

Example:

```java
class Employee {

    Employee() {

        this("Unknown");

    }

    Employee(String name) {

        this.name = name;

    }

}
```

Execution:

```
Employee()

↓

Employee(String)

↓

Object Initialized
```

This avoids duplicate initialization logic.

> **Rule**
>
> `this(...)` must be the first statement in a constructor.

---

# Initialization Order

When creating an object,

initialization occurs in this order:

```
Memory Allocation

↓

Default Field Values

↓

Field Initializers

↓

Constructor

↓

Object Ready
```

Understanding this order helps explain initialization behavior in inheritance, which we'll study later.

---

# Constructors vs Methods

| Constructor | Method |
|-------------|--------|
| Initializes an object | Performs an operation |
| Same name as class | Any valid name |
| No return type | Has a return type (or `void`) |
| Called automatically | Invoked explicitly |
| Executes once per object creation | Can execute many times |

---

# Spring Boot Perspective

In enterprise applications,

objects are often created by frameworks rather than directly using `new`.

For example:

```java
@Service
public class PaymentService {

}
```

Spring invokes the constructor and injects dependencies automatically.

This is why constructor design is important even when you don't instantiate objects yourself.

Constructor injection is generally preferred over field injection because it promotes immutability and makes dependencies explicit.

---

# Production Perspective

Consider:

```java
class DatabaseConnection {

    String url;

}
```

Nothing prevents creating:

```java
new DatabaseConnection();
```

where:

```
url = null
```

A better design:

```java
DatabaseConnection(String url)
```

ensures every object starts in a valid state.

Well-designed constructors prevent invalid objects from existing.

---

# Interview Focus

### Question

What is the difference between a constructor and a method?

### Strong Senior-Level Answer

> A constructor initializes a newly created object and executes automatically during object creation. It has the same name as the class and does not declare a return type. A method defines behavior, may return a value, and can be invoked multiple times throughout the object's lifecycle.

---

# Common Mistakes

❌ Adding a return type to a constructor.

❌ Assuming the compiler always generates a default constructor.

❌ Duplicating initialization logic across constructors.

❌ Leaving objects in an invalid state after construction.

---

# Engineering Insight

Constructors should establish an object's **valid initial state**.

A useful design principle is:

> If an object cannot function without certain data,
> require that data in the constructor.

This makes invalid states impossible rather than relying on callers to initialize the object correctly later.

Modern Java frameworks also encourage immutable objects with constructor-based initialization for this reason.

---

# Engineering Takeaways

- Fields represent object state.
- Methods define object behavior.
- Constructors initialize objects.
- Prefer parameterized constructors for required data.
- Use constructor chaining to avoid duplicated logic.

---

✓ Fields store data.

✓ Methods perform actions.

✓ `this` refers to the current object.

# Senior Interview Challenge

Consider the following class:

```java
class User {

    String username;

    String email;

}
```

Developers frequently create:

```java
new User();
```

and forget to initialize the fields, causing `NullPointerException`s later.

### How would you redesign this class?

**Expected Discussion Points**

- Introduce a parameterized constructor requiring mandatory fields.
- Keep the object in a valid state immediately after creation.
- Consider making required fields `final` if they should not change.
- Validate constructor arguments where appropriate.
- Prefer constructor injection over setter-based initialization for mandatory data.

Senior engineers use constructors not just to assign values, but to enforce object correctness from the moment an instance is created.
