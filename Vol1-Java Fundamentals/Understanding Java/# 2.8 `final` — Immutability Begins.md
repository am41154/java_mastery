# 2.8 `final` — Immutability Begins Here

## Why This Matters

The `final` keyword is often introduced as:

> "A final variable cannot be changed."

While true, this explanation is incomplete.

The meaning of `final` depends on **where it is applied**:

- Variable
- Reference
- Method
- Class

Understanding these differences helps you write safer, more predictable, and easier-to-maintain code.

---

## Real Interview

An interviewer writes:

```java
final List<String> names = new ArrayList<>();

names.add("Alice");
```

Then asks:

> **"Will this compile?"**

Many candidates answer:

> "No, because the variable is final."

The correct answer is:

> **Yes.**

The list can still be modified.

Only the **reference** is final—not the object.

---

## `final` Variables

A final primitive can be assigned only once.

```java
final int MAX_RETRY = 3;

// MAX_RETRY = 5; ❌ Compilation Error
```

Once initialized, its value cannot change.

---

## `final` References

Consider:

```java
final Trade trade = new Trade();
```

This prevents reassignment.

```java
trade = new Trade();   // ❌ Not Allowed
```

However, modifying the object's state is perfectly valid.

```java
trade.setStatus("COMPLETED");   // ✓ Allowed
```

> **Key Insight**
>
> `final` protects the reference, not the object.

---

## `final` Methods

A final method cannot be overridden.

```java
class PaymentService {

    public final void validate() {
    }

}
```

Subclasses inherit the method but cannot provide their own implementation.

This is useful when certain behavior must remain consistent across all subclasses.

---

## `final` Classes

A final class cannot be extended.

```java
public final class SecurityToken {
}
```

One well-known example is `String`.

Making `String` final ensures its behavior cannot be altered through inheritance, which contributes to its immutability and security.

---

## `final` vs Immutability

These terms are related but not identical.

```java
final List<String> list = new ArrayList<>();

list.add("Java");
```

The reference never changes.

The object does.

A class is **immutable** only if its internal state cannot change after construction.

Using `final` fields is one step toward immutability, but it is not sufficient by itself.

---

## Production Perspective

A shared configuration object is accidentally modified at runtime, causing inconsistent behavior across services.

The fix is not to make the reference `final`.

The fix is to make the **object itself immutable**.

This distinction prevents many subtle bugs in concurrent applications.

---

## Interview Focus

**Question**

What is the difference between a final reference and an immutable object?

**Strong Senior-Level Answer**

> A final reference cannot point to a different object after initialization. An immutable object cannot have its internal state changed after it is created. A final reference may still refer to a mutable object.

---

## Common Mistakes

❌ Assuming `final` makes an object immutable.

❌ Using mutable objects as constants.

❌ Overusing `final` on every local variable without improving readability.

---

## Senior Engineering Insight

`final` is most valuable when it communicates **intent**.

It tells other developers that a value, reference, or behavior is not expected to change.

Used thoughtfully, it improves code clarity and reduces accidental modifications.

---

## Engineering Takeaways

- `final` variables can be assigned only once.
- `final` references cannot point to another object.
- `final` methods cannot be overridden.
- `final` classes cannot be extended.
- `final` does not automatically make an object immutable.

---
