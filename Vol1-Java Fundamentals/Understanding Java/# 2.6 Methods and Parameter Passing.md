# 2.6 Methods and Parameter Passing — Is Java Really Pass-by-Value?

## Why This Matters

Few Java topics generate as much confusion as parameter passing.

Ask ten developers:

> **"Is Java pass-by-value or pass-by-reference?"**

You'll likely hear multiple answers.

Some will confidently say:

> "Objects are passed by reference."

Others will argue:

> "Primitives are pass-by-value, objects are pass-by-reference."

Both answers are incorrect.

The Java Language Specification is unambiguous:

> **Java is always pass-by-value.**

Understanding this concept is essential for:

- Debugging unexpected object modifications
- Designing immutable APIs
- Working with collections
- Understanding method behavior
- Explaining Java confidently in interviews

---

## Real Interview

An interviewer writes the following code.

```java
public void updateTrade(Trade trade) {
    trade.setStatus("COMPLETED");
}

Trade trade = new Trade("T1001");

updateTrade(trade);

System.out.println(trade.getStatus());
```

The output is:

```
COMPLETED
```

The interviewer asks:

> "If Java is pass-by-value, why did the object change?"

The candidate replies:

> "Because objects are passed by reference."

The interviewer smiles.

> "Then explain the next example."

```java
public void updateTrade(Trade trade) {
    trade = new Trade("T2001");
}
```

After the method returns,

the original object remains unchanged.

The contradiction exposes a common misunderstanding.

---

## What Does Pass-by-Value Mean?

Pass-by-value simply means:

> **The method receives a copy of the argument's value.**

The important word is **value**.

The value depends on the type of variable.

For primitive variables,

the value is the primitive itself.

For object variables,

the value is the reference.

Java copies the value stored in the variable.

It never copies the object automatically.

---

## Primitive Example

Consider the following code.

```java
public void increment(int number) {

    number++;

}

int value = 10;

increment(value);

System.out.println(value);
```

Output:

```
10
```

Why?

Because the method receives a copy.

```
Caller

value = 10

          │ Copy
          ▼

Method

number = 10
```

Incrementing the copy does not affect the original variable.

---

## Object Example

Now consider an object.

```java
Trade trade = new Trade("T1001");

updateTrade(trade);
```

Memory before the method call:

```
Caller

trade
   │
   ▼

+----------------------+
| Trade                |
| status = NEW         |
+----------------------+
```

When the method is invoked,

the reference is copied.

```
Caller

trade ─────┐
           │
           ▼

      Trade Object

           ▲
           │

Method

trade ─────┘
```

Notice something important.

There are **two references**,

but **only one object**.

Both references point to the same object.

---

## Modifying the Object

Inside the method,

```java
trade.setStatus("COMPLETED");
```

The object changes.

```
Trade Object

status = COMPLETED
```

Since both references point to the same object,

the caller observes the modification.

The object changed.

The reference did not.

---

## Reassigning the Reference

Now consider:

```java
public void updateTrade(Trade trade) {

    trade = new Trade("T2001");

}
```

Initially,

```
Caller

trade ─────┐
           │
           ▼

Trade T1001

           ▲
           │

Method

trade ─────┘
```

After reassignment,

```
Caller

trade ─────► Trade T1001



Method

trade ─────► Trade T2001
```

The copied reference now points somewhere else.

The caller's reference remains unchanged.

This explains why the original object is unaffected.

---

## A Helpful Analogy

Imagine writing someone's home address on two separate pieces of paper.

```
Paper A

123 Main Street

Paper B

123 Main Street
```

Changing the house itself affects everyone visiting that address.

Changing the address written on one piece of paper affects only that paper.

Java behaves the same way.

The object is the house.

The reference is the address.

Methods receive a copy of the address—not the house.

---

## Mutable vs Immutable Objects

Consider a `String`.

```java
String name = "Java";

change(name);
```

Inside the method,

```java
name = name.toUpperCase();
```

The original variable remains unchanged.

Why?

Because `String` is immutable.

No existing object is modified.

Instead,

a new object is created,

and the copied reference points to it.

We'll study String immutability in detail later.

## Production Perspective

A REST endpoint receives a request object.

A validation method modifies one of its fields.

Later,

another service unexpectedly reads the modified value.

The bug is difficult to reproduce because no explicit return value exists.

The problem isn't parameter passing.

The problem is shared mutable state.

Many production bugs occur because developers unintentionally modify objects passed between services instead of treating them as immutable.

## Interview Focus

**Question**

Is Java pass-by-value or pass-by-reference?

**Strong Senior-Level Answer**

> Java is always pass-by-value. For primitive types, the value copied is the primitive itself. For objects, the value copied is the reference to the object. As a result, methods can modify the state of the shared object but cannot change the caller's reference to point to a different object.

This explanation demonstrates a precise understanding of the language.

---

## Common Misconceptions

### ❌ "Objects are passed by reference."

Incorrect.

Only the reference value is copied.

---

### ❌ "Java supports both pass-by-value and pass-by-reference."

Incorrect.

Java supports only pass-by-value.

The copied value may happen to be a reference.

---

### ❌ "Assigning a new object inside a method changes the caller."

Incorrect.

It changes only the local copy of the reference.

---

## Senior Engineering Insight

Many APIs become difficult to reason about because methods modify objects supplied by their callers.

Experienced engineers prefer designs where methods either:

- Return a new object, or
- Clearly document that the supplied object will be mutated.

Reducing hidden side effects leads to code that is easier to understand, test, and maintain.

---

## Engineering Takeaways

- Java always uses pass-by-value.
- Primitive values are copied directly.
- Object references are copied, not objects.
- Multiple references can point to the same object.
- Reassigning a copied reference does not affect the caller.
- Shared mutable objects are a common source of production bugs.

---
