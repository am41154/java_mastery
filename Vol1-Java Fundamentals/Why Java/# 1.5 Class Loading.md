# 1.5 Class Loading — How Java Finds Your Code

## Why This Matters

Imagine you start a Spring Boot application containing:

- More than 2,000 classes
- Hundreds of third-party libraries
- Multiple starter dependencies
- Thousands of configuration files

Yet your application starts with a single command.

```bash
java -jar trading-service.jar
```

How does the JVM know where every class is located?

How does it avoid loading duplicate classes?

Why doesn't every application load the entire JDK into memory?

The answer lies in one of the JVM's most elegant mechanisms:

**Class Loading.**

Understanding class loading explains many seemingly unrelated topics, including:

- Spring Boot startup
- JDBC driver discovery
- Reflection
- Dependency Injection
- Plugin architectures
- `ClassNotFoundException`
- `NoClassDefFoundError`

Many production startup failures are, at their core, class loading problems.

---

## What Is a Class Loader?

A **Class Loader** is a JVM component responsible for locating, loading, and making Java classes available during runtime.

When the JVM encounters a class for the first time, it performs three high-level steps:

1. Locate the class.
2. Load it into memory.
3. Make it available for execution.

Without class loading, a compiled `.class` file is simply another file stored on disk.

---

## The Class Loading Process

Every Java class follows the same journey.

```
Class Reference
       │
       ▼
Locate Class
       │
       ▼
Read Bytecode
       │
       ▼
Create Class Object
       │
       ▼
Verify & Link
       │
       ▼
Initialize Class
       │
       ▼
Ready for Execution
```

This process happens automatically.

As developers, we rarely need to trigger it manually.

---

## The Three Built-in Class Loaders

Modern JVMs use a hierarchy of class loaders.

```
Application Class Loader
            │
            ▼
Platform Class Loader
            │
            ▼
Bootstrap Class Loader
```

Each loader has a specific responsibility.

### Bootstrap Class Loader

The Bootstrap Class Loader loads the core Java runtime classes.

Examples include:

- `java.lang.String`
- `java.lang.Object`
- `java.util.List`

Without these classes, no Java application could execute.

This loader is implemented as part of the JVM itself rather than as a normal Java class.

---

### Platform Class Loader

The Platform Class Loader loads platform modules provided by the JDK.

These include APIs used by many Java applications but which are separate from the core runtime classes.

Examples include:

- SQL APIs
- XML APIs
- Security modules

---

### Application Class Loader

The Application Class Loader is responsible for loading your application classes.

This includes:

- Business logic
- Spring Boot classes
- Third-party libraries
- Maven and Gradle dependencies

Whenever you write a new Java class, this is usually the loader responsible for loading it.

---

## Parent Delegation Model

One of the most important design decisions in the JVM is the **Parent Delegation Model**.

Rather than immediately loading every requested class, a class loader first asks its parent.

```
Application Loader
        │
        ▼
Platform Loader
        │
        ▼
Bootstrap Loader
```

If the parent can load the class, the child does nothing.

Only if the parent cannot find the class does the child attempt to load it.

This simple strategy prevents multiple versions of core Java classes from being loaded and improves runtime stability.

> **Key Insight**
>
> A child class loader delegates the loading request to its parent before attempting to load the class itself.

---

## Why Parent Delegation Matters

Imagine an application accidentally includes its own implementation of:

```java
java.lang.String
```

If class loaders loaded classes independently, your application could replace one of Java's most fundamental classes.

That would introduce serious security and stability issues.

Because of parent delegation, the Bootstrap Class Loader always loads core Java classes first.

Your application cannot override them.

This design is one of the reasons the JVM remains secure and predictable.

---

## Class Loading vs Object Creation

These two concepts are often confused.

Loading a class into memory does **not** create an object.

For example:

```java
TradeService service = new TradeService();
```

Before this statement executes:

1. `TradeService.class` is loaded (if it hasn't been loaded already).
2. The class is verified and initialized.
3. Only then is a new object created on the heap.

Class loading typically happens once.

Object creation may happen millions of times.

Understanding this distinction becomes important when discussing memory management later in the book.

## Production Perspective

A microservice starts successfully on a developer's laptop.

The same application immediately fails in production with:

```text
java.lang.ClassNotFoundException:
com.company.payment.PaymentClient
```

The application code hasn't changed.

After investigation, the team discovers that a required dependency was excluded from the production Docker image during packaging.

The JVM searched for the class.

The Application Class Loader couldn't find it.

Application startup failed.

The problem wasn't Java.

It was the application's runtime classpath.

> **Production Perspective**
>
> Many startup failures originate from missing classes, incorrect dependencies, or packaging mistakes rather than defects in business logic.

## Interview Focus

**Question**

What is the purpose of the Class Loader?

**Strong Senior-Level Answer**

> The Class Loader is responsible for locating, loading, linking, and initializing Java classes at runtime. The JVM uses a hierarchy of class loaders—Bootstrap, Platform, and Application—that follow the Parent Delegation Model. This design improves security, avoids duplicate loading of core classes, and ensures consistent runtime behaviour.

## Engineering Takeaways

- Classes are loaded on demand during runtime.
- The JVM uses three primary built-in class loaders.
- Parent Delegation prevents duplicate loading of core Java classes.
- Class loading and object creation are different operations.
- Many application startup issues are caused by missing or incorrectly packaged classes.
