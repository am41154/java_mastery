# 1.3 From Source Code to Running Application

## Why This Matters

Every Java developer has executed one of the following commands countless times:

```bash
mvn spring-boot:run
```

```bash
java -jar application.jar
```

```bash
./gradlew bootRun
```

The application starts.

Logs appear on the console.

A few seconds later, the REST endpoints become available.

From a developer's perspective, the process feels almost instantaneous.

Behind the scenes, however, dozens of important steps occur before the first line of your business logic is executed.

Understanding this execution pipeline helps explain many real-world problems, including:

- Slow application startup
- Class loading failures
- Version incompatibilities
- JVM warm-up behaviour
- Performance differences between the first and subsequent requests

Before diving into JVM internals, it's important to build a clear mental model of this journey.

---

## The Complete Journey

A Java application goes through multiple stages before it reaches the CPU.

```
Java Source (.java)
        │
        ▼
     Compiler
      (javac)
        │
        ▼
Bytecode (.class)
        │
        ▼
 Class Loader
        │
        ▼
Bytecode Verification
        │
        ▼
Execution Engine
(Interpreter + JIT)
        │
        ▼
 Machine Code
        │
        ▼
       CPU
```

Each stage has a specific responsibility.

Understanding these responsibilities makes advanced JVM concepts much easier to grasp later.

---

## Step 1 – Writing the Source Code

Every application begins as a collection of Java source files.

```java
public class TradeService {

    public void processTrade() {
        System.out.println("Trade Processed");
    }

}
```

At this point, the file is simply text.

Humans understand it.

The processor does not.

Before the CPU can execute your program, the source code must be translated into instructions that the machine understands.

---

## Step 2 – Compilation

The Java compiler (`javac`) reads the source code and performs several checks.

Among them are:

- Syntax validation
- Type checking
- Access modifier validation
- Annotation processing
- Generic type verification

If compilation succeeds, the compiler generates one or more `.class` files.

These files contain **bytecode** rather than native machine instructions.

This distinction is one of Java's defining characteristics.

Instead of compiling separately for Windows, Linux, or macOS, Java compiles once into a platform-independent format.

---

## Step 3 – Class Loading

When the application starts, the JVM first needs to locate every class required to run the program.

This responsibility belongs to the **Class Loader**.

It searches for classes in locations such as:

- Application JAR files
- External dependencies
- Java runtime libraries
- Module path (Java 9 and above)

Only after the required classes have been located and loaded can execution continue.

> **Key Insight**
>
> Before any Java code executes, the JVM must first know where every required class is located.

---

## Step 4 – Bytecode Verification

Loading a class into memory is not enough.

The JVM must ensure that the bytecode is valid and safe to execute.

During verification, the JVM checks that the bytecode:

- Follows JVM specifications
- Maintains type safety
- Does not perform invalid memory operations
- Has not been corrupted

This verification process is one of the reasons Java applications are significantly more secure than executing arbitrary native binaries.

---

## Step 5 – Execution

Once verification is complete, execution begins.

Initially, the JVM interprets the bytecode instruction by instruction.

As the application continues running, it identifies methods that are executed frequently.

These methods are known as **hot methods**.

Instead of interpreting them repeatedly, the JVM compiles them into optimized machine code using the **Just-In-Time (JIT) Compiler**.

```
Bytecode
    │
    ▼
Interpreter
    │
Frequently Executed?
    │
   Yes
    ▼
JIT Compiler
    │
    ▼
Optimized Machine Code
```

This process explains why many Java applications become faster after they have been running for some time.

---

## Production Perspective

Consider a Spring Boot application deployed to Kubernetes.

Immediately after deployment, users notice that the first few requests are slower than expected.

The application is healthy.

The database is responsive.

CPU utilisation is low.

The reason is often JVM warm-up.

During startup, the JVM is:

- Loading classes
- Initialising the Spring context
- Interpreting bytecode
- Identifying frequently executed methods

Only after sufficient execution does the JIT compiler optimise the hottest parts of the application.

Understanding this behaviour prevents teams from drawing incorrect conclusions during performance testing.

---

## Interview Focus

**Question**

What happens internally when you execute a Java application?

A concise senior-level answer should describe the complete execution pipeline:

1. Source code is compiled into bytecode.
2. The JVM loads the required classes.
3. Bytecode is verified for safety.
4. The interpreter begins execution.
5. Frequently executed methods are compiled by the JIT compiler.
6. Optimised machine code is executed by the CPU.

Notice that this explanation focuses on the lifecycle rather than individual components.

---

## Engineering Takeaways

- Java source code is never executed directly.
- The compiler generates platform-independent bytecode.
- The Class Loader locates and loads required classes.
- Bytecode verification protects the JVM from invalid code.
- The interpreter enables immediate execution.
- The JIT compiler improves performance over time by generating optimised machine code.

---

## Quick Revision

✓ Java source code becomes bytecode during compilation.

✓ The JVM loads and verifies classes before execution.

✓ Applications begin execution through interpretation.

✓ Frequently executed methods are compiled into native machine code by the JIT compiler.

✓ The CPU ultimately executes machine instructions—not Java source code.
