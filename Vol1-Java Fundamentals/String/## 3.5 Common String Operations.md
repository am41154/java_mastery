## 3.5 Common String Operations and Performance Best Practices

### Why This Matters

Every Java developer uses String APIs daily.

However, writing correct code is only part of the job.

Senior engineers understand:

- Which operations are expensive.
- Which methods create new objects.
- Which APIs are best suited for specific use cases.

Small improvements in String handling can significantly reduce memory allocation in high-throughput applications.

---

## Frequently Used String APIs

Some of the most commonly used methods include:

| Method | Purpose |
|---------|----------|
| `length()` | Returns the number of characters |
| `isEmpty()` | Checks if the string is empty |
| `isBlank()` | Checks if the string contains only whitespace (Java 11+) |
| `contains()` | Checks for a substring |
| `startsWith()` / `endsWith()` | Prefix and suffix matching |
| `substring()` | Extracts part of a string |
| `replace()` | Replaces characters or text |
| `split()` | Splits a string using a regular expression |
| `trim()` / `strip()` | Removes surrounding whitespace |

> **Note**
>
> Every method above returns a **new String** (unless no change is required). The original object remains unchanged.

---

## Prefer `isBlank()` Over Manual Checks

Instead of writing:

```java
if (value == null || value.trim().isEmpty()) {
    ...
}
```

Prefer:

```java
if (value == null || value.isBlank()) {
    ...
}
```

`isBlank()` improves readability and correctly handles all Unicode whitespace characters.

---

## Be Careful with `split()`

Many developers forget that `split()` accepts a **regular expression**, not a plain delimiter.

```java
String text = "A.B.C";

text.split(".");
```

Unexpectedly returns an empty result because `"."` is a regex meaning **any character**.

The correct code is:

```java
text.split("\\.");
```

This is a common interview and production pitfall.

---

## Avoid Unnecessary String Creation

Consider:

```java
String status = new String("SUCCESS");
```

The `new` keyword forces creation of another object.

Simply writing:

```java
String status = "SUCCESS";
```

allows the JVM to reuse the pooled String.

Creating unnecessary String objects increases memory usage without providing any benefit.

---

## Logging Best Practices

Avoid constructing log messages manually.

```java
logger.debug("User: " + userId + " logged in");
```

If debug logging is disabled, the String may still be constructed.

Instead, use parameterized logging.

```java
logger.debug("User: {} logged in", userId);
```

This avoids unnecessary String creation and improves performance.

---

## Production Perspective

A service receives 50,000 requests per minute.

Profiling reveals millions of temporary String objects created from:

- Repeated concatenation
- Unnecessary `split()` calls
- Manual log message construction

After replacing these patterns with `StringBuilder`, parameterized logging, and more efficient parsing, heap usage decreases and garbage collection pauses become noticeably shorter.

Sometimes performance gains come from eliminating unnecessary allocations rather than introducing complex optimizations.

---

## Interview Focus

**Question**

What are some common String-related performance issues you've encountered?

**Strong Senior-Level Answer**

> Common issues include repeated String concatenation inside loops, unnecessary object creation using `new String()`, excessive use of regex-based `split()`, and manual String concatenation in logging statements. These patterns increase object allocation and garbage collection pressure in high-throughput applications.

---

## Common Mistakes

❌ Assuming `split()` uses a plain delimiter.

❌ Using `new String()` without a valid reason.

❌ Building log messages with the `+` operator.

❌ Forgetting that most String methods create new objects.

---

## Engineering Takeaways

- Learn the behavior of frequently used String APIs.
- Prefer readability without sacrificing performance.
- Remember that `split()` uses regular expressions.
- Use parameterized logging to avoid unnecessary String creation.
- Reduce temporary objects in performance-critical code.
