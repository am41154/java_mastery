## 3.4 `StringBuilder` vs `StringBuffer`

### Why This Matters

One of the easiest ways to write inefficient Java code is to repeatedly concatenate Strings.

Consider the following code:

```java
String result = "";

for (int i = 0; i < 1000; i++) {
    result += i;
}
```

The code is correct.

The performance is not.

Every concatenation creates a new `String`, resulting in unnecessary object creation and increased garbage collection.

Understanding when to use `StringBuilder` and `StringBuffer` is essential for writing efficient backend applications.

---

## Why String Concatenation Is Expensive

Since `String` is immutable, every concatenation creates a new object.

```java
String s = "Java";

s = s + " Backend";
```

Conceptually, the JVM performs:

```
"Java"

↓

Create new object

↓

"Java Backend"
```

The original object remains unchanged and eventually becomes eligible for garbage collection.

Inside loops, this process repeats many times, leading to unnecessary allocations.

---

## `StringBuilder`

`StringBuilder` is designed for efficient string manipulation.

```java
StringBuilder builder = new StringBuilder();

builder.append("Java");
builder.append(" ");
builder.append("Backend");

String result = builder.toString();
```

Instead of creating a new object after every operation, `StringBuilder` modifies an internal character buffer.

This significantly reduces object creation and improves performance.

> **Use `StringBuilder` whenever string content is built incrementally within a single thread.**

---

## `StringBuffer`

`StringBuffer` provides the same functionality as `StringBuilder` but is **thread-safe**.

Its methods are synchronized.

```java
StringBuffer buffer = new StringBuffer();

buffer.append("Java");
buffer.append(" Backend");
```

Synchronization makes concurrent access safe but introduces additional overhead.

Unless multiple threads are modifying the same instance, `StringBuilder` is usually the better choice.

---

## Which One Should You Use?

| Scenario | Recommended Choice |
|----------|--------------------|
| Fixed text | `String` |
| Building strings in a single thread | `StringBuilder` |
| Shared mutable string across multiple threads | `StringBuffer` |

For most backend applications, `StringBuilder` is the preferred option.

---

## Does the Compiler Optimize String Concatenation?

Yes—in certain situations.

For example:

```java
String message = firstName + " " + lastName;
```

The Java compiler automatically converts this into code using `StringBuilder`.

Therefore, simple concatenation is perfectly acceptable.

The real problem occurs inside loops.

```java
for (...) {
    result += value;
}
```

Although modern JVMs perform additional optimizations, repeatedly appending using `StringBuilder` remains the recommended approach for clarity and predictable performance.

---

## Production Perspective

A service generates large CSV reports by concatenating thousands of rows using the `+` operator.

Under heavy load, response times increase and garbage collection becomes more frequent.

Replacing repeated concatenation with a single `StringBuilder` reduces object allocation dramatically and improves throughput without changing business logic.

---

## Interview Focus

**Question**

When would you choose `StringBuilder` over `String`?

**Strong Senior-Level Answer**

> Since `String` is immutable, repeated concatenation creates many temporary objects. `StringBuilder` maintains a mutable buffer, making it much more efficient for incremental string construction. `StringBuffer` offers the same functionality but adds synchronization for thread safety.

---

## Common Mistakes

❌ Using `+` repeatedly inside loops.

❌ Choosing `StringBuffer` when thread safety is unnecessary.

❌ Assuming every use of `+` is inefficient.

❌ Forgetting to call `toString()` after building the final result.

---

## Engineering Takeaways

- `String` is immutable.
- `StringBuilder` is mutable and optimized for single-threaded use.
- `StringBuffer` is synchronized and thread-safe.
- Prefer `StringBuilder` for repeated concatenation.
- Simple concatenation outside loops is generally optimized by the compiler.

---
