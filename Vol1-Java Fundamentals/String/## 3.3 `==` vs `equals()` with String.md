## 3.3 `==` vs `equals()` with Strings

### Why This Matters

One of the most common Java interview questions is:

> **"What is the difference between `==` and `equals()` for Strings?"**

Although the concept was introduced in Chapter 1, Strings deserve special attention because the **String Pool** often makes `==` appear to work correctly.

This leads to one of the most common bugs seen in Java code.

---

## `==` Compares References

The `==` operator checks whether two references point to the **same object**.

Consider the following example.

```java
String s1 = "Java";
String s2 = "Java";

System.out.println(s1 == s2);
```

Output:

```
true
```

At first glance, it appears that `==` compares the text.

It doesn't.

Both variables point to the same pooled object.

```
        String Pool

        +--------+
        | "Java" |
        +--------+
          ▲    ▲
          │    │
         s1   s2
```

Since both references are identical, `==` returns `true`.

---

## `equals()` Compares Content

Now consider:

```java
String s1 = new String("Java");
String s2 = new String("Java");

System.out.println(s1.equals(s2));
```

Output:

```
true
```

Although two separate objects exist, their contents are identical.

That is exactly what `equals()` checks.

```
Heap

+--------+      +--------+
| "Java" |      | "Java" |
+--------+      +--------+
     ▲               ▲
     │               │
    s1              s2
```

---

## Why Does This Confuse Developers?

Consider:

```java
String s1 = "Java";
String s2 = new String("Java");
```

Now evaluate both expressions.

```java
s1 == s2
```

returns

```
false
```

while

```java
s1.equals(s2)
```

returns

```
true
```

The contents match.

The objects do not.

This is the key distinction.

---

## Which One Should You Use?

As a rule:

- Use `equals()` when comparing text.
- Use `==` only when intentionally checking object identity.

For example:

```java
if ("SUCCESS".equals(status)) {
    // Process successful response
}
```

This approach is both correct and null-safe.

---

## Production Perspective

A payment service compares transaction statuses using:

```java
if (status == "SUCCESS") {
    ...
}
```

The code works during testing because the values are string literals.

Later, the status is read from a database.

The comparison suddenly fails.

Why?

The database returns a new `String` object whose contents are `"SUCCESS"`.

The references differ, so `==` returns `false`.

Using `equals()` would have avoided the bug.

---

## Interview Focus

**Question**

Why does `==` sometimes work correctly for Strings?

**Strong Senior-Level Answer**

> It works only when both references point to the same object, which often happens with pooled string literals. This behavior is a consequence of the String Pool, not because `==` compares string contents. To compare text, always use `equals()`.

---

## Common Mistakes

❌ Using `==` to compare user input.

❌ Assuming `==` compares characters.

❌ Forgetting that values read from files, databases, or APIs are usually different objects.

❌ Ignoring null safety when calling `equals()`.

---

## Best Practice

Prefer writing:

```java
"SUCCESS".equals(status)
```

instead of:

```java
status.equals("SUCCESS")
```

If `status` is `null`, the first expression safely returns `false`.

The second throws a `NullPointerException`.

---

## Engineering Takeaways

- `==` compares object references.
- `equals()` compares string contents.
- String Pooling can make `==` appear to work.
- Never rely on `==` for business logic involving text.
- Use constant-first comparisons for better null safety.

---

## Quick Revision

✓ `==` checks object identity.

✓ `equals()` checks content.

✓ String literals may share the same pooled object.

✓ Values from external systems are usually different objects.

✓ Prefer `"constant".equals(variable)` to avoid `NullPointerException`.
