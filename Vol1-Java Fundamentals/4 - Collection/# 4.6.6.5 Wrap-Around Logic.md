# 4.6.6.5 Wrap-Around Logic

The previous sections introduced the idea that `ArrayDeque` treats its internal array as a circular buffer.

A natural question now arises:

> **How does the implementation move from the last array index back to index `0`?**

This process is known as **wrap-around**.

Without wrap-around, an array-based queue would quickly run out of space even though unused positions exist at the beginning of the array.

The wrap-around mechanism allows `ArrayDeque` to reuse those positions efficiently without moving existing elements.

---

# The Problem

Suppose our internal array has a capacity of **8**.

Initially:

```text
Index

0 1 2 3 4 5 6 7

+---+---+---+---+---+---+---+---+

| A | B | C | D |   |   |   |   |

+---+---+---+---+---+---+---+---+

head = 0

tail = 4
```

Now remove the first four elements.

```text
removeFirst()

removeFirst()

removeFirst()

removeFirst()
```

Result:

```text
Index

0 1 2 3 4 5 6 7

+---+---+---+---+---+---+---+---+

|   |   |   |   |   |   |   |   |

+---+---+---+---+---+---+---+---+

head = 4

tail = 4
```

The deque is empty again.

Notice that the pointers have moved, even though the capacity has not changed.

---

# Filling the End of the Array

Insert four elements.

```java
addLast(E);

addLast(F);

addLast(G);

addLast(H);
```

Result:

```text
Index

0 1 2 3 4 5 6 7

+---+---+---+---+---+---+---+---+

|   |   |   |   | E | F | G | H |

+---+---+---+---+---+---+---+---+

head = 4

tail = 0
```

Notice what happened.

After inserting into index **7**, the next insertion position became **0**.

The pointer wrapped around.

---

# Why Wrap-Around Is Necessary

Imagine that wrap-around did not exist.

After reaching index 7:

```text
tail = 8
```

But index 8 does not exist.

The implementation would have only two choices.

### Option 1

Shift every element back to index 0.

```text
O(n)
```

---

### Option 2

Allocate a larger array.

```text
Expensive
```

Both approaches waste CPU time.

Instead, the deque simply restarts from index 0.

---

# Circular Movement

Conceptually, the indices behave like a clock.

```text
             0

        7         1

      6             2

        5         3

             4
```

Moving forward from:

```text
7
```

takes us to:

```text
0
```

Moving backward from:

```text
0
```

takes us to:

```text
7
```

The array behaves as though its ends are connected.

---

# Using Modulo Arithmetic

The simplest mathematical way to implement wrap-around is:

```java
next = (current + 1) % capacity;
```

Example:

```text
Capacity = 8
```

Current index:

```text
7
```

Calculation:

```text
(7 + 1) % 8

=

8 % 8

=

0
```

Exactly what we need.

---

Another example:

```text
Current = 5
```

```text
(5 + 1) % 8

=

6
```

Normal movement continues.

---

# Moving Backward

Removing from the rear requires moving backwards.

Conceptually:

```java
previous = (current - 1 + capacity) % capacity;
```

Example:

Current:

```text
0
```

Calculation:

```text
(0 - 1 + 8) % 8

=

7 % 8

=

7
```

The pointer wraps from the beginning to the end.

---

# Why Doesn't OpenJDK Use Modulo?

Modulo is correct.

However, modulo division is relatively expensive compared to simple arithmetic or bitwise operations.

Since queue operations are executed millions of times in high-performance applications, even small optimizations matter.

The OpenJDK implementation therefore uses a faster technique.

---

# Power-of-Two Capacity

One of the key implementation details of `ArrayDeque` is that its internal array capacity is maintained as a power of two.

Examples:

```text
8

16

32

64

128

256
```

Not:

```text
10

25

50

100
```

This design choice enables a very efficient wrap-around calculation.

---

# Bit Masking

When the capacity is a power of two:

```text
Capacity = 8
```

the mask becomes:

```text
capacity - 1

=

7
```

Binary representation:

```text
Capacity

8

=

1000₂
```

Mask:

```text
7

=

0111₂
```

Instead of:

```java
(index + 1) % 8
```

OpenJDK can compute:

```java
(index + 1) & 7
```

The result is identical.

---

# Example

Suppose:

```text
Current Index = 7
```

Binary:

```text
7

=

0111₂
```

Move forward:

```text
7 + 1

=

8

=

1000₂
```

Apply the mask:

```text
1000

AND

0111

--------

0000
```

Result:

```text
0
```

Exactly the desired wrap-around.

---

Another example.

Current:

```text
5

=

0101₂
```

Increment:

```text
6

=

0110₂
```

Mask:

```text
0110

AND

0111

=

0110
```

Result:

```text
6
```

Again, correct.

---

# Why Bit Masking Is Faster

Modern CPUs execute bitwise operations extremely efficiently.

Compared to division required for modulo:

```text
(index + 1) % capacity
```

the operation:

```text
(index + 1) & (capacity - 1)
```

is significantly cheaper.

Although the improvement for a single operation is tiny, it becomes meaningful when queue operations occur millions or billions of times.

# Internal Visualization

Suppose:

```text
Capacity = 8
```

Current state:

```text
Index

0 1 2 3 4 5 6 7

F G . . . . A B

head = 6

tail = 2
```

Logical order:

```text
A

↓

B

↓

F

↓

G
```

Now:

```java
addLast(H);
```

The element is stored at index 2.

The `tail` advances to index 3.

No element movement occurs.

The circular structure naturally reuses previously freed positions.

---

# Benefits of Wrap-Around

- Reuses freed array slots
- Eliminates element shifting
- Maintains O(1) insertion
- Maintains O(1) removal
- Reduces memory copying
- Improves CPU cache utilization

---

# Common Misconceptions

## "Wrap-around copies elements."

Incorrect.

Only the indices change.

Elements remain where they are.

---

## "Modulo is always used."

Conceptually yes.

In practice, OpenJDK uses bit masking because the capacity is maintained as a power of two.

---

## "The array grows whenever the tail reaches the end."

No.

Growth occurs only when the deque becomes full.

Reaching the last index simply causes the pointer to wrap around.

---

# Interview Questions

### Q1. Why does `ArrayDeque` use wrap-around?

Wrap-around allows the deque to reuse free positions at the beginning of the array without shifting elements, enabling amortized O(1) insertion and removal.

---

### Q2. Why are capacities maintained as powers of two?

Power-of-two capacities allow wrap-around calculations to use efficient bit masking instead of modulo division.

---

### Q3. Why is `(index + 1) & (capacity - 1)` equivalent to modulo for power-of-two capacities?

When the capacity is a power of two, the bit mask retains only the lower bits representing valid indices, producing the same result as modulo arithmetic.

---

### Q4. Does reaching the last index mean the deque must resize?

No.

The pointer simply wraps back to index 0.

Resizing occurs only when the deque has no remaining free slots.

---

# Senior Interview Perspective

A common interview discussion is:

> **Interviewer:** "Why does the OpenJDK implementation of `ArrayDeque` use bitwise AND instead of the modulo operator?"

A strong answer is:

> "Modulo arithmetic correctly implements circular indexing, but division is relatively expensive. Since `ArrayDeque` always maintains a power-of-two capacity, OpenJDK can replace `% capacity` with `& (capacity - 1)`, which produces the same result using a much faster bitwise operation. This optimization reduces the cost of every queue operation while preserving correctness."

This demonstrates an understanding of both algorithm design and low-level performance optimization.

---

# Key Takeaways

- Wrap-around allows `ArrayDeque` to reuse freed positions without moving elements.
- Conceptually, wrap-around can be implemented using modulo arithmetic.
- OpenJDK optimizes this further by maintaining power-of-two capacities and using bit masking instead of modulo.
- Bit masking is faster because it avoids division while producing identical results for power-of-two capacities.
- Efficient wrap-around is one of the key reasons `ArrayDeque` delivers excellent real-world performance.
