# 4.6.5 Deque Interface

The `Deque` interface (pronounced **"deck"**) stands for:

> **Double-Ended Queue**

It extends the `Queue` interface by allowing insertion, removal, and examination of elements at **both the front and the rear**.

Unlike a traditional queue, which primarily supports FIFO processing, a deque is significantly more flexible.

A deque can behave as:

- A Queue (FIFO)
- A Stack (LIFO)
- A Double-Ended Queue

Because of this versatility, the `Deque` interface is widely used throughout modern Java applications.

---

# Position in the Collections Framework

The inheritance hierarchy is:

```text
                Iterable

                    │

               Collection

                    │

                 Queue

                    │

                 Deque

              ┌─────┴─────┐

              │           │

        ArrayDeque   LinkedList
```

Notice that:

- Every `Deque` is also a `Queue`.
- Every `Queue` is also a `Collection`.

However, the reverse is not true.

---

# Why Was Deque Introduced?

Before Java 6, developers commonly used:

```java
Stack<Integer> stack = new Stack<>();
```

or

```java
LinkedList<Integer> queue = new LinkedList<>();
```

These approaches had limitations.

### Stack

- Legacy class
- Extends `Vector`
- Unnecessary synchronization
- Poor API design by modern standards

### LinkedList

Although it implements `Deque`, it incurs additional memory overhead due to node-based storage.

Java introduced the `Deque` interface to provide a modern, unified abstraction for both queue and stack operations.

---

# Conceptual Model

Unlike a queue, which has one insertion point and one removal point, a deque allows operations at both ends.

```text
           Front

             ↓

+-----------------------------+

A → B → C → D

+-----------------------------+

             ↑

            Rear
```

Supported operations:

```text
Insert Front

Insert Rear

Remove Front

Remove Rear

Peek Front

Peek Rear
```

Both ends are treated equally.

---

# Method Families

The API is intentionally symmetric.

| Front | Rear |
|--------|------|
| addFirst() | addLast() |
| offerFirst() | offerLast() |
| removeFirst() | removeLast() |
| pollFirst() | pollLast() |
| getFirst() | getLast() |
| peekFirst() | peekLast() |

This symmetry makes the interface intuitive and expressive.

---

# Insertion Methods

## Front

```java
deque.addFirst("A");

deque.offerFirst("A");
```

Both insert an element at the front.

Difference:

- `addFirst()` throws an exception if insertion fails.
- `offerFirst()` returns `false`.

---

## Rear

```java
deque.addLast("D");

deque.offerLast("D");
```

These insert at the rear.

---

Example:

```java
Deque<String> deque = new ArrayDeque<>();

deque.addFirst("B");

deque.addFirst("A");

deque.addLast("C");

deque.addLast("D");
```

Result:

```text
Front

↓

A → B → C → D

                ↑

              Rear
```

---

# Removal Methods

Removing from the front:

```java
deque.removeFirst();

deque.pollFirst();
```

Removing from the rear:

```java
deque.removeLast();

deque.pollLast();
```

Again, each operation comes in two variants:

- Exception-based
- Safe return-value-based

---

# Peek Operations

Front:

```java
deque.peekFirst();
```

Rear:

```java
deque.peekLast();
```

Neither removes the element.

---

# Stack Operations

One of the most valuable features of `Deque` is its ability to replace the legacy `Stack` class.

The interface provides familiar stack methods.

```java
push()

pop()

peek()
```

Example:

```java
Deque<Integer> stack = new ArrayDeque<>();

stack.push(10);

stack.push(20);

stack.push(30);

System.out.println(stack.pop());
```

Output:

```text
30
```

Internally:

```text
Push

↓

30

20

10

↓

Pop
```

Exactly the behavior expected from a stack.

---

# Queue Operations

The same deque can also operate as a FIFO queue.

```java
Deque<Integer> queue = new ArrayDeque<>();

queue.offerLast(10);

queue.offerLast(20);

queue.offerLast(30);

System.out.println(queue.pollFirst());
```

Output:

```text
10
```

Simply choosing different methods changes the behavior.

---

# Why ArrayDeque Implements Deque

The `ArrayDeque` class was specifically designed around the `Deque` interface.

Its internal circular array supports:

- Fast insertion at the front
- Fast insertion at the rear
- Fast removal from the front
- Fast removal from the rear

without shifting elements.

This makes it an excellent implementation of the `Deque` abstraction.

---

# Deque vs Queue

| Feature | Queue | Deque |
|----------|--------|--------|
| Insert Front | ❌ | ✅ |
| Insert Rear | ✅ | ✅ |
| Remove Front | ✅ | ✅ |
| Remove Rear | ❌ | ✅ |
| Stack Operations | ❌ | ✅ |
| Queue Operations | ✅ | ✅ |

A deque is essentially a more capable queue.

---

# Deque vs Stack

| Feature | Stack | Deque |
|----------|--------|--------|
| Legacy API | Yes | No |
| Extends Vector | Yes | No |
| Supports Queue | No | Yes |
| Synchronization | Always | Depends on Implementation |
| Recommended by JDK | No | Yes |

This is why the JDK documentation recommends using `Deque` instead of `Stack` for stack implementations.

# Enterprise Use Cases

## Browser History

A browser maintains navigation history.

```text
Current Page

↓

Previous Pages

↓

Earlier Pages
```

Operations occur at one end, making stack behavior appropriate.

---

## Undo / Redo

Editors often maintain two deques.

```text
Undo Stack

↓

Redo Stack
```

Operations at one end allow efficient reversal and restoration of actions.

---

## Sliding Window Algorithms

Many algorithmic problems maintain a deque of candidate elements.

Examples include:

- Maximum in a sliding window
- Minimum in a sliding window
- Monotonic queue algorithms

These rely heavily on efficient operations at both ends.

---

## Work-Stealing Frameworks

The Java `ForkJoinPool` uses deque-like structures.

Worker threads typically:

- Push tasks to one end.
- Pop tasks from the same end.
- Idle workers steal tasks from the opposite end.

This minimizes contention while balancing workloads.

---

# Common Misconceptions

## "Deque is just another Queue."

Incorrect.

A deque is a generalized queue supporting operations at both ends.

---

## "`Deque` exists only for algorithms."

No.

It is widely used in production systems for stacks, queues, work scheduling, parsing, caching, and concurrency frameworks.

---

## "`Stack` should still be used for stacks."

No.

The JDK documentation recommends `Deque` implementations, particularly `ArrayDeque`, for stack behavior.

# Interview Questions

### Q1. What does Deque stand for?

Deque stands for **Double-Ended Queue**, meaning elements can be inserted and removed from both the front and rear.

---

### Q2. Can a Deque replace both Queue and Stack?

Yes.

By choosing the appropriate methods, a deque can provide FIFO queue behavior or LIFO stack behavior.

---

### Q3. Why does the JDK recommend `ArrayDeque` over `Stack`?

`Stack` is a legacy class extending `Vector`, introducing unnecessary synchronization and outdated APIs. `ArrayDeque` provides a more efficient, modern implementation.

---

### Q4. Why does `Deque` provide both `addFirst()` and `offerFirst()`?

Like the `Queue` interface, it supports both exception-based and return-value-based programming styles, making it suitable for both bounded and unbounded implementations.

---

### Q5. Why is `Deque` important for concurrent algorithms?

Many high-performance scheduling algorithms and work-stealing frameworks require efficient insertion and removal at both ends, making deque semantics ideal.

# Senior Interview Perspective

A common interview discussion is:

> **Interviewer:** "If you need a stack in Java today, which class would you use?"

A strong answer is:

> "I would typically use `ArrayDeque` through the `Deque` interface. It provides efficient stack operations (`push`, `pop`, and `peek`), avoids the legacy synchronization overhead of `Stack`, and follows the JDK's recommended approach for implementing LIFO behavior."

This demonstrates knowledge of both API evolution and performance considerations.

# Key Takeaways

- `Deque` extends `Queue` by supporting operations at both ends.
- A deque can efficiently implement both FIFO and LIFO processing.
- The API is intentionally symmetric, with corresponding methods for the front and rear.
- `ArrayDeque` is the preferred implementation for most stack and queue use cases in single-threaded applications.
- Understanding the `Deque` interface is essential before studying `ArrayDeque`, `LinkedList`, and concurrent deque implementations.
