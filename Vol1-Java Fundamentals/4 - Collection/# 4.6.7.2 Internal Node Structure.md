# 4.6.7.2 Internal Node Structure

The foundation of Java's `LinkedList` is the internal **`Node<E>`** class.

Unlike `ArrayDeque`, which stores elements directly inside an array, every element inserted into a `LinkedList` is wrapped inside its own `Node` object.

Understanding this node structure is essential because almost every operation in `LinkedList`—insertion, removal, traversal, and iteration—is implemented by manipulating node references.

---

# The Internal Node Class

The OpenJDK implementation defines an internal static class similar to the following:

```java
private static class Node<E> {

    E item;

    Node<E> next;

    Node<E> prev;

    Node(Node<E> prev, E element, Node<E> next) {
        this.item = element;
        this.next = next;
        this.prev = prev;
    }
}
```

> **Note:** The code shown above is simplified for educational purposes. The actual OpenJDK implementation may include additional modifiers and implementation details.

---

# Understanding Each Field

Each node contains three fields.

```text
+----------------------------------+

| prev | item | next |

+----------------------------------+
```

---

## item

Stores the actual element.

Example:

```java
Node<String>

↓

item = "Apple"
```

The node itself is **not** the data.

It simply wraps the data.

---

## next

Points to the next node in the list.

```text
A

↓

B
```

If there is no next node:

```text
next = null
```

---

## prev

Points to the previous node.

```text
B

↑

A
```

For the first node:

```text
prev = null
```

---

# Visualizing a Node

Suppose the list contains:

```text
A ⇄ B ⇄ C
```

The middle node looks conceptually like:

```text
                +-------------------------+
Previous Node ← | prev  |  B  |  next | → Next Node
                +-------------------------+
```

Each node knows both of its neighbors.

---

# First Node

The first node has no predecessor.

```text
null ← [A] ⇄ [B] ⇄ [C]
```

Internally:

```text
prev = null

item = A

next = Node(B)
```

---

# Last Node

The last node has no successor.

```text
[A] ⇄ [B] ⇄ [C] → null
```

Internally:

```text
prev = Node(B)

item = C

next = null
```

---

# Middle Node

A middle node contains both references.

```text
[A] ⇄ [B] ⇄ [C]
```

Node **B**:

```text
prev → A

item → B

next → C
```

This allows traversal in both directions.

---

# Why Wrap Every Element in a Node?

Suppose elements were stored directly.

```text
A

↓

B

↓

C
```

Removing **B** would require shifting surrounding elements if they were stored contiguously.

Instead, `LinkedList` simply reconnects neighboring nodes.

Before removal:

```text
A ⇄ B ⇄ C
```

After removal:

```text
A ⇄ C
```

No element copying is required.

Only references change.

---

# Node Creation

Whenever a new element is inserted:

```java
list.add("Java");
```

A new node is allocated.

Conceptually:

```java
new Node<>(previousNode, "Java", nextNode);
```

Unlike `ArrayDeque`, which typically performs only an array assignment, `LinkedList` creates a new object for every insertion.

---

# Memory Layout

Unlike arrays:

```text
[A][B][C][D]
```

nodes are independent heap objects.

Conceptually:

```text
Node A

↓

Heap Address 1024

Node B

↓

Heap Address 84560

Node C

↓

Heap Address 219
```

The JVM may place neighboring nodes anywhere in memory.

---

# Per-Node Memory Overhead

Each node stores more than just the element.

Conceptually, a node contains:

- Object header
- Reference to previous node
- Reference to next node
- Reference to the element

This additional metadata means that a `LinkedList` consumes significantly more memory than an array-based collection storing the same number of references.

The exact memory footprint depends on factors such as:

- JVM implementation
- Object alignment
- Pointer compression (Compressed OOPs)
- Architecture (32-bit vs 64-bit)

---

# One Million Elements

Suppose a list contains:

```text
1,000,000 Elements
```

Internally:

```text
1,000,000 Node Objects
```

Each node is an independent object.

By contrast:

```text
ArrayDeque

↓

One Backing Array
```

This difference has a noticeable impact on memory usage and garbage collection.

---

# Why Is Node Static?

The OpenJDK implementation declares `Node` as a **static nested class**.

This design avoids maintaining an implicit reference to the enclosing `LinkedList` instance.

Benefits include:

- Lower memory usage per node
- Simpler object structure
- Reduced risk of unnecessary object retention

Because a node only needs references to its neighbors and stored element, it has no need to reference the enclosing list.

---

# Common Misconceptions

## "A node stores only the element."

Incorrect.

Every node stores the element **and** references to neighboring nodes.

---

## "Nodes are stored sequentially in memory."

Incorrect.

The JVM is free to allocate node objects anywhere on the heap.

---

## "Every node has both previous and next references."

Not always.

The first node has no previous node, and the last node has no next node.

Those references are `null`.

---

# Interview Questions

### Q1. Why does every insertion into a `LinkedList` allocate a new object?

Because each element is wrapped inside its own `Node` object, which stores the element along with references to the previous and next nodes.

---

### Q2. Why does each node contain both `prev` and `next` references?

This enables efficient bidirectional traversal and constant-time insertions and removals at both ends of the list.

---

### Q3. Why is the internal `Node` class declared `static`?

Making the nested class `static` prevents each node from holding an implicit reference to the enclosing `LinkedList` instance, reducing memory overhead.

---

### Q4. Does a `LinkedList` allocate one array internally?

No.

Each inserted element results in a separate `Node` object allocated on the heap.

---

# Senior Interview Perspective

A common interview discussion is:

> **Interviewer:** "Why is the internal `Node` class declared `static` in OpenJDK's `LinkedList` implementation?"

A strong answer is:

> "The node does not require access to members of the enclosing `LinkedList` instance. Declaring it as a static nested class avoids storing an implicit reference to the outer object in every node, reducing memory overhead and preventing unnecessary object retention."

This demonstrates an understanding of Java nested classes and the memory implications of implementation choices.

---

# Key Takeaways

- Every element in a `LinkedList` is stored inside an independent `Node` object.
- Each node maintains references to both its predecessor and successor.
- Nodes are allocated independently on the heap and are not stored contiguously.
- The `Node` class is declared `static` to minimize per-node memory overhead.
- The flexibility of node-based storage comes at the cost of increased memory usage, object allocation, and garbage collection overhead.
