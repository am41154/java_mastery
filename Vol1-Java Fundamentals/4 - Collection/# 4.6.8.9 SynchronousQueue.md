# 4.6.8.9 SynchronousQueue

`SynchronousQueue` is a **thread-safe blocking queue with zero capacity**.

Unlike every other queue implementation in Java, a `SynchronousQueue` **does not store elements**.

Instead, every insertion operation must wait for a corresponding removal operation, and every removal operation must wait for a corresponding insertion.

Because producers and consumers exchange elements directly without any intermediate storage, `SynchronousQueue` is often called a **handoff queue** or **rendezvous queue**.

# Why Do We Need SynchronousQueue?

Most queues temporarily store work.

For example:

```text
Producer

↓

Queue

↓

Consumer
```

If producers generate work faster than consumers process it:

```text
Queue

↓

Stores Tasks
```

Sometimes this is undesirable.

In certain systems, a producer should continue **only when another thread is immediately ready to process the work**.

That is exactly what `SynchronousQueue` provides.

---

# Zero Capacity

Suppose a producer executes:

```java
queue.put("Task-1");
```

Can the queue store it?

No.

The capacity is:

```text
0
```

The producer waits.

```text
Producer

↓

WAITING
```

When a consumer executes:

```java
queue.take();
```

the element is transferred immediately.

```text
Producer

↓

Task

↓

Consumer
```

At no point is the element stored inside the queue.

---

# Direct Handoff

Conceptually:

```text
Producer

↓

Task

↓

Consumer
```

There is no intermediate buffer.

Compare this with an ordinary queue:

```text
Producer

↓

Queue

↓

Consumer
```

This direct exchange is the defining characteristic of `SynchronousQueue`.

---

# Producer Behavior

Example:

```java
SynchronousQueue<String> queue =
        new SynchronousQueue<>();

queue.put("Hello");
```

If no consumer is waiting:

```text
Producer

↓

WAITING
```

The producer remains blocked until another thread calls:

```java
take();
```

---

# Consumer Behavior

Suppose the consumer executes first.

```java
queue.take();
```

If no producer is waiting:

```text
Consumer

↓

WAITING
```

When a producer later calls:

```java
put("Hello");
```

the value is transferred immediately.

---

# Timeline Example

Consumer starts first.

```text

↓

```

Two seconds later:

```text

↓

```

Immediately:

```text

```

# offer() Behavior

```java
queue.offer(task);
```

Unlike other queues:

```text
Returns false
```

unless a consumer is already waiting.

The element is never buffered.

---

# poll() Behavior

```java
queue.poll();
```

Returns:

```text
null
```

unless a producer is already waiting.

---

# Fairness Policy

`SynchronousQueue` supports two scheduling modes.

Default constructor:

```java
new SynchronousQueue<>();
```

Uses a **non-fair** policy.

Waiting threads are not guaranteed to be served in arrival order.

Fair version:

```java
new SynchronousQueue<>(true);
```

Uses FIFO ordering for waiting producer and consumer threads.

Fairness reduces starvation but may reduce overall throughput.

---

# Internal Implementation

Internally, the JDK uses different algorithms depending on the fairness mode.

### Non-Fair Mode

Uses a **LIFO stack**-based transfer algorithm.

Benefits:

- Higher throughput
- Better cache locality
- Lower contention under heavy workloads

### Fair Mode

Uses a **FIFO queue**-based transfer algorithm.

Benefits:

- Predictable ordering
- Reduced thread starvation

This implementation detail is hidden behind the same public API.

---

# Time Complexity

Ignoring blocking time:

| Operation | Complexity |
|-----------|------------|
| put() | O(1) |
| take() | O(1) |
| offer() | O(1) |
| poll() | O(1) |

The dominant cost is waiting for the matching producer or consumer.

# Typical Use Cases

`SynchronousQueue` is commonly used for:

- Direct thread handoff
- Worker thread coordination
- High-performance thread pools
- Task dispatching
- Pipeline stage synchronization
- Exchange between producer and consumer threads

---

# ThreadPoolExecutor Example

One of the most important uses of `SynchronousQueue` is with cached thread pools.

```java
ExecutorService executor =
        Executors.newCachedThreadPool();
```

Internally, this executor uses:

```text
SynchronousQueue
```

Workflow:

```text
Submit Task

↓

Idle Worker Exists?

↓

Yes → Hand Off Task

↓

No → Create New Worker
```

Since the queue cannot store tasks, the executor creates new worker threads when no idle thread is available (subject to its configuration).

---

# Advantages

- No memory used for buffering
- Immediate producer-consumer synchronization
- Excellent for direct task handoff
- Simple coordination model
- High throughput in handoff-oriented systems

---

# Limitations

- Cannot buffer tasks
- Producers and consumers must rendezvous
- Unsuitable for bursty workloads where tasks should be queued
- Throughput depends on producer-consumer availability

---

# SynchronousQueue vs LinkedBlockingQueue

| Feature | SynchronousQueue | LinkedBlockingQueue |
|---------|------------------|---------------------|
| Capacity | 0 | Configurable |
| Stores Elements | No | Yes |
| Producer Blocking | Always until matched | Only when full |
| Consumer Blocking | Until producer arrives | Only when empty |
| Typical Use | Direct handoff | Buffered work queue |

# Common Misconceptions

## "`SynchronousQueue` is just a queue with size 1."

Incorrect.

A queue of size 1 stores one element.

A `SynchronousQueue` stores **no elements at all**.

---

## "`offer()` always succeeds."

Incorrect.

It succeeds only if another thread is already waiting to receive the element.

---

## "`take()` removes an element from the queue."

Not exactly.

There is no stored element to remove.

The consumer receives the element directly from the producer.

---

# Interview Questions

### Q1. Why is `SynchronousQueue` called a zero-capacity queue?

Because it never stores elements. Every insertion must be matched by a corresponding removal.

---

### Q2. Why is `SynchronousQueue` used by `Executors.newCachedThreadPool()`?

Because cached thread pools prefer direct task handoff. If no idle worker is available to accept the task immediately, the executor creates a new worker thread instead of queuing the task.

---

### Q3. What is the difference between a queue of size one and a `SynchronousQueue`?

A queue of size one can temporarily buffer an element.

A `SynchronousQueue` never buffers any element.

---

### Q4. What is the purpose of the fairness parameter?

It determines whether waiting threads are matched in FIFO order (`true`) or using the default non-fair strategy (`false`), trading throughput for fairness.

---

# Senior Interview Perspective

A common interview discussion is:

> **Interviewer:** "Why would a cached thread pool use `SynchronousQueue` instead of `LinkedBlockingQueue`?"

A strong answer is:

> "A cached thread pool is designed to execute tasks immediately rather than accumulate them. `SynchronousQueue` forces a direct handoff between the submitting thread and a worker. If no idle worker is available, the executor creates a new thread instead of storing the task in a queue. This minimizes queuing latency and allows the pool to grow dynamically under load."

This answer demonstrates an understanding of both the queue's behavior and its integration with Java's executor framework.

---

# Key Takeaways

- `SynchronousQueue` is a thread-safe blocking queue with **zero capacity**.
- It never stores elements; producers and consumers exchange data directly.
- Both producers and consumers block until a matching thread is available.
- It supports optional fairness, allowing FIFO or higher-throughput non-fair scheduling.
- It is widely used for direct thread handoff and forms the foundation of cached thread pools.
