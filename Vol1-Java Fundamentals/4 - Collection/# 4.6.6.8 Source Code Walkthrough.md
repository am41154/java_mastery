# 4.6.6.8 Source Code Walkthrough (OpenJDK)

By this point, we understand the concepts behind `ArrayDeque`:

- Circular array
- Head and tail pointers
- Wrap-around
- Dynamic resizing

Now let's see how these ideas appear in the actual OpenJDK implementation.

> **Note:** The code shown below is intentionally simplified for educational purposes. The real OpenJDK implementation contains additional optimizations, safety checks, and utility methods.

---

# Internal Fields

At its core, an `ArrayDeque` maintains only three important fields.

Conceptually:

```java
transient Object[] elements;

transient int head;

transient int tail;
```

## elements

Stores every element in the deque.

```text
+----+----+----+----+----+----+

| A  | B  | C  | D  |    |    |

+----+----+----+----+----+----+
```

---

## head

Points to the first logical element.

```text
↓

A B C D
```

---

## tail

Points to the next insertion position.

```text
A B C D

        ↑
```

This design avoids additional calculations during insertion.

---

# Simplified addLast()

Conceptually:

```java
public void addLast(E e) {

    elements[tail] = e;

    tail = nextIndex(tail);

    if (tail == head) {
        grow();
    }
}
```

Let's examine each step.

---

## Step 1 — Store the Element

```java
elements[tail] = e;
```

If:

```text
tail = 4
```

then:

```text
elements[4] = e;
```

No shifting.

No traversal.

Just one array assignment.

---

## Step 2 — Advance the Tail

```java
tail = nextIndex(tail);
```

Conceptually:

```java
tail = (tail + 1) % capacity;
```

OpenJDK uses a faster bit-mask operation internally because the capacity is always a power of two.

---

## Step 3 — Detect Full Capacity

If:

```java
tail == head
```

then the insertion has consumed the final available slot.

The deque immediately grows its backing array before returning control to the caller.

This ensures that the ambiguous "full" state is never externally visible.

---

# Simplified addFirst()

Insertion at the front follows the reverse process.

Conceptually:

```java
head = previousIndex(head);

elements[head] = e;

if (head == tail) {
    grow();
}
```

Notice the difference.

For front insertion:

- Move the pointer first.
- Then write the element.

---

# Simplified pollFirst()

Removing from the front:

```java
Object x = elements[head];

elements[head] = null;

head = nextIndex(head);

return x;
```

Three important operations occur.

---

## Read

Retrieve the first logical element.

---

## Clear the Slot

```java
elements[head] = null;
```

This line is surprisingly important.

Many developers overlook it.

---

## Advance the Pointer

The logical beginning of the deque moves forward.

---

# Why Set Removed Elements to null?

Suppose the array contains:

```text
A B C D
```

Remove:

```text
A
```

If the array still contains a reference to `A`:

```text
A B C D
```

the garbage collector considers the object reachable.

It cannot reclaim the memory.

Instead:

```java
elements[head] = null;
```

Result:

```text
. B C D
```

Now the object is no longer referenced by the deque and becomes eligible for garbage collection (assuming no other references exist).

This prevents **memory leaks caused by obsolete references**.

---

# Simplified pollLast()

Conceptually:

```java
tail = previousIndex(tail);

Object x = elements[tail];

elements[tail] = null;

return x;
```

Notice the order.

Unlike `pollFirst()`:

- Move pointer first.
- Read element.
- Clear slot.

This is because `tail` points to the next insertion position rather than the current last element.

---

# Simplified Grow Operation

The resize algorithm is conceptually:

```java
allocate bigger array

↓

copy elements

↓

head = 0

↓

tail = size
```

Internally:

```text
Old

↓

Wrapped

↓

New

↓

Linear
```

The implementation removes any wrap-around during the copy process.

---

# Efficient Copying

Suppose the old array contains:

```text
Index

0 1 2 3 4 5 6 7

F G H . . . A B
```

The logical order is:

```text
A

↓

B

↓

F

↓

G

↓

H
```

The new array becomes:

```text
Index

0 1 2 3 4

A B F G H
```

The implementation performs at most two contiguous copy operations:

1. From `head` to the end of the old array.
2. From index `0` to `tail`.

This is much more efficient than copying one element at a time.

---

# Why the Fields Are transient

The OpenJDK declaration uses:

```java
transient Object[] elements;
```

and similarly for `head` and `tail`.

`ArrayDeque` provides custom serialization logic.

Instead of serializing unused array slots and implementation details, it serializes only the logical sequence of elements.

This produces a smaller and cleaner serialized form.

---

# Why ArrayDeque Rejects null

Unlike `LinkedList`, `ArrayDeque` does not permit `null` elements.

Example:

```java
deque.add(null);
```

throws:

```text
NullPointerException
```

Why?

Because many methods use:

```java
null
```

to indicate:

```text
"No element present."
```

For example:

```java
pollFirst()
```

returns:

```text
null
```

when the deque is empty.

Allowing `null` values would make it impossible to distinguish:

```text
Stored null
```

from

```text
Empty deque
```

---

# Performance Characteristics

A normal insertion performs:

```text
Array Write

↓

Pointer Update

↓

Done
```

Only three or four machine-level operations are required.

This explains why `ArrayDeque` is one of the fastest general-purpose queue implementations in the JDK.

---

# Common Misconceptions

## "`ArrayDeque` never clears removed elements."

Incorrect.

Removed array slots are explicitly assigned `null` to avoid retaining unnecessary object references.

---

## "`tail` points to the last element."

No.

It points to the next insertion position.

This simplifies both insertion and resize logic.

---

## "The resize operation copies the physical array exactly as it appears."

No.

The implementation copies elements in **logical order**, producing a contiguous layout in the new array.

---

# Interview Questions

### Q1. Why does `pollFirst()` assign `null` to the removed slot?

To eliminate obsolete object references so the garbage collector can reclaim memory when appropriate.

---

### Q2. Why does `ArrayDeque` reject `null` elements?

Because methods such as `pollFirst()` use `null` as a sentinel value to indicate that the deque is empty.

---

### Q3. Why are `head`, `tail`, and `elements` declared `transient`?

Because `ArrayDeque` performs custom serialization, storing only the logical sequence of elements rather than its internal representation.

---

### Q4. How does the resize algorithm preserve logical order?

It copies elements starting from `head`, wraps around if necessary, and places the first logical element at index `0` of the new array.

---

# Senior Interview Perspective

A common interview discussion is:

> **Interviewer:** "Why does `ArrayDeque` explicitly assign `null` to removed array slots? Doesn't advancing `head` or `tail` already remove the element logically?"

A strong answer is:

> "Advancing the pointers removes the element logically, but the backing array would still hold a strong reference to the removed object. By assigning `null` to the vacated slot, `ArrayDeque` eliminates obsolete references, allowing the garbage collector to reclaim the object if no other references exist. This avoids memory leaks caused by lingering references."

This demonstrates an understanding of both data structure implementation and JVM garbage collection.

---

# Key Takeaways

- The OpenJDK implementation of `ArrayDeque` is remarkably small, relying primarily on an array and two indices.
- Most operations consist of an array access and a pointer update, making them extremely efficient.
- Removed elements are explicitly set to `null` to avoid retaining obsolete references.
- Resizing copies elements in logical order, removing any wrap-around in the new array.
- Understanding the implementation decisions behind `ArrayDeque` provides valuable insight into both collection design and JVM memory management.
