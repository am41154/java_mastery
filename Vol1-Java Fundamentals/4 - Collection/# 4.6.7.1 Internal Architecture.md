# 4.6.7.1 Internal Architecture

Unlike `ArrayDeque`, which stores elements inside a contiguous array, `LinkedList` stores each element in a separate object called a **node**.

Each node contains:

- The actual data
- A reference to the previous node
- A reference to the next node

These nodes are connected together to form a **doubly linked list**.

This design eliminates the need for resizing but introduces additional memory overhead and pointer management.

---

# High-Level Structure

Conceptually, a `LinkedList` looks like this:

```text
+--------+     +--------+     +--------+

| Node A | --> | Node B | --> | Node C |

+--------+     +--------+     +--------+
```

Unlike an array:

- Elements are **not contiguous** in memory.
- Each node may reside anywhere on the JVM heap.
- Nodes are connected only through references.

---

# Doubly Linked List

Java's `LinkedList` is **doubly linked**, meaning every node maintains two references.

```text
           next
+------+ ---------> +------+ ---------> +------+
| Node |            | Node |            | Node |
+------+ <--------- +------+ <--------- +------+
         previous          previous
```

Every node knows:

- Its previous neighbor
- Its next neighbor

This enables efficient traversal in both directions.

---

# Internal Fields

Conceptually, the `LinkedList` class maintains three important fields.

```java
transient Node<E> first;

transient Node<E> last;

transient int size;
```

---

## first

Points to the first node.

```text
↓

[A] ⇄ [B] ⇄ [C]
```

Every traversal from the front begins here.

---

## last

Points to the final node.

```text
[A] ⇄ [B] ⇄ [C]

             ↑
```

This allows O(1) insertion at the end.

---

## size

Tracks the number of elements.

Unlike `ArrayDeque`, which derives its logical size from pointer positions, `LinkedList` maintains an explicit counter.

```text
size = 3
```

This makes the `size()` method a constant-time operation.

---

# Empty LinkedList

Initially:

```text
first = null

last = null

size = 0
```

Visual representation:

```text
first

↓

null


last

↓

null
```

No nodes exist.

---

# After Adding One Element

Suppose:

```java
list.add("A");
```

The structure becomes:

```text
first

↓

+----------------------+

| prev | A | next |

+----------------------+

↑

last
```

Notice:

```text
first == last
```

because only one node exists.

---

# After Adding Multiple Elements

Suppose:

```java
list.add("A");

list.add("B");

list.add("C");
```

The internal structure becomes:

```text
first

↓

+-----+     +-----+     +-----+

|  A  | ⇄ |  B  | ⇄ |  C  |

+-----+     +-----+     +-----+

                             ↑

                            last
```

Each node is linked to its neighbors in both directions.

---

# Why Use a Doubly Linked List?

Imagine removing the last element.

With a singly linked list:

```text
A → B → C
```

To find the previous node:

```text
Traverse

A

↓

B

↓

C
```

Complexity:

```text
O(n)
```

---

With a doubly linked list:

```text
A ⇄ B ⇄ C
```

The last node already knows its predecessor.

```text
C.previous

↓

B
```

Complexity:

```text
O(1)
```

This is one of the primary reasons Java chose a doubly linked implementation.

---

# Node Independence

Unlike arrays, nodes are independent objects.

Conceptually:

```text
Node A

↓

Heap Address 1056

Node B

↓

Heap Address 90412

Node C

↓

Heap Address 187
```

The JVM makes no guarantee that neighboring nodes occupy neighboring memory locations.

This has important performance implications, which we will explore later.

---

# Comparison with ArrayDeque

ArrayDeque:

```text
+---+---+---+---+

| A | B | C | D |

+---+---+---+---+
```

One array.

Contiguous memory.

---

LinkedList:

```text
[A] ⇄ [B] ⇄ [C] ⇄ [D]
```

Multiple node objects.

References connect the nodes.

---

# Advantages of This Architecture

- Constant-time insertion at both ends
- Constant-time removal at both ends
- No array resizing
- Efficient bidirectional traversal
- Supports both `List` and `Deque` interfaces

---

# Disadvantages

- Higher memory consumption
- One object allocation per element
- Poor cache locality
- Additional pointer updates during modifications
- Increased garbage collection pressure

---

# Common Misconceptions

## "`LinkedList` stores elements in an array."

Incorrect.

Every element resides in its own `Node` object.

---

## "`first` and `last` store the actual data."

No.

They store references to the first and last nodes.

The element is stored inside each node.

---

## "`LinkedList` is singly linked."

Incorrect.

The JDK implementation uses a **doubly linked list**.

Every node contains references to both its predecessor and successor.

---

# Interview Questions

### Q1. Why does Java use a doubly linked list instead of a singly linked list?

A doubly linked list enables constant-time insertion and removal at both ends and efficient backward traversal without searching for predecessor nodes.

---

### Q2. Why does `LinkedList` maintain both `first` and `last` references?

These references allow constant-time access to both ends of the list, making operations such as `addFirst()`, `addLast()`, `removeFirst()`, and `removeLast()` efficient.

---

### Q3. Why does `LinkedList` maintain an explicit `size` field while `ArrayDeque` does not?

`LinkedList` can increment or decrement the size during every insertion or removal with negligible cost, making `size()` O(1). `ArrayDeque` instead derives its logical size from the positions of `head` and `tail`.

---

### Q4. Does Java allocate one array for a `LinkedList`?

No.

Each inserted element creates a separate node object, which may reside anywhere in heap memory.

---

# Senior Interview Perspective

A common interview discussion is:

> **Interviewer:** "Why is Java's `LinkedList` implemented as a doubly linked list rather than a singly linked list?"

A strong answer is:

> "A doubly linked list allows constant-time insertions and removals at both the front and the rear because each node maintains references to both its predecessor and successor. It also enables efficient backward traversal and simplifies operations such as `removeLast()` without requiring a full traversal to locate the previous node. The trade-off is additional memory overhead for the extra reference stored in every node."

This answer demonstrates an understanding of the design trade-offs behind the implementation.

---

# Key Takeaways

- `LinkedList` is implemented as a **doubly linked list**.
- Each element is stored in an independent `Node` object.
- The list maintains references to both the first and last nodes.
- An explicit `size` field enables O(1) `size()` operations.
- The architecture favors insertion and removal flexibility over memory efficiency and cache locality.
