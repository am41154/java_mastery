# 4.6.8.6 ArrayBlockingQueue

`ArrayBlockingQueue` is a **thread-safe, bounded FIFO blocking queue** backed by a **fixed-size circular array**.

Unlike `LinkedBlockingQueue`, which allocates a new node for each inserted element, `ArrayBlockingQueue` stores elements directly inside a preallocated array.

Because its capacity is fixed, it never grows dynamically. This predictable memory usage makes it a popular choice for systems that require strict resource control and built-in backpressure.

# Internal Architecture

Internally, `ArrayBlockingQueue` uses a **circular array**.

Conceptually:

```text
+---+---+---+---+---+
| A | B | C |   |   |
+---+---+---+---+---+
```

Instead of moving elements after every removal, it maintains two indices:

- `putIndex`
- `takeIndex`

These indices wrap around the array when they reach the end.

---

# Circular Buffer

Suppose the capacity is:

```text
5
```

Initially:

```text
+---+---+---+---+---+
|   |   |   |   |   |
+---+---+---+---+---+
 ^
putIndex
 ^
takeIndex
```

After inserting three elements:

```text
+---+---+---+---+---+
| A | B | C |   |   |
+---+---+---+---+---+
             ^
         putIndex

 ^
takeIndex
```

After removing two elements:

```text
+---+---+---+---+---+
|   |   | C |   |   |
+---+---+---+---+---+
         ^
     takeIndex

             ^
         putIndex
```

The freed slots are reused instead of shifting existing elements.

---

# Fixed Capacity

Unlike `LinkedBlockingQueue`, capacity must be specified during construction.

Example:

```java
BlockingQueue<String> queue =
        new ArrayBlockingQueue<>(100);
```

The queue can never hold more than:

```text
100 Elements
```

Attempting to exceed the capacity results in blocking or failure, depending on the method used.

# Producer Operations

## put()

```java
queue.put(task);
```

If the queue has available space:

```text
Insert Element
```

If the queue is full:

```text
Producer

↓

WAITING
```

The producer blocks until a consumer removes an element.

---

## offer()

```java
queue.offer(task);
```

If the queue is full:

```text
false
```

is returned immediately.

---

## offer(timeout)

```java
queue.offer(task,
            5,
            TimeUnit.SECONDS);
```

Behavior:

```text
Wait up to 5 Seconds

↓

Space Available?

↓

Yes → Insert

No → Return false
```

---

# Consumer Operations

## take()

```java
String task = queue.take();
```

If the queue is empty:

```text
Consumer

↓

WAITING
```

The thread remains blocked until a producer inserts an element.

---

## poll()

```java
String task = queue.poll();
```

Returns:

```text
null
```

immediately if the queue is empty.

---

## poll(timeout)

```java
String task =
        queue.poll(5,
                   TimeUnit.SECONDS);
```

Behavior:

```text
Wait up to 5 Seconds

↓

Element Available?

↓

Yes → Return Element

No → Return null
```

# Locking Strategy

Unlike `LinkedBlockingQueue`, `ArrayBlockingQueue` uses **a single `ReentrantLock`** to protect the entire queue.

Conceptually:

```text
Producer

↓

Shared Lock

↓

Circular Array

↓

Shared Lock

↓

Consumer
```

Both producers and consumers compete for the same lock.

This simplifies the implementation but can increase contention under heavy concurrent workloads.

---

# Fairness Policy

One unique feature of `ArrayBlockingQueue` is its optional fairness policy.

Example:

```java
BlockingQueue<String> queue =
        new ArrayBlockingQueue<>(100, true);
```

The second parameter enables **fair scheduling**.

With fairness enabled:

```text
Thread A Waiting

↓

Thread B Waiting

↓

Thread C Waiting
```

Threads acquire the lock in approximately first-come, first-served order.

Without fairness (the default):

```text
Any Waiting Thread
```

may acquire the lock first, which generally improves throughput.

# Memory Usage

Because the backing array is allocated once:

```text
Capacity = 1000

↓

One Array
```

No additional node objects are created for each element.

Compared to `LinkedBlockingQueue`:

- Lower memory overhead
- Better cache locality
- Reduced garbage collection pressure

# Advantages

- Fixed memory usage
- Excellent cache locality
- No per-element object allocation
- Built-in backpressure
- Constant-time operations
- Predictable performance

---

# Limitations

- Capacity cannot be changed after creation.
- Uses a single lock for all operations.
- Producers and consumers contend for the same lock.
- Large capacities allocate large arrays upfront.

---

# Typical Use Cases

`ArrayBlockingQueue` is commonly used for:

- Producer-consumer pipelines
- Bounded work queues
- Real-time systems
- Rate-limited processing
- Systems with strict memory limits
- High-performance applications requiring predictable resource usage

---

# ArrayBlockingQueue vs LinkedBlockingQueue

| Feature | ArrayBlockingQueue | LinkedBlockingQueue |
|---------|--------------------|---------------------|
| Internal Structure | Circular Array | Linked Nodes |
| Capacity | Fixed | Optional |
| Memory Usage | Lower | Higher |
| Dynamic Growth | No | Yes (up to configured capacity) |
| Locking | Single Lock | Separate Put/Take Locks |
| Cache Locality | Excellent | Moderate |
| Object Allocation | None per Element | One Node per Element |
| Producer/Consumer Contention | Higher | Lower |

There is no universally superior implementation.

The best choice depends on the application's workload and resource requirements.

---

# Common Misconceptions

## "`ArrayBlockingQueue` automatically expands."

Incorrect.

Its capacity is fixed for the lifetime of the queue.

---

## "`ArrayBlockingQueue` is always faster."

Not necessarily.

Its contiguous memory layout improves cache locality, but the single shared lock may become a bottleneck under high producer-consumer concurrency.

---

## "`Fairness` improves performance."

Usually the opposite.

Fairness reduces thread starvation but generally decreases overall throughput because of stricter scheduling.

# Interview Questions

### Q1. Why is `ArrayBlockingQueue` considered bounded?

Because its maximum capacity is fixed when the queue is created and cannot be changed later.

---

### Q2. Why does `ArrayBlockingQueue` generally use less memory than `LinkedBlockingQueue`?

Because it stores elements in a preallocated array rather than allocating a separate node object for each element.

---

### Q3. What is the purpose of the fairness parameter?

It controls how waiting threads acquire the lock. When enabled, threads are served approximately in FIFO order, reducing starvation at the cost of lower throughput.

---

### Q4. When would you choose `ArrayBlockingQueue` over `LinkedBlockingQueue`?

When predictable memory usage, bounded capacity, and lower allocation overhead are more important than maximizing producer-consumer concurrency.

# Senior Interview Perspective

A common interview discussion is:

> **Interviewer:** "How would you choose between `ArrayBlockingQueue` and `LinkedBlockingQueue`?"

A strong answer is:

> "If I need predictable memory usage and a fixed-capacity queue, I would choose `ArrayBlockingQueue`. It allocates a single backing array, provides excellent cache locality, and avoids per-element object allocation. If the workload has many concurrent producers and consumers, `LinkedBlockingQueue` often provides higher throughput because it uses separate locks for insertion and removal. The choice depends on whether resource predictability or concurrency is the primary concern."

This answer demonstrates an understanding of memory layout, locking strategies, and real-world performance trade-offs.

---

# Key Takeaways

- `ArrayBlockingQueue` is a thread-safe, bounded FIFO blocking queue backed by a circular array.
- It uses a fixed-size array and never resizes dynamically.
- A single `ReentrantLock` protects both producer and consumer operations.
- It provides predictable memory usage, excellent cache locality, and low allocation overhead.
- It is an excellent choice for bounded producer-consumer systems where resource usage must remain tightly controlled.
