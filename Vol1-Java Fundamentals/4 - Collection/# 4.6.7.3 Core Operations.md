# 4.6.7.3 Core Operations (linkFirst(), linkLast(), unlink())

Every insertion and removal operation in `LinkedList` ultimately manipulates **node references**.

Unlike array-based collections, no elements are shifted during insertion or deletion.

Instead, the implementation updates a small number of references to connect or disconnect nodes.

OpenJDK organizes these operations into internal helper methods such as:

- `linkFirst()`
- `linkLast()`
- `linkBefore()`
- `unlinkFirst()`
- `unlinkLast()`
- `unlink()`

Although these methods are private implementation details, understanding them makes the behavior of `LinkedList` much easier to understand.

---

# Adding the First Element

Suppose the list is empty.

```text
first = null

last = null
```

Now execute:

```java
list.addFirst("A");
```

A new node is created.

```text
+------+
|  A   |
+------+
```

Since the list was empty:

```text
first

↓

[A]

↑

last
```

Both `first` and `last` point to the same node.

---

# Simplified linkFirst()

Conceptually:

```java
Node<E> newNode = new Node<>(null, e, first);

first = newNode;

if (last == null) {
    last = newNode;
} else {
    oldFirst.prev = newNode;
}

size++;
```

The algorithm performs four steps:

1. Create a new node.
2. Make it the first node.
3. Update the previous first node (if one exists).
4. Increment the size.

No existing elements are copied.

---

# Adding at the End

Suppose the list contains:

```text
[A] ⇄ [B]
```

Now execute:

```java
list.addLast("C");
```

A new node is created.

```text
[A] ⇄ [B] ⇄ [C]
```

Only a few references change.

---

# Simplified linkLast()

Conceptually:

```java
Node<E> newNode = new Node<>(last, e, null);

last = newNode;

if (first == null) {
    first = newNode;
} else {
    oldLast.next = newNode;
}

size++;
```

Again, no traversal is required because the list already maintains a reference to the last node.

Complexity:

```text
O(1)
```

---

# Inserting in the Middle

Suppose the list contains:

```text
A ⇄ C
```

We insert **B** before **C**.

Before:

```text
A ⇄ C
```

After:

```text
A ⇄ B ⇄ C
```

The implementation adjusts four references.

```text
A.next = B

B.prev = A

B.next = C

C.prev = B
```

No other nodes are affected.

---

# Simplified linkBefore()

Conceptually:

```java
Node<E> predecessor = successor.prev;

Node<E> newNode =
    new Node<>(predecessor, e, successor);

predecessor.next = newNode;

successor.prev = newNode;

size++;
```

This operation inserts a node without moving any existing elements.

---

# Removing the First Node

Suppose the list contains:

```text
[A] ⇄ [B] ⇄ [C]
```

After:

```java
removeFirst();
```

The result becomes:

```text
[B] ⇄ [C]
```

The first node is detached.

The new first node updates:

```text
prev = null
```

The removed node is no longer reachable from the list.

---

# Simplified unlinkFirst()

Conceptually:

```java
Node<E> firstNode = first;

Node<E> next = firstNode.next;

first.item = null;

first.next = null;

first = next;

if (next == null)
    last = null;
else
    next.prev = null;

size--;
```

Notice that references inside the removed node are cleared before it becomes unreachable.

This helps the garbage collector reclaim memory.

---

# Removing the Last Node

Before:

```text
[A] ⇄ [B] ⇄ [C]
```

After:

```java
removeLast();
```

Result:

```text
[A] ⇄ [B]
```

The last node is disconnected.

The new last node updates:

```text
next = null
```

---

# Simplified unlinkLast()

Conceptually:

```java
Node<E> lastNode = last;

Node<E> previous = lastNode.prev;

last.item = null;

last.prev = null;

last = previous;

if (previous == null)
    first = null;
else
    previous.next = null;

size--;
```

Again, only a few references are modified.

---

# Removing a Middle Node

Suppose:

```text
A ⇄ B ⇄ C
```

Remove:

```text
B
```

Before:

```text
A ⇄ B ⇄ C
```

After:

```text
A ⇄ C
```

The implementation reconnects the neighboring nodes.

```text
A.next = C

C.prev = A
```

Then it clears the removed node's references.

```text
B.prev = null

B.next = null

B.item = null
```

The removed node is now eligible for garbage collection if no other references exist.

---

# Why Clear References?

Consider:

```text
Removed Node

↓

Still references neighbors
```

Those references unnecessarily keep parts of the object graph connected.

Instead, OpenJDK clears them.

```java
node.item = null;

node.prev = null;

node.next = null;
```

Benefits:

- Faster garbage collection
- Eliminates obsolete references
- Reduces risk of accidental memory retention

---

# Reference Updates vs Element Shifting

Array-based collections often require shifting elements.

```text
A B C D

↓

Remove B

↓

A C D
```

Several elements may move.

---

`LinkedList` simply reconnects nodes.

```text
A ⇄ B ⇄ C

↓

A ⇄ C
```

Only references change.

This is the key advantage of linked structures.

---

# Complexity Summary

| Operation | Complexity |
|-----------|------------|
| linkFirst() | O(1) |
| linkLast() | O(1) |
| unlinkFirst() | O(1) |
| unlinkLast() | O(1) |
| unlink(Node) | O(1) *(once the node is known)* |

> **Important:** Finding the node by index is a separate operation and may require O(n) traversal.

---

# Common Misconceptions

## "`LinkedList` moves elements during insertion."

Incorrect.

Nodes remain in place.

Only references are updated.

---

## "Removing a node automatically frees memory."

Not immediately.

The node becomes **eligible** for garbage collection after it is no longer reachable and all obsolete references have been cleared.

---

## "`remove(index)` is always O(1)."

Incorrect.

Finding the node may require traversing the list.

Only the unlinking itself is O(1).

---

# Interview Questions

### Q1. Why doesn't `LinkedList` shift elements during insertion?

Because each element resides in an independent node. Insertions and removals modify node references rather than moving existing elements.

---

### Q2. Why are removed node references explicitly cleared?

To eliminate obsolete references, allowing the garbage collector to reclaim the removed node and any objects that are no longer reachable.

---

### Q3. Is `unlink(Node)` always O(1)?

Yes, provided the target node is already known. Locating the node by index may still require O(n) traversal.

---

### Q4. Which operation is more expensive: locating a node or unlinking it?

Locating the node is usually more expensive because it may require traversing the list. Once the node has been found, unlinking requires only a few reference updates.

---

# Senior Interview Perspective

A common interview discussion is:

> **Interviewer:** "People often say `LinkedList` provides O(1) deletion. Is that always true?"

A strong answer is:

> "The deletion itself is O(1) because it only updates neighboring node references. However, if the node must first be located using an index or a search, the traversal cost is O(n). Therefore, operations such as `remove(index)` are generally O(n), while `unlink(Node)` is O(1) once the node is known."

This distinction demonstrates a deeper understanding of algorithm analysis than simply quoting Big-O values.

---

# Key Takeaways

- `LinkedList` modifications work by reconnecting nodes rather than shifting elements.
- Internal helper methods such as `linkFirst()`, `linkLast()`, and `unlink()` form the basis of almost every insertion and removal operation.
- Only a small number of references change during each modification.
- Removed nodes have their references cleared to improve garbage collection.
- The cost of locating a node is often greater than the cost of inserting or removing it.
