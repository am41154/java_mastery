# 4.6.6.10 ArrayDeque vs LinkedList vs Stack

Java provides multiple classes that can be used as a **queue**, **deque**, or **stack**.

The three most commonly encountered implementations are:

- `ArrayDeque`
- `LinkedList`
- `Stack`

Although they expose similar operations, their internal implementations, memory usage, performance characteristics, and recommended use cases differ significantly.

Choosing the right implementation requires understanding these trade-offs rather than simply memorizing APIs.

---

# Quick Comparison

| Feature | ArrayDeque | LinkedList | Stack |
|---------|------------|------------|-------|
| Underlying Structure | Circular Array | Doubly Linked List | Dynamic Array (`Vector`) |
| Implements `Deque` | ✓ | ✓ | ✗ |
| Implements `Queue` | ✓ | ✓ | ✗ |
| Stack Operations | ✓ | ✓ | ✓ |
| Queue Operations | ✓ | ✓ | ✗ |
| Allows `null` | ✗ | ✓ | ✓ |
| Thread Safe | ✗ | ✗ | ✓ (synchronized) |
| Random Access | ✗ | ✗ | ✓ (`Vector`) |
| Memory Overhead | Low | High | Low |
| Cache Locality | Excellent | Poor | Excellent |
| Resize Required | Yes | No | Yes |
| JDK Recommendation | **Preferred** | Situational | Legacy |

---

# Internal Architecture

## ArrayDeque

```text
+----+----+----+----+----+

| A  | B  | C  | D  | E  |

+----+----+----+----+----+
```

Uses a contiguous circular array.

Advantages:

- Excellent cache locality
- Very few object allocations
- High throughput
- Low memory overhead

Disadvantages:

- Occasional resize
- No random access API

---

## LinkedList

```text
+------+     +------+     +------+

| A | •| --> | B | •| --> | C | •|

+------+     +------+     +------+
```

Each element is stored in its own node.

Advantages:

- No resizing
- Constant-time insertion/removal at ends
- Allows `null`
- Can act as both a `List` and a `Deque`

Disadvantages:

- High memory overhead
- Poor cache locality
- Frequent object allocation
- Higher GC pressure

---

## Stack

Internally:

```text
Vector

↓

Object[]
```

It extends the legacy `Vector` class.

Every operation is synchronized.

Advantages:

- Thread-safe

Disadvantages:

- Legacy API
- Synchronization overhead
- Not recommended for new code

---

# Memory Usage

Suppose we store one million integers.

### ArrayDeque

```text
One Object[]

↓

1,000,000 References
```

Memory overhead is relatively small.

---

### LinkedList

Each element requires:

```text
Node Object

+

Previous Pointer

+

Next Pointer

+

Object Header
```

Memory consumption is significantly higher.

---

### Stack

Similar to `ArrayList`.

Requires only the backing array.

However, synchronization adds runtime overhead.

---

# Performance Comparison

| Operation | ArrayDeque | LinkedList | Stack |
|-----------|------------|------------|-------|
| Push | O(1) amortized | O(1) | O(1) |
| Pop | O(1) | O(1) | O(1) |
| Enqueue | O(1) amortized | O(1) | Not Supported |
| Dequeue | O(1) | O(1) | Not Supported |
| Peek | O(1) | O(1) | O(1) |
| Iteration | Excellent | Moderate | Excellent |
| Resize | O(n) | Not Required | O(n) |

Although the asymptotic complexities are similar, `ArrayDeque` usually achieves the best real-world performance.

---

# Cache Locality

### ArrayDeque

```text
A B C D E F G
```

Elements are adjacent in memory.

Excellent cache utilization.

---

### LinkedList

```text
A

↓

Memory 8123

↓

B

↓

Memory 421

↓

C
```

Nodes may be scattered across the heap.

The CPU spends more time waiting for memory.

---

# Garbage Collection

### ArrayDeque

Few object allocations.

Occasional array allocation during resize.

---

### LinkedList

Every insertion creates a new node.

Every removal eventually creates additional GC work.

---

### Stack

Comparable to `ArrayList`, but synchronization adds CPU overhead.

---

# API Comparison

## Using ArrayDeque as a Stack

```java
Deque<String> stack = new ArrayDeque<>();

stack.push("A");

stack.push("B");

stack.pop();
```

---

## Using ArrayDeque as a Queue

```java
Queue<String> queue = new ArrayDeque<>();

queue.offer("A");

queue.offer("B");

queue.poll();
```

---

## Using LinkedList

```java
Deque<String> deque = new LinkedList<>();

deque.addFirst("A");

deque.addLast("B");
```

---

## Using Stack

```java
Stack<String> stack = new Stack<>();

stack.push("A");

stack.pop();
```

Although this works, modern Java code generally prefers `Deque`.

---

# JDK Recommendation

The official Java documentation recommends:

> **Use `Deque` implementations (typically `ArrayDeque`) instead of the legacy `Stack` class for stack behavior.**

Similarly, for most queue implementations:

> **`ArrayDeque` is preferred over `LinkedList` unless list-specific features are required.**

These recommendations are based on performance, memory efficiency, and API design.

---

# When Should You Use Each?

## Use ArrayDeque When

- Implementing queues
- Implementing stacks
- BFS
- DFS
- Sliding window algorithms
- Expression evaluation
- Task scheduling
- High-throughput systems
- General-purpose FIFO/LIFO operations

This is the default choice in most applications.

---

## Use LinkedList When

- Frequent insertion/removal using a `ListIterator`
- You specifically need both `List` and `Deque` interfaces
- Stable node positions are important (rare in typical Java applications)

Even in these cases, evaluate whether another collection better matches the requirements.

---

## Use Stack When

Almost never in new code.

Only:

- Maintaining legacy applications
- Working with APIs that explicitly require `Stack`

Otherwise, prefer:

```java
Deque<E> stack = new ArrayDeque<>();
```

---

# Common Misconceptions

## "`LinkedList` is always faster because it never resizes."

Incorrect.

The cost of pointer chasing and object allocation generally outweighs occasional resizing.

---

## "`Stack` is recommended because it is thread-safe."

Not necessarily.

Synchronization introduces overhead.

If thread safety is required, choose a collection designed for concurrent use, such as `ConcurrentLinkedDeque` or `LinkedBlockingDeque`, depending on your requirements.

---

## "`ArrayDeque` cannot be used as a stack."

Incorrect.

The `Deque` interface explicitly provides stack operations such as `push()`, `pop()`, and `peek()`.

---

# Interview Questions

### Q1. Why does the JDK recommend `ArrayDeque` instead of `Stack`?

Because `Stack` is a legacy class extending `Vector`, which synchronizes every operation. `ArrayDeque` provides a cleaner API and better performance for single-threaded use.

---

### Q2. Why is `ArrayDeque` usually faster than `LinkedList`?

Because contiguous storage improves cache locality, reduces object allocations, and minimizes pointer chasing.

---

### Q3. When would you choose `LinkedList` over `ArrayDeque`?

Only when list-specific behavior or efficient insertions/removals through a `ListIterator` are essential. For pure queue or stack semantics, `ArrayDeque` is generally the better choice.

---

### Q4. What should you use instead of `Stack` in modern Java?

Use the `Deque` interface with an `ArrayDeque` implementation:

```java
Deque<E> stack = new ArrayDeque<>();
```

---

# Senior Interview Perspective

A common interview discussion is:

> **Interviewer:** "For a high-throughput order processing system, would you choose `ArrayDeque`, `LinkedList`, or `Stack` to implement an in-memory queue?"

A strong answer is:

> "I would choose `ArrayDeque`. It provides amortized O(1) queue operations, excellent cache locality, minimal object allocation, and lower garbage collection pressure. Although `LinkedList` also provides O(1) insertions and removals at the ends, its per-node allocation and poor cache behavior generally make it slower in practice. I would avoid `Stack` because it is a legacy synchronized class intended primarily for backward compatibility."

This answer demonstrates an understanding of both algorithmic complexity and JVM runtime behavior.

# Decision Matrix

| Requirement | Recommended Choice |
|------------|--------------------|
| Fast queue | ArrayDeque |
| Fast stack | ArrayDeque |
| Lowest memory overhead | ArrayDeque |
| Best cache locality | ArrayDeque |
| Legacy stack API | Stack |
| Thread-safe deque | Concurrent collections (`ConcurrentLinkedDeque`, `LinkedBlockingDeque`, etc.) |
| Need `List` + `Deque` in one class | LinkedList |

---

# Key Takeaways

- `ArrayDeque` is the preferred general-purpose implementation for both stacks and queues.
- `LinkedList` remains useful when list-specific behavior is required but is rarely the best choice for pure queue/deque operations.
- `Stack` is a legacy class and should generally be avoided in new applications.
- Real-world performance depends on memory layout, cache locality, allocation patterns, and garbage collection—not just Big-O complexity.
- Choosing the appropriate collection is an important design decision that directly affects application performance and maintainability.
