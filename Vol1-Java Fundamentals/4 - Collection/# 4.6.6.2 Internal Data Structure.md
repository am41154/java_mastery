# 4.6.6.2 Internal Data Structure

The defining characteristic of `ArrayDeque` is that it is implemented using a **resizable circular array**.

Unlike `LinkedList`, which stores each element in a separate node, `ArrayDeque` stores all elements in a contiguous block of memory.

Conceptually, it looks like this:

```text
+------+------+------+------+------+------+------+------+

|      |      |      |      |      |      |      |      |

+------+------+------+------+------+------+------+------+
```

Internally, the deque maintains:

- An array to store elements
- A pointer to the front (`head`)
- A pointer to the rear (`tail`)

These three components are sufficient to implement all queue and stack operations efficiently.

---

# Core Internal Components

An `ArrayDeque` can be visualized as:

```java
class ArrayDeque<E> {

    Object[] elements;

    int head;

    int tail;

}
```

> **Note:** This is a simplified representation for learning purposes. The actual OpenJDK implementation contains additional methods and implementation details.

Each field has a specific responsibility.

| Field | Purpose |
|--------|----------|
| `elements` | Stores the elements |
| `head` | Index of the first element |
| `tail` | Position where the next element will be inserted at the rear |

Unlike a linked list, there are **no node objects** and **no next/previous pointers**.

---

# The Internal Array

Suppose the deque has a capacity of 8.

Initially:

```text
Index

 0    1    2    3    4    5    6    7

+----+----+----+----+----+----+----+----+

|    |    |    |    |    |    |    |    |

+----+----+----+----+----+----+----+----+

head = 0

tail = 0
```

The array is empty.

Notice that both `head` and `tail` point to the same position.

---

# Adding the First Element

```java
deque.addLast("A");
```

Internally:

```text
Index

 0    1    2    3    4    5    6    7

+----+----+----+----+----+----+----+----+

| A  |    |    |    |    |    |    |    |

+----+----+----+----+----+----+----+----+

head = 0

tail = 1
```

Important observation:

`tail` does **not** point to the last element.

Instead, it points to the **next available insertion position**.

This design simplifies insertion logic.

---

# Adding More Elements

Insert:

```java
addLast("B");

addLast("C");

addLast("D");
```

Result:

```text
Index

 0    1    2    3    4    5    6    7

+----+----+----+----+----+----+----+----+

| A  | B  | C  | D  |    |    |    |    |

+----+----+----+----+----+----+----+----+

head = 0

tail = 4
```

Again:

- `head` points to the first element.
- `tail` points one position beyond the last element.

---

# Removing an Element

Suppose we remove the front element.

```java
deque.removeFirst();
```

Conceptually:

```text
Index

 0    1    2    3    4    5    6    7

+----+----+----+----+----+----+----+----+

|    | B  | C  | D  |    |    |    |    |

+----+----+----+----+----+----+----+----+

head = 1

tail = 4
```

Notice something important.

The remaining elements were **not shifted**.

Instead:

- `head` simply moved one position to the right.

This is the key reason removal remains O(1).

---

# Why Not Shift Elements?

Imagine one million elements.

Removing the first element by shifting:

```text
Before

A B C D E ...
```

After:

```text
B C D E ...
```

Every element would need to be copied.

Complexity:

```text
O(n)
```

Instead, `ArrayDeque` performs:

```text
head++
```

Complexity:

```text
O(1)
```

This simple pointer update avoids potentially millions of memory operations.

---

# The Meaning of Head

The `head` index always points to:

> **The first valid element in the deque.**

Example:

```text
Index

 0    1    2    3    4    5    6    7

+----+----+----+----+----+----+----+----+

|    | B  | C  | D  |    |    |    |    |

+----+----+----+----+----+----+----+----+

      ↑

    head
```

Removing from the front simply advances this pointer.

---

# The Meaning of Tail

The `tail` index points to:

> **The next available position for insertion at the rear.**

It does **not** point to the last element.

Example:

```text
Index

 0    1    2    3    4    5    6    7

+----+----+----+----+----+----+----+----+

|    | B  | C  | D  |    |    |    |    |

+----+----+----+----+----+----+----+----+

                     ↑

                   tail
```

The next call to:

```java
addLast("E");
```

places the element directly at index 4.

---

# Logical vs Physical Order

This distinction often confuses developers.

Suppose:

```text
head = 5
```

The physical array may look like:

```text
Index

0 1 2 3 4 5 6 7

F G . . . A B C
```

Physically:

```text
F G A B C
```

Logically:

```text
A → B → C → F → G
```

The logical order begins at `head` and wraps around the array as needed.

Applications interact with the logical sequence, while the physical storage may appear fragmented.

---

# Empty Deque

A deque is empty when:

```text
head == tail
```

Example:

```text
head = 3

tail = 3
```

No valid elements exist between them.

The array itself may still contain obsolete object references internally, but logically the deque contains no elements.

---

# Full Deque

The internal array also needs to detect when it is full.

Rather than overwriting existing elements, `ArrayDeque` grows its capacity.

Conceptually:

```text
Capacity = 8

Elements = 8
```

Next insertion triggers:

```text
Resize

↓

Allocate Larger Array

↓

Copy Elements

↓

Continue
```

We'll examine this resizing process in detail later.

---

# Why This Design Is Efficient

The internal representation provides several important advantages.

### Constant-Time Updates

Insertion and removal usually modify only:

- `head`
- `tail`

No traversal is required.

---

### Contiguous Storage

Elements remain inside a single array.

Benefits include:

- Better CPU cache locality
- Fewer allocations
- Faster iteration
- Reduced garbage collection pressure

---

### Minimal Metadata

Each deque stores only:

- One array
- Two integer indices

Compared to a linked list, this significantly reduces memory overhead.

---

# Common Misconceptions

## "`tail` points to the last element."

Incorrect.

`tail` points to the **next insertion position**, not the current last element.

---

## "Removing from the front shifts elements."

No.

The `head` index advances.

Elements remain in place until they are eventually overwritten or the array is resized.

---

## "The physical array is always in logical order."

Not after wrap-around.

The logical sequence begins at `head` and may continue from the start of the array after reaching the end.

---

# Interview Questions

### Q1. What are the primary internal fields of `ArrayDeque`?

A simplified implementation consists of an internal array (`elements`) and two indices (`head` and `tail`).

---

### Q2. Why does `tail` point to the next insertion position instead of the last element?

This simplifies insertion logic and allows efficient detection of the insertion location without additional calculations.

---

### Q3. Why doesn't `ArrayDeque` shift elements during `removeFirst()`?

Shifting would require O(n) time. Instead, it advances the `head` pointer, keeping removal O(1).

---

### Q4. What is the advantage of contiguous storage over linked nodes?

Contiguous storage improves cache locality, reduces memory overhead, minimizes object allocations, and generally results in better runtime performance.

---

# Senior Interview Perspective

A common interview discussion is:

> **Interviewer:** "How does `ArrayDeque` achieve O(1) removal from the front without shifting array elements?"

A strong answer is:

> "`ArrayDeque` stores elements in a circular array and maintains `head` and `tail` indices. Removing the front element simply advances the `head` pointer to the next valid element. Since no elements are physically moved, the operation completes in constant time."

This answer demonstrates an understanding of the underlying implementation rather than just the API.

---

# Key Takeaways

- `ArrayDeque` stores elements in a contiguous array instead of linked nodes.
- The implementation is based on three core components: an internal array, a `head` index, and a `tail` index.
- `head` points to the first valid element, while `tail` points to the next insertion position.
- Front removals do not shift elements; they simply advance the `head` index.
- This design enables efficient O(1) insertions and removals while maintaining excellent memory and cache efficiency.
