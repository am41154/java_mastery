# 4.6.8.3 ConcurrentLinkedDeque

`ConcurrentLinkedDeque` is a **thread-safe, non-blocking, unbounded double-ended queue (Deque)** provided by the `java.util.concurrent` package.

Like `ConcurrentLinkedQueue`, it uses **lock-free algorithms** based on **Compare-And-Swap (CAS)** instead of explicit locks.

The key difference is that elements can be inserted and removed from **both the front and the rear**, making it suitable for workloads that require more flexibility than a traditional FIFO queue.

---

# Package

```java
import java.util.concurrent.ConcurrentLinkedDeque;
```

---


Unlike `ConcurrentLinkedQueue`, this class implements the full `Deque` interface.

# Why Do We Need a Concurrent Deque?

A normal queue supports insertion and removal at only one end.

```text
offer()

↓

A → B → C

↓

poll()
```

Some applications require operations at both ends.

Examples include:

- Work-stealing schedulers
- Undo/redo history
- Breadth-first and depth-first traversal
- Double-ended task scheduling
- Sliding window algorithms

A concurrent deque supports these patterns safely in multi-threaded environments.

---

# Internal Architecture

Internally, `ConcurrentLinkedDeque` is implemented as a **lock-free doubly linked list**.

Conceptually:

```text
Head
 ↓
[A] ⇄ [B] ⇄ [C]
              ↑
             Tail
```

Each node maintains references to:

- Previous node
- Next node

Unlike `LinkedList`, updates are coordinated using CAS operations instead of locks.

---

# Lock-Free Operations

Suppose two producer threads execute:

```java
addFirst(task);
```

simultaneously.

Both attempt to update:

```text
Head
```

Only one CAS operation succeeds.

The other thread detects the change and retries.

Similarly, concurrent calls to:

```java
addLast(task);
```

coordinate updates to the tail without blocking.

This allows multiple threads to make progress concurrently.

---

# Basic Operations

## Add at the Front

```java
ConcurrentLinkedDeque<String> deque =
        new ConcurrentLinkedDeque<>();

deque.addFirst("A");
deque.addFirst("B");
```

Result:

```text
Front

↓

B ⇄ A
```

---

## Add at the Rear

```java
deque.addLast("C");
```

Result:

```text
Front

↓

B ⇄ A ⇄ C

↑

Rear
```

---

## Remove from the Front

```java
String item = deque.pollFirst();
```

Removes:

```text
B
```

Remaining:

```text
A ⇄ C
```

If the deque is empty:

```java
pollFirst();
```

returns:

```java
null
```

---

## Remove from the Rear

```java
String item = deque.pollLast();
```

Removes:

```text
C
```

Remaining:

```text
A
```

Again, no blocking occurs.

---

# Queue Mode

A deque can behave as a FIFO queue.

```java
deque.offerLast("A");
deque.offerLast("B");
deque.offerLast("C");

deque.pollFirst();
```

Processing order:

```text
A

↓

B

↓

C
```

---

# Stack Mode

A deque can also behave as a LIFO stack.

```java
deque.push("A");
deque.push("B");
deque.push("C");

deque.pop();
```

Processing order:

```text
C

↓

B

↓

A
```

This flexibility makes `ConcurrentLinkedDeque` more versatile than `ConcurrentLinkedQueue`.

---

# Weakly Consistent Iterators

Like other concurrent collections, iterators are **weakly consistent**.

They:

- Do not throw `ConcurrentModificationException`
- May observe some concurrent updates
- Continue iteration safely while other threads modify the deque

Example:

```java
for (String task : deque) {
    process(task);
}
```

The iteration proceeds without locking the entire data structure.

---

# Time Complexity

| Operation | Complexity |
|-----------|------------|
| addFirst() | O(1) |
| addLast() | O(1) |
| pollFirst() | O(1) |
| pollLast() | O(1) |
| peekFirst() | O(1) |
| peekLast() | O(1) |
| size() | O(n) |

As with `ConcurrentLinkedQueue`, `size()` traverses the structure because maintaining an exact count during concurrent updates would require additional synchronization.

---

# Advantages

- Thread-safe
- Lock-free
- Supports both FIFO and LIFO operations
- High scalability
- Multiple producers and consumers
- Weakly consistent iteration
- No explicit locking required

---

# Limitations

- Unbounded capacity
- No blocking operations
- `size()` is O(n)
- More complex implementation than a standard queue
- Slightly higher memory overhead than `ConcurrentLinkedQueue` because each node stores both `prev` and `next` references

---

# Typical Use Cases

`ConcurrentLinkedDeque` is commonly used for:

- Work-stealing algorithms
- Concurrent task scheduling
- Undo/redo history
- Concurrent stack implementations
- Event processing pipelines
- Double-ended work queues

It is particularly useful when both FIFO and LIFO access patterns are required.

---

# ConcurrentLinkedQueue vs ConcurrentLinkedDeque

| Feature | ConcurrentLinkedQueue | ConcurrentLinkedDeque |
|---------|-----------------------|-----------------------|
| FIFO | Yes | Yes |
| LIFO | No | Yes |
| Deque Operations | No | Yes |
| Head Operations | Yes | Yes |
| Tail Operations | Limited | Yes |
| Lock-Free | Yes | Yes |
| Blocking | No | No |
| Capacity | Unbounded | Unbounded |

If only FIFO behavior is required, `ConcurrentLinkedQueue` is generally simpler.

Choose `ConcurrentLinkedDeque` when double-ended access is necessary.

---

# Common Misconceptions

## "`ConcurrentLinkedDeque` is simply a synchronized `LinkedList`."

Incorrect.

It uses lock-free algorithms based on CAS rather than mutual exclusion locks.

---

## "`push()` and `pop()` make it a stack instead of a deque."

No.

A deque supports queue and stack operations simultaneously.

The API simply exposes both usage patterns.

---

## "`size()` is O(1)."

Incorrect.

Like `ConcurrentLinkedQueue`, determining the exact size requires traversal.

# Interview Questions

### Q1. Why would you choose `ConcurrentLinkedDeque` instead of `ConcurrentLinkedQueue`?

When the application requires insertion or removal at both ends of the collection, such as work-stealing or concurrent stack behavior.

---

### Q2. Why doesn't `ConcurrentLinkedDeque` use locks?

It relies on lock-free algorithms using CAS operations, reducing contention and improving scalability.

---

### Q3. Can `ConcurrentLinkedDeque` be used as a stack?

Yes.

Methods such as `push()`, `pop()`, and `peek()` provide LIFO behavior.

---

### Q4. Why is `size()` O(n)?

Because maintaining an exact count during concurrent updates would require additional synchronization, reducing scalability.

# Key Takeaways

- `ConcurrentLinkedDeque` is a thread-safe, lock-free implementation of the `Deque` interface.
- It supports insertion and removal from both the front and the rear.
- It uses CAS operations instead of locks to coordinate concurrent updates.
- Queue operations never block; unsuccessful operations return immediately.
- It is well suited for concurrent applications requiring both FIFO and LIFO semantics.
