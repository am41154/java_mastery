# 4.6.7.6 LinkedList vs ArrayDeque (Implementation Comparison)

`LinkedList` and `ArrayDeque` are the two primary general-purpose implementations of the `Deque` interface in Java.

Both support insertion and removal from both ends in constant time, but they achieve this using fundamentally different internal data structures.

Understanding these implementation differences is essential for choosing the appropriate collection in real-world applications.

---

# Internal Architecture

## ArrayDeque

Internally, `ArrayDeque` uses a circular array.

```text
+---+---+---+---+---+---+
| A | B | C | D | E | F |
+---+---+---+---+---+---+
```

The collection maintains two indices:

- `head`
- `tail`

Elements are stored contiguously in memory.

---

## LinkedList

Internally, `LinkedList` uses a doubly linked list.

```text
[A] ⇄ [B] ⇄ [C] ⇄ [D]
```

Each element is wrapped inside an independent `Node` object containing:

- Element
- Previous reference
- Next reference

---

# Memory Layout

### ArrayDeque

```text
One Backing Array

↓

[A][B][C][D]
```

Only one primary object stores all element references.

Neighboring elements are adjacent in memory.

---

### LinkedList

```text
Node A

↓

Heap Address 1024

Node B

↓

Heap Address 81720

Node C

↓

Heap Address 3540
```

Each node is allocated independently.

Neighboring nodes may reside at completely different memory locations.

---

# Object Allocation

## ArrayDeque

Typical insertion:

```text
Write Element

↓

Advance Head/Tail
```

No additional object allocation occurs unless the backing array needs to grow.

---

## LinkedList

Every insertion performs:

```text
Allocate Node

↓

Initialize References

↓

Link Node
```

Each inserted element results in a new object allocation.

---

# Memory Overhead

| Characteristic | ArrayDeque | LinkedList |
|---------------|------------|------------|
| Per Element Object | No | Yes |
| Additional References | No | Previous + Next |
| Object Header Per Element | No | Yes |
| Overall Memory Usage | Low | High |

The node-based architecture makes `LinkedList` significantly more memory intensive.

---

# Cache Locality

## ArrayDeque

```text
A B C D E F
```

Sequential memory layout allows CPUs to efficiently prefetch neighboring elements.

This results in excellent cache utilization.

---

## LinkedList

```text
A

↓

Memory 4210

↓

B

↓

Memory 98240

↓

C
```

Traversal requires repeatedly following references.

This increases cache misses and memory latency.

---

# Garbage Collection

### ArrayDeque

Typical allocations:

- One backing array
- Occasional larger array during resize

Garbage collection pressure is relatively low.

---

### LinkedList

Typical allocations:

```text
One Node

↓

Per Element
```

Every removed node eventually becomes eligible for garbage collection.

Large linked lists therefore create substantially more GC work.

---

# Traversal

## ArrayDeque

Iteration accesses neighboring memory locations.

```text
A → B → C → D
```

This access pattern is highly efficient.

---

## LinkedList

Iteration repeatedly dereferences node references.

```text
A

↓

B

↓

C

↓

D
```

Although both implementations require O(n) time for iteration, `ArrayDeque` generally has much lower constant overhead.

---

# Indexed Access

### ArrayDeque

The `Deque` interface intentionally does not expose indexed operations.

If random indexed access is required, `ArrayList` is usually the appropriate choice.

---

### LinkedList

`LinkedList` implements the `List` interface and therefore supports:

```java
get(index);
```

However, indexed access requires traversal and has O(n) complexity.

Supporting indexed access does not make it efficient.

---

# Dynamic Growth

### ArrayDeque

When the backing array becomes full:

```text
16

↓

32

↓

64
```

A larger array is allocated and existing elements are copied.

Resizing is infrequent and results in amortized O(1) insertion.

---

### LinkedList

No resizing is ever required.

Each insertion simply allocates a new node.

This avoids copying existing elements but increases allocation overhead.

---

# Practical Performance

| Operation | ArrayDeque | LinkedList |
|-----------|------------|------------|
| addFirst() | Excellent | Excellent |
| addLast() | Excellent | Excellent |
| removeFirst() | Excellent | Excellent |
| removeLast() | Excellent | Excellent |
| Iteration | Excellent | Moderate |
| Memory Usage | Low | High |
| Cache Locality | Excellent | Poor |
| GC Pressure | Low | High |
| Indexed Access | Not Supported | O(n) |

Although the asymptotic complexities appear similar, `ArrayDeque` generally delivers better throughput due to its cache-friendly design and reduced allocation overhead.

---

# When to Choose ArrayDeque

`ArrayDeque` is usually the best choice for:

- Queue implementations
- Stack implementations
- Breadth-First Search (BFS)
- Depth-First Search (DFS)
- Sliding window algorithms
- Task scheduling
- Producer/consumer buffers (single-threaded)
- General-purpose FIFO and LIFO operations

It should be considered the default implementation for the `Deque` interface.

---

# When to Choose LinkedList

`LinkedList` is appropriate when:

- Both `List` and `Deque` behavior are required from the same collection.
- Frequent insertions and removals occur using an existing `ListIterator`.
- Bidirectional navigation through a list is important.

These scenarios are less common than general queue or stack usage.

---

# JDK Recommendation

The Java Platform documentation recommends using `ArrayDeque` for most queue and stack implementations.

`LinkedList` remains a valid implementation of `Deque`, but it is not the preferred default because of its higher memory overhead and poorer cache locality.

---

# Common Misconceptions

## "`LinkedList` is always faster because it never resizes."

Incorrect.

The cost of object allocation, pointer chasing, and cache misses often exceeds the occasional resize cost of `ArrayDeque`.

---

## "`ArrayDeque` wastes too much memory."

`ArrayDeque` reserves spare capacity to reduce resizing frequency.

In practice, this modest overhead is usually outweighed by improved cache locality and lower allocation costs.

---

## "`LinkedList` is the best implementation for queues."

Historically this was a common assumption.

Modern JVMs and hardware typically favor `ArrayDeque` for queue implementations because of its superior memory layout and runtime characteristics.

---

# Interview Questions

### Q1. Why does the JDK generally recommend `ArrayDeque` over `LinkedList` for queues?

Because `ArrayDeque` provides better cache locality, lower memory overhead, fewer object allocations, and reduced garbage collection pressure while offering the same amortized O(1) operations at both ends.

---

### Q2. If both support O(1) insertion at the ends, why is `ArrayDeque` usually faster?

Because Big-O ignores constant factors such as cache behavior, object allocation, and pointer traversal, all of which favor `ArrayDeque`.

---

### Q3. When would `LinkedList` still be a reasonable choice?

When an application genuinely requires both `List` and `Deque` behavior in the same collection or performs frequent insertions and removals using a `ListIterator`.

---

### Q4. Which implementation would you recommend for a stack?

`ArrayDeque`, using the `Deque` interface:

```java
Deque<String> stack = new ArrayDeque<>();
```

This is the approach recommended by the JDK for modern Java applications.

---

# Senior Interview Perspective

A common interview discussion is:

> **Interviewer:** "You're implementing an in-memory task queue for a high-throughput backend service. Would you choose `LinkedList` or `ArrayDeque`?"

A strong answer is:

> "I would choose `ArrayDeque`. Both implementations provide constant-time insertion and removal at the ends, but `ArrayDeque` stores elements in a contiguous array, resulting in much better cache locality and significantly fewer object allocations. This reduces garbage collection pressure and generally provides higher throughput. I would consider `LinkedList` only if I specifically needed both `List` and `Deque` semantics or frequent modifications through a `ListIterator`."

This response demonstrates an understanding of JVM internals, hardware behavior, and practical engineering trade-offs.

---

# Key Takeaways

- `ArrayDeque` and `LinkedList` implement the same `Deque` interface using completely different internal data structures.
- `ArrayDeque` stores elements in a contiguous circular array, while `LinkedList` stores each element in a separate node.
- `ArrayDeque` generally provides better memory efficiency, cache locality, and runtime performance.
- `LinkedList` offers additional flexibility through the `List` interface but incurs higher allocation and garbage collection costs.
- For most modern queue and stack implementations, `ArrayDeque` is the preferred choice.
