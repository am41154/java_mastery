# 4.6.8.5 LinkedBlockingQueue

`LinkedBlockingQueue` is a **thread-safe, optionally bounded FIFO blocking queue** provided by the `java.util.concurrent` package.

It is one of the most widely used implementations of the `BlockingQueue` interface and forms the foundation of many producer-consumer systems, asynchronous processing pipelines, and thread pool implementations.

Internally, it uses a **linked-node structure** combined with **separate locks for producers and consumers**, allowing higher concurrency than implementations using a single lock.

# Internal Architecture

Unlike `ConcurrentLinkedQueue`, which is lock-free, `LinkedBlockingQueue` uses **locks and conditions** to coordinate producers and consumers.

Internally, it is implemented as a **singly linked list**.

```text
Head
 ↓
[A] → [B] → [C]
              ↑
             Tail
```

Each node contains:

- The element
- A reference to the next node

Unlike `LinkedList`, there is no reference to the previous node.

---

# Optional Capacity

A `LinkedBlockingQueue` may be created with or without a capacity limit.

### Unbounded Queue

```java
BlockingQueue<String> queue =
        new LinkedBlockingQueue<>();
```

Internally:

```text
Capacity

↓

Integer.MAX_VALUE
```

Although often called "unbounded", the queue is still limited by available JVM memory.

---

### Bounded Queue

```java
BlockingQueue<String> queue =
        new LinkedBlockingQueue<>(100);
```

Now the queue can contain at most:

```text
100 Elements
```

Once full:

```java
put()
```

blocks until space becomes available.

---

# Producer Operations

### put()

```java
queue.put(task);
```

If the queue has space:

```text
Insert Immediately
```

If the queue is full:

```text
Producer

↓

WAITING
```

The thread remains blocked until a consumer removes an element.

---

### offer()

```java
queue.offer(task);
```

If the queue is full:

```text
false
```

is returned immediately.

No waiting occurs.

---

### offer(timeout)

```java
queue.offer(task,
            5,
            TimeUnit.SECONDS);
```

Behavior:

```text
Wait up to 5 Seconds

↓

Space Available?

↓

Yes → Insert

No → Return false
```

---

# Consumer Operations

### take()

```java
Task task = queue.take();
```

If the queue is empty:

```text
Consumer

↓

WAITING
```

The thread sleeps until another thread inserts an element.

---

### poll()

```java
Task task = queue.poll();
```

Returns:

```text
null
```

immediately if the queue is empty.

---

### poll(timeout)

```java
Task task =
        queue.poll(10, TimeUnit.SECONDS);
```

Behavior:

```text
Wait up to 10 Seconds

↓

Element Available?

↓

Yes → Return Element

No → Return null
```

# Separate Locks

One of the most important implementation details is that `LinkedBlockingQueue` uses **two independent locks**.

Conceptually:

```text
Producer

↓

putLock

↓

Queue

↓

takeLock

↓

Consumer
```

This means:

- Producers primarily compete with other producers.
- Consumers primarily compete with other consumers.
- Producers and consumers can often proceed concurrently.

Compared to a single-lock implementation, this significantly improves throughput.

---

# Condition Variables

Each lock is associated with a condition.

```text
Queue Full

↓

notFull
```

```text
Queue Empty

↓

notEmpty
```

When a producer inserts an element:

```text
Signal notEmpty
```

Waiting consumers wake up.

When a consumer removes an element:

```text
Signal notFull
```

Waiting producers wake up.

This coordination happens automatically.

---

# Size Tracking

Unlike `ConcurrentLinkedQueue`, `LinkedBlockingQueue` maintains its size efficiently.

Internally:

```text
Atomic Counter

↓

Current Size
```

Therefore:

```java
queue.size();
```

runs in:

```text
O(1)
```

This is possible because updates occur while holding the appropriate locks.

# Time Complexity

| Operation | Complexity |
|-----------|------------|
| put() | O(1) |
| take() | O(1) |
| offer() | O(1) |
| poll() | O(1) |
| peek() | O(1) |
| size() | O(1) |

Blocking time is not included in these complexity measurements.

# Advantages

- Thread-safe
- Automatic producer-consumer coordination
- Optional bounded capacity
- Separate producer and consumer locks
- Constant-time insertion and removal
- Constant-time size lookup
- High throughput under mixed producer-consumer workloads

---

# Limitations

- Uses locks instead of lock-free algorithms
- Higher memory usage than array-based implementations
- One node object allocated per element
- Unbounded queues can grow until memory is exhausted if producers outpace consumers

---

# Typical Use Cases

`LinkedBlockingQueue` is commonly used for:

- Thread pools
- Producer-consumer systems
- Asynchronous task processing
- Logging pipelines
- Event processing
- Background job execution
- Work queues

It is often the default choice when producers and consumers operate at different speeds.

---

# ThreadPoolExecutor Example

One of the most common uses is within a thread pool.

```java
ExecutorService executor =
        Executors.newFixedThreadPool(4);
```

Internally, this executor uses a:

```text
LinkedBlockingQueue
```

to hold submitted tasks until worker threads are ready to execute them.

# LinkedBlockingQueue vs ConcurrentLinkedQueue

| Feature | LinkedBlockingQueue | ConcurrentLinkedQueue |
|---------|---------------------|-----------------------|
| Thread Safe | Yes | Yes |
| Blocking | Yes | No |
| Lock-Free | No | Yes |
| Capacity | Optional | Unbounded |
| Producer Waiting | Yes | No |
| Consumer Waiting | Yes | No |
| size() | O(1) | O(n) |

Choose `LinkedBlockingQueue` when automatic thread coordination is required.

Choose `ConcurrentLinkedQueue` when maximum throughput and non-blocking behavior are more important.

---

# Common Misconceptions

## "`LinkedBlockingQueue` is always unbounded."

Incorrect.

It can be created with a fixed capacity.

---

## "`put()` never blocks."

Incorrect.

It blocks whenever a bounded queue reaches its capacity.

---

## "`LinkedBlockingQueue` is lock-free."

Incorrect.

It uses `ReentrantLock` and `Condition` objects internally.

---

# Interview Questions

### Q1. Why does `LinkedBlockingQueue` use two locks?

To allow producers and consumers to operate concurrently whenever possible, reducing contention compared to a single shared lock.

---

### Q2. Why is `size()` O(1) in `LinkedBlockingQueue` but O(n) in `ConcurrentLinkedQueue`?

Because `LinkedBlockingQueue` maintains an atomic count while coordinating updates with locks, whereas `ConcurrentLinkedQueue` avoids maintaining an exact count to preserve its lock-free design.

---

### Q3. When would you choose `LinkedBlockingQueue` over `ConcurrentLinkedQueue`?

When producers and consumers need blocking behavior, bounded capacity, or automatic coordination.

---

### Q4. What happens when `put()` is called on a full bounded queue?

The producer thread blocks until another thread removes an element and space becomes available.

---

# Senior Interview Perspective

A common interview discussion is:

> **Interviewer:** "Why does `LinkedBlockingQueue` use separate locks for producers and consumers instead of a single lock?"

A strong answer is:

> "Using separate `putLock` and `takeLock` allows producers and consumers to make progress concurrently. For example, while one producer inserts an element, a consumer can simultaneously remove another element. A single shared lock would unnecessarily serialize these operations, reducing throughput under concurrent workloads."

This demonstrates an understanding of concurrent data structure design rather than simply knowing the API.

---

# Key Takeaways

- `LinkedBlockingQueue` is a thread-safe FIFO queue with blocking operations.
- It uses a linked-node structure with separate locks for producers and consumers.
- It supports both bounded and effectively unbounded capacities.
- `put()` and `take()` automatically coordinate producers and consumers without busy waiting.
- It is one of the most widely used queue implementations in Java concurrency and underpins many thread pool implementations.
