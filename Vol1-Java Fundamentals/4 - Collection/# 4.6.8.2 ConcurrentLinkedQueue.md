# 4.6.8.2 ConcurrentLinkedQueue

`ConcurrentLinkedQueue` is a **thread-safe, non-blocking, unbounded FIFO queue** provided by the `java.util.concurrent` package.

Unlike `LinkedList` or `ArrayDeque`, it is specifically designed for **highly concurrent environments**, where multiple producer and consumer threads operate on the queue simultaneously.

Instead of using locks (`synchronized` or `ReentrantLock`), `ConcurrentLinkedQueue` relies on **lock-free algorithms** built on **Compare-And-Swap (CAS)** operations.

This design allows multiple threads to access the queue concurrently with minimal contention, making it one of the highest-throughput queue implementations in the JDK.

---

```java
import java.util.concurrent.ConcurrentLinkedQueue;
```

# Why Not Use synchronized?

One possible implementation of a thread-safe queue is:

```java
synchronized (queue) {
    queue.offer(task);
}
```

While correct, this approach has limitations.

When one thread holds the lock:

```text
Producer A

↓

Owns Lock
```

Every other producer or consumer must wait.

```text
Producer B

↓

Waiting
```

```text
Consumer C

↓

Waiting
```

As the number of threads increases, lock contention grows, reducing scalability.

`ConcurrentLinkedQueue` avoids this bottleneck by using a **lock-free algorithm**.

---

# Internal Architecture

Internally, `ConcurrentLinkedQueue` is implemented as a **singly linked list**.

Conceptually:

```text
Head
 ↓
[A] → [B] → [C]
              ↑
             Tail
```

Each node stores:

- The element
- A reference to the next node

Unlike `LinkedList`, there is **no previous reference**, reducing memory overhead.

---

# Lock-Free Design

The queue does **not** use:

- `synchronized`
- `ReentrantLock`

Instead, it relies on **CAS (Compare-And-Swap)** operations provided by the JVM.

Conceptually:

```text
Read Current Value

↓

Compare Expected Value

↓

Update Only If Unchanged
```

If another thread modified the value first:

```text
CAS Fails

↓

Retry
```

No thread is blocked while waiting for a lock.

---

# Compare-And-Swap (CAS)

Suppose the queue's tail currently points to:

```text
Tail

↓

Node C
```

Two producers attempt to add elements simultaneously.

Both expect:

```text
Tail = Node C
```

Producer A successfully performs the CAS operation.

```text
Tail

↓

Node D
```

Producer B now discovers that the tail has changed.

Its CAS operation fails.

Instead of blocking:

```text
Retry
```

The operation repeats using the updated tail.

This retry mechanism is fundamental to lock-free algorithms.

# Basic Operations

## Adding Elements

```java
ConcurrentLinkedQueue<String> queue =
        new ConcurrentLinkedQueue<>();

queue.offer("A");
queue.offer("B");
queue.offer("C");
```

Queue:

```text
Front

↓

A → B → C

↑

Rear
```

---

## Removing Elements

```java
String item = queue.poll();
```

Result:

```text
A
```

Remaining queue:

```text
B → C
```

If the queue is empty:

```java
poll();
```

returns:

```java
null
```

The calling thread does **not** wait.

---

## Peeking

```java
String first = queue.peek();
```

Returns the first element without removing it.

If the queue is empty:

```java
null
```

is returned.

---

# FIFO Ordering

Elements leave the queue in the same order they were inserted.

```text
Offer

A

↓

B

↓

C
```

Removal order:

```text
Poll

↓

A

↓

B

↓

C
```

This FIFO guarantee is preserved even when multiple threads access the queue concurrently.

---

# Weakly Consistent Iterator

Iterators returned by `ConcurrentLinkedQueue` are **weakly consistent**.

This means:

- They do not throw `ConcurrentModificationException`.
- They may reflect some, all, or none of the modifications made after iteration begins.
- They provide a stable traversal without locking the entire collection.

Example:

```java
for (String task : queue) {
    System.out.println(task);
}
```

If another thread adds or removes elements during iteration, the iterator continues safely.

---

# Time Complexity

| Operation | Complexity |
|-----------|------------|
| offer() | O(1) |
| poll() | O(1) |
| peek() | O(1) |
| isEmpty() | O(1) |
| size() | O(n) |

Unlike most collections, `size()` is **not constant time**.

The queue must traverse its nodes to determine the current number of elements because concurrent modifications make maintaining an exact size counter expensive and potentially contentious.

# Advantages

- Thread-safe without explicit locking
- Excellent scalability under contention
- High throughput
- Multiple producers and consumers
- No blocking during queue operations
- No `ConcurrentModificationException` during iteration

---

# Limitations

- Unbounded capacity
- No blocking operations (`put()`, `take()`)
- `size()` is O(n)
- Iterators provide only weak consistency
- Not suitable when producers or consumers must wait

---

# Typical Use Cases

`ConcurrentLinkedQueue` is commonly used for:

- Asynchronous task queues
- Event processing pipelines
- Logging systems
- Actor-style messaging
- Work distribution
- Lock-free communication between threads

It is an excellent choice when **throughput** is more important than blocking coordination.

---

# Common Misconceptions

## "`ConcurrentLinkedQueue` uses locks internally."

Incorrect.

Its implementation is based on lock-free algorithms using CAS operations.

---

## "`size()` is O(1)."

Incorrect.

The queue must traverse its nodes because maintaining an exact count during concurrent updates would introduce additional synchronization overhead.

---

## "`poll()` waits for an element."

No.

If the queue is empty, `poll()` immediately returns `null`.

Blocking behavior is provided by implementations such as `LinkedBlockingQueue`.

---

# Interview Questions

### Q1. Why is `ConcurrentLinkedQueue` considered lock-free?

Because it coordinates concurrent updates using Compare-And-Swap (CAS) operations instead of mutual exclusion locks.

---

### Q2. Why is `size()` O(n)?

Because the queue does not maintain an exact element count during concurrent updates. It traverses the linked structure to calculate the current size.

---

### Q3. Does `ConcurrentLinkedQueue` block when the queue is empty?

No.

Methods such as `poll()` return immediately with `null`.

---

### Q4. Why doesn't its iterator throw `ConcurrentModificationException`?

Because it is weakly consistent and is designed to tolerate concurrent modifications without locking the collection.

# Senior Interview Perspective

A common interview discussion is:

> **Interviewer:** "Why would you choose `ConcurrentLinkedQueue` instead of wrapping a `LinkedList` with `Collections.synchronizedList()`?"

A strong answer is:

> "A synchronized wrapper serializes access using a single lock, which limits scalability as thread contention increases. `ConcurrentLinkedQueue` uses a lock-free algorithm based on CAS operations, allowing multiple producers and consumers to make progress concurrently with much lower contention. This generally provides significantly better throughput in highly concurrent applications."

This demonstrates an understanding of the difference between **thread safety through locking** and **thread safety through non-blocking algorithms**.

---

# Key Takeaways

- `ConcurrentLinkedQueue` is a thread-safe, lock-free FIFO queue.
- It uses a singly linked list internally and coordinates updates using CAS operations.
- Queue operations never block; unsuccessful operations return immediately.
- Iterators are weakly consistent and safe for concurrent traversal.
- It is an excellent choice for high-throughput producer-consumer scenarios where waiting is not required.
