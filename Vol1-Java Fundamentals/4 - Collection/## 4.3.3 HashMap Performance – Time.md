## 4.3.3 HashMap Performance ‚Äì Time Complexity and Common Pitfalls

### Why This Matters

Many developers memorize the following table:

| Operation | Complexity |
|-----------|-----------:|
| `put()` | O(1) |
| `get()` | O(1) |
| `remove()` | O(1) |

While technically correct, these answers are incomplete.

A senior engineer knows that performance depends on several factors:

- Quality of the hash function
- Number of collisions
- Load factor
- Distribution of keys
- Resizing
- Key immutability

Understanding these trade-offs is what separates implementation knowledge from engineering knowledge.

---

## Average Case vs Worst Case

Interviewers often ask:

> **"What is the time complexity of HashMap?"**

A strong answer is:

| Operation | Average | Worst Case |
|-----------|---------:|-----------:|
| `put()` | O(1) | O(n) |
| `get()` | O(1) | O(n) |
| `remove()` | O(1) | O(n) |

Notice the distinction.

**Average performance** assumes keys are evenly distributed across buckets.

**Worst-case performance** occurs when many keys collide into the same bucket.

## Why Good `hashCode()` Matters

Suppose every object returns the same hash code.

```java
@Override
public int hashCode() {
    return 1;
}
```

What happens?

Every key is placed into the same bucket.

```
Bucket 1

A
‚Üì

B
‚Üì

C
‚Üì

D
‚Üì

E
‚Üì

...
```

At this point, the `HashMap` behaves more like a linked structure than a hash table.

The lookup advantage disappears.

> **Engineering Insight**
>
> A poor `hashCode()` implementation can make a perfectly correct program perform terribly under load.

## Mutable Keys ‚Äì A Hidden Bug

Consider the following example.

```java
Employee employee = new Employee(101);

map.put(employee, "Developer");
```

Later:

```java
employee.setId(102);
```

If `id` is part of the `hashCode()` calculation, the object's logical bucket changes.

However, the `HashMap` still stores it in the original bucket.

Now:

```java
map.get(employee);
```

may return:

```
null
```

even though the object exists.

This is one of the most common production bugs involving `HashMap`.

---

## Null Keys and Null Values

Unlike some other map implementations, `HashMap` allows:

- One `null` key.
- Multiple `null` values.

```java
Map<String, String> map = new HashMap<>();

map.put(null, "Default");

map.put("A", null);
```

This behavior is sometimes tested in interviews.

> **Note**
>
> `ConcurrentHashMap` does **not** allow `null` keys or values. We'll discuss why later in this chapter.

---

## Choosing the Right Key

Not every object makes a good key.

Good keys should be:

- Immutable
- Have a well-distributed `hashCode()`
- Correctly implement `equals()`
- Represent a stable identity

Excellent examples:

- `String`
- `UUID`
- `Long`
- `Integer`

Custom mutable objects require careful design.

---

## Production Perspective

An e-commerce application uses a mutable `Product` object as a cache key.

After the product price is updated, cache lookups begin failing unexpectedly.

The issue isn't the cache implementation.

The problem is that the key's state changed after insertion, resulting in a different hash code.

Replacing the mutable key with an immutable identifier (`productId`) resolves the issue completely.

---

## Interview Focus

### Question

Can a `HashMap` lose an object?

### Strong Senior-Level Answer

> No, the object still exists in the map. However, if a mutable key changes after insertion and its `hashCode()` changes, future lookups may search a different bucket and fail to locate the entry. The object becomes effectively unreachable through normal lookups.

---

## Common Mistakes

‚ùå Returning a constant value from `hashCode()`.

‚ùå Using mutable objects as keys.

‚ùå Assuming `HashMap` always performs in O(1).

‚ùå Ignoring collision handling.

‚ùå Forgetting to override `equals()` together with `hashCode()`.

---

## Engineering Insight ‚Äì Real Performance Is More Than Big-O

Many developers focus exclusively on algorithmic complexity.

In production systems, performance also depends on:

- Memory layout
- CPU cache locality
- Object allocation
- Garbage collection
- Collision distribution

Two `HashMap` implementations with the same theoretical complexity can behave very differently under heavy load depending on key quality and usage patterns.

This is why senior engineers evaluate both algorithmic complexity and implementation details.

---

## Engineering Takeaways

- `HashMap` offers O(1) average-case lookup.
- Poor hashing increases collisions.
- Mutable keys can make entries unreachable.
- Strings are excellent keys because they are immutable.
- Performance depends on distribution, not just data structure.

---

## Quick Revision

‚úì Average lookup is O(1).

‚úì Worst-case lookup is O(n), improved to O(log n) for treeified buckets in Java 8+.

‚úì Good `hashCode()` implementations distribute keys evenly.

‚úì Never modify fields involved in `equals()` or `hashCode()` after insertion.

‚úì Prefer immutable keys for predictable behavior.

## Senior Interview Challenge

A payment gateway experiences a gradual increase in response time over several months.

Profiling shows that a heavily used `HashMap` now spends significantly more time performing lookups than it did after the initial release.

No functional changes have been made to the lookup logic.

### Discussion Points

- Has the number of entries grown substantially?
- Are collisions increasing because of poor key distribution?
- Is the load factor causing more frequent bucket sharing?
- Has a recent code change affected `hashCode()` or `equals()`?
- Are mutable objects being used as keys?

A senior engineer approaches this problem by understanding the internal mechanics of `HashMap`, not by immediately replacing it with another data structure.
