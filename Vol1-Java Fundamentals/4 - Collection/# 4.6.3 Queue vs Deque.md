# 4.6.3 Queue vs Deque

At first glance, **Queue** and **Deque** appear to be very similar.

Both store elements in sequence.

Both support insertion and removal.

Both are part of the Java Collections Framework.

However, they are designed for different levels of flexibility.

A **Queue** allows insertion at one end and removal from the other.

A **Deque (Double-Ended Queue)** allows insertion and removal from **both ends**.

Understanding this distinction is important because many Java collections, concurrent utilities, and enterprise frameworks are built around the `Deque` interface rather than the older `Stack` class.

---

# Queue

A traditional queue follows the FIFO (First In, First Out) principle.

Elements are inserted only at the **rear**.

Elements are removed only from the **front**.

```text
Enqueue

↓

+---------------------------------+

A → B → C → D

+---------------------------------+

↑                 ↑

Front           Rear
```

Operations:

```text
Insert  → Rear

Remove  → Front
```

There is only one valid direction for insertion and removal.

---

# Deque

Deque stands for:

> **Double-Ended Queue**

Unlike a normal queue, a deque allows operations at **both ends**.

```text
+---------------------------------+

A → B → C → D

+---------------------------------+

↑                 ↑

Front           Rear
```

Supported operations:

```text
Insert Front

Insert Rear

Remove Front

Remove Rear
```

Both ends are equally important.

---

# Why Was Deque Introduced?

Suppose we need a data structure that behaves like:

- A Queue today
- A Stack tomorrow

Without `Deque`, we would need separate implementations.

For example:

```text
Queue

↓

LinkedList
```

and

```text
Stack

↓

Stack
```

Instead, Java introduced a more general abstraction.

```text
Deque

↓

ArrayDeque
```

The same data structure can now support:

- FIFO operations
- LIFO operations

This simplifies APIs and improves performance.

---

# Queue Operations

The `Queue` interface primarily exposes FIFO-oriented methods.

| Operation | Purpose |
|-----------|----------|
| offer() | Insert at rear |
| poll() | Remove from front |
| peek() | View front element |

Example:

```java
Queue<String> queue = new ArrayDeque<>();

queue.offer("A");
queue.offer("B");
queue.offer("C");

System.out.println(queue.poll());
```

Output:

```text
A
```

FIFO is preserved.

---

# Deque Operations

A `Deque` exposes operations for both ends.

Front operations:

```java
addFirst()

offerFirst()

removeFirst()

pollFirst()

peekFirst()
```

Rear operations:

```java
addLast()

offerLast()

removeLast()

pollLast()

peekLast()
```

Example:

```java
Deque<String> deque = new ArrayDeque<>();

deque.addFirst("B");

deque.addFirst("A");

deque.addLast("C");

deque.addLast("D");
```

Result:

```text
Front

↓

A → B → C → D

                ↑

              Rear
```

Both ends remain accessible.

---

# Deque as a Queue

A deque can behave exactly like a normal queue.

Example:

```java
Deque<Integer> deque = new ArrayDeque<>();

deque.offerLast(10);

deque.offerLast(20);

deque.offerLast(30);

System.out.println(deque.pollFirst());
```

Result:

```text
10
```

FIFO behavior.

---

# Deque as a Stack

A deque can also behave like a stack.

```java
Deque<Integer> stack = new ArrayDeque<>();

stack.push(10);

stack.push(20);

stack.push(30);

System.out.println(stack.pop());
```

Output:

```text
30
```

LIFO behavior.

Internally:

```text
Push

↓

30

20

10

↓

Pop
```

Exactly like a stack.

---

# Queue vs Deque

| Feature | Queue | Deque |
|----------|--------|--------|
| Insert Front | ❌ | ✅ |
| Insert Rear | ✅ | ✅ |
| Remove Front | ✅ | ✅ |
| Remove Rear | ❌ | ✅ |
| FIFO Support | ✅ | ✅ |
| LIFO Support | ❌ | ✅ |
| Can Replace Stack | ❌ | ✅ |

Deque is therefore a superset of Queue.

---

# Why ArrayDeque Is So Popular

The Java documentation recommends:

> Use `ArrayDeque` instead of `Stack`.

Why?

Because `ArrayDeque`:

- Supports both queue and stack operations
- Has no legacy synchronization overhead
- Uses a circular array internally
- Provides excellent cache locality
- Offers amortized O(1) insertion and removal at both ends

As a result, it is the preferred general-purpose implementation for both FIFO and LIFO processing in single-threaded applications.

---

# Internal View

Conceptually, an `ArrayDeque` maintains two pointers.

```text
            Circular Buffer

+----+----+----+----+----+----+----+----+

|    | A  | B  | C  |    |    |    |    |

+----+----+----+----+----+----+----+----+

      ↑              ↑

    Front          Rear
```

Both pointers move independently.

Insertion or removal at either end simply adjusts the corresponding pointer.

No element shifting is required.

This is one of the reasons `ArrayDeque` is significantly faster than many developers expect.

# Enterprise Use Cases

## Queue Mode

Commonly used for:

- Request processing
- Producer-consumer workflows
- Task scheduling
- Event processing
- Breadth-First Search (BFS)

---

## Stack Mode

Commonly used for:

- Expression evaluation
- Undo functionality
- Depth-First Search (DFS)
- Parsing algorithms
- Backtracking

A single `Deque` implementation supports both categories efficiently.

---

# Common Misconceptions

## "Deque is only a double-ended queue."

Not entirely.

Although its name emphasizes double-ended operations, a deque can also act as:

- A Queue
- A Stack

This versatility is the reason it is widely used in modern Java applications.

---

## "Queue is obsolete because Deque exists."

No.

The `Queue` interface remains valuable because it communicates intent.

If an API accepts a `Queue`, callers know that FIFO semantics are expected.

Using the most specific abstraction improves readability and maintainability.

---

## "LinkedList is the best Deque implementation."

Not necessarily.

For most single-threaded scenarios, `ArrayDeque` outperforms `LinkedList` due to lower memory overhead and better CPU cache locality.

# Interview Questions

### Q1. What is the difference between Queue and Deque?

A Queue supports insertion at the rear and removal from the front. A Deque supports insertion and removal at both the front and rear.

---

### Q2. Can a Deque be used as a Queue?

Yes.

Using methods such as `offerLast()` and `pollFirst()`, a deque behaves exactly like a FIFO queue.

---

### Q3. Can a Deque replace the Stack class?

Yes.

The Java Collections Framework recommends `ArrayDeque` instead of the legacy `Stack` class for most stack use cases.

---

### Q4. Why is `ArrayDeque` generally preferred over `LinkedList` for stack and queue operations?

`ArrayDeque` stores elements in a contiguous circular array, resulting in better cache locality, lower memory overhead, and generally better performance than the node-based `LinkedList`.

---

### Q5. Is every Queue implementation also a Deque?

No.

For example, `PriorityQueue` implements `Queue` but not `Deque`, because its ordering is based on priority rather than front/rear operations.

---

# Senior Interview Perspective

A common interview discussion is:

> **Interviewer:** "If `Deque` can do everything `Queue` can, why does the `Queue` interface still exist?"

A strong answer is:

> "`Deque` is a more general abstraction, but APIs should expose the narrowest interface that satisfies their requirements. If an application only requires FIFO semantics, accepting a `Queue` communicates intent more clearly and avoids exposing unnecessary operations. This follows the Interface Segregation Principle and leads to more maintainable code."

---

# Key Takeaways

- A Queue supports insertion at the rear and removal from the front.
- A Deque supports insertion and removal at both ends.
- A Deque can efficiently implement both FIFO and LIFO processing.
- `ArrayDeque` is the preferred general-purpose implementation for stack and queue operations in single-threaded applications.
- Choosing `Queue` or `Deque` should be based on the operations an application requires, not simply on implementation availability.
