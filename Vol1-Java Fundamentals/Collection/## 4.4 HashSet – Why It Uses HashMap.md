## 4.4 HashSet – Why It Uses HashMap Internally

### Why This Matters

One of the most common interview questions is:

> **"How is `HashSet` implemented internally?"**

Many developers answer:

> "It stores unique elements."

That is true.

But it doesn't explain **how** uniqueness is achieved.

The interesting fact is that **HashSet doesn't implement its own hashing algorithm**.

Instead, it leverages one of the most optimized data structures in the JDK:

> **`HashMap`**

Understanding this relationship gives you a much deeper understanding of both collections.

---

## What is a HashSet?

A `HashSet` is a collection that stores **unique elements**.

Unlike a `List`:

- Duplicate elements are not allowed.
- Order is not guaranteed.
- Lookup is very fast.

Example:

```java
Set<String> technologies = new HashSet<>();

technologies.add("Java");
technologies.add("Spring");
technologies.add("Kafka");
technologies.add("Java");
```

Result:

```
Java
Spring
Kafka
```

The duplicate `"Java"` is ignored.

---

## But How Does HashSet Know an Element Already Exists?

Internally, `HashSet` maintains a `HashMap`.

Conceptually,

```
HashSet

↓

HashMap
```

Every element added to the set becomes the **key** in the underlying map.

The value is a dummy object that never changes.

Conceptually:

```
HashMap

Key        Value

Java   -> PRESENT

Spring -> PRESENT

Kafka  -> PRESENT
```

The values are irrelevant.

Only the keys matter.

In the JDK implementation, the dummy value is a single shared object, avoiding unnecessary memory allocation.

---

## Visual Walkthrough

When you write:

```java
set.add("Java");
```

Internally, it behaves approximately like:

```java
map.put("Java", PRESENT);
```

Now consider adding the same element again.

```java
set.add("Java");
```

Internally:

```java
map.put("Java", PRESENT);
```

Since the key already exists, `HashMap` simply replaces the existing value with the same dummy object.

The map size does not increase.

Therefore, the set still contains only one `"Java"`.

---

## Why Doesn't HashSet Allow Duplicates?

Suppose the set contains:

```
Java

Spring

Kafka
```

Adding another `"Java"` follows the same process as any `HashMap` insertion:

```
"Java"

↓

hashCode()

↓

Bucket

↓

equals()

↓

Existing key found

↓

No new entry created
```

The uniqueness guarantee comes entirely from `HashMap`'s handling of keys.

---

## Performance Characteristics

Since `HashSet` delegates to `HashMap`, its performance characteristics are nearly identical.

| Operation | Average | Worst Case |
|-----------|---------:|-----------:|
| `add()` | O(1) | O(n) |
| `contains()` | O(1) | O(n) |
| `remove()` | O(1) | O(n) |

As with `HashMap`, Java 8+ improves heavy collision scenarios using treeified buckets.

---

## Why Are `equals()` and `hashCode()` Still Important?

Consider a custom class.

```java
class Employee {

    int id;

}
```

Now:

```java
Set<Employee> employees = new HashSet<>();

employees.add(new Employee(101));
employees.add(new Employee(101));
```

Will the duplicate be removed?

No.

Both objects have different default identities.

Without overriding `equals()` and `hashCode()`, the set considers them different.

To define logical equality, both methods must be implemented correctly.

---

## Production Perspective

A backend service imports customer records from multiple systems.

To eliminate duplicates, developers use:

```java
Set<Customer> customers = new HashSet<>();
```

Unexpectedly, duplicate customers still appear.

The issue is not with `HashSet`.

The `Customer` class does not override `equals()` and `hashCode()`, so logically identical customers are treated as distinct objects.

Once these methods are implemented correctly, duplicate elimination works as expected.

---

## HashSet vs ArrayList

Suppose you need to check whether a user ID already exists.

Using an `ArrayList`:

```java
list.contains(userId);
```

The list scans elements one by one.

Complexity:

```
O(n)
```

Using a `HashSet`:

```java
set.contains(userId);
```

The lookup uses hashing.

Average complexity:

```
O(1)
```

For large datasets, the difference can be substantial.

## Interview Focus

### Question

How is `HashSet` implemented internally?

### Strong Senior-Level Answer

> `HashSet` is internally backed by a `HashMap`. Each element in the set is stored as a key in the map, while a single shared dummy object is used as the value. The uniqueness guarantee comes from the fact that `HashMap` does not allow duplicate keys.

---

## Common Mistakes

❌ Assuming `HashSet` has its own hashing implementation.

❌ Forgetting to override `equals()` and `hashCode()` for custom objects.

❌ Expecting insertion order to be preserved.

❌ Using `HashSet` when duplicate values are required.

---

## Engineering Insight

A common optimization in backend systems is replacing:

```java
List<String> allowedCountries
```

with

```java
Set<String> allowedCountries
```

when the primary operation is membership testing.

Checking:

```java
allowedCountries.contains(countryCode)
```

thousands of times per second is significantly more efficient with a `HashSet`.

This simple design decision can noticeably reduce CPU utilization in high-throughput services.

---

## Engineering Takeaways

- `HashSet` is implemented using `HashMap`.
- Elements are stored as keys.
- A shared dummy object is used as the value.
- Correct `equals()` and `hashCode()` implementations are essential.
- Choose `HashSet` when fast uniqueness checks are required.
