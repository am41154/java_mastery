## 4.7 Concurrent Collections – Thread-Safe Data Structures

### Why This Matters

Modern backend applications rarely execute on a single thread.

Consider a typical Spring Boot application.

```
              Client Requests

      Request 1     Request 2     Request 3
            │            │             │
            ▼            ▼             ▼
        Thread-1     Thread-2      Thread-3
                \        |        /
                 \       |       /
                  Shared Collection
```

Multiple threads may simultaneously:

- Read data
- Insert records
- Remove entries
- Update cached values

Standard collections like `ArrayList` and `HashMap` are **not thread-safe**.

Without proper synchronization, concurrent modifications can result in:

- Lost updates
- Data corruption
- Inconsistent reads
- Unexpected exceptions

The Java Collections Framework provides specialized concurrent collections to address these challenges.

---

## Are Standard Collections Thread-Safe?

No.

Consider the following example:

```java
Map<Integer, String> cache = new HashMap<>();
```

Two threads execute simultaneously:

```java
cache.put(101, "Alice");
```

```java
cache.put(102, "Bob");
```

If both threads modify the internal bucket structure at the same time, the map can become inconsistent.

The exact outcome depends on timing, making such bugs difficult to reproduce and debug.

---

## Synchronization Isn't Always the Best Solution

One approach is:

```java
Map<Integer, String> map =
    Collections.synchronizedMap(new HashMap<>());
```

This makes individual operations thread-safe.

However, it achieves safety by synchronizing access.

```
Thread 1

↓

LOCK

↓

HashMap

↓

UNLOCK

↓

Thread 2
```

Only one thread can access the map at a time.

Under heavy load, this becomes a bottleneck.

Modern applications often require better scalability.

---

# ConcurrentHashMap

`ConcurrentHashMap` is designed for high-concurrency environments.

```java
ConcurrentHashMap<Integer, User> cache =
    new ConcurrentHashMap<>();
```

Unlike a synchronized `HashMap`, it allows multiple threads to perform reads and many updates concurrently.

This significantly improves throughput in multi-threaded applications.

---

## How Does ConcurrentHashMap Achieve Better Performance?

The internal implementation has evolved over Java versions, but the key idea remains the same:

**Reduce contention.**

Instead of locking the entire map for every operation, `ConcurrentHashMap` minimizes the scope of locking and uses fine-grained synchronization where necessary.

Conceptually:

```
Thread 1

↓

Bucket A

Thread 2

↓

Bucket B

Thread 3

↓

Bucket C
```

Independent operations can proceed concurrently without blocking each other.

> **Interview Tip**
>
> In Java 8+, `ConcurrentHashMap` no longer uses fixed segment-based locking. Instead, it relies on finer-grained synchronization and lock-free techniques (such as CAS) for many operations.

---

## Null Keys and Values

Unlike `HashMap`:

```java
HashMap

✓ One null key

✓ Multiple null values
```

`ConcurrentHashMap` does **not** allow either.

```java
map.put(null, value);
```

Throws:

```
NullPointerException
```

Why?

Because a `null` return value would become ambiguous in concurrent operations:

- Does the key not exist?
- Or is the value actually `null`?

Disallowing `null` avoids this ambiguity.

---

## CopyOnWriteArrayList

Another useful concurrent collection is:

```java
CopyOnWriteArrayList<String> list =
    new CopyOnWriteArrayList<>();
```

Whenever the collection is modified,

Java creates a **new internal array**.

```
Original

A B C

↓

Add D

↓

New Array

A B C D
```

Readers continue using the old array while writers create the new one.

This eliminates the need for readers to acquire locks.

---

## When Should You Use CopyOnWriteArrayList?

It is ideal when:

- Reads are much more frequent than writes.
- The collection is relatively small.
- Iteration occurs frequently.

Examples include:

- Application configuration
- Feature flags
- Registered listeners
- Static reference data

It is **not** suitable for collections that are updated frequently, as every write copies the entire array.

---

## BlockingQueue

A `BlockingQueue` is designed for producer-consumer systems.

```
Producer Thread

↓

BlockingQueue

↓

Consumer Thread
```

Unlike a normal queue, consumers can wait until data becomes available.

```java
BlockingQueue<Task> queue =
    new LinkedBlockingQueue<>();
```

Useful methods include:

```java
queue.put(task);
```

and

```java
Task task = queue.take();
```

If the queue is empty,

`take()` waits instead of returning immediately.

This behavior makes `BlockingQueue` ideal for worker-thread architectures.

---

## Choosing the Right Concurrent Collection

| Requirement | Recommended Collection |
|-------------|------------------------|
| Thread-safe key-value store | `ConcurrentHashMap` |
| Read-heavy list | `CopyOnWriteArrayList` |
| Producer-consumer queue | `LinkedBlockingQueue` |
| Lock-free queue | `ConcurrentLinkedQueue` |

There is no universally "best" concurrent collection.

The correct choice depends on the application's access pattern.

---

## Production Perspective

A Spring Boot application maintains an in-memory cache of customer sessions.

Initially, developers use a `HashMap`.

Under concurrent traffic, occasional cache inconsistencies appear.

Replacing the cache with a `ConcurrentHashMap` eliminates the issue without introducing coarse-grained synchronization, allowing the application to scale more effectively under load.

The change is small in code but significant in reliability.

---

## Interview Focus

### Question

Why would you choose `ConcurrentHashMap` over `Collections.synchronizedMap()`?

### Strong Senior-Level Answer

> `Collections.synchronizedMap()` synchronizes every operation on a single lock, limiting concurrency. `ConcurrentHashMap` uses finer-grained synchronization and lock-free techniques, allowing multiple threads to access different parts of the map concurrently. This improves scalability while maintaining thread safety.

---

## Common Mistakes

❌ Using `HashMap` for shared mutable state.

❌ Assuming synchronized collections scale well under heavy concurrency.

❌ Using `CopyOnWriteArrayList` for write-heavy workloads.

❌ Expecting `ConcurrentHashMap` to allow `null` keys or values.

---

## Engineering Insight

Choosing a concurrent collection is about **access patterns**, not just thread safety.

Ask these questions:

- Is the application read-heavy or write-heavy?
- How many threads access the collection?
- Is ordering required?
- Will consumers block while waiting for work?

Answering these questions leads to the correct data structure more reliably than memorizing APIs.

---

## Engineering Takeaways

- Standard collections are not thread-safe.
- `ConcurrentHashMap` is preferred for shared key-value data.
- `CopyOnWriteArrayList` excels in read-heavy scenarios.
- `BlockingQueue` is ideal for producer-consumer architectures.
- Choose concurrent collections based on workload characteristics.
