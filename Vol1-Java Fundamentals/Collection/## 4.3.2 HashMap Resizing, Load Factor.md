## 4.3.2 HashMap Resizing, Load Factor, and Rehashing

### Why This Matters

A `HashMap` is designed to provide fast lookups.

However, as more entries are added, the number of collisions naturally increases.

If the internal bucket array never grew, performance would gradually degrade from:

```
O(1)
```

towards

```
O(n)
```

To prevent this, `HashMap` periodically **resizes** itself.

Understanding **when** resizing happens and **what actually occurs during resizing** is one of the most common senior-level interview topics.

---

## Initial Capacity

When you create a `HashMap` without specifying a size,

```java
Map<Integer, String> map = new HashMap<>();
```

Java creates an internal bucket array with a **default capacity** (currently 16).

```
Buckets

0
1
2
...
15
```

Initially, most buckets are empty.

As entries are inserted, they begin occupying these buckets.

---

## What Is Load Factor?

The **Load Factor** determines **how full** a `HashMap` is allowed to become before it expands.

By default:

```
Load Factor = 0.75
```

This means the map resizes when approximately **75% of the buckets are occupied**.

For a capacity of 16:

```
16 × 0.75 = 12
```

After the 12th entry (assuming an even distribution), the next insertion triggers a resize.

> **Interview Tip**
>
> The load factor is a balance between memory usage and lookup performance. A lower value reduces collisions but consumes more memory, while a higher value saves memory but increases collision probability.

---

## What Happens During Resizing?

When the threshold is reached, Java allocates a **new bucket array** with double the capacity.

```
Before Resize

Capacity = 16
```

```
0
1
2
...
15
```

↓

```
After Resize

Capacity = 32
```

```
0
1
2
...
31
```

However, simply copying entries isn't enough.

Each key's bucket depends on the array size.

Changing the array size changes the bucket calculation.

Therefore, every existing entry must be redistributed.

This process is called **rehashing**.

---

## Rehashing

Suppose a key originally maps to:

```
Bucket 5
```

After resizing,

the same hash may now map to:

```
Bucket 21
```

Every entry is recalculated and placed into its new bucket.

Conceptually:

```
Old Table

Bucket 5

A

B

↓

Resize

↓

New Table

Bucket 5

A

Bucket 21

B
```

Although this operation is expensive, it happens infrequently.

The cost is distributed across many insertions.

This is why insertion remains **O(1) amortized** rather than O(n).

---

## Why Isn't Resizing Done More Frequently?

Resizing involves:

- Allocating a larger array.
- Recalculating bucket positions.
- Moving every existing entry.

If resizing occurred after every insertion, performance would be terrible.

Instead, `HashMap` chooses a threshold that balances:

- Lookup speed
- Memory usage
- Resize frequency

The default load factor of **0.75** has proven to be a good compromise for most applications.

---

## Can We Avoid Frequent Resizing?

Yes.

If you know approximately how many entries the map will hold, initialize it with an appropriate capacity.

Instead of:

```java
Map<Integer, User> users = new HashMap<>();
```

Prefer:

```java
Map<Integer, User> users = new HashMap<>(10_000);
```

This minimizes expensive resize operations during bulk inserts.

A common example is loading:

- Customer records
- Product catalogs
- Configuration data
- Cache entries

where the approximate size is already known.

---

## Production Perspective

A startup's caching service loads nearly one million configuration entries during application startup.

The developers create the cache using the default constructor.

During startup, the `HashMap` resizes multiple times.

Each resize requires allocating a larger array and rehashing all existing entries, significantly increasing startup time.

After initializing the map with an estimated capacity, startup time improves noticeably because most resize operations are eliminated.

This optimization requires no algorithmic changes—only a better understanding of how `HashMap` works.

---

## Interview Focus

### Question

Why is resizing expensive in a `HashMap`?

### Strong Senior-Level Answer

> During resizing, HashMap allocates a larger bucket array and redistributes every existing entry because bucket positions depend on the table size. This process, known as rehashing, is computationally expensive but occurs infrequently, keeping average insertion complexity close to O(1).

---

## Common Mistakes

❌ Believing resizing only allocates a larger array.

❌ Forgetting that every entry must be rehashed.

❌ Ignoring initial capacity when the expected size is known.

❌ Assuming `HashMap` performance is always constant regardless of load.

---

## Engineering Insight

### Capacity vs Size

These terms are often confused.

**Size**

The number of key-value pairs currently stored.

```
size = 8
```

**Capacity**

The number of buckets available internally.

```
capacity = 16
```

A `HashMap` with:

```
capacity = 1024

size = 10
```

is mostly empty.

Similarly,

```
capacity = 16

size = 12
```

is close to triggering a resize.

Understanding this distinction is useful when tuning memory usage in large applications.

---

## Engineering Takeaways

- `HashMap` expands automatically as it grows.
- The default load factor is **0.75**.
- Resizing doubles the bucket array capacity.
- Rehashing redistributes every existing entry.
- Specify an initial capacity when storing a large number of entries.

---
