## 4.5 TreeMap & TreeSet – Sorted Collections and Red-Black Trees

### Why This Matters

Most collections we've discussed so far prioritize **fast lookups**.

However, some applications require data to remain **sorted at all times**.

Examples include:

- Ranking users by score
- Maintaining timestamps in chronological order
- Generating reports in sorted order
- Finding the next or previous element efficiently

This is where `TreeMap` and `TreeSet` become valuable.

Unlike `HashMap` and `HashSet`, these collections trade some performance for guaranteed ordering.

---

## What Is a TreeMap?

A `TreeMap` stores key-value pairs in **sorted order**.

```java
Map<Integer, String> rankings = new TreeMap<>();

rankings.put(3, "Charlie");
rankings.put(1, "Alice");
rankings.put(2, "Bob");
```

Output:

```
1 → Alice

2 → Bob

3 → Charlie
```

Notice that the elements are **not stored in insertion order**.

They are automatically sorted according to their keys.

---

## What Is a TreeSet?

`TreeSet` behaves similarly.

```java
Set<Integer> numbers = new TreeSet<>();

numbers.add(30);
numbers.add(10);
numbers.add(20);
```

Result:

```
10

20

30
```

Like `HashSet`, duplicates are not allowed.

Unlike `HashSet`, elements are always maintained in sorted order.

---

# Internal Working

Both `TreeMap` and `TreeSet` are implemented using a **Red-Black Tree**.

A Red-Black Tree is a self-balancing Binary Search Tree.

Conceptually:

```
          20
         /  \
       10    40
      /      / \
     5      30  50
```

Whenever a new element is inserted or removed, the tree automatically rebalances itself.

This ensures that the tree height remains close to logarithmic, providing consistent performance even as the dataset grows.

> **Interview Tip**
>
> You are **not** expected to explain Red-Black Tree rotations in most Java interviews. Knowing that it is a self-balancing Binary Search Tree with O(log n) operations is usually sufficient.

---

## Performance Characteristics

Because elements are stored in a balanced tree rather than a hash table, operations are slightly slower than `HashMap`.

| Operation | TreeMap / TreeSet |
|-----------|------------------:|
| Search | O(log n) |
| Insert | O(log n) |
| Remove | O(log n) |

Compare this with `HashMap`:

| Collection | Average Lookup |
|------------|---------------:|
| HashMap | O(1) |
| TreeMap | O(log n) |

The additional cost provides automatic sorting.

---

## Natural Ordering

By default, sorting is based on **natural ordering**.

For numbers:

```
10

20

30
```

For strings:

```
Apple

Banana

Orange
```

Natural ordering is provided through the `Comparable` interface.

Example:

```java
public class Employee implements Comparable<Employee> {

    @Override
    public int compareTo(Employee other) {
        return Integer.compare(this.id, other.id);
    }
}
```

---

## Custom Sorting with Comparator

Sometimes natural ordering is not sufficient.

Suppose employees should be sorted by salary instead of ID.

```java
TreeSet<Employee> employees =
    new TreeSet<>(Comparator.comparing(Employee::getSalary));
```

Now the ordering depends on salary rather than the natural order defined in the class.

This flexibility is one of the strengths of tree-based collections.

---

## HashMap vs TreeMap

| Feature | HashMap | TreeMap |
|---------|---------|---------|
| Ordering | No | Sorted by key |
| Internal Structure | Hash Table | Red-Black Tree |
| Lookup | O(1) Average | O(log n) |
| Null Key | One allowed | Not allowed* |

> *A `TreeMap` using natural ordering cannot store a `null` key because comparisons with `null` are not possible.*

---

## HashSet vs TreeSet

| Feature | HashSet | TreeSet |
|---------|----------|----------|
| Ordering | No | Sorted |
| Internal Structure | HashMap | Red-Black Tree |
| Lookup | O(1) Average | O(log n) |
| Typical Use | Fast membership checks | Sorted unique elements |

---

## Production Perspective

An online trading platform maintains active orders sorted by price.

Using a `HashMap` would require sorting the entire dataset before every display.

Instead, a `TreeMap` keeps the orders sorted continuously, allowing the system to efficiently retrieve the highest, lowest, next, or previous price without repeatedly sorting the collection.

Although insertions are slightly slower, the overall solution is more efficient because sorting work is avoided later.

---

## Interview Focus

### Question

When would you choose `TreeMap` over `HashMap`?

### Strong Senior-Level Answer

> I would choose `TreeMap` when sorted keys are required or when navigation operations such as `firstKey()`, `lastKey()`, `higherKey()`, or `lowerKey()` are important. If ordering is unnecessary and lookup performance is the priority, `HashMap` is usually the better choice because it provides O(1) average lookup.

---

## Common Mistakes

❌ Using `TreeMap` simply because data "looks nicer."

❌ Assuming `TreeMap` preserves insertion order.

❌ Forgetting that sorting introduces additional overhead.

❌ Ignoring the need for `Comparable` or `Comparator`.

---

## Engineering Insight

A common anti-pattern is storing data in a `HashMap` and then sorting it after every insertion.

If the application requires continuously sorted data, a `TreeMap` is often the more appropriate choice despite its O(log n) operations.

Choosing the right data structure can eliminate unnecessary sorting logic and simplify the overall design.

---

## Engineering Takeaways

- `TreeMap` stores sorted key-value pairs.
- `TreeSet` stores sorted unique elements.
- Both are implemented using Red-Black Trees.
- Operations execute in O(log n).
- Use them when ordering is a functional requirement, not just a presentation concern.

---
