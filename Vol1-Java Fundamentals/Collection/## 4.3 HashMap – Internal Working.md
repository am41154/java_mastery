## 4.3 HashMap – Internal Working (Deep Dive)

### Why This Matters

If there is one class that almost every Java interviewer asks about, it is **`HashMap`**.

At first glance, it appears to be a simple key-value store.

```java
Map<String, User> users = new HashMap<>();

users.put("A101", user);
User u = users.get("A101");
```

Simple API.

Complex implementation.

Understanding how `HashMap` works demonstrates knowledge of several core Java concepts:

- Object equality
- `hashCode()`
- Buckets
- Collision handling
- Arrays
- Linked Lists
- Red-Black Trees
- Performance optimization

This section is one of the most important in the entire handbook.

---

## What is a HashMap?

A `HashMap` stores data as **key-value pairs**.

```
EmployeeId  → Employee

CustomerId  → Customer

OrderId     → Order
```

Unlike a `List`, elements are not retrieved by position.

Instead, they are retrieved using a **key**.

The objective is simple:

> **Find a value without searching every element.**

That is why `HashMap` provides **O(1)** average lookup time.

The keyword here is **average**.

Understanding why requires a look inside the implementation.

---

# Internal Structure

Internally, a `HashMap` is backed by an **array**.

Each position in the array is called a **bucket**.

```
Bucket Array

0
1
2
3
4
5
6
7
...
15
```

When you insert a key,

```java
map.put(key, value);
```

Java determines **which bucket** should store that key.

The question is:

> How?

The answer is:

**Hashing.**

---

## Step 1 — Calculate the Hash Code

Every Java object inherits:

```java
hashCode()
```

For example,

```java
String key = "USER123";

int hash = key.hashCode();
```

This produces an integer.

Example:

```
18923455
```

The actual number is not important.

The important point is that the same object always produces the same hash code (unless its state changes—which is why immutable keys are preferred).

---

## Step 2 — Determine the Bucket

The hash code is converted into a bucket index.

Conceptually,

```
Bucket Index

hash % arraySize
```

Suppose:

```
Hash Code = 18923455

Array Size = 16
```

```
18923455 % 16 = 15
```

Therefore,

```
Bucket 15
```

```
Bucket Array

0

1

2

...

15  →  USER123
```

Java's actual implementation uses bitwise operations for better performance, but the idea remains the same—the hash determines where the entry is stored.

---

## Why Is Lookup So Fast?

Suppose we now execute:

```java
map.get("USER123");
```

Java does **not** search every element.

Instead, it:

1. Computes the hash code.
2. Determines the bucket.
3. Looks only in that bucket.

This avoids scanning the entire collection.

That is the reason `HashMap` is extremely efficient for lookups.

---

## What Is a Collision?

Now consider another key.

```
ORDER987
```

Unfortunately,

after hashing,

it also maps to

```
Bucket 15
```

Now we have:

```
Bucket 15

USER123

ORDER987
```

This situation is called a **collision**.

A collision simply means:

> Two different keys produce the same bucket index.

Collisions are expected.

A good hash function minimizes them—it does not eliminate them.

---

## How Does HashMap Handle Collisions?

Before Java 8, collisions were handled using a **Linked List**.

```
Bucket 15

USER123
    │
    ▼
ORDER987
    │
    ▼
ITEM555
```

When searching,

Java traverses the list until it finds the matching key.

This changes lookup performance from

```
O(1)
```

to

```
O(n)
```

for that bucket.

If collisions become excessive, performance degrades significantly.

Fortunately, modern Java improves this behavior.

---

## Java 8 Treeification

Starting with Java 8,

if too many entries accumulate in the same bucket,

the linked list is converted into a **Red-Black Tree**.

```
        USER123
        /     \
 ORDER987   ITEM555
     \
   PROD999
```

Searching in a balanced tree is much faster.

Complexity improves from:

```
O(n)
```

to:

```
O(log n)
```

This optimization protects `HashMap` from severe performance degradation caused by excessive collisions.

> **Interview Tip**
>
> A bucket is treeified only when the number of entries crosses a threshold (currently 8) and the table has reached a minimum capacity. This avoids creating trees unnecessarily for small maps.

---

## Engineering Insight

A common misconception is:

> "HashMap always provides O(1) lookup."

A more accurate statement is:

> **HashMap provides O(1) average-case lookup, but performance depends on a good hash distribution and proper implementations of `hashCode()` and `equals()`.**

Understanding this distinction is a hallmark of a senior engineer.

---

## Quick Revision

✓ `HashMap` stores key-value pairs.

✓ Internally, it is backed by an array of buckets.

✓ The bucket is selected using the key's hash.

✓ Multiple keys can map to the same bucket.

✓ Collisions are normal and handled internally.

✓ Java 8+ converts heavily populated buckets into Red-Black Trees for better performance.
