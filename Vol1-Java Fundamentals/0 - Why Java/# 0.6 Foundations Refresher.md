# 1.6 Foundations Refresher — The Java You Already Know

## Why This Matters

A typical day involves:

- Building REST APIs with Spring Boot
- Writing JPA repositories
- Consuming Kafka messages
- Deploying to Kubernetes
- Investigating production incidents

Very little time is spent thinking about variables, objects, parameter passing, or access modifiers.

Yet these are exactly the topics interviewers often use to assess the depth of your understanding.

## Variables and References

One of the most common misconceptions is that Java variables store objects.

They do not.

Variables store either:

- A primitive value, or
- A reference to an object.

```java
int count = 10;

Trade trade = new Trade();
```

In the first example, `count` directly stores the value `10`.

In the second example, `trade` stores a reference to a `Trade` object created on the heap.

This distinction explains why modifying an object through one reference is visible through another reference pointing to the same object.

> **Key Insight**
>
> Objects live in memory. Variables simply hold references to those objects.

## Primitive Types

Java provides eight primitive data types.

| Type | Size | Typical Use |
|------|------|-------------|
| `byte` | 8-bit | Small numeric values |
| `short` | 16-bit | Memory-constrained data |
| `int` | 32-bit | Default integer type |
| `long` | 64-bit | Large numeric values |
| `float` | 32-bit | Decimal values (rare in enterprise systems) |
| `double` | 64-bit | Default floating-point type |
| `char` | 16-bit | Unicode characters |
| `boolean` | JVM-defined | True or false |

In enterprise applications:

- `int` is the default choice for whole numbers.
- `long` is commonly used for identifiers and timestamps.
- `double` should generally be avoided for financial calculations due to precision limitations.

For monetary values, `BigDecimal` is usually the preferred option.

---

## Primitive vs Object

Primitive values are not objects.

This means they:

- Cannot be `null`.
- Do not contain methods.
- Occupy less memory.
- Are generally faster to access.

Wrapper classes provide object representations of primitive types.

| Primitive | Wrapper |
|-----------|----------|
| `int` | `Integer` |
| `long` | `Long` |
| `double` | `Double` |
| `boolean` | `Boolean` |

Wrappers become necessary when working with:

- Collections
- Generics
- Framework APIs
- Nullable values

---

## Autoboxing and Unboxing

Java automatically converts between primitives and wrapper classes.

```java
Integer quantity = 10;   // Autoboxing

int value = quantity;    // Unboxing
```

Although convenient, automatic conversions create additional objects in some situations.

In performance-critical code, unnecessary boxing should be avoided.

## How Java Passes Parameters

This is one of the most frequently misunderstood interview topics.

Java always uses **pass-by-value**.

Consider the following example.

```java
void updateTrade(Trade trade) {
    trade.setStatus("SETTLED");
}
```

Many developers incorrectly conclude that Java passes objects by reference.

What actually happens is more subtle.

Java copies the **reference value** into the method.

Both variables point to the same object.

The reference itself is copied—not the object.

This explains why the object's state can change while reassigning the parameter inside the method has no effect on the caller.

> **Interview Focus**
>
> Java is always pass-by-value.
>
> For objects, the copied value happens to be a reference.

## Arrays

Arrays provide a fixed-size collection of elements of the same type.

```java
int[] prices = {100, 120, 140};
```

Arrays are:

- Fast
- Memory efficient
- Indexed
- Fixed in size

Most enterprise applications use collections such as `ArrayList` instead of raw arrays because collections provide dynamic resizing and richer APIs.

Understanding arrays remains important because many collection implementations are built on top of them.

## Access Modifiers

Access modifiers control visibility.

| Modifier | Same Class | Same Package | Subclass | Other Package |
|----------|------------|--------------|----------|---------------|
| `public` | ✓ | ✓ | ✓ | ✓ |
| `protected` | ✓ | ✓ | ✓ | ✗* |
| *(via inheritance in another package)* |
| package-private | ✓ | ✓ | ✗ | ✗ |
| `private` | ✓ | ✗ | ✗ | ✗ |

Choosing the most restrictive access level possible improves encapsulation and reduces accidental coupling.

---

## Enums

Enums represent a fixed set of constants.

```java
public enum OrderStatus {
    NEW,
    EXECUTED,
    CANCELLED
}
```

Unlike integer constants, enums provide:

- Type safety
- Better readability
- Compile-time validation

Enums are commonly used for workflow states, event types, and configuration values.

---

## Records (Java 16+)

Many enterprise applications use classes whose primary purpose is to hold immutable data.

Instead of writing boilerplate code, Java provides **records**.

```java
public record Trade(
    String tradeId,
    BigDecimal amount
) {}
```

The compiler automatically generates:

- Constructor
- Getters
- `equals()`
- `hashCode()`
- `toString()`

Records are particularly useful for DTOs, API responses, and event payloads.

---

## Annotations

Annotations provide metadata to the compiler or runtime.

For example:

```java
@RestController
public class TradeController {
}
```

Spring uses annotations extensively to enable:

- Dependency Injection
- REST APIs
- Transaction Management
- Validation
- Security

Annotations themselves contain no business logic.

Frameworks interpret them to provide additional behaviour.

## Production Perspective

A payment service unexpectedly throws a `NullPointerException`.

The root cause isn't Spring, Kafka, or the database.

A wrapper type (`Integer`) is automatically unboxed into an `int`, but the wrapper contains `null`.

```java
Integer quantity = null;

int value = quantity;
```

The application fails during automatic unboxing.

The issue isn't complex.

It's a forgotten language fundamental.

This is a recurring theme in production systems.

Many difficult-looking failures originate from surprisingly simple Java concepts.

These questions appear regularly in senior Java interviews:

- What's the difference between `int` and `Integer`?
- When should you use `BigDecimal` instead of `double`?
- Why do collections require wrapper classes?
- When would you choose a record over a traditional class?
- What problem do annotations solve?

Although these questions cover language fundamentals, interviewers often use them to judge how precisely a candidate explains core concepts.

---

## Engineering Takeaways

- Variables store values or object references—not objects themselves.
- Java always uses pass-by-value.
- Wrapper classes enable primitives to work with collections and frameworks.
- Arrays are fixed-size; collections provide dynamic behaviour.
- Use the most restrictive access modifier that satisfies the design.
- Records reduce boilerplate for immutable data.
- Annotations describe metadata that frameworks interpret at compile time or runtime.
